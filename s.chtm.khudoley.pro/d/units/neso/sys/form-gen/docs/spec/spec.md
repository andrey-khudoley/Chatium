### Принцип распределения

Пункт попадает в волну по **цене позднего решения**, а не по важности. Три класса:

| Класс           | Что стоит поздний ответ                                                                                    | Крайний срок                      |
| --------------- | ---------------------------------------------------------------------------------------------------------- | --------------------------------- |
| **Необратимое** | форма или смысл уже записанной строки, факт удаления данных. Правится миграцией, а при неудаче — потерей   | до первой боевой записи в таблицу |
| **Ломающее**    | внешний контракт: набор операций, коды ошибок, права. Правится согласованным обновлением всех потребителей | до первого внешнего потребителя   |
| **Аддитивное**  | ничего: добавляется новая операция, новое поле с дефолтом, новая таблица, новый выход                      | в любой момент                    |

**Правило, запрещающее сдвиг:** пункт первых двух классов нельзя перенести в более позднюю волну ради скорости. Всё аддитивное, наоборот, отодвигается сознательно.

Порядок волн: **прототип** ничего не замораживает, **MVP** замораживает форму строк, **beta** замораживает внешний контракт, **релиз** только добавляет.

## 1. Назначение

`form-gen` — **модуль генерации форм заказа для страниц юнита neso**. Он связывает три уже существующие вещи:

1. **Предложения аккаунта GetCourse** — с MVP получаются через getcourse-гейтвей (операция `getOffers`, статус `enabled`; спека гейтвея — `d/gateways/getcourse/docs/spec/`); в прототипе — прямыми запросами (§0.2, волна 1).
2. **Форму** — генерируемый артефакт, привязанный к выбранному предложению.
3. **Страницы-потребители** — любые: клоны Gamma `p/units/neso/pages/aNNN/`, прочие страницы аккаунта и **внешние сайты** (расширено 23-07-2026, [ADR-0001](../ADR/0001-client-script-widget.md)). Подключение — вставка `<script src>`-виджета (§5.1). Устройство страниц `aNNN` (зафиксировано 23-07-2026): обработчик серверный (`app.get('/', ctx => ctx.resp.html(PAGE_HTML))`), `PAGE_HTML` — статическая строка, автосгенерированная из Gamma («do not hand-edit») — поэтому у них строка `<script>` вставляется через `index.tsx` (инъекция в `PAGE_HTML`), а не правкой `pageHtml.ts`, которую стирает регенерация.

Сценарий использования: пользователь выбирает подходящее предложение из аккаунта GetCourse → модуль генерирует форму под это предложение → форма подключается к любой странице юнита импортом, параметры передаются при подключении.

**Чего `form-gen` НЕ делает** _(решено 23-07-2026)_:

- Не редактирует сами страницы `aNNN` — подключение формы остаётся ручной операцией автора страницы (вставка строки `<script>`, §5.1; для `aNNN` — через `index.tsx`-инъекцию, не правкой `pageHtml.ts`).
- Не заменяет getcourse-гейтвей — с волны 2 (MVP) все обращения к GetCourse идут через него, своих ключей и прямых вызовов GC-API у модуля нет. Исключение — волна 1: прототип ходит в GC напрямую ради скорости проверки идеи (§0.2).
- Не строит аналитику форм (воронки, конверсия, A/B-сравнение, дашборды). Модель логирования по волнам: **волна 1 — общее отладочное логирование** всех событий формы (аналог debug) через `ctx.account.log()`; **с волны 3** важные события публикуются в брокер, отладочные логи выключаются — **остаётся только логирование ошибок** (§0.2). Анализ данных — забота потребителей событий брокера, не модуля.
- Не отправляет уведомления о заказах (email, SMS, мессенджеры) — ни на одной волне. Понадобятся уведомления — их сделает отдельный микросервис, читающий события из брокера (волна 3).
- Не поддерживает A/B-варианты одной формы — **один `formID` = одна конфигурация**; разные варианты предложения (цена, текст, вид) заводятся отдельными формами. Управление показом вариантов — вне модуля.

## 2. Доменная модель

**Решено (23-07-2026).** Модуль мыслит четырьмя сущностями; своя (хранимая) из них одна — форма.

1. **Форма (`formID`)** — конфигурация в Heap (§3.1): **набор предложений** (одно или несколько — при нескольких посетитель выбирает в форме) + настройки внешнего вида. Создаётся генератором админ-страницы (§5.3), правится там же. **Жизненный цикл без статусов: создана → настроена (сразу активна) → удалена.** Статус `active`/`disabled` отвергнут (решение пользователя 23-07-2026): ненужная форма просто удаляется, промежуточное состояние не нужно — меньше поле в необратимой схеме, меньше ветвлений в виджете и эндпоинте. Следствие-обязательство: **тихая деградация виджета** — `widget?form=<удалённый formID>` отдаёт no-op JS (страница не ломается, якорь-`<div>` остаётся пустым, обработчик модалки не регистрируется), сабмит по несуществующему `formID` возвращает `{ ok: false, error }`.
2. **Предложение GetCourse** — внешняя сущность; форма ссылается по id и хранит **снимок отображения** (`title`, `price`, `currency` в `offers`, §3.1) — задаётся автором в генераторе. Виджет при рендере в GC не ходит; актуальность снимка — ответственность автора формы.
3. **Заявка/заказ** — **не наша сущность**: форма создаёт заказ сразу в GC (§5.2, §5.4) и у себя не хранит. След заявки — логи (и, с волны 3, событие в брокере). Модуль — генератор форм, не CRM заказов (§1).
4. **Настройка** — key-value запись служебной таблицы (§3).

## 3. Хранилище (Heap)

**Состав решён (23-07-2026, вторая редакция — [ADR-0001](../ADR/0001-client-script-widget.md)):** две таблицы. Первая редакция («конфиги форм в Heap не хранятся, конфиг в коде страницы») отменена разворотом на клиентский виджет: у внешней страницы нет серверного кода воркспейса, конфигу больше негде жить, кроме как на стороне form-gen. Реестр «какая форма на какой странице» не ведётся — таблица форм хранит конфигурации, а не привязки к страницам.

### 3.1. Таблица форм

**Идентификатор (зафиксирован 23-07-2026, необратимое):** `t__form-gen__forms__stage_2xnqat` (dev-копия; при переносе в прод сегмент `__stage_` → `__prod_` штатной трансформацией `/to-prod`). Хвост сгенерирован через shell.

**Схема (решено 23-07-2026):**

| Поле         | Тип    | Назначение                                                                                                                                          |
| ------------ | ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| `slug`       | string, **плоское корневое** | `formID` (§5.1) — случайный слаг с префиксом (nanoid). Единственный горячий фильтр: по нему виджет-роут и эндпоинт отправки находят форму; плоским (индексируемым) должен быть только он |
| `offers`     | any    | JSON-массив предложений формы; каждый элемент: `{ offerId, title, price, currency }` — id предложения GC, отображаемое в форме название, цена, валюта. **Форма мультиофферная:** при нескольких элементах посетитель выбирает предложение в форме; заказ создаётся по выбранному |
| `appearance` | any    | настройки внешнего вида формы; **контракт поля откладывается сознательно** (§0.2, волна 2) — тип any позволяет определить его позже без миграции      |

Обоснование `offers` как any-JSON: состав предложения (цена, валюта, подпись) — снимок отображения, который правится в генераторе как единое целое; реляционное разложение по строкам не даёт ничего, кроме лишних таблиц. Следствие: **виджет не ходит в GC при рендере** — всё для отрисовки лежит в записи формы (правка §2: снимок есть).

### 3.2. Таблица настроек

**Идентификатор (зафиксирован 23-07-2026, необратимое):** `t__form-gen__settings__stage_x2plui` (правило переноса — то же).

**Схема:** key-value по образцу `BrokerSettings` брокера (`d/system/broker/tables/settings.table.ts`): `key` (string, плоское) → `value` (any). Содержимое: настройки подключения GC (волна 1 — **три поля: URL школы, ключ школы, ключ разработчика**; с волны 2 — параметры гейтвея), дефолты форм (§5.3).

**Три поля настроек GC волны 1 (решено 23-07-2026):** `gc_school_url` (URL/хост тестовой школы), `gc_school_key` (ключ школы — используется в Legacy-импорте `deals/add`), `gc_developer_key` (**ключ разработчика — в волне 1 только сохраняется, кодом не читается**; задел на волну 2: REST API v2 / гейтвей). GC-клиент прототипа (§5.2) обращается только по `gc_school_url` + `gc_school_key`.

**Разделение по окружениям (решено 23-07-2026):** записи настроек — **данные Heap, а не код**, поэтому `/to-prod` их не переносит (переносится только код, §0). Каждое окружение заполняет свою таблицу настроек через свою админ-страницу (§5.3), и GC-креды у них разные: **dev-копия (`__stage_`) настроена на тестовую школу GetCourse** — прототип волны 1 создаёт там любые тестовые заказы, боевой аккаунт не затрагивается; **prod-копия (`__prod_`) — на боевую школу**. Это снимает вопрос «тестовых данных GC» без технических предложений и меток: полигон прототипа — отдельная школа.

## 5. Бизнес-логика

### 5.1. Встраивание формы в страницу

**Решено (23-07-2026, заменяет решение от того же дня — [ADR-0001](../ADR/0001-client-script-widget.md)):** подключение — **клиентский `<script src=...>`-виджет**. Страница любого происхождения (aNNN воркспейса, любая другая, включая внешние домены — §5.2 CORS) вставляет в HTML одну строку:

```html
<script src="https://<домен>/<form-gen>/widget?form=<formID>" async></script>
```

**URL-сегмент — `widget`, без `.js` (уточнено 23-07-2026 по итогам прогона прототипа):** исходно предполагался сегмент `widget.js`, но sync-агент Chatium отклоняет каталог с расширением в имени («Directory name cannot end with .js/.ts»), поэтому роут живёт в каталоге `widget/` и отдаётся по `.../widget?form=`. Расширение `.js` в `<script src>` необязательно — тип определяется заголовком `Content-Type: application/javascript` ответа, а не URL.

Форма — сущность с идентификатором `formID`, её конфигурация живёт в Heap (§3), создаётся генератором админ-страницы (§5.3). Два режима отображения при загрузке скрипта:

1. **Inline:** если на странице есть якорь `<div id="<formID>">` — скрипт заменяет его формой, встроенной в поток страницы.
2. **Модалка:** скрипт регистрирует обработчик вызова модального окна с формой; страница вызывает его со своих кнопок.

Свойства решения:

- Фрагмент самодостаточен: стили с префиксом `fg-` (изоляция от стилей страницы), разметка формы, логика отправки (§5.2).
- Для страниц `aNNN` подключение упрощается до вставки строки `<script>` (через `index.tsx`-инъекцию в `PAGE_HTML` — правки `pageHtml.ts` стирает регенерация, §1); импорта из form-gen больше не нужно.
- Правка формы (тексты, предложение) — в админке по `formID`, страницы не трогаются.

**Отменённое решение:** импорт серверной функции `attachOrderForm` с инъекцией в `PAGE_HTML`. Причина замены: требование расширилось — формы должны подключаться на **любых** страницах, включая внешние домены, а серверный импорт работал только внутри воркспейса. Цена клиентского виджета (загрузка формы после загрузки страницы) принята как плата за универсальность. `app.function` как канал подключения — не нужен по той же причине.

**Контракт виджета (решено 23-07-2026, обкатывается прототипом, замораживается волной 2):**

1. **Доставка конфига — запечённый server-side.** `widget.js` — не статичный файл, а серверный роут: по `?form=<formID>` читает запись формы (§3) и отдаёт JS с конфигом, вшитым внутрь (название, цена, тексты). Один сетевой запрос, форма рисуется без мигания. **Отвергнут** статичный скрипт с fetch за конфигом: лишний сетевой цикл.
2. **Поведение при загрузке.** Скрипт подтягивает CSS (стили с префиксом `fg-`) и делает две вещи: (а) заменяет **все** вхождения `<div id="<formID>"></div>` на разметку inline-формы; (б) регистрирует обработчик вызова модального окна. **Вызов обработчика — забота страницы** (программный вызов из её кода, когда и откуда ей нужно); при вызове появляется уже стилизованное модальное окно с формой. **Отвергнут** data-атрибут на кнопках как основной механизм: страница сама решает, когда открывать модалку.
3. **Формат `formID` — короткий случайный слаг с префиксом** (вида `fg7Kx2mQ`): генерируется при создании формы (nanoid), хранится в её записи, валиден как DOM-id, внутренних идентификаторов не раскрывает.

_(Открыто: имя и сигнатура регистрируемого обработчика модалки — деталь, снимается прототипом, фиксируется с контрактом в волне 2 — §0.2.)_

#### 5.1.1. Конфигурация формы (сущность `formID`)

**Решено (23-07-2026, вторая редакция — мультиофферность):** конфигурация формы — запись в Heap-таблице форм (§3.1), создаётся и правится генератором (§5.3). Состав:

| Поле         | Обязательность | Значение                                                                                                     |
| ------------ | -------------- | ------------------------------------------------------------------------------------------------------------- |
| `offers`     | обязательное   | массив предложений `{ offerId, title, price, currency }`; при нескольких — посетитель выбирает предложение в форме, заказ создаётся по выбранному |
| `appearance` | опциональное   | настройки внешнего вида; контракт поля откладывается сознательно (§0.2, волна 2)                              |

Первая редакция (одно предложение + заголовок + текст кнопки как поля конфигурации) заменена: форма мультиофферная, а всё оформление (заголовки, тексты кнопок и прочий внешний вид) уезжает в `appearance` — его контракт определяется позже без миграции (тип any, §3.1).

- **Состав полей посетителя (имя/email/телефон) — константа модуля**, не настройка: пока вызов GC не проверен прототипом, параметризация полей — гадание; добавить в конфигурацию можно аддитивно позже.
- **UTM-метки** из URL страницы скрипт собирает и отправляет **всегда автоматически** — поведение, а не настройка.

### 5.2. Отправка формы и редирект на оплату

**Решено (23-07-2026, уточнено):** отправка — **`fetch` POST**, тело запроса — **`application/x-www-form-urlencoded`**, ответ сервера — **JSON**; редирект на клиенте.

Цикл: заполнение формы в модалке → `fetch POST` (тело `x-www-form-urlencoded`) на эндпоинт form-gen → сервер валидирует, вызывает GetCourse (волна 1 — напрямую, с волны 2 — через getcourse-гейтвей), получает ссылку на оплату → ответ JSON `{ ok, redirectUrl }` → клиент выполняет `location.href = redirectUrl`. Ошибка валидации/GC → JSON `{ ok: false, error }` → сообщение внутри модалки, состояние формы сохраняется.

**Транспорт запроса — `x-www-form-urlencoded`, а не JSON (решено 23-07-2026):** это CORS **simple request** — браузер не шлёт preflight `OPTIONS`, поэтому кросс-доменная отправка с внешних сайтов работает без отдельного OPTIONS-роута (в Chatium file-based роутинге `app.options` не документирован — 002-routing). `application/json`-тело сделало бы запрос непростым и потребовало preflight. Ответ при этом остаётся JSON. **Отвергнуто:** классический form-POST с 302 — модалке всё равно нужен JS, а fetch даёт показ ошибок без потери введённого. Ключи GC — только на сервере; клиент видит только эндпоинт form-gen и `formID` (id предложения наружу не отдаётся без необходимости).

**Внешние домены (решено 23-07-2026, CORS уже в волне 1):** формы вставляются и на сторонних сайтах; эндпоинт отправки отвечает с **`Access-Control-Allow-Origin: *` на всех ветках** (и `{ok:true}`, и `{ok:false}` — иначе внешний домен не прочитает тело ошибки). Обоснование: CORS — защита браузера от кросс-доменного чтения, а не защита сервера; эндпоинт публичный по определению (форму заполняет анонимный посетитель), cookies/credentials в цикле нет — `*` без credentials безопасен. Заголовок ставится явно через return-конверт роута (`{ statusCode, rawHttpBody, headers }`, 002-routing); при этом волна 1 **проверяет рантаймом**, не ставит ли платформа Chatium `ACAO: *` сама по умолчанию (048-probes) — если да, явный заголовок избыточен, но не конфликтует. **Отвергнут** whitelist доменов: против прямых POST (боты, curl) origin-ограничения бессильны, а операционная цена (каждый новый сайт — правка настроек) ненулевая. Раздача самого скрипта CORS не требует (`<script src>` работает кросс-доменно). Защита от спам-сабмитов (создают реальные заказы в GC) — отдельная тема, решается rate-limit'ом/honeypot'ом, добавляется аддитивно (§0.2, волна 3).

**Источник ссылки на оплату (решено 23-07-2026):** операция GC Legacy-импорта `deals`, `action=add` — в гейтвее это `createDeal` (`p/gateways/getcourse/api/v1/createDeal.ts`): `POST https://<school>/pl/api/deals`, form-body `key`/`action`/`params` (Base64 от JSON), системный параметр **`return_payment_link`** возвращает ссылку на оплату в ответе. В каталоге гейтвея `createDeal` уже `enabled`.

**Волна 1 — временный дубль (решено 23-07-2026):** прототип воспроизводит эту обвязку прямым запросом по образцу `legacyGcImportClient.ts` + `legacyGcFormBody.ts` гейтвея. Дубль одноразовый: удаляется при переходе на гейтвей в MVP (волна 2) — форма зовёт `createDeal` через гейтвей, своего GC-клиента у form-gen не остаётся. Обоснование: обвязка обкатана в гейтвее, копия дешевле интеграции на этапе проверки идеи; риск расхождения снят коротким сроком жизни дубля.

### 5.3. Админ-страница: настройки, подбор предложения, генератор сниппетов

**Решено (23-07-2026):** у модуля есть одна админ-страница (`requireAccountRole(ctx, 'Admin')` первой строкой обработчика — роль с заглавной `A`, 003-auth) с двумя функциями:

1. **Настройки подключения GC** — значения живут в Heap-таблице настроек (§3), не в коде. Волна 1: **три поля — URL школы, ключ школы, ключ разработчика** (последний только сохраняется, кодом волны 1 не читается — §3.2). С волны 2 прямой ключ школы уходит (ключи школ хранит гейтвей), остаются параметры обращения к гейтвею и дефолты форм. Настройки — **per-environment** (§3.2): dev-админка указывает **тестовую школу GC** (полигон прототипа), prod-админка — боевую; `/to-prod` настройки не переносит. Форма настроек и создания формы в админке ходят на свои API через `fetch` (same-origin, сессия админа прокидывается) — не через `.run()`, поэтому `// @shared-route` этим API не нужен.
2. **Генератор форм (с волны 2)** — список предложений аккаунта через гейтвей (`getOffers`, `enabled`), выбор предложения, настройка конфигурации (§5.1.1) → **создание записи формы** в таблице форм (§3) и выдача двух готовых сниппетов: `<script src=".../widget?form=<formID>">` и якорь `<div id="<formID>">` (для inline-режима). Закрывает исходный сценарий «выбрать предложение → сгенерировать форму»; правка формы — здесь же, по `formID`, без правки страниц.

**Источник `offerId` по волнам (решено 23-07-2026):** волна 1 — id предложения указывается **вручную** (автор берёт его из аккаунта GC и вписывает в конфигурацию прототипной формы) — прототип проверяет сквозной цикл, а не удобство; с волны 2 — через генератор (см. выше).

**Модель генерации (вторая редакция, [ADR-0001](../ADR/0001-client-script-widget.md)):** «генератор» — инструмент создания **сущности формы** в Heap; страница получает только сниппеты подключения. Первая редакция («конфиг в коде страницы, в Heap не хранится») отменена разворотом на клиентский виджет (§5.1). Реестр «какая форма на какой странице» по-прежнему не ведётся (§3).

### 5.4. Маппинг полей формы → `params` Legacy-импорта

**Маппинг полей формы → `params` импорта (решено 23-07-2026, статус — рабочая гипотеза до подтверждения прототипом):** форма запроса подтверждена схемой гейтвея (`legacy_api_schema.json`: form-body `key`/`action=add`/`params` = Base64(JSON) с секциями user, deal, system, session «по help GC»); состав секций — гипотеза:

```jsonc
// params (до Base64)
{
  "user":    { "email": "...", "phone": "...", "first_name": "..." },   // поля формы — константа модуля (§5.1.1)
  "deal":    { "offer_id": "<числовой id из offers>", "deal_cost": "<price из снимка offers>" }, // предложение по id + цена (подтв. 24-07)
  "system":  { "refresh_if_exists": 1, "return_payment_link": 1 },      // не плодить дубли пользователей; вернуть ссылку
  "session": { "utm_source": "...", "utm_medium": "..." }               // UTM из URL страницы — всегда автоматом (§5.1.1)
}
```

Ссылка на оплату ожидается в `result.payment_link` тела ответа. **Статус (решено 23-07-2026):** для волны 1 гипотеза принята за рабочую истину — прототип и существует, чтобы её проверить. Подтверждение по результатам прогона — открытый пункт волны 2 (§0.2): код MVP не опирается на пункты, не подтверждённые прототипом; расхождения фиксируются правкой этого раздела. Вариант «сначала вычитать help GC целиком» отвергнут: прогон прототипа всё равно обязателен, гипотеза даёт ему готовый план проверки.

**Результаты прогона прототипа (24-07-2026) — гипотеза подтверждена сквозным заказом:**

- **Транспорт и структура ответа — подтверждены.** GC на `deals/add` отвечает HTTP 200 с телом `{ success, action: "add", result: { success, user_id, deal_id, payment_link, ... } }`. При успехе ссылка — в `result.payment_link`; при отказе `result.success:false` + `error_message`.
- **Предложение привязывается по `deal.offer_id` (числовой id), НЕ `offer_code`.** Первый прогон с `offer_code: 5948989` дал `result.success:false`, `error_message: "Предложение с кодом 5948989 не найдено"` (по `offer_code` GC ищет символьный код). После замены на `deal.offer_id` тот же числовой id `5948989` создал сделку. Итог: в `offers[].offerId` (§3.1) хранится **числовой id предложения GetCourse**, маппинг кладёт его в `deal.offer_id`.
- **Сквозной цикл работает:** сделка создана (`deal_id` присвоен), `return_payment_link:1` вернул ссылку — ответ `{ ok:true, redirectUrl }`, `redirectUrl` вида `https://<gc-домен>/sales/shop/dealPay/id/<dealId>/hash/<hash>`. Секции `user`/`system` подтверждены (заказ создался, ссылка получена). `session`/UTM — механика та же, отдельным прогоном с UTM не гонялась.
- **Цена сделки — `deal.deal_cost` из снимка формы (подтверждено).** Стоимость выбранного оффера берётся на сервере из `offers[].price` (§3.1, не с клиента) и переопределяет цену предложения: на странице оплаты GC сумма = цена из формы (прогон с `price=100` → «Сумма для оплаты: 100 руб.»).
- **Обработка ответа:** `createDeal` различает `result.success:false` → `GC_DEAL_REJECTED: <error_message GC>` (причина видна в форме); успех без ссылки → `GC_NO_PAYMENT_LINK`; HTTP≠200/невалидный JSON → `GC_HTTP_*`/`GC_BAD_RESPONSE`.
- **Диагностика снята:** временный лог `gc.diag` удалён после подтверждения (остались штатные `gc.request`/`gc.response`/`gc.error`).
