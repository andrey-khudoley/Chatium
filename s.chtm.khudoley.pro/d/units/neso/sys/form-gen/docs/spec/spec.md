### Принцип распределения

Пункт попадает в волну по **цене позднего решения**, а не по важности. Три класса:

| Класс           | Что стоит поздний ответ                                                                                    | Крайний срок                      |
| --------------- | ---------------------------------------------------------------------------------------------------------- | --------------------------------- |
| **Необратимое** | форма или смысл уже записанной строки, факт удаления данных. Правится миграцией, а при неудаче — потерей   | до первой боевой записи в таблицу |
| **Ломающее**    | внешний контракт: набор операций, коды ошибок, права. Правится согласованным обновлением всех потребителей | до первого внешнего потребителя   |
| **Аддитивное**  | ничего: добавляется новая операция, новое поле с дефолтом, новая таблица, новый выход                      | в любой момент                    |

**Правило, запрещающее сдвиг:** пункт первых двух классов нельзя перенести в более позднюю волну ради скорости. Всё аддитивное, наоборот, отодвигается сознательно.

Порядок волн: **прототип** ничего не замораживает, **MVP** замораживает форму строк, **beta** замораживает внешний контракт, **релиз** только добавляет.

## 1. Назначение

`form-gen` — **модуль генерации форм заказа для страниц юнита neso**. Он связывает три уже существующие вещи:

1. **Предложения аккаунта GetCourse** — с MVP получаются через getcourse-гейтвей (операция `getOffers`, статус `enabled`; спека гейтвея — `d/gateways/getcourse/docs/spec/`); в прототипе — прямыми запросами (§0.2, волна 1).
2. **Форму** — генерируемый артефакт, привязанный к выбранному предложению.
3. **Страницы `p/units/neso/pages/aNNN/`** — HTML-клоны из Gamma, куда форма подключается **обычным импортом с передачей параметров**. Устройство страницы (зафиксировано 23-07-2026): обработчик серверный (`app.get('/', ctx => ctx.resp.html(PAGE_HTML))`), но рендеринга нет — `PAGE_HTML` в `pageHtml.ts` — готовая статическая строка, **автосгенерированная** из Gamma («do not hand-edit; regenerate via build script»). Следствия: (а) серверная инъекция фрагмента формы в строку перед `resp.html()` возможна и естественна; (б) точка встраивания обязана жить в `index.tsx` — правки `pageHtml.ts` стирает регенерация; (в) в момент инъекции доступен `ctx` — данные предложения можно подтянуть server-side, форма уходит клиенту уже с ценой/названием.

Сценарий использования: пользователь выбирает подходящее предложение из аккаунта GetCourse → модуль генерирует форму под это предложение → форма подключается к любой странице юнита импортом, параметры передаются при подключении.

**Чего `form-gen` НЕ делает** _(черновик, уточняется)_:

- Не редактирует сами страницы `aNNN` — подключение формы к странице остаётся ручной операцией автора страницы (импорт).
- Не заменяет getcourse-гейтвей — с волны 2 (MVP) все обращения к GetCourse идут через него, своих ключей и прямых вызовов GC-API у модуля нет. Исключение — волна 1: прототип ходит в GC напрямую ради скорости проверки идеи (§0.2).

## 5. Бизнес-логика

### 5.1. Встраивание формы в страницу

**Решено (23-07-2026):** подключение — **импорт серверной функции** в `index.tsx` страницы; серверная инъекция фрагмента в `PAGE_HTML` перед `</body>`.

```tsx
import { attachOrderForm } from '../../sys/form-gen/shared/attach'

export const a475PageRoute = app.get('/', async (ctx) => {
  const html = await attachOrderForm(ctx, PAGE_HTML, {
    offerId: 12345,               // предложение GetCourse
    trigger: 'a[href="#order"]',  // CSS-селектор кнопок-триггеров страницы
  })
  return ctx.resp.html(html)
})
```

Свойства решения:

- Фрагмент самодостаточен: `<style>` (все классы с префиксом `fg-` — изоляция от стилей Gamma), скрытая модалка с формой, `<script>`-биндер (клик по `trigger` → `preventDefault` → открыть модалку).
- Данные предложения (название, цена) подтягиваются **на сервере** в момент рендера — форма уходит клиенту преднастроенной, без клиентского fetch за данными.
- URL эндпоинта отправки запекается server-side через `withProjectRoot(route.url())` — хардкода URL нет (инвариант платформы).
- Точка встраивания живёт в `index.tsx` — правки `pageHtml.ts` стирает регенерация из Gamma (§1).

**Сигнатура конфига (решено 23-07-2026):** минимальное обязательное ядро + опции с дефолтами. Принцип эволюции: новая настройка допускается **только** как опциональное поле с дефолтом — аддитивно, существующие вызовы не ломаются (§8); расширение обязательного ядра — ломающее, через правку спеки.

```ts
type AttachOrderFormConfig = {
  // обязательные — то, без чего форма бессмысленна
  offerId: number       // предложение GetCourse
  trigger: string       // CSS-селектор кнопок-триггеров страницы

  // опциональные, с дефолтами
  title?: string        // заголовок модалки; дефолт — название предложения из GC
  submitText?: string   // текст кнопки отправки; дефолт «Оплатить»
  fields?: Array<'name' | 'email' | 'phone'>  // дефолт — все три
  utm?: boolean         // прокидывать UTM-метки из URL страницы в заказ; дефолт true
}
```

**Отвергнуто:** `app.function` как основной канал — страницы и form-gen в проде живут в одном каталоге `p/units/neso/`, развязка между окружениями не нужна, типизированный импорт проще; сигнатура держится совместимой, `app.function`-обёртка добавляется аддитивно при потребности (§8). Клиентский `<script>`-виджет с fetch за конфигом — отвергнут как основной путь: лишний сетевой цикл и голый экран до загрузки; остаётся запасным для страниц вне воркспейса.

#### 5.1.1. Сигнатура конфига

**Решено (23-07-2026):** фиксируется **минимальное ядро**, всё остальное добавляется аддитивно опциональными параметрами с дефолтами (§8):

| Параметр      | Обязательность | Значение                                                                    |
| ------------- | -------------- | --------------------------------------------------------------------------- |
| `offerId`     | обязательный   | id предложения GetCourse                                                    |
| `trigger`     | обязательный   | CSS-селектор кнопок-триггеров страницы                                      |
| `title?`      | опциональный   | заголовок модалки; дефолт — название предложения из GC                      |
| `buttonText?` | опциональный   | текст кнопки отправки; дефолт — «Оформить заказ»                            |

- **Состав полей формы — константа модуля**, не параметр: пока вызов GC не проверен прототипом, параметризация полей — гадание; вынести в конфиг можно аддитивно позже. Вариант «`fields` в конфиге сразу» отвергнут: проектирование до проверки GC-вызова — риск переделки.
- **UTM-метки** из URL страницы скрипт собирает и отправляет **всегда автоматически** — это поведение, а не параметр: забыть невозможно, конфиг не раздувается.

### 5.2. Отправка формы и редирект на оплату

**Решено (23-07-2026):** отправка — **`fetch` + JSON**, редирект на клиенте.

Цикл: заполнение формы в модалке → `fetch POST` на эндпоинт form-gen → сервер валидирует, вызывает GetCourse (волна 1 — напрямую, с волны 2 — через getcourse-гейтвей), получает ссылку на оплату → ответ `{ ok, redirectUrl }` → клиент выполняет `location.href = redirectUrl`. Ошибка валидации/GC → `{ ok: false, error }` → сообщение внутри модалки, состояние формы сохраняется.

**Отвергнуто:** классический form-POST с 302 — модалке всё равно нужен JS, а fetch даёт показ ошибок без потери введённого. Ключи GC — только на сервере; клиент видит только эндпоинт form-gen.

**Источник ссылки на оплату (решено 23-07-2026):** операция GC Legacy-импорта `deals`, `action=add` — в гейтвее это `createDeal` (`p/gateways/getcourse/api/v1/createDeal.ts`): `POST https://<school>/pl/api/deals`, form-body `key`/`action`/`params` (Base64 от JSON), системный параметр **`return_payment_link`** возвращает ссылку на оплату в ответе. В каталоге гейтвея `createDeal` уже `enabled`.

**Волна 1 — временный дубль (решено 23-07-2026):** прототип воспроизводит эту обвязку прямым запросом по образцу `legacyGcImportClient.ts` + `legacyGcFormBody.ts` гейтвея. Дубль одноразовый: удаляется при переходе на гейтвей в MVP (волна 2) — форма зовёт `createDeal` через гейтвей, своего GC-клиента у form-gen не остаётся. Обоснование: обвязка обкатана в гейтвее, копия дешевле интеграции на этапе проверки идеи; риск расхождения снят коротким сроком жизни дубля.

_(Открыто: маппинг полей формы → `params` импорта и разбор ответа GC — проверяется прототипом, §0.2 волна 1.)_

### 5.3. Подбор предложения

**Решено (23-07-2026):** источник `offerId` зависит от волны.

- **Волна 1 (прототип):** id предложения указывается **вручную** — автор берёт его из аккаунта GetCourse и вписывает в вызов `attachOrderForm`. Никакого инструмента подбора нет — прототип проверяет сквозной цикл, а не удобство.
- **С волны 2 (MVP):** подбор — через **getcourse-гейтвей**: он предоставляет функционал запроса существующих предложений (`getOffers`, `enabled`) и выбора нужного. Ручной поиск id по аккаунту GC уходит.

Форма инструмента подбора (админ-страница со списком? скрипт, выдающий сниппет вызова?) и нужен ли Heap-конфиг форм — открытые пункты §0.2 волны 2.
