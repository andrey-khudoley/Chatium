# Spec-as-source: `getcourse-gateway`

Статус: в разработке (пересборка спецификации действующего проекта).
Последнее обновление: 22-07-2026.
Область действия: весь каталог проекта пересборки — dev-копия `s.chtm.khudoley.pro/d/gateways/getcourse` (окружение stage) и будущая prod-копия `p/gateways/getcourse` (§1, §3 «Окружения»). ## Словарь терминов

Англоязычные термины ниже используются намеренно — это устоявшиеся термины индустрии или платформы, точного однословного русского эквивалента у них нет. Все прочие английские выражения в тексте переведены на русский.

| Термин             | Значение                                                                                                                                                                                                            |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **gateway**        | Тонкий HTTP-фасад между приложениями Chatium и API GetCourse: принимает `/v1/{op}`, проксирует один исходящий вызов к школе, нормализует ответ и ошибки.                                                             |
| **Heap**           | Встроенное хранилище данных платформы Chatium (имя собственное).                                                                                                                                                    |
| **spec-as-source** | Методология, при которой спецификация — источник истины: код не меняется, пока изменение не описано здесь (§0).                                                                                                     |
| **op**             | Стабильный `camelCase`-идентификатор операции gateway (`addUser`, `getAllGroups`); сегмент публичного пути `/v1/{op}`. Канон — реестр [`gc-unified-op-registry-v0`](../gateway/gc-unified-op-registry-v0.md).       |
| **контур**         | Одна из двух независимых половин API GetCourse: **`new`** (Tech API, JSON, `/pl/api/v1`, Bearer) и **`legacy`** (импорт/экспорт, форма `key`+`action`+base64 `params` / GET-query, `/pl/api`).                      |
| **школа**          | Аккаунт GetCourse клиента: пара «хост» (`X-Gc-School-Host`) + «ключ API школы» (`X-Gc-School-Api-Key`). Gateway ключи продакшен-школ **не хранит** (§5.5).                                                          |
| **availability**   | Операционный статус операции в каталоге: `enabled` \| `beta` \| `disabled` \| `unsupported` (§5.6).                                                                                                                 |
| **каталог**        | Машиночитаемый SSOT опубликованных операций (метаданные + схема `args`), зашитый в код и отдаваемый через `GET /v1/operations` (§5.2).                                                                              |
| **upstream**       | Исходящий HTTP-вызов gateway к GetCourse (ровно один на входящий запрос, §5.7).                                                                                                                                     |
| **raw-журнал**     | Heap-таблица со снимком входящего запроса (`gatewayRequestLog`) или исходящего вызова (`gatewayUpstreamLog`) после PII-маскирования (§3.5, §5.8).                                                                    |
| **PII**            | Персональные данные; в raw-журналы и логи попадают только после маскирования (`redactRaw`).                                                                                                                          |
| **requestId**      | UUID v4, генерируется на каждый входящий запрос `/v1/*`; сквозная корреляция: JSON-ответ + заголовок `X-Gateway-Request-Id` + логи + raw-журналы (§5.1).                                                            |
| **SSOT**           | Single source of truth — единственный источник истины по конкретному вопросу.                                                                                                                                       |
| **UGC**            | Код приложения Chatium, публикуемый как бандл; среда исполнения — изоляты V8 без гарантий межзапросного состояния.                                                                                                  |
| **teardown**       | Завершающая фаза интеграционного прогона: откат/маркировка созданных в школе сущностей (§9).                                                                                                                        |
| **shim**           | Временная прослойка совместимости (`handleV1OpRoute.ts`), сохраняющая старый вход при новой архитектуре до завершения миграции.                                                                                     |

## 0. Правило предварительной спецификации

Редактировать код проекта пересборки запрещено, если требуемое изменение ещё не описано в этой спецификации.

**Иерархия источников.** Эта спецификация — верхний источник истины проекта. Ей подчинены унаследованные нормативы в [`docs/gateway/`](../gateway/): [`gateway-operation-manual.md`](../gateway/gateway-operation-manual.md) (детальный протокол: жизненный цикл, коды ошибок §10, семантические правила §2.8, схемы), [`gc-unified-op-registry-v0.md`](../gateway/gc-unified-op-registry-v0.md) (реестр 59 `op`), [`gateway-testing-strategy.md`](../gateway/gateway-testing-strategy.md) (дизайн тестов) и машиночитаемые артефакты (`new_api_schema.json`, `legacy_api_schema.json`, `gc-op-http-mapping.json`, `gc-required-fields-by-op.json`). Они сохраняют силу нормативов в той части, на которую спека ссылается; **при расхождении приоритет за spec.md**, затем правится manual. Дублировать сюда протокольные таблицы manual не требуется — спека фиксирует решения и структуру, manual — развёрнутый контракт.

**Отношение к действующей реализации.** В отличие от брокера, готовая реализация существует: `p/gateways/getcourse` (59 роутов, конвейер `handleV1Op`, рукописный каталог, raw-журналы, панель — ADR-0004). Она — **референс поведения**, но не источник истины: спека переописывает её решения явно, и код пересборки обязан соответствовать спеке, а не «историческому коду». Расхождение кода с этой спекой — дефект кода либо осознанная правка спеки, третьего нет.

**Способ переноса — гибрид (решено 22-07-2026, [ADR-0005](../ADR/0005-hybrid-transfer-core-copy.md)).** Ядро переносится **копией с ревизией по спеке**: `lib/gateway/` (конвейер, каталог, клиенты контуров, семантическая классификация), `api/v1/` (59 роутов + `operations`), `shared/`-валидации и wire-типы — эта часть обкатана живыми вызовами, переписывание рискует тонкими регрессиями без архитектурного выигрыша. Обвязка **пересобирается** под решения волны 1 (§0.2): каркас, логгер, таблицы, тесты, панель — по необходимости. Точная граница «копия/пересборка» уточняется остальными решениями волны 1 и фиксируется в ADR-0005.

**Каркас.** Стартовый каркас пересборки — канонический `inner/samples/new_project` (структура, admin API, тесты) с оговорками волны 1 (§0.2): судьба Heap-таблицы логов и модель тестов решаются до первого кода. Исторический каркас `p/template_project` — только происхождение действующей реализации, спека на него не опирается.

## 0.2. Волны разработки и чек-лист готовности

Спецификация закрывается не целиком, а волнами: пока пишется код очередной волны, закрываются вопросы следующей. Ниже — все открытые пункты, разложенные по волнам. **Закрытые пункты из списка удаляются**, а не остаются отмеченными: решение живёт в соответствующем разделе спеки, здесь — только оставшаяся работа.

### Принцип распределения

Пункт попадает в волну по **цене позднего решения**, а не по важности. Три класса:

| Класс           | Что стоит поздний ответ                                                                                     | Крайний срок                      |
| --------------- | ----------------------------------------------------------------------------------------------------------- | --------------------------------- |
| **Необратимое** | форма или смысл уже записанной строки, факт удаления данных. Правится миграцией, при неудаче — потерей      | до первой боевой записи в таблицу |
| **Ломающее**    | внешний контракт: набор операций, коды ошибок, заголовки. Правится согласованным обновлением потребителей   | до первого внешнего потребителя   |
| **Аддитивное**  | ничего: новая операция, новое поле с дефолтом, новый экран                                                  | в любой момент                    |

**Особенность проекта: внешний контракт уже опубликован.** Слой `/v1/*` (пути, заголовки `X-Gc-*`, конверт `ok`/`error`/`requestId`, коды §10 manual, инвариант «200 ⇔ ok:true») существует и имеет потребителей (SDK `p/units/chatiumclub/sdk`, клиенты client1–4). Для пересборки весь этот контракт — **данность класса «ломающее», уже замороженная**: наследуется как есть, любое ломающее изменение — только через `/v2` + ADR (§8). Поэтому волна 2 здесь не «замораживает» контракт, а **воспроизводит** его.

### Волна 1 — Каркас и решение о переносе

**Код:** каркас проекта в `d/gateways/getcourse` без боевых данных. Здесь принимались все необратимые решения фундамента — до первой строки в таблицах.

**Гейт закрыт — все решения приняты 22-07-2026:** перенос — гибрид, ядро копией / обвязка пересборкой (§0, [ADR-0005](../ADR/0005-hybrid-transfer-core-copy.md)); таблицы — пары stage/prod со статическим селектором, данные старых таблиц — чистый лист без миграции (§3.6); скоуп `gw-gc`, идентификаторы всех пяти пар зафиксированы (§3); логирование — ClickHouse без своей таблицы (§5.10); тесты — штатная система Chatium с машинным гейтом (§9).

**Открытых пунктов нет — волна закрыта по решениям.** Осталась реализация каркаса; приёмка — §9.5.1.

### Волна 2 — Ядро `/v1` (MVP пересборки)

**Код:** конвейер `handleV1Op` (§5.1), каталог 59 операций + `GET /v1/operations` (§5.2), исходящие клиенты обоих контуров (§5.3), классификация ответов (§5.4), секреты и настройки (§5.5), raw-журналы (§3.5, §5.8), базовые тесты ядра. С первой строки в боевой таблице данные живут — гейт волны: всё, что меняет форму или смысл строки.

- [ ] **Семантические юниты L1–L4 / N1–N3** — реализовать полный набор (объём зафиксирован 22-07-2026, §9.1): 7 правил × (позитив с проверкой `error.details.gcRule` + негатив) + кейс нормализации булева Legacy. Действующая реализация покрывает частично (план §2.2 не закрыт с мая); без этого набора волна 2 не зелёная
- [ ] **Прицельный юнит Bearer-склейки** (подтверждён 22-07-2026, §9.1): точное строковое равенство `Authorization: Bearer {dev}_{school}` + краевые случаи (trim секретов до склейки, подчёркивания внутри ключей). Перенос незакрытого пункта плана §1.6
- [ ] **Ликвидация shim** (подтверждено 22-07-2026): `handleV1OpRoute.ts` + `v1AddUserHandler.ts` в новый проект **не переносятся**; тест-раннер и интеграционный кейс переводятся на реестр `v1OpHandlers` / `handleV1Op` напрямую при переносе ядра (последствие ADR-0004, часть ревизии ADR-0005)
- [ ] **Фактическое значение `workspace_path` gateway** (§5.10) — подтвердить прогоном в каждом окружении: записать лог из боевого кода и прочитать обратно без фильтра по пути. Тот же открытый пункт у брокера (его §0.2, волна 2) — проверка методически общая, значения свои (`d/gateways/getcourse` stage / `p/gateways/getcourse` prod, предположительно)

**Чего в волне сознательно нет:** панель и доступы (волна 3) — ядро проверяемо тестами и curl без UI; ревизия `availability` (волна 3–4) — стартовое состояние каталога наследуется из действующей реализации (6 `enabled`: `createDeal`, `getAllGroups`, `getDealFields`, `getOffers`, `getUserFields`, `getUserGroups`; 53 `disabled`; политика «тестим по одному и включаем», решение 09-05-2026).

**Приёмка:** §9.5.2.

### Волна 3 — Beta: наблюдаемость, панель, полный сьюит

**Код:** панель мониторинга и доступы (§5.9), KPI/фильтры (§5.8), полный интеграционный сьюит по фазам 1–4 со teardown (§9), включение retention. Гейт — всё, что фиксирует контракт админ-слоя, и всё, что удаляет.

- [ ] **Полный интеграционный сьюит зелёный** (определение зафиксировано 22-07-2026, §9.2): для каждой операции в `enabled`/`beta` на момент приёмки сценарий зелёный; teardown отработал; префлайт включённых чистый (без `warn-*`); 1 rps. Уровень A (`gc_developer_api_key`, `gc_test_school_api_key`, `gc_test_school_host`) + `gc_itest_*` уровня B; сьюит растёт вместе с включением операций
- [ ] **Ревизия `availability` (начало)** — по правилу §5.6 (зафиксировано 22-07-2026): включение только при зелёном сценарии сьюита; объём волны 3 — операции сценариев A–D (`addUser` первым) + каталоги-производители фазы 2; остальные — волна 4
- [ ] **Пороговые уведомления** (контракт зафиксирован 22-07-2026, §5.8): 15-минутный джоб, два триггера (dev-ключ сломан; доля 5xx > 50 % при ≥ 20 вызовах/час), анти-спам 6 ч, пороги — константы кода
- [ ] **KPI-сводка панели** (решено 22-07-2026, §5.8): тяжёлые метрики (p95/avg `durationMs`, topErrorCode) — из ClickHouse через `queryAi` по записям `v1_op_completed`; дешёвые счётчики — `countBy` по Heap; скан `DASHBOARD_SCAN_LIMIT` не воспроизводится. Реализация: SQL-запросы + деградация без `@traffic/sdk`

**Приёмка:** §9.5.3.

### Волна 4 — Релиз

**Код:** всё аддитивное поверх работающего gateway. Гейта нет — каждый пункт добавляет, ничего не переформулируя.

- [ ] **Ревизия `availability` (завершение)** — критерий «ноль дефолтов» (зафиксирован 22-07-2026, §5.6): у каждой из 59 операций осознанный статус с записанной причиной; `enabled`/`beta` — только через зелёный тест

## 1. Назначение

`getcourse-gateway` — **тонкий stateless HTTP-фасад к API GetCourse**: единый публичный слой `/v1/{op}` над двумя разнородными контурами платформы GetCourse (`new` Tech API и `legacy` импорт/экспорт), с единым конвертом ответа, каталогом операций, нормализацией ошибок (включая семантические ошибки GC при HTTP 2xx) и наблюдаемостью каждого вызова. Клиенты gateway — приложения Chatium (через тонкий SDK или прямой HTTP) и ручные вызовы; gateway избавляет их от знания контуров, форм авторизации, base64-форм Legacy и разбора зоопарка ответов GC.

Принципиальные границы (все — осознанные решения, зафиксированы в manual и наследуются):

- **Мультитенантность по заголовкам, без хранения секретов клиентов.** Школа адресуется парой заголовков `X-Gc-School-Host` + `X-Gc-School-Api-Key` на каждом запросе; ключи продакшен-школ в Heap **не** хранятся (§5.5). В Heap живут только dev-ключ GetCourse и тестовые ключи для автотестов.
- **Один входящий → максимум один исходящий.** Серверные ретраи и идемпотентность **запрещены** (manual §8.6, §12.4 — жёсткое решение, не «отложено»): gateway не читает `Idempotency-Key`, не ведёт журналов дедупликации, не ставит фоновых повторов. Повторы — ответственность клиента.
- **Входящих вебхуков нет.** Gateway не принимает HTTP от GetCourse (manual §6, ADR-0004 внешнего контура): вебхуки принимают прикладные приложения на свои URL; единственный канал данных от GC — ответы на исходящие запросы. Регистрация вебхука (`setUri`) — обычная исходящая операция.
- **Без сессии Chatium на `/v1/*`.** Публичный слой не требует пользователя платформы; профиль доступа — конфигурационный (dev-ключ настроен + валидные заголовки школы, §7).
- **Stateless между запросами.** Никакой опоры на межзапросный кэш изолята; каталог — статический TS-модуль в бандле (§5.2).

Проект живёт в категории `gateways` воркспейса. Окружения: dev-копия `s.chtm.khudoley.pro/d/gateways/getcourse` (stage) и prod-копия `p/gateways/getcourse`; модель переноса — копирование кода, выбор набора таблиц — по пути копии (селектор — §3; вопрос пар таблиц — волна 1, §0.2).

## 2. Доменная модель

Домен gateway минимален — он не владеет доменом GetCourse, а транслирует его:

| Сущность              | Природа                                                                 | Где живёт                                                    |
| --------------------- | ----------------------------------------------------------------------- | ------------------------------------------------------------ |
| **Операция (`op`)**   | Публичная единица каталога: имя, метод, контур, доступность, схема `args`, шаблон пути к GC | Каталог в коде (§5.2); реестр-канон в [`docs/gateway/`](../gateway/gc-unified-op-registry-v0.md) |
| **Вызов**             | Один входящий запрос `/v1/{op}` с его `requestId`, исходом и таймингом  | Не хранится как сущность; след — raw-журналы + логи (§3.5, §5.8) |
| **Upstream-вызов**    | Исходящий HTTP к школе (0 или 1 на вызов)                               | След — `gatewayUpstreamLog` (§3.5)                           |
| **Школа**             | Пара «хост + ключ API» из заголовков; gateway её не персистит           | Только в контексте запроса                                   |
| **Настройка**         | Строка key-value: dev-ключ, тестовые ключи, `gc_itest_*`, операционные ключи панели | `settings` (§3.1)                                            |
| **Доступ к панели**   | Право не-Admin видеть панель: грант, выданный по инвайту                | `panelAccess` / `panelInvites` (§3.3, §3.4)                  |

Состояний с жизненным циклом у gateway два: `availability` операции (операционный enum на записи каталога, меняется публикацией кода — §5.6) и статус инвайта (выдан → принят/отозван/истёк — §5.9). Всё остальное — журналы фактов.

## 3. Хранилище (Heap)

**Ограничения платформы** (наследуются из практики воркспейса, `008-heap.md`): индексами управлять нельзя; индексируются только плоские корневые поля; фильтруешь на горячем пути — поле корневое и плоское; лимит таблицы 1 млн строк закладывается retention'ом, а не надеждой. Подсчёты — `countBy`/`where`, не `findAll().length`.

**Идентификаторы таблиц (зафиксированы 22-07-2026) — парами stage/prod** (§3.6), формат `t__gw-gc__<таблица>__stage_<хвост>` / `…__prod_<хвост>`, хвост общий на пару:

| `settings`           | `setting`     | `UqaXHv`   | Настройки key-value               |
| `panelAccess`        | `paccess`     | `sZlSgN`   | Гранты доступа к панели           |

| `gatewayUpstreamLog` | `gups`        | `hbI66D`   | Raw-журнал исходящих вызовов GC   |

Пример: `t__gw-gc__setting__stage_UqaXHv` (dev-копия) / `t__gw-gc__setting__prod_UqaXHv` (prod-копия). Таблицы `logs` нет (§3.2). Старые одиночные `t__saas-gw-gc__*` действующей реализации (`setting__W3r8Ys`, `log__K7n2Tp`, `paccess__GDgDvk`, `pinvite__U7FaXD`, `greq__Gr9Qm2`, `gups__Up7Mn3`) — архив, новым кодом не используются (§3.6).

### 3.1. `settings` — настройки (key-value)

Generic key-value (`key: string`, `value: any`) — паттерн воркспейса: новая настройка = новая строка, форма таблицы не замораживается. Фиксированные ключи: `gc_developer_api_key` (секрет, §5.5), `gc_test_school_api_key` (секрет), `gc_test_school_host`, `log_level`, `logs_limit`, `log_webhook`, `panel_date_filter`, `dashboard_reset_at`, `project_name`/`project_title` + произвольные `gc_itest_*` (уровень B тестов, manual §5.8–5.9). Валидация при записи: непустые строки после `trim` для ключей; хост — правила manual §2.5 (без схемы, опциональный `:порт` 1–65535, ≤253 символов). Секреты в логах маскируются (`[redacted]`). Доступ — только Admin (§7).

### 3.2. `logs` — упразднена

**Решено (22-07-2026):** собственной таблицы серверных логов в проекте пересборки **нет** — логирование через ClickHouse без своей таблицы (§5.10). Пара stage/prod для логов не заводится. Действующая `t__saas-gw-gc__log__K7n2Tp` остаётся архивом по правилу «чистого листа» (§3.6).

### 3.3. `panelAccess` — гранты доступа к панели

`userId` (searchable), `grantedAt`, `grantedBy`. Наличие строки = активный грант (Admin в таблице не нуждается). Отзыв — удаление строки (Admin).

### 3.4. `panelInvites` — инвайты

`token` (searchable), `issuedAt`, `expiresAt` (TTL 7 дней), `consumedAt`, `consumedByUserId`, `revokedAt`, `createdBy`. Приём инвайта — под `runWithExclusiveLock` (гонка двойного приёма). Классификация: активен / принят / отозван / истёк.

### 3.5. `gatewayRequestLog` / `gatewayUpstreamLog` — raw-журналы

Введены ADR-0004 вместо workspace-событий: полный контроль хранения и запросов, аналитика нативными Heap-примитивами.

- **`gatewayRequestLog`** — строка на **каждый** входящий `/v1/{op}`: `requestId` (searchable), `op` (searchable), `contour`, `method`, `schoolHost`, `rawArgs`, `rawHeadersSafe`, `clientHttpStatus`, `errorCode`, `durationMs`, `requestedAt`. Пишется в `finally` конвейера — всегда, независимо от исхода.
- **Схема зафиксирована (22-07-2026): поля действующей реализации + новое плоское поле `schoolHost`** в `gatewayRequestLog` — валидированное значение заголовка школы. Хост — **не секрет** (manual §2.5, §5.7); плоское корневое поле фильтруемо: разбор инцидентов «у школы X» и фундамент метрик по школам (волна 4). Из `rawHeadersSafe` маска по-прежнему удаляет заголовок хоста — значение живёт только в выделенном поле, без дублирования. Свои отметки времени (`requestedAt`, `sentAt`) — `number` (epoch ms), по конвенции воркспейса.
- **`gatewayUpstreamLog`** — строка на **каждый** исходящий вызов GC (если до отправки дошло): `requestId` (searchable), `op`, `upstreamKind`, `rawGcJson`, `gcHttpStatus`, `semanticRule`, `durationMs`, `sentAt`.
- Связь 1:0..1 по `requestId`. Перед записью — PII-маскирование `redactRawDeep` (лимит 64 KiB). Секреты (ключи, `Authorization`) в журналы не попадают by construction (§5.5).
- **Retention (решено 22-07-2026): 30 дней, без архива.** Ежедневный `app.job` в полночь по образцу брокера: самоперепланирование `scheduleJobAt(00:00)` **первым действием** (переросший бюджет проход обрывается после тяжёлой операции — перепланирование в конце остановило бы цепочку навсегда); затем по одному `deleteAll({ limit: null, hard: true })` на каждую таблицу по условию возраста (`hard` обязателен — иначе строки остаются и лимит достигается при работающей чистке; числовой `limit` — предохранитель, не размер партии); парный лог `start`/`done` + `durationMs` (непарность = переросший бюджет). Порог — константа `RAW_LOG_RETENTION_DAYS = 30`, общая обеим таблицам и обоим исходам; меняется только правкой спеки. Холодного архива нет: raw-журналы — операционная диагностика, не бизнес-факты; длинные KPI при потребности строятся из ClickHouse. Включение джоба — волна 3 (удаление держится вне волны, где схема оседает).

## 5. Бизнес-логика

### 5.1. Конвейер `/v1/{op}` — `handleV1Op`

Единый обработчик `handleV1Op(ctx, req, op, handler)`; per-op хендлер в файле роута делает только явный выбор клиента GC. Порядок проверок нормативен (manual §2.6, §2.11) — каждая ступень отсекает до следующей, до GC доходит только полностью валидный запрос:

1. `requestId` (UUID v4) — генерируется первым, живёт до ответа;
2. запись каталога для `op` — есть файл роута без записи → `INVOKE_INTERNAL_ERROR` (рассогласование сборки); неизвестный путь — платформенный 404 → `INVOKE_OP_UNKNOWN`;
3. HTTP-метод = `entry.httpMethod`, иначе `INVOKE_HTTP_METHOD_NOT_ALLOWED` (405);
4. `availability`: `disabled` → 503, `unsupported` → 501 — **до** заголовков школы и Heap (выключенная операция не зависит от конфигурации);
5. лимит тела POST — `GW_MAX_REQUEST_BODY_BYTES = 1 MiB` → `INVOKE_BODY_TOO_LARGE` (413) до парсинга;
6. заголовки школы: `X-Gc-School-Host` (валидация хоста manual §2.5) + `X-Gc-School-Api-Key` (непустой);
7. `gc_developer_api_key` из Heap — пусто → `GATEWAY_DEV_KEY_NOT_CONFIGURED` (503), проверяется для **обоих** контуров (единый профиль конфигурации);
8. для POST — `Content-Type: application/json` и тело-объект (объект и есть `args`, обёртки нет); для GET — `args` из query (дубликат ключа → последнее значение);
9. валидация `args` схемой каталога (`safeParse` билдера `s` из `@app/schema`) → `INVOKE_ARGS_SCHEMA_VIOLATION`;
10. подстановка `{path-параметров}` из `args` по `pathTemplate`;
11. **один** исходящий HTTP к GC (§5.3) → классификация ответа (§5.4) → конверт клиенту; в `finally` — raw-журналы (§3.5) при любом исходе.

### 5.2. Каталог операций

**Рукописный статический массив** 59 записей (ADR-0004: codegen упразднён — каталог меняется редко, рукописный файл правится без инструментальной цепочки, нет расхождений источника и артефакта). Запись: `op`, `httpMethod`, `contour`, `availability`, `pathTemplate`, `legacyImportAction?`, `argsValidator` (рантайм-объект `s.object(...)`), `argsSchema.fields[]` (plain-описатель для клиентов). Канон имён и состав — реестр v0 (`../gateway/gc-unified-op-registry-v0.md`): 51 `new` + 8 `legacy`; расширение — §8.

### 5.3. Исходящий транспорт к GetCourse

Только `@app/request`; произвольный `fetch` запрещён. Базовый URL: `https://{host из заголовка}` + `/pl/api/v1` (new) / `/pl/api` (legacy). Таймаут — жёсткая константа `GW_OUTBOUND_TIMEOUT_MS = 10_000` (без настройки; пересмотр — по p99 статистике, manual §12.2).

- **Контур `new`:** метод/путь/тело по OpenAPI; авторизация `Authorization: Bearer {devKey}_{schoolApiKey}` (склейка двух секретов через `_`).
- **Контур `legacy`:** импорт — POST `application/x-www-form-urlencoded` с полями `key`, `action`, `params` (стандартный Base64 UTF-8-байтов `JSON.stringify` объекта действия; собственный кодек `utf8Base64`, глобалы платформы запрещены — `047-base64.md`); экспорт — GET с `key` и фильтрами в query. Dev-ключ на провод Legacy **не** уходит.

Сбои: таймаут → `INVOKE_GC_TIMEOUT` (504); сеть/TLS/DNS → `INVOKE_GC_NETWORK_ERROR` (502). Ретраев нет (§1).

### 5.4. Классификация ответа GC

Два уровня (manual §2.8 — нормативная таблица правил):

1. **Транспортный:** статус вне 2xx → `INVOKE_GC_UPSTREAM_ERROR` (502); 2xx+JSON → парсинг (ошибка парсинга → 502); 2xx не-JSON → успех с `data = { gcRawBody, gcContentType, gcRawBodyTruncated? }` (обрезка 512 KiB).
2. **Семантический** (GC умеет отвечать 2xx с ошибкой в теле): Legacy — правила **L1–L4** (`success` ложный по нормализации булева; строковый `error`; `result.error === true|1`; `result.success` ложный); new — **N1–N3** (`status === false`; числовой `code !== 200`; `data.result === false` при успешной обёртке). Срабатывание → `INVOKE_GC_SEMANTIC_ERROR` (502, **не** 200), `error.details.gcRule` — из фиксированного перечня семи значений. Инвариант «200 ⇔ ok:true» держится именно этой ступенью.

### 5.5. Секреты

| Секрет                     | Хранение                                     | На проводе к GC                                          |
| -------------------------- | -------------------------------------------- | -------------------------------------------------------- |
| Ключ разработчика GC       | Heap `gc_developer_api_key` (Admin-only)     | только контур `new`, первая часть Bearer                 |
| Ключ API школы клиента     | **нигде** — заголовок каждого запроса        | `new`: вторая часть Bearer; `legacy`: поле/query `key`   |
| Тестовый ключ школы        | Heap `gc_test_school_api_key` (для автотестов) | тесты подставляют его в тот же заголовок               |

Запреты логирования (manual §5.7): значения обоих ключей, полный `Authorization`, полные тела GC — не попадают ни в логи, ни в raw-журналы, ни в `error.details`. Допустимы: `op`, хост школы (не секрет), коды ошибок, длины, `requestId`.

### 5.6. `availability`

Операционный enum на записи каталога: `enabled` (стабильна) / `beta` (успех обязан нести `warnings` с `GATEWAY_OP_BETA_UNSTABLE`) / `disabled` (503 до GC) / `unsupported` (501 до GC). Меняется правкой каталога + публикацией (окно рассогласования внутри одной версии UGC исключено сборкой).

**Правило включения (зафиксировано 22-07-2026):** перевод операции из `disabled` — **только** при зелёном сценарии этой операции в интеграционном сьюите на тестовой школе (+ юнит-покрытие по стратегии §7.3). Выбор статуса: `enabled` — тест зелёный и оговорок нет (типично чтение и каталоги); `beta` — тест зелёный, но есть известные оговорки (нестабильность GC, неполные фикстуры) — клиент получает `warnings`. Включение до зелёного теста («понадобилось, потом протестируем») запрещено.

**Объём волны 3** — операции реальных потребителей и цепочек сьюита: сценарии A–D — `addUser` (первым: интеграционный кейс уже существует, текущий `disabled` — аномалия), `setUri`, `addCommentToDialog`, `updateUserCustomFields` (+ уже включённый `createDeal`); каталоги-производители фазы 2 — `getAllWebinars`, `getAllPersonalManagers`, `getTrainings` (+ уже включённые `getAllGroups`, `getOffers`, `getDealFields`, `getUserFields`, `getUserGroups`). Остальные ~40 — волна 4, по одному, тем же правилом.

**Релизный критерий — «ноль дефолтов» (зафиксирован 22-07-2026; волна 4):** ревизия завершена, когда у **каждой** из 59 операций статус — осознанное решение с записанной причиной: `enabled` — тест зелёный; `beta` — тест зелёный, оговорки записаны; `disabled` — причина записана (нет воспроизводимой фикстуры / нет спроса / ждёт очереди); `unsupported` — зафиксирована неподдержка со стороны GC. Для `disabled`/`unsupported` в записи каталога заводится поле-комментарий причины — каталог становится реестром решений, а не исторических умолчаний.

### 5.7. Жёсткие ограничения слоя

Сводка решений, которые при пересборке **запрещено** «улучшать» кодом без правки спеки: ретраи и идемпотентность (§1); входящие вебхуки (§1); хранение ключей школ (§5.5); межзапросный кэш как контракт (UGC-изоляты — каталог только статическим импортом); прямые `ctx.account.log`/`console.log` в прикладном коде (только обёртка логгера); расширение Content-Type сверх JSON; второй исходящий вызов в одном обработчике.

### 5.8. Наблюдаемость

Три слоя, все связаны `requestId`:

1. **Raw-журналы Heap** (§3.5) — полный маскированный след каждого вызова и upstream; источник панели (ленты, карточки, KPI `gatewayCounts` через `countBy` по глобальному фильтру дат).
2. **Серверные логи** — непрерывный поток по веткам конвейера (manual §7.2.1: каждая существенная ветка даёт запись с кодом этапа + `message` + безопасным `payload`); итоговая запись `v1_op_completed` с минимальным набором полей (`requestId`, `op`, `contour`, `availability`, `schoolHost`, `clientHttpStatus`, `ok`, `errorCode`, `gcHttpStatus`, `durationMs`). Хранение — ClickHouse (§5.10).
3. **Панель** (`HomePage.vue`): сводка администратора, таблицы-превью, вкладки журналов, форма «Создать запрос» (имитация внешнего вызова с подстановкой тестовых значений; `gc_developer_api_key` клиенту не выдаётся — добавляется сервером).

**Источники KPI-сводки (решено 22-07-2026).** Дешёвые счётчики (total, ok-доля, upstream-счётчики) — `countBy`/`where` по raw-журналам Heap: точно и дёшево. **Тяжёлые метрики (p95/avg `durationMs`, topErrorCode) — из ClickHouse** через `queryAi` по итоговым записям `v1_op_completed` (`quantile`/`avg`/`topK` над `JSONExtract` из `json_str`, окно — фильтр дат панели): перцентили Heap не умеет, а скан строк с лимитом (`DASHBOARD_SCAN_LIMIT` действующей реализации) дорожает с ростом журнала и молча врёт при обрезке — не воспроизводится. Следствия: **(а)** ключи `payload` записи `v1_op_completed` (`durationMs`, `op`, `errorCode`, `ok`, `clientHttpStatus`, `gcHttpStatus`, `schoolHost`) — **контракт**: переименование ломает SQL панели и требует правки спеки; **(б)** деградация обязательна — `@traffic/sdk` недоступен или запрос упал → сводка показывает Heap-счётчики с пометкой «расширенные метрики недоступны», панель не падает.

**Экран «по школам» (решено 22-07-2026; волна 4):** те же метрики с группировкой по школе — счётчики через `countBy` по плоскому `schoolHost` журнала (§3.5), тяжёлые — CH-запрос с `GROUP BY` по ключу `schoolHost` из `v1_op_completed`. **SLA** (uptime, целевой p95) формулируется по триггеру «первая боевая школа» — цифры берутся из фактической статистики тестового периода (недельное окно), не выдумываются заранее; до того SLA не нормируется.

Workspace-события `@start/sdk` **не** используются (ADR-0004; соответствующие разделы manual §7.3–§7.4 в части `writeWorkspaceEvent` — исторические).

**Пороговые уведомления (контракт зафиксирован 22-07-2026; реализация — волна 3).** Периодический `app.job` каждые 15 минут (самоперепланирование первым действием — паттерн retention §3.5) считает по `gatewayRequestLog` через `countBy` два триггера: **(1) сломана конфигурация** — ≥ 1 запись с `GATEWAY_DEV_KEY_NOT_CONFIGURED` за последние 15 минут (каждый вызов new-контура отваливается — всегда авария); **(2) массовые ошибки** — доля ответов 5xx > 50 % при ≥ 20 вызовах за скользящий час. Третий триггер (волна 4): **(3) подозрение на перебор ключей** — > 30 ошибок аутентификации GC (`INVOKE_GC_UPSTREAM_ERROR` / `INVOKE_GC_SEMANTIC_ERROR` с 401/403-признаками) с одного `schoolHost` за скользящий час — один `countBy` по плоскому `schoolHost` (§3.5). Срабатывание любого триггера → `sendNotificationToAccountOwners` (`@user-notifier/sdk`): краткий текст без секретов + ссылка на панель. **Анти-спам:** не чаще одного уведомления на триггер в 6 часов (отметка последней отправки — строка в `settings`). Пороги — **константы кода** (`NOTIFY_*`), меняются правкой спеки; вынос в настройки — аддитивно при доказанной потребности.

**Активный rate-limit против перебора ключей не вводится (решено 22-07-2026).** Ключ школы проверяет GetCourse, gateway ключей не хранит и оракулом не является — стоимость атаки задают лимиты самого GC; самодельный лимит на платформе либо дорог (Heap-счётчик на каждый вызов — удвоение стоимости запроса), либо ненадёжен (in-memory счётчики не переживают изоляты, §5.7). Перебор не блокируется, но детектируется триггером (3). Вопрос переоткрывается по факту инцидента или появления платформенного примитива.

### 5.9. Доступы к панели

Двухуровневая модель: роль **Admin** аккаунта ИЛИ **активный грант** (`panelAccess`). Гейты: `requireInternalAccess` (страницы; отказ → `/web/forbidden`), `guardInternalApi` (API; отказ → 403). Инвайты: Admin генерирует токен (TTL 7 дней) → пользователь принимает на `/web/access/invite` (`consumeInvite` под `runWithExclusiveLock`) → грант; отзыв инвайта/гранта — Admin. Страница тестов — только Admin (грант не достаточен).

### 5.10. Логирование

**Решено (22-07-2026): модель брокера — ClickHouse, своей таблицы логов нет.** `ctx.account.log` уже пишет в ClickHouse `chatium_ai.account_logs`; Heap-копия была бы вторым экземпляром тех же данных (таблица `logs` упразднена — §3.2).

- **Единая точка** — `writeServerLog(ctx, { level, message, payload? })`; прямые `ctx.account.log` / `ctx.log` / `console.log` в прикладном коде запрещены (обходят отсечку и живой монитор). `throwLoggedServerError` сохраняется поверх неё.
- **Уровни — платформенные** (`error`/`warn`/`info`/`debug`); syslog-шкала 0–7 действующей реализации сужается маппингом: 7→`debug`, 6–5→`info`, 4→`warn`, ≤3→`error`. Своя шкала не переносится (довод брокера: отображение при записи + разбор при чтении без выигрыша).
- **Отсечка** — настройка `log_level` (`Disable`/`Error`/`Warn`/`Info`/`Debug`, дефолт `Info`) **до** вызова платформы.
- **Payload:** в ClickHouse и dev-консоль — всегда; в WebSocket живого монитора — только на `Debug`. Живой монитор админки сохраняется.
- **Чтение истории** — `queryAi` из `@traffic/sdk` (динамический `import` + `@ts-ignore`, модуль внедряется платформой); обязательный фильтр `workspace_path` (значение подтвердить прогоном — §0.2 волна 2), `total` — отдельным `count()`.
- **Запрет рекурсии:** чтение настроек логгера внутри логгера не логируется.
- Вебхук `log_webhook` действующей реализации **не переносится** — при потребности возвращается аддитивно (волна 4).
- Запреты содержимого — §5.5 (секреты) без изменений. Платформенная механика — `inner/docs/050-logging.md`; реализация-образец — логгер брокера.

## 8. Правила эволюции

- **Версия публичного API — единственная, `/v1`.** Ломающее изменение (переименование/удаление полей конверта, смена типов, семантики `requestId`, кодов/статусов §10, требований к заголовкам, инварианта «200 ⇔ ok:true») — только новым префиксом `/v2` + ADR + политика параллельной поддержки. До того — запрещено (manual §9.3, §12.5).
- **Аддитивно и безопасно:** новые `op` в каталоге, новые опциональные поля ответа, новые `error.details.*`, новые экраны панели, новые настройки-строки.
- **Имена `op` стабильны** (canon-реестр); обратная несовместимость `args` конкретной операции — предпочтительно новое имя `op`.
- **Расширение реестра `op` — процесс из 5 шагов (зафиксирован 22-07-2026):** (1) триггер — реальный запрос школы или сценария, не впрок; (2) имя — по правилам нейминга реестра (`camelCase`, глагол + сущность, без `snake_case`); (3) ADR — только если операция меняет границу ответственности gateway (состояние, входящие вызовы), обычное добавление ADR не требует; (4) при первом расширении реестр версионируется в `gc-unified-op-registry-v1.md`; обновляются локальные OpenAPI-снимки (при расхождении), запись каталога, файл роута, схема `args`, юниты; (5) новая операция рождается `disabled` — включение только по правилу §5.6 (зелёный сценарий сьюита → `beta`/`enabled`).
- **`availability` — операционный рычаг**, не версия: выключение операции не ломает контракт (клиент обязан обрабатывать 503/501 по кодам).
- **`catalogSchemaVersion`** двигается при изменении формы каталога; deprecation-поля добавляются в `operations[]` как расширение.
- Контрактные изменения manual выполняются синхронно с этой спекой (§0: сначала spec, затем manual, затем код).

## 9. Тестирование и приёмка

**Модель исполнения — штатная система тестов Chatium (решено 22-07-2026;** `020-testing.md`, образец — брокер ADR-0015): единый реестр тестов, `./web/tests` — страница для человека, `./web/tests/ai` — JSON-выдача; **«зелёный» = машинный факт `summary.success = true`** — именно этим запросом проверяется приёмка волн (§9.3), гейт повторяем (curl/расписание/CI). Дизайн тестов наследует [`gateway-testing-strategy.md`](../gateway/gateway-testing-strategy.md) как норматив — фазы, цепочки, throttle, teardown, префлайт сохраняются; меняется только каркас исполнения и репортинга.

**Интеграционный сьюит `/v1/{op}` в машинный автопрогон не входит** — он бьёт по живой школе GetCourse; повторяемый гейт не имеет права спамить школу. Он живёт отдельной категорией реестра с ручным запуском (и уважает 1 rps); в автопрогон входят юниты и локальная интеграция без внешних вызовов. Состояния префлайта (`ready` / `blocked-availability` / `warn-heap` / `warn-deps`) воспроизводятся на штатной странице.

### 9.1. Юниты

Без сети и без Heap тестовой школы: сборка исходящего HTTP по `(op, contour)` (путь, метод, Bearer vs форма `key`), сериализация `params` (base64-кодек), валидация `args` до сети, коды ошибок и конверт, целостность каталога (каждая запись ↔ файл роута; у каждой записи схема).

**Bearer-склейка — прицельный кейс (подтверждён 22-07-2026):** точное строковое равенство собранного `Authorization: Bearer {dev}_{school}` (один пробел после `Bearer`, без пробелов вокруг `_`) + краевые случаи: секреты с пробелами по краям (trim до склейки), подчёркивания внутри самих ключей. Ошибка склейки ломает авторизацию всех new-операций разом — косвенных проверок заголовков недостаточно.

**Классификация ответов — полный набор (объём зафиксирован 22-07-2026):** по юниту на **каждое** из 7 правил L1–L4 / N1–N3 в позитиве (фикстура срабатывания → `INVOKE_GC_SEMANTIC_ERROR` 502 + верное значение `error.details.gcRule`) **и** негативе (чистый ответ того же контура не срабатывает) + отдельный кейс нормализации булева Legacy (`true`/`1`/`"yes"` против `"0"`/`"false"`/`"no"`, с регистрами). Табличные кейсы на фикстурах JSON; чек-лист покрытия — таблица зависимостей стратегии §3.4.

### 9.2. Интеграционный сьюит `/v1/{op}`

Реальные вызовы на **тестовой школе** (уровень A из Heap). Фазовая модель стратегии §3: фаза 1 — операции на ручных ключах (`gc_itest_*`); фаза 2 — производители идентификаторов (`createDeal` → `dealId`, `addUser` → `userId`, каталоги); фаза 3 — потребители из контекста прогона (память процесса, не Heap); фаза 4 — деструкторы в `finally` (сделка → `updateDealFields status:"false"`, группы → `removeUserGroups`, …). Правила: одна сделка на сьюит; email всегда `tester@khudoley.pro` (хардкод); артефакты в школе — с префиксом `[gateway-itest]`; ≥1000 мс между исходящими (1 rps); запуск интеграции только для `enabled`/`beta` — для `disabled`/`unsupported` запуск запрещён при любой конфигурации. Префлайт без сети считает готовность каждого `op` (`ready` / `blocked-availability` / `warn-heap` / `warn-deps`).

**«Полный сьюит зелёный» (определение зафиксировано 22-07-2026):** для **каждой** операции в статусе `enabled`/`beta` на момент прогона её сценарий зелёный; teardown отработал; префлайт включённых операций чистый; 1 rps соблюдён. Набор растёт вместе с включением операций (§5.6). «Все 59 зелёные» гейтом не являются: операции без воспроизводимых фикстур (`dialogId`, Telegram `chatId`, `templateId` диплома и т.п.) остаются `disabled`/`unsupported` осознанно — включённое покрыто, невключённое не имитируется.

### 9.3. Приёмка по волнам

#### 9.5.1. Волна 1 — Каркас

Все решения гейта волны 1 записаны в спеку (пункты §0.2 удалены), каркас проекта поднят, главная страница открывается, типы/стиль зелёные. Боевых записей в таблицах нет.
