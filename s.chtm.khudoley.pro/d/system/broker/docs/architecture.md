# Architecture

Обзор архитектуры брокера. Источник истины по контрактам и бизнес-правилам — [`docs/spec/spec.md`](spec/spec.md) (карта разделов — [`docs/spec/spec-map.md`](spec/spec-map.md)); при расхождении приоритет у спеки. Данный документ — карта слоёв и потоков, без дублирования контрактов.

## Назначение

Глобальный брокер событий уровня аккаунта: модули публикуют события по типам, брокер материализует их в доставки подписчикам и раздаёт через pull-модель с жизненным циклом claim/ack/dead. Волна 2 (MVP) реализована полностью — см. [`README.md`](../README.md) («Текущее состояние»).

| **Web / Pages / Shared** | `web/`, `pages/`, `shared/` | `web/tests/` — страница тестов (интерактивная + JSON-гейт `ai.tsx`); `pages/tests/UnitTestsPage.vue` — Vue-страница по образцу `new_project` (импорт `// @shared-route`-роутов + `.run(ctx, …)`); `shared/tests/test-definitions.ts` — реестр 39 тестов, 6 категорий.     |

**Слой `repos/` в проекте отсутствует** — осознанное упрощение MVP, `lib/` обращается к `tables/` напрямую; инвариант «одна активная таблица на сущность» держит экспорт-селектор внутри `*.table.ts`, а не отдельный репозиторный слой (см. «Ключевые решения» ниже).

Поток вызова: HTTP/`.run()` → `api/` → `lib/broker/*` → `tables/*.table.ts` (Heap, окружение выбрано статически).

## Поток данных: публикация события

Фазы согласно spec §5.8:

1. **Publish (фаза 1)** — `lib/broker/publish.ts`: аутентификация модуля → проверка активности → запрет публикации в зарезервированный префикс `broker.*` (кроме служебного аудита) → валидация паттернов → потолок размера payload (8 КБ) → дедупликация по `idempotencyKey` под замком `lockKey('dedup', …)` → запись строки в журнал `BrokerEvents` с `dispatchedAt: null`.
2. **Журнал** — `BrokerEvents` (`tables/events.table.ts`) хранит неразобранные события; хвост publish (после `create`) в `try/catch` планирует асинхронный разбор и пишет лог, не блокируя ответ модулю.
3. **Asap-триггер** — `scheduleJobAsap` немедленно ставит в очередь `jobs/fanout-drainer.ts` вместо ожидания расписания.
4. **Дренер (фаза 2)** — `jobs/fanout-drainer.ts` под `tryRunWithExclusiveLock(lockKey('drainer'), 250, …)` вызывает `runFanoutPass` (`lib/broker/fanout.ts`): скан событий с `dispatchedAt: null` батчами по 200, резолв подписчиков по `allowedSubscribeTypes` (`$includes`/`$any`, с warn-логируемым фолбэком на полный скан при недоступности индексного пути), материализация доставок под per-event замком `lockKey('fanout', id)` с повторной проверкой активности подписчика внутри замка. Идемпотентно по паре `(eventId, subscriberModuleKey)`. `dispatchedAt` проставляется событию только после обработки всех его подписчиков; ошибка по одному событию не блокирует остальные (per-event изоляция).
5. **Доставки** — материализуются в `BrokerDeliveries` (`tables/deliveries.table.ts`) со статусом `pending`.
6. **Pull (CAS)** — `lib/broker/pull.ts`, `fetchDeliveriesCore`: клэмп лимита (`1..200`, дефолт 50), единый cutoff, выборка составным условием `$and`/`$or`, застолбление через CAS `updateAll` с проверкой `won === 1`, инкремент `claimCount`, выдача `claimToken`.
7. **Ack/Dead** — `ackDeliveryCore`/`deadDeliveryCore`: CAS-first обновление статуса; при `won === 0` — `retryRead` с предикатом «противоречие проигрышу CAS» для классификации причины (гонка / already-acked / already-dead / stale claimToken), с warn-логом на устаревший алиас.

Push-триггер подписчикам в волне 2 не реализован (debug-лог, заглушка — см. «Остаточные риски» в README, работы волны 3).

Связанные ADR: [0004](ADR/0004-transport-pull-driven-push-trigger.md) (транспорт pull-driven с push-триггером), [0005](ADR/0005-two-tier-event-storage.md) (двухуровневое хранение событий), [0006](ADR/0006-fat-delivery-payload-snapshot.md) (fat-payload доставки), [0009](ADR/0009-operational-audit-events.md) (операционные аудит-события), [0011](ADR/0011-event-history-hybrid-heap-clickhouse.md) (гибрид Heap+ClickHouse для истории), [0012](ADR/0012-fetch-deliveries-atomic-poll-claim.md) (атомарный poll-claim), [0014](ADR/0014-claim-race-per-row-cas.md) (per-row CAS для гонки claim).

## Паттерн окружений

Каждая из 5 сущностей брокера (`modules`, `events`, `deliveries`, `events-archive`, `settings`) хранится в паре Heap-таблиц stage/prod с общим объектом полей; наружу экспортируется только репозиторий, выбранный статическим селектором `IS_PROD` (вычисляется из `PROJECT_ROOT` в `config/env.ts`). Тот же принцип применён к служебным именам:

- **`LOCK_NS`** — `'broker'` (prod) / `'broker-stage'` (stage), подставляется в первый элемент ключа замка.
- **`lockKey(purpose, ...parts)`** — единственная точка построения ключей замков, форма `['<LOCK_NS>-<purpose>', ...parts]`. Критично: первый элемент ключа должен быть уникален **на назначение** замка (`register`, `dedup`, `fanout`, `drainer`), а не общим для всех замков окружения — иначе при конкуренции захват деградирует до ~5-секундных поллов (см. «Ключевые решения» ниже).
- **`LOG_SOCKET_CHANNEL`** — свой канал live-логов на окружение, чтобы stage и prod не пересекались в веб-сокет-мониторе.
- **`WORKSPACE_PATH`** — используется фильтром `readLogs`/`readLogsByMark` (`lib/log/read-logs.ts`) для выборки логов конкретного окружения из ClickHouse.

Текущий каталог (`s.chtm.khudoley.pro/d/system/broker`) — окружение stage; при переносе в `p/system/broker` тот же код без изменений адресуется к `_prod`-таблицам и `_prod`-именам через тот же селектор.

## Ключевые решения и уроки RV

1. **Форма ключей замков.** Первый элемент ключа замка обязан быть уникален на назначение (`'<LOCK_NS>-<purpose>'`) — общий первый элемент под конкуренцией даёт деградацию захвата до ~5-секундных поллов (наблюдение Runtime Verification). Зафиксировано в spec §5.8 и в платформенном `inner/docs/028-sync.md`. `lockKey()` в `config/env.ts` — единственная точка построения ключей, других мест их сборки в коде быть не должно.
2. **Цепочку дренера ведёт только держатель замка.** Перепланирование следующего прохода `jobs/fanout-drainer.ts` происходит внутри замка, до тяжёлой части обработки; при неудачном захвате (skip) джоб завершается быстро и без постановки преемника. Нарушение этого порядка (перепланирование вне замка или на skip) даёт шторм параллельных джобов.
3. **Слой `repos/` отсутствует осознанно.** Для MVP `lib/` обращается к `tables/` напрямую; инвариант «одна активная таблица окружения на сущность» держит экспорт-селектор внутри `*.table.ts`. Это отклонение от типового 4-слойного разделения (`tables/ → repos/ → lib/ → api/`, см. `template_project`), обоснованное объёмом волны 2 — при усложнении в волне 3 слой можно ввести отдельным рефакторингом.
4. **Heap.Enum — объектная форма.** Поля `source`, `status` в `BrokerModules` и `status` в `BrokerDeliveries` заданы как объектная форма `Heap.Enum` — отклонение от массивной формы, заложенной в исходном плане; выбор типо-корректен и не меняет контракт спеки.
5. **403 на отказ аутентификации подтверждён Runtime Verification.** User-thrown `AccessDeniedError` (`lib/broker/auth.ts`) на проводе даёт HTTP 403 — единственное исключение из общего конверта `BrokerResult`/HTTP 200 (см. spec §5.9.1, §5.9.5). Подтверждено зелёным тестом `auth_wrong_token_403` с проверкой и статуса, и тела ответа.
6. **`WORKSPACE_PATH` stage подтверждён равным `'d/system/broker'`** тестом `log_two_phase` — важно для корректной фильтрации `readLogs` по окружению (spec §5.10.7).
7. **Бюджет полного прогона тестов** — `web/tests/ai` укладывается в ~9–10 с при лимите 20 с; параметр `?category=` — запасной ход для более узкого прогона, если полный набор вырастет и подойдёт к границе бюджета.
8. **Двухфазный лог-тест.** `log_two_phase` (через `api/tests/log-probe.ts`) пишет пробу в начале прогона (`fireLogProbe`, гейт одноразовым `probeKey` в `BrokerSettings`) и читает её в конце — интервал между фазами равен длительности всего прогона и служит окном ожидания видимости записи в ClickHouse. `finally` независимо восстанавливает исходный уровень логирования и удаляет `probeKey` вне зависимости от исхода теста. См. [ADR-0015](ADR/0015-testing-strategy-native-row-isolation.md).

## Остаточные риски

- **Клаттер от упавших транспортных register-тестов.** Тестовые модули с ключом `brokertest-*` удаляются в `try/finally` (`cleanupTransportModule`), но при убитом процессе (не штатное падение теста) уборка не выполняется — остаётся мусорная строка в `BrokerModules`. Не конфликт с продовыми данными (grep-префикс `test-`/`brokertest-` зарезервирован, см. spec §5.2), но требует периодической ручной чистки.
- **Параллельные прогоны `run-all` взаимно разрушительны.** Набор тестов делит общий sweep тестовых строк, общий `probeKey` лог-теста и общий `log_level` — правило «один прогон за раз» (см. README, «Тесты») архитектурное, не организационное.
- **Частичный отказ на публикации системного аудит-события.** Если `publishSystemEvent` (`broker.*`, см. [ADR-0009](ADR/0009-operational-audit-events.md)) не срабатывает после успешного создания модуля в `register.ts`, модуль зарегистрирован, но аудит-событие может отсутствовать — токен возвращается вызывающему в любом случае (приоритет отдан доставке токена, не полноте аудита).
- **`readLogs` (полная версия, `lib/log/read-logs.ts`) в волне 2 не вызывается ниоткуда** — потребитель появится в админке волны 3. Формат интерполяции временных меток (ts64) требует проверки при первом реальном подключении.
- **Накопление stage-данных.** Лимиты аккаунта общие для stage и prod (см. «Паттерн окружений» выше); retention-джоб для `BrokerDeliveries`/`BrokerEvents` запланирован на волну 3 (spec §3.5) — до тех пор чистка stage периодическая и ручная.
