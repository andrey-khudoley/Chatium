# ADR-0011: Гибрид хранения истории событий — Heap-якорь + ClickHouse-зеркало

Дата: 2026-07-19

## Контекст

Хранение неизменяемого журнала `BrokerEvents` (§3.2) строилось как многоуровневое ([ADR-0005](0005-two-tier-event-storage.md)): горячая `BrokerEvents` → инлайновый архив `BrokerEventsArchive`, а поверх них проектировалась выгрузка старых партий архива в файлы `ctx.storage`. На это давят два фактора:

1. **Выгрузка в файлы — самая сложная и хрупкая часть модуля**: оркестрация, семплинг границ диапазонов, десяток параллельных джобов, файлы в десятки мегабайт, store-then-delete под лимит 60 с на джоб. При этом её порог срабатывания (сотни тысяч строк архива ≈ десятки млн событий) для личной BPM **практически недостижим — в норме no-op**. Основную сложность несёт предохранитель, который почти никогда не срабатывает.
2. **Нужна аналитика брокера** (throughput по `eventType`, объём fan-out, доля dead-letter, распределение лага «публикация → доставка», тренды продюсеров). Heap этого нативно не даёт: агрегаты по истории — это скан архива/файлов в коде, неудобно и медленно.

Отдельно рассматривался вопрос: **не сделать ли ClickHouse основным хранилищем событий** (журнал в ClickHouse, Heap — только под доставки). Отвергнут — ClickHouse в Chatium (`inner/docs/049-clickhouse.md`) не подходит на роль источника истины OLTP-журнала:

- **нет UPDATE** — поле `dispatchedAt` (§3.2, разовый `null → timestamp`; durable work-queue fan-out) поставить нельзя;
- **нет локов и атомарного «проверить → вставить»** — дедуп события `(producerModuleKey, idempotencyKey)` под `runWithExclusiveLock` (§5.8) невозможен;
- **запись best-effort/эвентуальна, без подтверждения коммита** (`writeWorkspaceEvent`) — нельзя строить at-least-once на «событие durably в журнале после Фазы 1»;
- **нет дедупа на записи** — повторный проход fan-out (recovery) задублировал бы событие в журнале;
- фиксированная схема, только `SELECT` через `queryAi`, лимит ответа ~30 КБ.

Пункты про `dispatchedAt` и синхронный fan-out означают: событие **обязано** лежать в Heap как минимум до завершения fan-out. Значит ClickHouse физически может владеть только историей, но не горячей фазой.

## Решение

**Гибрид: Heap остаётся источником истины истории, ClickHouse добавляется вторым best-effort зеркалом для аналитики; выгрузка архива в файлы не строится.**

1. **Heap — якорь durability (источник истины).** Оба уровня из [ADR-0005](0005-two-tier-event-storage.md) **сохраняются без изменений**: горячая `BrokerEvents` (§3.2) + инлайновый архив `BrokerEventsArchive` (§3.4, строки ~8 КБ). Запись подтверждённая; на них держатся at-least-once, дедуп, `dispatchedAt`, а корректностно-важные операции (ре-материализация нового подписчика, реплей, аудит с гарантией) читают **отсюда** — `payload` сохранён точно (`any`).

2. **ClickHouse — вторичное best-effort зеркало.** Fan-out-дренер (§5.8), уже держащий событие в руке, **дополнительно** зеркалит его в ClickHouse (`writeWorkspaceEvent`/`writeMetricEvent`) одним best-effort вызовом. Используется **только** для дашбордов/метрик, ad-hoc-аудита и удобного (не гарантирующего корректность) реплея. **Никогда не источник истины.** Может расходиться с Heap (потеря/дубль) — это допустимо by design: точные числа берут из Heap, для CH — дедуп по `eventId` на чтении. `payload` пишется одним непрозрачным JSON-полем (для лосслесс-реплея из CH; агрегация по полям `payload` — вне задач зеркала).

3. **Выгрузка архива в файлы не строится.** Инлайновый архив один держит порядка десятков млн событий, то есть на масштабе FLOW лимит таблицы недостижим. Механизм и его дизайн из спецификации **удалены**, чтобы не тащить в неё конструкцию, которая не реализуется. Если архив когда-нибудь начнёт подходить к лимиту, вынос за пределы Heap проектируется заново — по обстоятельствам того времени, а не по сегодняшним догадкам (пункт в §0.2).

**Почему гибрид безопасен там, где «ClickHouse как основное хранилище» — нет:** мы **не удаляем** событие из Heap, полагаясь на ClickHouse. Якорная durable-копия всегда в Heap (горячая → инлайновый архив, подтверждённые записи). CH-зеркало best-effort — его потеря/дубль на корректность брокера не влияют, потому что «правда» в Heap. Именно неподтверждённость записи в ClickHouse (нет сигнала коммита из workspace) и запрещает делать его единственной копией.

## Последствия

Плюсы:

- **уходит самый сложный код**: выгрузка архива в файлы вместе с её оркестрацией и третьим проходом retention — не пишется;
- retention упрощается до двух проходов (доставки + архивация в инлайн);
- **аналитика/дашборды брокера — бесплатно** (события уже зеркалятся в ClickHouse, читаются через `queryAi`);
- durability истории остаётся **аудиторского** качества (якорь — подтверждённые Heap-записи, store-then-delete в архивации);
- расхождение/потеря CH-зеркала безвредны — оно не источник истины.

Минусы / границы:

- **две копии** истории (Heap-архив + CH-зеркало) — небольшое дублирование хранения; роли строго разделены (одна для правды, вторая для отчётов);
- CH-зеркало **может дрейфовать** (best-effort запись + повтор дренера → дубли): для точных чисел — эталон в Heap, для CH — дедуп по `eventId` (`argMax`/`GROUP BY`) на чтении;
- **+1 best-effort вызов** записи в CH на событие внутри дренера (async, вне горячего пути `publishEvent`);
- `payload` в CH — непрозрачный JSON: агрегировать по его полям без `JSON`-extract нельзя (для FLOW реплей и так идёт из Heap, так что это не критично);
- **у роста архива нет заранее готового ответа**: если объём когда-нибудь превысит ёмкость инлайнового архива, механизм придётся проектировать с нуля. Принято сознательно — заранее спроектированный запас устареет раньше, чем понадобится;
- корректностно-важный реплей по-прежнему читает из Heap, не из CH.
