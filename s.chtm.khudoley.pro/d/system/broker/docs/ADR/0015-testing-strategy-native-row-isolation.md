# ADR-0015: Стратегия тестирования — штатная система Chatium, изоляция строками

Дата: 2026-07-22

## Контекст

Спека закрывается волнами (§0.1), и каждая волна обязана доказать, что её механизмы работают в Chatium как задумано, не роняя уже проверенное. Для этого нужен **повторяемый** способ проверки — «зелёный/красный» как машинный факт, а не ручной прогон с чтением логов на глаз.

Первое предположение — «в Chatium нет npm, значит нет и тест-раннера, обвязку придётся писать самим» — **оказалось неверным**. Платформа несёт штатную систему тестов (`inner/docs/020-testing.md`): реестр категорий как единый источник, две страницы из него (`./web/tests` для человека, `./web/tests/ai` — JSON для агента/CI), обязательные категории (БД/API/функциональные/интеграционные), планка покрытия и правило «обновляй тесты при каждом изменении».

Три обстоятельства брокера сужают выбор способа изоляции тестовых данных:

1. **Таблицы брокера — с фиксированными идентификаторами** (§3): пары `t__broker__modules__stage_wI7S9L` / `…__prod_wI7S9L` и прочие; пару выбирает окружение копии (селектор по пути, §3). Они боевые и общие на аккаунт.
2. **Heap не поддерживает динамические имена таблиц** — нельзя завести таблицу на прогон с вычисленным именем.
3. **Брокер конкурентен** (CAS-захват, идемпотентный fan-out — [ADR-0014](0014-claim-race-per-row-cas.md)/[ADR-0012](0012-fetch-deliveries-atomic-poll-claim.md)) — изоляция должна допускать **реальную** параллельность в тестах, иначе главный класс багов непроверяем.

## Решение

**Тестируем на штатной системе, изолируем строками, регрессию ведём волнами.** Три части:

1. **Штатная система, не самодельная.** Набор описывается один раз в реестре и отдаётся `./web/tests` (+ `/ai`); «зелёный» = `./web/tests/ai` вернул `summary.success = true`. Живёт в проекте (`web/tests`, `api/tests/*` от корня копии), версионируется с кодом. К четырём обязательным категориям добавлены две брокерские — **конкурентность** и **лимиты** (§9.3).
2. **Изоляция — неймспейс строк на боевых таблицах, а не отдельные таблицы.** Тест регистрирует модули `test-<uniq>`; всё под ними неймспейсится через `producerModuleKey`/`subscriberModuleKey`, чистится свипом по префиксу. Тесты бьют по тем же фиксированным §3-таблицам, что и гейтвей их окружения (пара stage/prod выбирается селектором по пути, §3), — то есть проверяют боевой путь, а не его копию.
3. **Регрессия привязана к волнам.** Волна N сдана, когда её приёмка (§9.5.N) зелёная И все прежние тесты зелёные. Набор только растёт; рождается в волне 2.

Обоснование:

1. **Не изобретать то, что есть.** Штатная система даёт повторяемый гейт (curl/CI) из коробки; своя обвязка дублировала бы её без выигрыша и разошлась бы с платформенными соглашениями.
2. **Проверяем боевой код.** Неймспейс строк оставляет путь исполнения тем же, что в проде: тот же гейтвей, тот же дренер, те же таблицы. Отдельные тест-таблицы потребовали бы параметризовать код набором таблиц — и тогда тест проверял бы вариант, а не прод.
3. **Динамические таблицы невозможны** (Heap не умеет вычисляемые имена) — значит изоляция обязана быть на уровне строк, других вариантов нет.
4. **Конкурентность тестируема.** Реальная параллельность достигается независимыми HTTP-запросами (`Promise.all`), как в прогоне [ADR-0014](0014-claim-race-per-row-cas.md) — неймспейс строк этому не мешает.

Отвергнутые альтернативы:

- **Отдельные статические тест-таблицы** (`BrokerModulesTest` и т.д.) + параметризация гейтвея набором таблиц. Работает, но тест перестаёт проверять боевой путь: код исполняется в тест-режиме с другой конфигурацией. Инвазивно и снижает ценность теста.
- **Динамические таблицы на прогон** — невозможно (см. Контекст, п.2).
- **Ручные приёмочные прогоны** (агент гоняет сценарий, человек читает логи) — не регрессия: не дают «зелёного» как факта и не ловят откаты в прежних волнах.
- **Своя обвязка с мок-`ctx`** — противоречит `020-testing.md` («реальный `ctx`, реальный код»); мок разошёлся бы с платформой.

## Последствия

Плюсы:

- повторяемый машинный гейт (`summary.success`) — годится для CI и мониторинга;
- тесты проверяют боевой код и боевые таблицы, а не их копии;
- изоляция дешёвая (неймспейс + свип), конкурентность проверяема реальной параллельностью;
- регрессия встроена в определение сдачи волны.

Минусы / границы:

- **тест-строки лежат в живых таблицах** — их видит настоящий fan-out-дренер (это и нужно для достоверности) и, с волны 3, retention-джоб;
- **`createdAt` строки задать нельзя** (§3) — «состаривание» для теста retention не подделать; этому тесту нужна другая тактика (тест-параметр отсечки / проверка формы запроса, §9.5.3);
- **read-back после `updateAll`/`update` может отставать** (`020-testing.md`) — проверки «застолбили → перечитали» оборачиваются retry (до 3 попыток), тогда как `create` читается сразу;
- набор нельзя сокращать ради скорости — удаление теста допустимо только вместе с удалением проверяемого.
