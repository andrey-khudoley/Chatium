# ADR-0006: «Толстая» доставка — снимок события в строке `BrokerDeliveries`

Дата: 2026-07-02

## Контекст

Чтобы обработать доставку (§3.3), модулю нужны данные события — как минимум `eventType` и `payload`. Строка доставки может быть:

- **тонкой** — только `eventId` + статус; данные модуль дочитывает из общего журнала `BrokerEvents` по `eventId`;
- **толстой** — при материализации (fan-out) в строку кладётся **снимок события** (`eventType` + `schemaVersion` + `payload`), и модуль обрабатывает её целиком из своей таблицы.

Тонкий вариант порождает проблему «сильно отставшего подписчика»: событие живёт в горячем журнале ~месяц, а `pending`/`claimed`-доставка подписчика, молчавшего дольше, переживёт архивацию события — и `findById` по `eventId` в горячей таблице промахнётся. Это тянет за собой кросс-уровневый lookup в архив ([ADR-0005](0005-two-tier-event-storage.md)), денормализацию времени события на доставку и границу гарантии «at-least-once только пока событие горячее».

Оценка объёмов FLOW: модулей немного, payload компактный (≤ ~8 КБ, §3.2), доставки живут коротко и чистятся retention (§3.5).

## Решение

**Толстая доставка.** При fan-out в каждую строку `BrokerDeliveries` копируется снимок события (`eventType` + `schemaVersion` + `payload`). Модуль обрабатывает доставку **целиком из своей таблицы доставок и к общей `BrokerEvents` на рантайме не обращается**. Журнал и его холодный архив становятся **чистой историей** — читаются только диагностикой/админкой (аудит, ре-материализация, ручной разбор), не на пути обработки.

Обоснование:

1. **Меньше запросов и точек отказа.** Модуль работает из одной своей таблицы; нет обращения к общей `BrokerEvents` на каждый обрабатываемый элемент.
2. **Проблема отставшего подписчика исчезает.** Payload едет вместе с доставкой — сколько бы подписчик ни молчал, его `pending`-строка самодостаточна. Не нужны ни кросс-уровневый lookup в архив, ни денормализация `eventCreatedAt`, ни граница «гарантия только в горячем окне».
3. **Чистое разделение ответственности.** Доставки — рабочий поток; журнал и архив — неизменяемая история. Пути не пересекаются.
4. **Порядок без доп. поля.** Best-effort порядок берётся по `createdAt` самой доставки (≈ время публикации, fan-out идёт при публикации) — денормализованное время события не нужно.

Отвергнут тонкий вариант: экономит место в доставках, но добавляет запросы к общей таблице, точку отказа и кросс-уровневый lookup — без выигрыша на объёмах FLOW.

## Последствия

Плюсы:

- самодостаточная обработка (одна таблица, без дочтения журнала);
- нет проблемы отставшего подписчика и кросс-уровневого lookup;
- at-least-once **не** ограничен горячим окном (payload не теряется при отставании);
- порядок — по `createdAt` доставки, без денормализации времени события.

Минусы / границы:

- **дублирование payload** — по копии на подписчика; но временное (доставки чистятся retention: `acked` 24ч, `dead` 30д) и на малых объёмах FLOW некритично;
- строка доставки несёт payload, поэтому подпадает под тот же оптимум записи ≤ ~8 КБ — крупный контент кладётся ссылкой на файловое хранилище, не инлайном (§3.2);
- `eventId` в доставке остаётся, но лишь как обратная ссылка на журнал для трассировки/идемпотентности, не для дозагрузки данных.
