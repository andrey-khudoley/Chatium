# ADR-0007: Выгрузка холодного архива в файлы под платформенные лимиты (tier 3)

Дата: 2026-07-02

> **Отложено [ADR-0011](0011-event-history-hybrid-heap-clickhouse.md) (2026-07-19).** Для масштаба FLOW инлайновый архив (tier 2) достаточен, а порог выгрузки (700k строк) недостижим — поэтому этот механизм **не реализуется сейчас**. Дизайн ниже сохранён как **описанный запас**: достраивается, только если архив приблизится к лимиту 1 млн строк.

## Контекст

Инлайновый архив `BrokerEventsArchive` (tier 2, [ADR-0005](0005-three-tier-event-storage.md)) — строки ~8 КБ, данные в самой таблице. Сама таблица ограничена платформой (`inner/docs/008-heap.md`, `005-jobs.md`, `002-routing.md`):

- **1 000 000 строк** на таблицу;
- **1000 строк** на один запрос `findAll` (поддерживает `offset`);
- **60 секунд** на запуск джоба, **20 секунд** на вызов эндпоинта.

Нужно выгружать старые строки в файлы **до** переполнения, но не «в лоб»: (а) нельзя выгружать только что заархивированное — иначе данные попадают в архив и тут же уезжают; (б) один джоб не успеет перелопатить сотни тысяч строк за 60 с и не удержит в памяти гигантский файл.

Прикидка: инлайновая строка ~8 КБ. «Файл на 50 000 строк» ≈ **400 МБ** — джоб держал бы в памяти ~0.8–1.2 ГБ (у `ctx.storage.uploadFile` аргумент `data` — целый блоб) и не уложился бы в 60 с. Значит размер **файла** надо развязать с размером **слайса джоба**.

## Решение

**Порог и «оперативный хвост».** Держим в архиве **200 000** новейших инлайновых строк как оперативные (запрашиваемые через `where`) — их не трогаем. Выгрузку запускаем при **700 000** строк всего (запас 300k до лимита 1 млн), выгружая **самые старые 500 000**.

**Параллелизм под 60 с.** Выгрузку делают **10 независимых джобов**, каждый — свой непересекающийся слайс ~50 000 строк. 50k строк на джоб по времени проходят (≈50 чтений + 50 удалений батчами по 1000 ≈ секунды), тяжёлое — только запись файлов.

**Слайсинг по диапазонам `batchFrom`, не по offset.** Чистый offset ломается при конкурентном удалении (удаление в слайсе 0 сдвигает offset слайса 5). Поэтому оркестратор (ночной retention-джоб) один раз, **в одном потоке**, считает 11 границ через offset-семплинг (`findAll limit 1 offset i·50000` по инлайновым строкам, `batchFrom asc`) и раздаёт джобам **диапазоны значений** `[from, to)`. Диапазоны стабильны под параллельным удалением.

**Файлы ~40 МБ (≈5000 строк), а не один на слайс.** Внутри своего диапазона джоб пишет несколько файлов по целевому размеру `FILE_TARGET` ~40 МБ (набор по размеру, как в tier 2, но уровнем выше). 40 МБ безопасны по памяти и времени; на всю выгрузку — ~100 файлов, каждый схлопывает ~5000 строк → 1 замороженную строку.

**Содержимое файла — плоский массив событий.** В инлайновых строках события лежат батчами (`events: any` = массив событий на строку). Перед записью батчи **разворачиваются и склеиваются** (`flatMap`), чтобы в файле был массив **событий**, а не массив батчей — иначе файл неудобно читать.

**Имя файла отражает период.** `broker-events_<дата первого события>_<дата последнего>.json` (из `batchFrom` первой и `batchTo` последней строки сегмента, в безопасном для имени виде) — так по имени видно, за какой интервал данные.

**Без потерь при сбое.** На каждый файл — store-then-delete: выгрузить файл → записать замороженную индексную строку (`fileId`, диапазон, суммарный `count`) → только потом удалить схлопнутые инлайновые строки.

## Последствия

Плюсы:

- выгрузка укладывается в платформенные лимиты (60 с × 10 джобов, файлы ~40 МБ, запросы ≤1000);
- оперативные 200k строк остаются запрашиваемыми; нет «архив→сразу выгрузка»;
- слайсинг по значениям безопасен под параллелизмом;
- файл самодостаточен (плоский массив событий) и самоописателен (период в имени);
- ёмкость растёт ещё на порядки (~5000 строк → 1 замороженная).

Минусы / границы:

- дальний предохранитель: 700k строк архива ≈ десятки млн событий, для личной BPM практически недостижимо;
- замороженные индексные строки тоже считаются в лимит 1 млн, но растут ничтожно (одна на ~5000 строк, каждая покрывает большой период);
- **проверить при реализации:** держит ли платформа файл ~40 МБ в памяти джоба и его выгрузку за 60 с (при риске — уменьшить `FILE_TARGET`); поведение `findAll { fileId: null }` (запрос по null — иначе дискриминатор `frozen`, [ADR-0005](0005-three-tier-event-storage.md)); санитизация `:` в имени файла;
- чтение замороженного сегмента (диагностика/админка) тянет весь файл — ещё одна причина держать файлы небольшими.
