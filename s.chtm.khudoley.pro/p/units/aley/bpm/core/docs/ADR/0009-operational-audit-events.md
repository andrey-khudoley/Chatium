# ADR-0009: Операционный аудит — системные `broker.*` события в журнале, без отдельной таблицы

Дата: 2026-07-07

## Контекст

`BrokerModules` хранит только **текущее** состояние строки (§3.1). Историю операций — регистрация, смена состояния (`status`, ADR-0010), решения модерации, изменения whitelist, удаление — на строке не удержать. В §5.3–§5.6 эти факты описаны как «событие брокера». Встал вопрос: складывать их в **отдельную операционную таблицу** (`BrokerOpsAudit`) или в **общий журнал** `BrokerEvents` системными типами.

## Решение

**Операционные факты = системные события типов `broker.*` в том же журнале `BrokerEvents`. Отдельной таблицы нет.**

- Типы в зарезервированном namespace `broker.*`: `broker.module.registered`, `broker.module.status-changed`, `broker.module.moderated`, `broker.module.deleted`, `broker.publish-types.changed`, `broker.subscribe-types.changed` и т.п.
- Публикует их **сам брокер как системный продюсер**: внутренним кодом (не модульным publish-API), под зарезервированным `producerModuleKey = 'broker'` (сентинел, **не** запись в `BrokerModules`), без токена и **в обход `assertCanPublish`** — брокер источник полномочий.
- **Namespace `broker.` зарезервирован** (и `moduleKey = broker` тоже): модуль публиковать `broker.`-типы **не может** — отклоняется независимо от `allowedPublishTypes` (даже `*`), а явный `broker.*` в publish-вайтлисте отвергается валидацией (§5.2/§5.3). **Подписка** на `broker.*` модулю разрешена (аудит-фид).
- Идут **обычным путём публикации**: `seq`, `dispatchedAt`, трёхуровневый retention/архив (§3.4/§3.5). Отдельной машинерии не заводим.
- **Fan-out — единообразный, без спец-кейса.** Обычно подписчиков по `broker.*` нет → ноль доставок. Но модуль (напр. админ-дашборд статуса системы) может подписаться на `broker.module.*` и получать аудит-фид тем же механизмом **бесплатно**.

## Обоснование

1. **Консистентность хранения.** У брокера одна модель — «факт → событие → журнал». Его собственные факты обязаны идти тем же путём; отдельная таблица породила бы **второй event-store** с иными правилами.
2. **Единая машинерия.** Публикация, retention, архив переиспользуются — ничего нового.
3. **Наблюдаемость бесплатно.** Дашборд/подписчик получает ленту через тот же fan-out; никакого отдельного API для чтения аудита.

## Последствия

Плюсы:

- один источник истины и одна машинерия хранения;
- аудит-фид для дашборда — обычной подпиской на `broker.module.*`;
- инвариант «`BrokerEvents` = журнал фактов» распространяется и на операционную мету.

Минусы / границы:

- операционные события архивируются как доменные — но объём низкий (модулей единицы, изменения редки), лимит строк не грозит;
- запрос истории по конкретному модулю может уйти в холодные уровни — но это редкие админ-сценарии, не горячий путь; а признак, нужный модерации (первичка vs ре-модерация), лежит в `status` строки (ADR-0010), поэтому **скана журнала для §5.6 не требуется**.

Отвергнуто:

- **Отдельная таблица `BrokerOpsAudit`** — нарушает «факт → событие → журнал», заводит второй event-store; под низкий объём операционных фактов выигрыша нет, только рассинхрон правил.
