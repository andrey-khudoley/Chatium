# ADR-0010: Состояние модуля — единый `status`, вместо `enabled` / `adminDisabled`

Дата: 2026-07-07

## Контекст

Прежняя модель `BrokerModules` держала состояние двумя булевыми:

- `enabled` — участие модуля в обмене;
- `adminDisabled` — независимый админ-стоп с приоритетом над `enabled`.

Проблемы:

1. **`enabled` тянул две роли** — «допущен» и «приостановлен до одобрения»: §5.3/§5.4 ставили `enabled = false` при ре-модерации расширения whitelist. Из-за этого `enabled` был **немонотонным** (активный модуль снова становился `false`).
2. Как следствие, §5.6 не мог по `enabled` различить **первичную регистрацию** и **ре-модерацию** — обе стоят `enabled = false`. Приходилось выводить «был ли модуль активен» из истории событий.
3. Два ортогональных булевых давали **двусмысленные комбинации** (`enabled = false` + `adminDisabled = true` — какую причину показывать?).

## Решение

> ⚠️ **Состав значений ниже устарел.** Значение `reModeration` упразднено поправкой от 2026-07-19 (в конце документа) — актуальны три: `onModeration`, `active`, `disabled`. Сам принцип «единый `status` вместо пары булевых» в силе.

**Заменить `enabled` + `adminDisabled` одним полем `status` (enum) с взаимоисключающими значениями:**

| Значение | Подпись | Смысл |
|---|---|---|
| `onModeration` | «На модерации» | `external`, первая заявка, ждёт первичного одобрения (§5.2) |
| `active` | «Активен» | участвует в обмене |
| `reModeration` | «Повторная модерация» | был активен, расширил whitelist, ждёт повторного одобрения (§5.3/§5.4) |
| `disabled` | «Выключен» | админ-стоп |

Правила:

- **`disabled` применяется только из `active`** (админ гасит работающий модуль). `onModeration`/`reModeration` и `disabled` не совмещаются.
- `internal` при регистрации → сразу `active`; `external` → `onModeration`.
- «Активен сейчас» = `status == active`.
- Переходы фиксируются `broker.*` событиями ([ADR-0009](0009-operational-audit-events.md)); на строке — только текущее состояние.

## Обоснование

1. **Различие первичка / ре-модерация — прямо в статусе** (`onModeration` vs `reModeration`), без «ever active» и без обращения к истории.
2. **Очередь модерации** = `status ∈ { onModeration, reModeration }`, одним запросом.
3. **Один атомарный признак-состояние** вместо двух ортогональных булевых — исчезают двусмысленные комбинации, консистентнее под гонки (одно поле, а не два).
4. **Расширяемость** — добавить значение проще, чем третий булев.

## Последствия

Плюсы:

- §5.6 читает решение (`approve`/`decline`, delete-vs-keep) прямо из `status`;
- не нужен ни флаг `everActive`, ни скан журнала для модерации;
- одно поле честно описывает «в каком состоянии модуль и почему».

Минусы / границы:

- теряется ортогональность «админ-стоп ⟂ состояние модерации» — но она не нужна: приостановленный на модерации модуль и так не участвует, «выключать» его поверх бессмысленно (отсюда правило «`disabled` только из `active`»);
- полная хронология статусов — в `broker.*` журнале (ADR-0009), не на строке.

Отвергнуто:

- **Пара булевых `enabled`/`adminDisabled`** — немонотонный `enabled`, двусмысленные комбо, нужда в выводе «ever active» из истории.
- **Отдельный флаг `everActive`** — денормализация исторического факта на строку, против принципа «строка — текущее, история — в событиях».

Связано: [ADR-0009](0009-operational-audit-events.md), §3.1, §5.2–§5.6.

## Поправка от 2026-07-19: значение `reModeration` упразднено

Основное решение (единый `status` вместо пары булевых) **остаётся в силе**. Пересматривается только состав значений: их три — `onModeration`, `active`, `disabled`.

### Что не сработало

`reModeration` кодировал два независимых факта сразу: «модуль работает» и «у модуля висит заявка на расширение». Второй факт при этом отменял первый — §5.3/§5.4 писали запрошенный набор прямо в боевое поле, и модуль приходилось приостанавливать, иначе fan-out начал бы слать ему неодобренные типы.

Побочный эффект обнаружился при проработке гейта pull-операций (§5.9, О5): модуль, попросивший **дополнительные** права, терял **уже одобренные**. Его очередь замирала до решения админа, хотя доставки в ней были нарезаны законно. Наказание за просьбу, а не за нарушение.

Это ровно та болезнь, с которой боролась исходная редакция ADR: поле, которое вместе с состоянием кодирует что-то ещё. Там это был немонотонный `enabled`, здесь — статус, означающий одновременно операционное состояние и наличие заявки.

### Что вместо

Оси разведены. `status` описывает **только** операционное состояние — работает модуль или нет. Наличие заявки живёт в отдельных полях `pendingPublishTypes` / `pendingSubscribeTypes` (§3.1), которые читает исключительно админка модерации; ни publish-проверка, ни fan-out их не видят.

`onModeration` на оси статуса **остаётся** и конфликта не создаёт: новичок-`external` действительно не работает — он ни разу не одобрен, доставок ему не создавалось. Это честное операционное состояние, а не заявка, притворяющаяся состоянием.

Правило «`disabled` только из `active`» сохраняется без изменений.

### Что это даёт

- **`active` снова означает ровно «работает»** — без исключений и без перечисления двух значений в каждом запросе. Гейт pull (§5.9, О5) и резолв подписчиков на fan-out (§3.1) остаются однозначным `status = 'active'`.
- **Модуль с заявкой продолжает работать** на одобренных правах; новые вступают в силу после approve.
- **Отклонение заявки стало полным откатом**: `pending*` очищается, боевой набор не пострадал. В прежней редакции отката не существовало — запрошенный набор затирал одобренный, и `declineModeration` оставлял модуль приостановленным до подачи новой корректной заявки.

### Чем платим

- **Очередь модерации больше не берётся одним условием по статусу.** Теперь это `status = 'onModeration'` **или** непустой `pending*` — запрос с `$or` (форма — §5.9, О4). Пункт 2 исходного обоснования («очередь = `status ∈ {...}`, одним запросом») этой поправкой отменяется.
- **Две новые колонки** в `BrokerModules` вместо одного значения enum.
- Различение «первичная заявка / расширение» по-прежнему не требует истории (пункт 1 обоснования в силе): первичная — по статусу, расширение — по непустому `pending*`.

Затронутые разделы: §3.1 (поля и значения статуса), §5.3/§5.4 (куда пишется запрошенный набор), §5.6 (очередь, approve, decline), §5.9 О5 (гейт).
