# ADR-0014: Гонка двойного claim — построчный CAS на `updateAll`

Дата: 2026-07-20

Пересматривает более раннее решение — эксклюзивную блокировку по подписчику, действовавшую 19–20-07-2026.

## Контекст

`fetchDeliveries` обязана быть атомарной: выбрать доставки и тут же застолбить их ([ADR-0012](0012-fetch-deliveries-atomic-poll-claim.md)). Изнутри это чтение плюс запись, и в зазор между ними попадают параллельные заборы одного подписчика — два его воркера либо плановый pull, совпавший с pull по push-триггеру. Оба читают базу до первой записи, оба видят одну строку, оба её столбят. Итог — **одновременная** обработка одной доставки, при которой идемпотентность потребителя не спасает: обе копии проверяют «уже обработано?» в один момент и обе получают «нет».

Первоначально это закрывалось `runWithExclusiveLock` по `subscriberModuleKey`, а оптимистичный CAS был отвергнут с обоснованием «условного апдейта в Heap нет». **Обоснование было неверным.** `updateAll(ctx, { patch, where, limit }): Promise<number>` существует — он просто не описан в `inner/docs/008-heap.md` (документация не покрывает всю поверхность API, и вывод «в доках нет ⇒ в API нет» оказался ошибкой).

Рантайм-проверка 20-07-2026 установила недостающие факты:

- `updateAll` возвращает число **реально изменённых** строк; повторный вызов с уже не совпадающим условием даёт `0`;
- **операция атомарна**: 10 параллельных попыток захватить одну строку условием `{ id, status: 'pending' }` дают ровно одного победителя (вернул `1`) и девять проигравших (вернули `0`), записанное значение соответствует победителю. Проверено двумя способами — `Promise.all` внутри одного обработчика и 10 независимых HTTP-запросов; результат совпал;
- `patch` принимает только литеральные значения — выражения вида «текущее + 1» невыразимы;
- параметра `order` у `updateAll` нет;
- `limit` — предохранитель от массовой операции, а не размер партии (по аналогии с `deleteAll`, где это подтверждено прямо).

## Решение

**Построчный CAS, без блокировки на пути забора.** `fetchDeliveries`:

1. читает кандидатов — свои `pending` плюс просроченные `claimed`, порядок `createdAt asc`, лимит из запроса;
2. по каждому кандидату выполняет условное обновление, где условие повторяет прочитанное состояние:

```typescript
const won = await BrokerDeliveries.updateAll(ctx, {
  patch: {
    status: 'claimed',
    claimedAt: now,
    claimToken,                      // новая метка захвата (§3.3)
    claimCount: row.claimCount + 1,  // вычислено в коде: patch литеральный
  },
  where: { id: row.id, status: 'pending' },
  limit: 1,
})
// для просроченной claimed условие иное:
// { id: row.id, status: 'claimed', claimedAt: { $lt: cutoff } }
```

3. `won === 1` — строка наша, кладём в результат; `won === 0` — перехватили, пропускаем и берём следующую.

Замок на этом пути **не используется**. Блокировки §5.8 (дедуп по `idempotencyKey`, идемпотентный fan-out) остаются без изменений — там условие вешать не на что, строки ещё не существует.

Обоснование:

1. **Гонка закрыта по-настоящему.** Условие `where` включает состояние, которое мы прочитали. Если строку перехватили — у `pending` сменился статус, у просроченной `claimed` обновился `claimedAt`, — условие не сойдётся, вернётся `0`. Атомарность подтверждена прогоном, а не выведена из документации.
2. **Предохранитель не мешает.** `where` содержит `id`, то есть попадает максимум в одну строку; `limit: 1` не может оказаться меньше числа совпадений. Свойство следует из уникальности `id`, отдельной проверки не требует.
3. **Заборы перестают выстраиваться в очередь.** Главный выигрыш против варианта с замком: воркеры одного модуля забирают доставки параллельно и расходятся по непересекающимся строкам. Число записей в Heap то же самое — меняется только отсутствие сериализации.
4. **Снимается связанное ограничение.** В редакции с замком размер батча приходилось держать скромным, потому что всё время N записей замок удерживался. Теперь верхняя граница батча определяется только тем, успеет ли модуль обработать пачку за `claimTimeoutMs`.

Отвергнут **батчевый CAS** (один `updateAll` на всю пачку с общим `claimToken`, затем выборка по этому токену — всего два запроса вместо N+1) по двум независимым причинам: у `updateAll` нет `order`, поэтому «взять N самых старых» не выразить и best-effort порядок по `createdAt` (§1) сломается; и `limit` как предохранитель означает, что вызов с `limit: N` при большем числе подходящих строк не возьмёт N штук, а упадёт.

## Последствия

Плюсы:

- одновременная обработка одной доставки исключена, как и с замком, но без сериализации заборов;
- параллельные воркеры одного модуля масштабируются;
- на один примитив меньше на горячем пути (замок остаётся только там, где он незаменим — §5.8);
- размер батча больше не связан с временем удержания замка.

Минусы / границы:

- **воркер может получить меньше строк, чем просил** — проигранные гонки просто пропускаются. Это нормально и саморегулируется: проигравший получил меньше работы ровно потому, что её взял другой;
- **N+1 запросов на забор** (чтение кандидатов плюс по условному обновлению на каждого). Столько же, сколько было с замком, но теперь это нижняя граница: удешевить батчеванием нельзя из-за отсутствия `order`;
- CAS требует, чтобы условие точно повторяло прочитанное состояние. Для просроченных `claimed` это `claimedAt < cutoff` с тем же `cutoff`, что использовался при выборке — иначе окно проверки разъедется;
- гонка закрыта только для **одновременного** захвата. Последовательные повторы (перезабор просроченной, §3.1) остаются штатными, идемпотентность потребителя по-прежнему обязательна (§1).
