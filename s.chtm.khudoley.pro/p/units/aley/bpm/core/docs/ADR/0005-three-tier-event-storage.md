# ADR-0005: Трёхуровневое хранение событий (горячая → инлайновый архив → файлы)

Дата: 2026-07-02

## Контекст

`BrokerEvents` — неизменяемый append-only журнал (§3.2): события не редактируются и не удаляются. При этом действуют платформенные ограничения (`inner/docs/008-heap.md`):

- **1 000 000 строк** на Heap-таблицу (тариф «Бизнес»);
- **оптимальный размер записи ≤ ~8 КБ** (крупнее — заметно медленнее на Postgres); пара мегабайт терпима, но нежелательна;
- **большие блобы — в файловое хранилище**, не в Heap.

Растущий неограниченно журнал упрётся в лимит строк, а удалять его нельзя. Нужна многоуровневая схема, снижающая **число строк** (лимит именно на них) без потери данных.

Отвергнутые ходы:

- **Не делать ничего** → упор в 1 млн строк, потеря новых событий.
- **Одна строка = одно событие в архиве** → архив тоже 1 млн строк, выигрыша по ёмкости нет.
- **Пачка событий одной большой строкой (напр. по 1000 ≈ ~1 МБ)** → нарушает оптимум записи ≤ 8 КБ, «толстые» строки медленны, это и есть «блоб в Heap», от которого предостерегает поддержка.
- **Сразу в файлы** → теряется запрос по содержимому через `where`, а объёмы FLOW этого не требуют.

## Решение

**Три уровня, данные мигрируют, но не теряются:**

1. **Горячая `BrokerEvents`** — 1 строка = 1 событие, ~последний месяц.
2. **Инлайновый архив `BrokerEventsArchive`** — события старше месяца упаковываются в строки, набранные **по размеру ~8 КБ** (не по фиксированному числу и не «под максимум»); данные лежат JSON-полем в самой таблице, доступны через `where` (по диапазону `batchFrom`/`batchTo`).
3. **Замороженные строки (файлы)** — когда сам архив близок к 1 млн строк, партия старых инлайновых строк консолидируется в один файл `ctx.storage`, вместо партии остаётся одна индексная строка с `fileId`. Так `K` строк → 1.

Ключевые правила механики (детали алгоритма — §3.4/§3.5):

- **Размер, а не число.** Строку архива набираем до ~8 КБ (оптимум записи Heap), считая размер сложением `JSON.stringify(event).length`. Так строки остаются «здоровыми», а ёмкость растёт в ≈ `8 КБ / размер события` раз.
- **Только полные строки.** Строка сбрасывается, лишь когда следующее событие в неё не влезло; неполный «хвост» ждёт в горячей — архив не засоряется мелкими строками.
- **Store-then-delete.** Сначала пишем архив/файл, только потом удаляем исходные строки — при сбое худшее это дубль/осиротевший файл, не утрата.
- **Консолидация в файлы схлопывает строки** (`K → 1`), потому что лимит — на число строк; блобы уходят в файловое хранилище, как советует поддержка.

## Последствия

Плюсы:

- неограниченный по смыслу журнал в рамках платформенных лимитов;
- инлайновый уровень (tier 2) остаётся запрашиваемым через `where`;
- строки в «здоровом» диапазоне ≤ ~8 КБ (быстрый Postgres);
- ёмкость: tier 2 — порядка единиц-десятков млн событий, tier 3 — ещё на порядки;
- отказоустойчивость перекладки (store-then-delete).

Минусы / границы:

- три прохода retention-джоба вместо простого удаления (проход 3 — дальний предохранитель, в норме no-op);
- байт-точность размера не гарантирована: меряем символы `JSON.stringify().length` (`Buffer`/`TextEncoder` в UGC может не быть, см. `047-base64.md`), поэтому таргет берём с запасом под 8 КБ;
- выборка инлайновых строк для консолидации требует запроса «`fileId` не задан» — если Heap не поддерживает запрос по null, завести дискриминатор `frozen`;
- чтение tier 2/3 — только диагностика/админка (рантайм-доставки журнал не читают, [ADR-0006](0006-fat-delivery-payload-snapshot.md)), поэтому кросс-уровневый lookup по содержимому неоптимизирован намеренно.

Связано: одиночное событие само по себе — маленький факт (обычно сотни байт; ~8 КБ — потолок записи Heap, не цель; крупное — ссылкой в файлы, §3.2), а строка архива здесь набирает до ~8 КБ **пачкой** таких событий. Разный масштаб, общий принцип — держать строку Heap ≤ ~8 КБ. Неизменяемость журнала и каскад удаления модуля — §3.2/§5.5.
