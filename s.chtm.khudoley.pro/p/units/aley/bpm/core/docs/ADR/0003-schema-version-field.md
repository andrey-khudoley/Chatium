# ADR-0003: Версия схемы `payload` — отдельным полем `schemaVersion`, а не внутри payload

Дата: 2026-07-02

## Контекст

`payload` события (§3.2) — произвольный JSON (`any`), структуру которого задаёт контракт типа — соглашение между продюсером и подписчиками в их коде, брокер его не хранит и не валидирует (§1, §8). Со временем формат payload для одного и того же `eventType` может меняться несовместимо, поэтому нужна **версия схемы**. Встал вопрос: хранить версию отдельным полем строки или внутри самого `payload` (`payload.version`).

Дополнительный контекст платформы: `payload` объявлен `any`, а надёжный запрос Heap `where` **внутрь** JSON-поля по вложенному полю не подтверждён (есть фильтр по вложенному `Object` и `$includes` по массиву скаляров, но не матч элемента массива-объектов; см. §3.4). То есть значение, лежащее внутри `payload`, для выборки/маршрутизации фактически недоступно без вычитывания и парсинга строки.

Рассмотренные варианты:

| Вариант | Запрос по версии | Гарантия присутствия | Связь с контрактом |
|---|---|---|---|
| Отдельное поле `schemaVersion` | `where({ eventType, schemaVersion })` | Всегда (default `1`) | `(eventType, schemaVersion)` — идентификатор контракта, сам контракт вне брокера |
| Внутри `payload` (`payload.version`) | Только после парсинга; `where` не достаёт | Конвенция, можно забыть | Версия — часть контента, который она же и описывает |
| В типе (`tasks.created.v2`) | По типу | Всегда | Ломает glob-вайтлисты, смешивает роутинг с версией |

## Решение

Хранить версию схемы **отдельным полем `schemaVersion`** (целое, `1` по умолчанию) на строке `BrokerEvents`; при fan-out версия копируется в снимок `BrokerDeliveries` рядом с `eventType`/`payload` (§3.3). Идентификатор контракта — пара `(eventType, schemaVersion)`; сам контракт (структура `payload` по этой паре) брокер не хранит — это соглашение продюсера и подписчиков в их коде (§1, §8).

Обоснование:

1. **Версия — это конверт, а не факт.** Она описывает, *как читать* payload; держать её внутри читаемого — курица-яйцо (чтобы прочитать версию, надо уже знать формат). Метаданным место снаружи, рядом с `eventType`.
2. **Запрашиваемость.** Фильтрация и маршрутизация по версии — обычным `where`, без парсинга `any`-поля (которое Heap по содержимому и не запрашивает). Это же делает дешёвыми будущие миграции («перекодировать v1→v2» = `where`-фильтр, а не обход всех payload).
3. **Единообразие.** Поле есть всегда (default `1`), тулинг и потребители могут на него опираться; вариант «внутри payload» — необязательная конвенция.
4. **Версия ≠ новый тип.** Бамп `schemaVersion` не меняет `eventType`, поэтому вайтлисты (glob по типу) и модерация (§5.6) не затрагиваются. Версия — вторичный дискриминатор внутри типа. Вариант «версия в имени типа» отвергнут именно потому, что ломал бы glob-матч подписок.

## Последствия

Плюсы:

- версия доступна для `where`-фильтрации/маршрутизации без чтения `payload`;
- гарантированное присутствие и единообразие (default `1`);
- дешёвые пакетные миграции по версии;
- ортогональность вайтлистам и модерации — бамп версии не требует ре-модерации.

Минусы / границы:

- одно дополнительное поле в `BrokerEvents` и в снимке `BrokerDeliveries` (при fan-out копируется);
- продюсер обязан корректно проставлять и бампать `schemaVersion` при несовместимом изменении формата (ответственность продюсера/контракта);
- сам брокер версию **не интерпретирует** — совместимость и реакция на незнакомую (более новую) версию это политика подписчика (отложить/отклонить); контракт версии — соглашение продюсера и подписчиков вне брокера (§1, §8), не таблица брокера.
