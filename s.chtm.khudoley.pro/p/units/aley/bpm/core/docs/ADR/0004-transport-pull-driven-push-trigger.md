# ADR-0004: Транспорт — pull-driven с push-триггером (гибрид), ретраи на стороне модуля

Дата: 2026-07-02

## Контекст

Брокер должен доставлять опубликованные события подписчикам. Платформа Chatium serverless: сам по себе ничего не крутится — рантайм модуля кто-то должен инициировать. Способ инициации (push прямым вызовом / pull по расписанию / push-триггер + pull) ортогонален модели хранения прогресса. Рассмотрены три модели:

| Модель | Где состояние прогресса | Ретраи / dead-letter | Сложность брокера |
|---|---|---|---|
| **Очередь** (SQS/RabbitMQ) | строка на пару «событие × подписчик» | ведёт **брокер** (retry, backoff, visibility timeout, DLQ) | высокая |
| **Лог + курсор** (Kafka) | один курсор на подписку | подписчик двигает курсор; «ядовитое» событие стопорит | низкая, но head-of-line blocking |
| **Pull-driven + push-триггер** (выбрано) | строка на пару «событие × подписчик» (статус) | ведёт **модуль** (перезабор просроченных `claimed`) | низкая |

Дополнительный факт: у подписчика в любом случае будет своя pull-логика (забрать свои данные и запустить обработку). Логично переиспользовать её и для доставки нового, и для догоняния, не строя отдельный механизм доставки.

## Решение

**Pull — основной механизм; push — best-effort триггер; ретраи ведёт модуль, не брокер.**

- Подписчик сам забирает свои доставки (`poll`), помечает взятыми (`claim`, с `claimedAt`) и подтверждает (`ack`). Той же логикой обрабатывает и новое, и догоняние.
- Push — тонкий сигнал «есть новое», чтобы подписчик запустил pull сразу, а не ждал планового опроса. Данные push не переносит, доставку не гарантирует; потерялся — плановый pull подберёт.
- Ретраи брокер **не** ведёт: взятую, но не подтверждённую дольше таймаута подписчика (`claimTimeoutMs`) доставку модуль перезабирает сам на следующем pull. Число попыток, backoff, внутреннюю очередь обработки и момент «сдаться» (терминальный `dead`) определяет сам модуль.
- Гранулярность состояния — **на пару «событие × подписчик»** (а не один курсор на подписку): сбойное событие изолировано в своей строке и не блокирует остальные (нет head-of-line blocking курсорной модели).

Аналогия — SQS: `receive` = claim (+ visibility timeout), `delete` = ack; не подтвердил вовремя — «всплывает». Отличие: «всплытие» инициирует не брокер, а сам подписчик, пересканируя свои просроченные `claimed`.

Обоснование:

1. **Простота брокера.** Уходят retry-цикл, backoff-планировщик, visibility-timeout джоб, dead-letter как ответственность брокера. Брокер = уведомитель + статусная доска.
2. **DRY.** Переиспользуется существующая pull-логика модуля; отдельного механизма доставки нет.
3. **Изоляция сбоев.** Per-delivery строки дают независимый исход на каждое событие (в отличие от курсора, где «ядовитое» событие стопорит подписку).
4. **Надёжность на дешёвом транспорте.** Push — только ускоритель; корректность держится на pull. Потеря push безопасна.
5. **Подходит масштабу FLOW.** Модулей немного, они доверенные, конкурентных воркеров одного подписчика не предвидится — цена «толстой» очереди (write-амплификация доставок) окупается простотой, а её недостатки некритичны.

## Последствия

Плюсы:

- минимальный брокер (нет retry/backoff/DLQ-механики);
- надёжность не зависит от push;
- изоляция сбойных событий;
- гарантия at-least-once сохраняется (перезабор просроченных `claimed`), потребитель обязан быть идемпотентным.

Минусы / границы:

- задержка ретрая ≈ период pull (для BPM приемлемо);
- нужен per-модульный таймаут перезабора (`claimTimeoutMs`, поле `BrokerModules`);
- гонка claim при совпадении push-триггера и планового pull — закрывается идемпотентностью и `runWithExclusiveLock` на пулл модуля;
- растёт таблица доставок (fan-out), но чистится retention (§3.5) — write-амплификация принята сознательно.
