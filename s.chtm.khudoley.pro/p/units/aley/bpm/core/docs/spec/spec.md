## 1. Назначение

`p/units/aley/bpm/core` — серверное ядро личной BPM-системы «FLOW». Модуль отвечает за хранение доменных данных, бизнес-логику и координацию взаимодействия между частями системы.

Центральная роль `core` — брокер событий для событийно-ориентированной архитектуры BPM-сервиса. Все части системы (задачи, журнал, диалоги, финансы, библиотека и др.) обмениваются фактами через события: производитель публикует, что произошло; `core` сохраняет событие в надёжном Heap-журнале, определяет потребителей и предоставляет им единый способ получить и обработать свои доставки.

Брокер не является внешней очередью сообщений. Он реализован поверх Heap Chatium и обеспечивает:

- неизменяемый журнал опубликованных событий;
- реестр участников (модулей-производителей и потребителей);
- реестр подписок с фильтрацией по типу события и источнику;
- материализованные строки доставок для каждого события и подписчика;
- жизненный цикл доставки: poll → claim → ack / fail с повторными попытками;
- best-effort уведомления подписчиков о появлении новых доставок (без постоянного опроса);
- dead-letter для событий, которые не удалось доставить после исчерпания попыток;
- административную диагностику состояния брокера.

Транспортная модель: **push/callback как основной канал** (брокер сам вызывает обработчик подписчика при появлении события) + **polling как резервный канал надёжности** (подписчик может запросить непрочитанные доставки, если push был потерян). Push не заменяет очередь — он лишь ускоряет обработку. Источником истины о состоянии доставки всегда является Heap-журнал.

Целевая гарантия — at-least-once. Exactly-once не обещается: потребители обязаны быть идемпотентными. Порядок доставки best-effort по времени публикации внутри одной подписки.

## 3. Хранилище (Heap)

Перечень Heap-таблиц брокера заполняется по мере проработки. Ниже зафиксированы таблицы, прошедшие согласование.

### 3.1. `BrokerModules` — реестр модулей-участников

Реестр производителей и потребителей событий. Модуль не пишет себя в таблицу напрямую — он регистрируется через гейтвей (флоу — §5.2); ключ upsert — `moduleKey`. Поведенческая логика (сверка токена аутентификации, принудительная проверка белых списков) описана в §5; здесь — схема записи.

| Поле | Тип | Назначение |
|------|-----|-----------|
| `moduleKey` | string | Уникальный ключ модуля (`tasks`, `journal`, `finance`). Ключ upsert. Естественный (бизнес-)ключ, а не системное `id` — обоснование выбора в [ADR-0001](../ADR/0001-modulekey-natural-key.md). |
| `displayName` | string | Человекочитаемое имя для админ-диагностики. |
| `source` | string | Источник модуля: `internal` (запрос пришёл внутренним `app.function`-вызовом) или `external` (запрос пришёл на внешний `app.html`/HTTP-эндпоинт). Проставляется гейтвеем по каналу регистрации, **не** принимается из тела запроса (§5.2). |
| `allowedPublishTypes` | string[] | Белый список glob-паттернов типов событий, которые модулю разрешено публиковать (`tasks.*`, `finance.payment.created`, `*`). Проверяется при каждом publish (`assertCanPublish`, §5). **Пустой массив = публикация запрещена** — модуль обязан явно объявить, что эмитит. |
| `allowedSubscribeTypes` | string[] | Белый список glob-паттернов типов, на которые модулю разрешено подписываться. Проверяется при регистрации подписки (`assertCanSubscribe`, §5). Пустой массив = подписка запрещена. |
| `enabled` | boolean | Активен ли модуль в обмене. Начальное значение задаёт гейтвей при регистрации по `source`: `internal` → `true` (активен сразу); `external` → `false`, до однократной модерации в админке гейтвея модуль в обмене не участвует (§5.2). `false` → модуль не участвует в обмене. |
| `adminDisabled` | boolean | Независимый административный стоп. Имеет приоритет над `enabled`: при `true` модуль отключён независимо от его собственного флага. Хранит только текущее состояние; каждое изменение этого флага (как и `enabled`) фиксируется отдельным событием брокера — «когда / кто / почему» живёт в событии, не на строке модуля. |
| `authTokenHash` | string | `SHA-256`-хэш токена доступа, **сгенерированного и выданного гейтвеем** при регистрации (реализация хэша — §5.1). Сам токен возвращается модулю один раз и больше нигде не хранится; здесь — только хэш. Сверяется при каждой операции publish/poll/ack/fail (§5, §7). |
| `metadata` | any | Произвольные дополнительные данные модуля (необязательно). |

В таблице выше — только **объявляемые** поля (второй аргумент `Heap.Table`).

**Системные поля.** Heap добавляет ко **всем** таблицам три зарезервированных поля автоматически; объявлять их в схеме нельзя (см. `inner/docs/008-heap.md`). Здесь они используются так:

| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | string | Системный первичный ключ строки. Бизнес-идентичность модуля ведётся по `moduleKey`, не по `id` ([ADR-0001](../ADR/0001-modulekey-natural-key.md)). |
| `createdAt` | number | Момент первой регистрации модуля (проставляет Heap при создании строки). |
| `updatedAt` | number | Момент последнего upsert (обновляет Heap автоматически). |

### 3.2. `BrokerEvents` — журнал опубликованных событий

Неизменяемый журнал фактов: каждая строка — одно опубликованное событие. Это центральное хранилище брокера и источник истины о том, что произошло (§1). Строки **только добавляются** (append-only): после публикации событие не редактируется и не удаляется — в том числе при удалении модуля-производителя его уже опубликованные события остаются (каскад §5.5). Материализация доставок по подписчикам — отдельная таблица `BrokerDeliveries` (в проработке); здесь хранится сам факт публикации, без состояния доставки.

| Поле | Тип | Назначение |
|------|-----|-----------|
| `eventType` | string | Доменный тип события (`tasks.created`, `finance.payment.created`). При публикации сверяется с `allowedPublishTypes` продюсера, при материализации доставок — с `allowedSubscribeTypes` подписчиков (glob, §5). Неизменяем. |
| `producerModuleKey` | string | `moduleKey` модуля-производителя — обычной строкой, **не** `RefLink` (связь с `BrokerModules` ведётся строковым ключом, §5.5). Один тип могут публиковать разные модули (multi-producer, §5.2). |
| `payload` | any | Полезная нагрузка — данные произошедшего факта (JSON-объект). Структуру задаёт контракт типа (`BrokerContracts`, в проработке). Неизменяема после публикации. |

В таблице выше — только **объявляемые** поля.

**Системные поля** (Heap добавляет автоматически ко всем таблицам, см. §3.1):

| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | string | Системный первичный ключ — он же идентификатор события, на который ссылаются строки `BrokerDeliveries` (тоже строкой, не `RefLink`). |
| `createdAt` | number | Момент публикации события. Служит отметкой времени для best-effort порядка доставки внутри подписки (§1). |
| `updatedAt` | number | Технически обновляется Heap, семантически не используется: журнал неизменяем (= `createdAt`). |

**Поля, фиксируемые вместе с flow публикации (§5, в проработке).** Вынесены отдельно, потому что зависят от ещё не описанных операций publish и материализации доставок — в объявляемую схему пока не входят:

- `idempotencyKey` (string, опционально) — ключ дедупликации от продюсера: повторная публикация того же факта (сетевые ретраи, at-least-once на стороне продюсера) не должна порождать дубль. Предполагаемая уникальность — в паре с `producerModuleKey`; механизм проверки (уникальный индекс / `runWithExclusiveLock`) — в §5.
- `dispatchedAt` (number, опционально) — отметка, что брокер материализовал доставки по событию; `null` = fan-out ещё не выполнен (краш-безопасность на разрыве «событие записано» ↔ «доставки созданы»). Точная семантика — вместе с flow публикации.
- Версионирование `payload` (`schemaVersion`) и явный порядковый номер публикации (монотонная последовательность вместо `createdAt` для строгого порядка) — открытые вопросы; решаются вместе с `BrokerContracts` и flow доставки.

**Граница с историей операций (открытый вопрос).** В §3.1/§5 изменения состояния модулей (`enabled`/`adminDisabled`) и решения модерации описаны как «событие брокера». Не зафиксировано, попадают ли они в этот же журнал `BrokerEvents` системными типами (вида `broker.module.registered`) или в отдельный операционный аудит `BrokerOpsAudit`. Решается при согласовании `BrokerOpsAudit`.

### 3.3. `BrokerDeliveries` — материализованные доставки

Рабочая очередь брокера: при публикации события (§3.2) на шаге материализации (fan-out) создаётся **по одной строке на каждого подписчика, чья подписка матчит `eventType`**. Здесь живёт состояние доставки **на пару «событие × подписчик»** — независимый жизненный цикл `poll → claim → ack / fail` (§1): один подписчик уже подтвердил обработку, другой ещё в очереди или упал с ошибкой. И push, и резервный polling работают по этим строкам; dead-letter — тоже признак строки, не события.

| Поле | Тип | Назначение |
|------|-----|-----------|
| `eventId` | string | `id` события из `BrokerEvents` (§3.2) — обычной строкой, **не** `RefLink`. |
| `eventType` | string | Денормализованный тип события: чтобы polling-выборка подписчика фильтровала по типу без джойна к журналу. |
| `subscriberModuleKey` | string | `moduleKey` подписчика — строкой, не `RefLink` (§5.5). |
| `status` | enum | Состояние доставки: `pending` (создана, ждёт обработки) → `claimed` (захвачена в обработку, visibility timeout) → `acked` (успех, терминальное) / `failed` (попытка не удалась, будет повтор) → `dead` (попытки исчерпаны, dead-letter, терминальное). |

В таблице выше — только **объявляемые** поля (минимальное ядро). Поля жизненного цикла попыток — `attempts`, `claimedAt`, `nextAttemptAt` (backoff), `lastError`, а также `subscriptionId` (какая подписка из `BrokerSubscriptions` матчнула) — фиксируются вместе с flow доставки (§5) и таблицей подписок; в ядро пока не включены.

**Системные поля** (Heap, §3.1): `id` — ключ строки доставки; `createdAt` — момент материализации (используется для очистки, см. ниже); `updatedAt` — момент последнего перехода статуса.

#### 3.3.1. Очистка устаревших доставок (джоб, ежедневно в полночь)

`BrokerDeliveries` — самая быстрорастущая таблица брокера: один fan-out превращает одно событие в N строк (по числу подписчиков). Heap-таблица в Chatium ограничена ~10 млн строк (платформенное ограничение), поэтому доставки нужно регулярно подчищать, иначе таблица упрётся в предел.

**Механизм.** Отдельный джоб `@app/jobs` (`app.job`) запускается **каждую полночь** и в конце сам планирует следующий запуск на ближайшие `00:00` через `scheduleJobAt` (самоперепланирование; полночь — в таймзоне аккаунта). Первый запуск ставится при деплое / разовым админ-действием (bootstrap); далее цепочка самоподдерживается.

**Что удаляется.** Все строки **старше 24 часов** (`createdAt < now − 24h`), **статус которых не требует сохранения**:

- **удаляются:** `acked` — успешно доставленные, ценности больше не несут;
- **сохраняются независимо от возраста:**
  - `pending` / `claimed` / `failed` — работа не завершена (в полёте / ждут повтора): удалить = потерять доставку;
  - `dead` — dead-letter: оставляем для диагностики и ручного разбора (статус «требует оставить»).

Итоговый предикат: `where({ status: 'acked', createdAt: { $lt: cutoff } })`.

**Батчами.** Удаление идёт партиями (`deleteAll` с явным `limit`, напр. 1000, в цикле до исчерпания — `deleteAll` по умолчанию `limit = 1` и блокируется при превышении, см. `inner/docs/008-heap.md`), с логом на каждую партию — чтобы не упереться в ограничения разовой операции и быть устойчивым к перезапуску.

```ts
const RETENTION_MS = 24 * 60 * 60 * 1000
const BATCH = 1000

export const cleanupDeliveriesJob = app.job('/broker/cleanup-deliveries', async (ctx) => {
  const cutoff = new Date(Date.now() - RETENTION_MS)

  let total = 0
  for (;;) {
    const deleted = await BrokerDeliveries.deleteAll(ctx, {
      where: { status: 'acked', createdAt: { $lt: cutoff } },
      limit: BATCH,
    })
    total += deleted
    if (deleted < BATCH) break
  }

  ctx.account.log('broker deliveries cleanup', { level: 'info', json: { deleted: total } })

  // следующий запуск — в ближайшую полночь (таймзона аккаунта)
  const next = new Date()
  next.setHours(24, 0, 0, 0)
  await cleanupDeliveriesJob.scheduleJobAt(ctx, next, {})
})
```

**Остаточный риск (открытый вопрос).** Этот джоб не чистит `dead` (dead-letter копится). Его собственная политика хранения — отдельная retention / выгрузка в выделенную DLQ-таблицу — фиксируется вместе с дизайном dead-letter.

Остальные таблицы (`BrokerContracts`, `BrokerSubscriptions`, `BrokerOpsAudit`) — в проработке, фиксируются по мере согласования полей.

## 5. Бизнес-логика

### Операции гейтвея и эндпоинты

> Имена ниже — предложенные (на ревью вместе с §5.3/§5.4). Гейтвей `core` — единственная точка записи в `BrokerModules`; модуль обращается только к этим операциям.

Каждая операция имеет **два транспортных эндпоинта** на одну и ту же логику; от того, какой из них вызван, гейтвей выводит `source` (§5.2):

| Операция | Назначение | internal — `app.function` | external — HTTP-роут |
|----------|------------|---------------------------|----------------------|
| `registerModule` | Регистрация, только create (§5.2) | `registerModule` | `POST /broker/register` |
| `updatePublishTypes` | Обновление `allowedPublishTypes`, только mutate (§5.3) | `updatePublishTypes` | `POST /broker/publish-types` |
| `updateSubscribeTypes` | Обновление `allowedSubscribeTypes`, только mutate (§5.4) | `updateSubscribeTypes` | `POST /broker/subscribe-types` |
| `deleteModule` | Удаление регистрации, только existing (§5.5) | `deleteModule` | `POST /broker/delete` |

- Вызов через internal-эндпоинт (`app.function`) → `source = internal`; через external HTTP-роут → `source = external`. На обновлениях (§5.3/§5.4) `source` берётся из строки и каналом **не** переопределяется — канал там влияет только на политику модерации.
- `registerModule` токен **не** принимает (только создаёт строку); `updatePublishTypes` / `updateSubscribeTypes` токен **требуют** и работают лишь по существующей строке. Объединять их в один эндпоинт с опциональным токеном решено **не** (conditional auth на границе доверия).

### 5.1. Хэширование auth-токена (SHA-256)

**Решение (зафиксировано):** `authTokenHash` вычисляется настоящим SHA-256 через `@npm/node-forge`. Обоснование выбора — что вообще доступно в Chatium и почему именно SHA-256, а не шифрование / FNV-1a / медленные KDF / HMAC — в [ADR-0002](../ADR/0002-token-hashing-sha256.md).

Правила реализации:

- **Зависимость.** В `package.json` workspace должен быть объявлен `"@npm/node-forge": "*"` (пакет в платформенном белом списке; по умолчанию в проекте не подключён — рядом объявлены только `@npm/date-fns*`). Доступность подтверждается типчеком/деплоем перед первым использованием.
- **Алгоритм.** `forge.md.sha256` → шестнадцатеричная строка из 64 символов.
- **Вход.** Хэшируется доменно-разделённая строка `broker-module-auth:<moduleKey>:<authToken>`: префикс отделяет пространство имён брокера, `moduleKey` привязывает хэш к конкретному модулю (защита от переноса хэша между модулями).
- **Хранение.** Результат пишется в выделенное поле `authTokenHash` строки `BrokerModules` (§3.1). Сам токен в открытом виде нигде не хранится и не логируется.

```ts
import * as forge from '@npm/node-forge'

function moduleAuthHash(moduleKey: string, authToken: string): string {
  const md = forge.md.sha256.create()
  md.update(`broker-module-auth:${moduleKey}:${authToken}`)
  return md.digest().toHex() // 64 hex-символа
}
```

### 5.2. Регистрация модуля

Модуль не пишет себя в `BrokerModules` напрямую — он регистрируется через гейтвей, который определяет источник, проверяет заявку и выдаёт токен доступа.

Шаги:

1. Модуль вызывает операцию `registerModule` (internal `app.function` или external `POST /broker/register`), передавая `moduleKey`, `allowedPublishTypes` и `allowedSubscribeTypes`. Токен не передаётся — операция только создаёт строку.
2. **Источник определяет гейтвей по каналу запроса, не из тела:**
   - запрос пришёл внутренним `app.function`-вызовом → `source = internal`;
   - запрос пришёл на внешний `app.html`/HTTP-эндпоинт → `source = external`.
3. Гейтвей валидирует заявку: корректность glob-синтаксиса заявленных `allowedPublishTypes` / `allowedSubscribeTypes`. **Эксклюзивности типа нет** — один и тот же тип события могут публиковать несколько модулей (multi-producer: например, гейты к разным банкам, эмитящие `payment.created`); событие несёт своего продюсера, подписчик при желании фильтрует по источнику. Отдельного правила «занятости» не вводится: спуфинг типов закрывается модерацией `external` (шаг 4) и запретом повторной регистрации (см. ниже).
4. При успешной валидации гейтвей:
   - генерирует случайный **токен доступа**, сохраняет в `authTokenHash` только его `SHA-256` (§5.1), а сам токен возвращает модулю один раз;
   - фиксирует строку модуля с проставленным `source` и начальным `enabled` по источнику:
     - `internal` → `enabled = true` (активен сразу);
     - `external` → `enabled = false`; заявка попадает на **однократную модерацию** в админке гейтвея, до одобрения модуль в обмене не участвует.
5. Токен модуль хранит у себя и предъявляет при каждой операции; гейтвей хэширует входящий токен и сверяет с `authTokenHash`. Хэш в БД при утечке рабочего креденциала не даёт.

Изменение набора разрешённых типов после регистрации — отдельные операции: публикации — §5.3, подписки — §5.4. **Повторная регистрация уже существующего `moduleKey` запрещена**: `registerModule` на занятый ключ → отказ; существующая строка только обновляется (§5.3/§5.4) или удаляется (§5.5). Это и снимает спуфинг — нельзя пере-зарегистрировать чужой модуль на новый токен/набор типов в обход модерации. Ротация токена (отдельно от update) — §5, в проработке.

### 5.3. Обновление `allowedPublishTypes`

Смена набора типов, которые модулю разрешено публиковать, — самостоятельная операция, а не повторная регистрация: `moduleKey`, `source`, `authTokenHash` и токен не меняются.

Шаги:

1. Модуль вызывает операцию `updatePublishTypes` (internal `app.function` или external `POST /broker/publish-types`) с новым полным набором `allowedPublishTypes` (полная замена, не дельта) и предъявляет свой токен. Операция применима только к **существующей** строке `moduleKey`; создание идёт исключительно через `registerModule` (§5.2).
2. **Аутентификация.** Гейтвей хэширует входящий токен и сверяет с `authTokenHash` строки `moduleKey` (§5.1). Чужой/неверный токен → отказ. `source` берётся из существующей строки, **не** переопределяется каналом запроса (в отличие от регистрации `source` неизменяем).
3. **Валидация.** Проверяется корректность glob-синтаксиса. Несколько модулей могут публиковать один тип (multi-producer — см. §5.2, шаг 3).
4. **Модерация по `source`:**
   - `internal` → изменение применяется сразу, `enabled` не трогается;
   - `external` · **сужение** (только удаление паттернов) → применяется сразу, без модерации;
   - `external` · **расширение** (есть паттерн, которого не было) → модуль уходит на **ре-модерацию**: гейтвей ставит `enabled = false` (модуль приостановлен) и пишет новый набор прямо в `allowedPublishTypes`; до повторного одобрения (§5.6) модуль в обмене не участвует. Отдельного «pending»-набора **нет** — признак заявки на модерацию это и есть `enabled = false` (§5.6).
5. **Применение.** Новый набор пишется в строку `BrokerModules`, `updatedAt` обновляется (при external-расширении — дополнительно `enabled = false`). Изменение фиксируется отдельным событием брокера (как `enabled`/`adminDisabled`, §3.1) — история живёт в событии, не на строке.

Пустой массив = публикация запрещена (как в §3.1): корректный способ полностью отозвать право на публикацию.

### 5.4. Обновление `allowedSubscribeTypes`

Смена набора типов, на которые модуль подписан, — самостоятельная операция; `moduleKey`, `source`, `authTokenHash` и токен не меняются.

Шаги:

1. Модуль вызывает операцию `updateSubscribeTypes` (internal `app.function` или external `POST /broker/subscribe-types`) с новым полным набором `allowedSubscribeTypes` (полная замена) и предъявляет токен. Операция применима только к **существующей** строке `moduleKey`; создание — исключительно через `registerModule` (§5.2).
2. **Аутентификация** — идентично §5.3 (сверка хэша токена; `source` из строки, не из канала).
3. **Валидация.** Проверяется корректность glob-паттернов.
4. **Модерация по `source`:**
   - `internal` → применяется сразу, `enabled` не трогается;
   - `external` · **сужение** → применяется сразу, без модерации;
   - `external` · **расширение** → ре-модерация (подписка на новый тип = доступ к новым данным, граница доверия): `enabled = false`, новый набор пишется прямо в `allowedSubscribeTypes`, до одобрения (§5.6) модуль приостановлен. Отдельного «pending»-набора нет.
5. **Применение** — идентично §5.3 (запись набора, `updatedAt`, при external-расширении — `enabled = false`, событие-история).

Пустой массив = подписка запрещена (§3.1).

### 5.5. Удаление регистрации модуля

Удаление снимает модуль с обмена и **освобождает `moduleKey`**: после удаления ключ снова можно занять через `registerModule` (§5.2). Это и есть способ «переоформить» модуль, раз повторная регистрация занятого ключа запрещена (§5.2).

Шаги:

1. Модуль вызывает операцию `deleteModule` (internal `app.function` или external `POST /broker/delete`), передавая `moduleKey` и предъявляя токен. Операция применима только к **существующей** строке.
2. **Аутентификация.** Гейтвей хэширует входящий токен и сверяет с `authTokenHash` строки `moduleKey` (§5.1). Чужой/неверный токен → отказ. Несуществующий `moduleKey` → отказ (нечего удалять).
3. **Без модерации.** Удаление — сокращение собственного присутствия модуля (как «сужение» в §5.3/§5.4), не расширение прав; и для `internal`, и для `external` применяется сразу. Админ-инициированное снятие — это `adminDisabled` (§3.1), отдельный механизм, не `deleteModule`.
4. **Применение.** Строка `BrokerModules` удаляется (состояние на строке — только текущее, §3.1). Факт удаления фиксируется отдельным событием брокера — история «когда / кто» живёт в событии, не на строке; сам `moduleKey` после этого свободен.

**Каскад при удалении (решено):** уже опубликованные события модуля остаются в неизменяемом журнале (исторический факт), а его подписки и незавершённые доставки снимаются. Точные технические детали каскада фиксируются вместе с дизайном `BrokerSubscriptions` / `BrokerEvents` / `BrokerDeliveries` (§3, пока не согласованы).

### 5.6. Модерация: очередь, аппрув и дезлайн

**Очередь модерации — это просто строки `BrokerModules` с `enabled = false`** (только `external`; у `internal` всё применяется сразу и `enabled` не выключается). Отдельной таблицы и `pending`-полей нет: «запрошенное, но не одобренное» состояние выражается приостановкой модуля. Идентификатор сущности — `moduleKey`.

Под `enabled = false` модуль попадает по двум причинам, и обе видны одним запросом:

| Причина | Откуда | Текущий набор в строке |
|---------|--------|------------------------|
| Первичная регистрация | §5.2 (`external` → `enabled = false`) | заявленный при регистрации |
| Ре-модерация после расширения типов | §5.3 / §5.4 (`external`-расширение) | уже новый, запрошенный набор |

Различие первичной регистрации и ре-модерации — по истории: модуль **ни разу не был активен** vs **был активен ранее** (выводится из событий брокера; конкретный признак фиксируется вместе с таблицей событий §3). Approve от этого не зависит, decline — зависит (см. ниже).

> **§5.6 ещё требует расширения:** фиксированные запросы каждой операции (Request/Response контракты) и примеры с реальными полями из `BrokerModules`; пока здесь только логика и псевдокод.

Операции (admin-only). Это не модульный API — у `internal`/`external` модулей к ним нет доступа в принципе. Вызываются из админ-страницы самого брокера: серверный роут с `// @shared-route`, клиент дёргает его напрямую через `.run()`, без `app.function` и без отдельного HTTP-эндпоинта для модулей.

| Операция | Назначение | Роут (`// @shared-route`) |
|----------|------------|----------------------------|
| `listPendingModeration` | Модули, ожидающие решения | `/broker/admin/moderation` |
| `approveModeration` | Одобрить модуль по `moduleKey` | `/broker/admin/moderation/approve` |
| `declineModeration` | Отклонить модуль по `moduleKey` | `/broker/admin/moderation/decline` |

- **`listPendingModeration`** — `where({ source: 'external', enabled: false })`: список приостановленных external-модулей с их текущим (запрошенным) набором типов, причиной (первичная / ре-модерация) и `updatedAt`.
- **`approveModeration(moduleKey)`** — ставит `enabled = true`. Единообразно для обеих причин: первичная регистрация — допуск в обмен; ре-модерация — возврат модуля в обмен на его уже записанном наборе. Решение фиксируется событием брокера (кто/когда — в событии, не на строке, §3.1).
- **`declineModeration(moduleKey, reason?)`** — оставляет модуль приостановленным, действие зависит от причины:
  - первичная регистрация → строка `BrokerModules` удаляется, `moduleKey` освобождается (модуль может подать заявку заново; ср. §5.5);
  - ре-модерация изменения → строка остаётся с `enabled = false`; чтобы вернуться в обмен, модуль подаёт корректный набор через §5.3/§5.4 (это перезапишет набор и снова оставит `enabled = false`) и проходит approve. Прежний одобренный набор отдельно **не хранится** — авто-отката к нему нет (следствие варианта с приостановкой через `enabled`).

  `reason` — необязательная причина для диагностики; решение фиксируется событием брокера.
