## Словарь терминов

Англоязычные термины ниже используются в спецификации намеренно — это устоявшиеся термины индустрии или платформы, точного однословного русского эквивалента у них нет. Все прочие английские выражения в тексте переведены на русский.

| Термин | Значение |
|--------|----------|
| **Heap** | Встроенное хранилище данных платформы Chatium (имя собственное), поверх которого реализован брокер. |
| **workspace** | Рабочее пространство Chatium — корень проекта с `package.json` и зависимостями (`@app/*`, `@npm/*`). |
| **spec-as-source** | Методология, при которой спецификация — источник истины: код не меняется, пока изменение не описано здесь (см. §0). |
| **upsert** | Атомарная операция «вставить запись или обновить существующую по ключу». |
| **push / callback** | Best-effort **сигнал-триггер**: брокер уведомляет подписчика о появлении события, чтобы тот сразу запустил свою pull-логику. Данные не переносит, доставку не гарантирует. |
| **pull (опрос)** | **Основной** механизм: подписчик сам забирает свои доставки у брокера, помечает взятыми и подтверждает обработку. Ретраи ведёт модуль, не брокер. |
| **poll → claim → ack** | Жизненный цикл доставки со стороны подписчика: запросить свои → пометить взятой (`claimed`, с временем) → подтвердить успех (`acked`). Незавершённые (просроченные `claimed`) подписчик перезабирает сам. |
| **fan-out** | Материализация одного опубликованного события в **множество** доставок: брокер создаёт по одной строке `BrokerDeliveries` на каждого подходящего активного подписчика (одно событие → N доставок, §3.3/§5). Термин из систем обмена сообщениями — один вход «разветвляется» на много получателей. |
| **at-least-once** | Гарантия доставки «хотя бы один раз»: возможны повторы, потребитель обязан быть идемпотентным. |
| **exactly-once** | Доставка «ровно один раз»; в брокере не обещается. |
| **best-effort** | «По мере возможности», без строгих гарантий (о порядке доставки и об уведомлениях). |
| **dead-letter** | Терминальный статус доставки, которую подписчик после своих ретраев признал необрабатываемой. Когда сдаться — решает модуль; брокер лишь хранит статус. |
| **glob** | Синтаксис шаблонов с подстановкой (`tasks.*`, `*`). |
| **SHA-256** | Алгоритм криптографического хэширования (для хэша токена аутентификации). |

## 1. Назначение

`p/units/aley/bpm/core` — серверное ядро личной BPM-системы «FLOW». Модуль отвечает за хранение доменных данных, бизнес-логику и координацию взаимодействия между частями системы.

Центральная роль `core` — брокер событий для событийно-ориентированной архитектуры BPM-сервиса. Все части системы (задачи, журнал, диалоги, финансы, библиотека и др.) обмениваются фактами через события: производитель публикует, что произошло; `core` сохраняет событие в надёжном Heap-журнале, определяет потребителей и предоставляет им единый способ получить и обработать свои доставки.

Брокер не является внешней очередью сообщений. Он реализован поверх Heap Chatium и обеспечивает:

- неизменяемый журнал опубликованных событий;
- реестр участников (модулей-производителей и потребителей);
- реестр разрешённых типов подписки (whitelist `allowedSubscribeTypes` на каждом модуле): маршрутизация **только по типу события**, продюсер для подписчика анонимен — подписчик получает факт по типу, не зная и не указывая, кто его опубликовал ([ADR-0008](../ADR/0008-subscription-routing-type-only.md));
- материализованные **самодостаточные** строки доставок для каждой пары «событие × подписчик» — со снимком события (`eventType` + `schemaVersion` + `payload`), по которым подписчик ведёт свой pull, не читая общий журнал;
- best-effort уведомления (push-триггер) подписчиков о появлении новых событий — чтобы те запустили свой pull, а не ждали планового опроса;
- терминальный статус `dead` для доставок, которые подписчик признал необрабатываемыми (ретраи и решение «сдаться» — на стороне модуля);
- административную диагностику состояния брокера.

Чего брокер **не** делает: не переносит данные события в push, сам не ретраит и не ведёт backoff, не решает, когда доставку признать мёртвой, **не владеет и не проверяет контракт `payload`** (структуру данных согласуют между собой продюсер и подписчики в своём коде — брокер лишь переносит `schemaVersion` непрозрачным тегом, не таблицей-реестром). Всё это — на стороне модуля-подписчика (см. транспорт ниже).

Транспортная модель — **pull-driven с push-триггером** (обоснование выбора pull/push/гибрид и отказ от очереди/курсора — [ADR-0004](../ADR/0004-transport-pull-driven-push-trigger.md)):

- **Pull — основной механизм.** Подписчик сам забирает свои доставки у брокера (`poll`), помечает взятыми (`claim`) и по завершении подтверждает (`ack`). Ту же pull-логику модуль использует и для нового, и для догоняния — отдельного механизма доставки не строим.
- **Push — только best-effort триггер.** При появлении события брокер уведомляет подписчика, чтобы тот сразу запустил свой pull, а не ждал планового опроса. Push не переносит данные и не гарантирует доставку; потерялся — плановый pull подберёт.
- **Ретраи ведёт модуль, не брокер.** Взятая, но не подтверждённая дольше таймаута (`claimTimeoutMs ?? 1 мин`, §3.1) доставка считается брошенной (падение) и возвращается подписчику на его следующем pull — просрочку вычисляет серверная выборка `fetchDeliveries` (§5.9), но инициатива остаётся за модулем: не пришёл за доставками — перезабора не будет, брокер сам ни статус не сбрасывает, ни повторный push не шлёт. Число попыток, backoff и внутреннюю очередь обработки определяет сам модуль; исчерпав попытки, он ставит доставке терминальный `dead`.

Источник истины о статусе доставки — таблицы брокера. Целевая гарантия — at-least-once: доставка самодостаточна (несёт снимок события), поэтому держится до обработки независимо от того, где сейчас исходное событие (горячий журнал / архив / файл) — отставание подписчика payload не теряет. Перезабор просроченных `claimed` может привести к повтору — потребитель обязан быть идемпотентным. Порядок — best-effort по `createdAt` доставки (≈ время публикации) внутри одной подписки.

**Двойная обработка — зона ответственности подписчика, не брокера (решено 20-07-2026).** Корень повтора в том, что брокер судит о жизни воркера по таймауту, а таймаут не отличает «умер» от «работает медленно». Развилка без третьего выхода: отдать доставку другому по истечении таймаута (жив был — двойная работа) либо не отдавать никому (мёртв был — доставка зависла навсегда). Выбор сделан в пользу дублей против зависаний — это и есть `at-least-once`; `exactly-once` брокер не обещает (словарь терминов).

Поэтому задача не в том, чтобы дубль не случался, а в том, чтобы он **ничего не стоил**: подписчик ведёт у себя отметку обработанных `eventId` и на повторе выходит сразу, не выполняя работу заново. Механика этой дедупликации — на стороне модуля-подписчика, брокер её не предоставляет и не регулирует (как и ретраи, backoff, число воркеров — §3.3).

Со стороны брокера дубль **сужается**, но не устраняется: метка захвата (§3.3, §5.9 О6) наводит порядок в том, чей отчёт принимать. Сузить окно сильнее брокер **не пытается** — для этого пришлось бы вводить операции сверх зафиксированных трёх (О1) и требовать от каждого модуля фоновой активности. Если окажется, что у живых подписчиков обработка регулярно перешагивает таймаут, вопрос открывается заново (§0.1).

## 3. Хранилище (Heap)

Перечень Heap-таблиц брокера заполняется по мере проработки. Ниже зафиксированы таблицы, прошедшие согласование.

**Системный `id` неперебираем.** Heap генерирует его как длинную случайную строку букв и цифр (в `inner/docs/008-heap.md` формат не описан — уточнено отдельно). Следствия: подобрать чужой `id` нельзя, поэтому доступ к строке по `id` не нуждается в дополнительном сокрытии её существования (§5.9.5); и наоборот — `id` пригоден как непрозрачная ссылка наружу, гадать по нему о соседних записях бессмысленно.

**Соглашение о типах времени.** Системные `createdAt`/`updatedAt` Heap проставляет сам, и их тип — **`Date`**, а не число: в `where` они сравниваются с объектом `Date` (`{ createdAt: { $lt: new Date(...) } }`, `inner/docs/008-heap.md`). Все **объявляемые** нами отметки времени (`claimedAt` §3.3, `dispatchedAt` §3.2, `batchFrom`/`batchTo` §3.4) — наоборот, **`number` (epoch ms)**: над ними идёт арифметика порогов (`Date.now() - timeoutMs`), и число здесь естественнее. Правило простое: **системное поле — `Date`, наше — `number`**. На стыке (когда наше поле заполняется из системного) конвертация **явная**: `pack[0].createdAt.getTime()`, а не присваивание «как есть» — иначе в числовое поле попадёт `Date`.

**Платформенное ограничение: индексами Heap управлять нельзя.** В API нет ни создания индексов (`createIndex`, поле `indexes` в опциях таблицы), ни хинтов у `findAll`, ни `explain()`/`plan()` у селект-билдера — подтверждено разбором типов `@app/heap`/`@app/types`/`@app/data`. Хранилище внутри — PostgreSQL, планы строит его planner, повлиять на выбор индекса из кода workspace невозможно. Следствие для всей главы: **производительность выборок нельзя «починить» индексом** — её закладывают формой данных (компактные строки, §3.2), объёмом таблиц (retention §3.5 держит `BrokerDeliveries` в размере рабочей очереди, а не журнала) и рабочим набором (крупные аналитические выборки идут не в Heap, а в ClickHouse-зеркало, §3.4). Там, где выборка на горячем пути деградировала, разбираться следует с объёмом таблицы, а не с текстом запроса.

### 3.1. `BrokerModules` — реестр модулей-участников

Реестр производителей и потребителей событий. Модуль не пишет себя в таблицу напрямую — он регистрируется через гейтвей (флоу — §5.2); ключ upsert — `moduleKey`. Поведенческая логика (сверка токена аутентификации, принудительная проверка белых списков) описана в §5; здесь — схема записи.

| Поле | Тип | Назначение |
|------|-----|-----------|
| `moduleKey` | string | Уникальный ключ модуля (`tasks`, `journal`, `finance`). Ключ upsert. Естественный (бизнес-)ключ, а не системное `id` — обоснование выбора в [ADR-0001](../ADR/0001-modulekey-natural-key.md). |
| `displayName` | string | Человекочитаемое имя для админ-диагностики. |
| `source` | string | Источник модуля: `internal` (запрос пришёл внутренним `app.function`-вызовом) или `external` (запрос пришёл на внешний HTTP-роут, `app.post`). Проставляется гейтвеем по каналу регистрации, **не** принимается из тела запроса — канал неподделываем, см. §5.2 шаг 2. Влияет только на политику модерации, на аутентификацию — нет (§5.9, О5). |
| `allowedPublishTypes` | string[] | **Одобренный, действующий** белый список glob-паттернов типов событий, которые модулю разрешено публиковать (`tasks.*`, `finance.payment.created`, `*`). Грамматика — **посегментная**, как у `allowedSubscribeTypes` (каждый dot-сегмент = литерал или `*` целиком; без частично-сегментных паттернов и регэкспов). Префикс `broker.` зарезервирован системой — публиковать/объявлять его модуль не может (§3.2). Проверяется при каждом publish (`assertCanPublish`, §5). **Пустой массив = публикация запрещена** — модуль обязан явно объявить, что эмитит. Запрошенное, но не одобренное сюда **не пишется** — для этого `pendingPublishTypes`. |
| `allowedSubscribeTypes` | string[] | **Одобренный, действующий** белый список glob-паттернов типов, на которые модулю разрешено подписываться. Проверяется при регистрации подписки (`assertCanSubscribe`, §5). **На fan-out подписчики события резолвятся из этого поля** ([ADR-0008](../ADR/0008-subscription-routing-type-only.md)): publish-код разворачивает `eventType` в конечный набор предков-глобов (`tasks.created` → `{ *, tasks.*, tasks.created }`) и делает один `where { status: 'active', allowedSubscribeTypes: { $includes: { $any: [...] } } }` — доставки создаются **только активным** подписчикам (`onModeration`/`disabled` их не получают); отдельной таблицы подписок нет, маршрутизация по типу, продюсер анонимен. Глобы посегментные (сегмент = литерал или `*`), чтобы набор предков был перечислим. Пустой массив = подписка запрещена. Запрошенное, но не одобренное — в `pendingSubscribeTypes`, на fan-out **не влияет**. |
| `pendingPublishTypes` | string[] \| null | **Запрошенный, ещё не одобренный** набор publish-типов; `null` — заявки нет. Заполняется при `external`-расширении (§5.3), читается **только админкой модерации** (§5.6) — ни publish-проверка, ни fan-out его не видят. Одобрение переносит набор в `allowedPublishTypes` и очищает поле; отклонение просто очищает. Для `internal` не используется (изменения применяются сразу). |
| `pendingSubscribeTypes` | string[] \| null | То же для подписок (§5.4). Держит запрошенное расширение в стороне от боевого `allowedSubscribeTypes`, поэтому **модуль с висящей заявкой продолжает работать на одобренных правах**: fan-out нарезает ему доставки по старому набору, pull их забирает (§5.9, О5). |
| `status` | enum | Текущее **операционное** состояние модуля — работает он в обмене или нет ([ADR-0010](../ADR/0010-module-status-enum.md) + поправка от 19-07-2026). Три взаимоисключающих значения: `onModeration` («На модерации») — `external`, первая заявка, **ещё ни разу не запущен**, ждёт первичного одобрения (§5.2/§5.6); `active` («Активен») — участвует в обмене; `disabled` («Выключен») — админ-стоп, применяется **только из `active`** (§5.7). «Активен сейчас» = `active`, без исключений и перечислений. При регистрации: `internal` → сразу `active`, `external` → `onModeration`. Только текущее состояние — переходы фиксируются `broker.*` событиями (§3.2, [ADR-0009](../ADR/0009-operational-audit-events.md)), не на строке. **Наличие заявки на модерацию статусом не выражается** — это отдельная ось (`pending*`-поля): работающий модуль, попросивший расширение, остаётся `active`. |
| `claimTimeoutMs` | number \| null | **Необязательное** переопределение таймаута «взятой» доставки: сколько миллисекунд `claimed`-доставка остаётся неподтверждённой, прежде чем считается брошенной (падение подписчика) и попадает в перезабор (§3.3). Пусто/не задано → действует **дефолт брокера `DEFAULT_CLAIM_TIMEOUT_MS = 60_000` (1 минута)**. Модуль вправе объявить своё значение при регистрации (§5.2) — например, если его обработка заведомо длиннее минуты. **Применяет таймаут сервер** в выборке `fetchDeliveries` (§5.9, О2), беря эффективное значение `claimTimeoutMs ?? DEFAULT_CLAIM_TIMEOUT_MS` из этой строки. |
| `authTokenHash` | string | `SHA-256`-хэш токена доступа, **сгенерированного и выданного гейтвеем** при регистрации (реализация хэша — §5.1). Сам токен возвращается модулю один раз и больше нигде не хранится; здесь — только хэш. Сверяется при каждой операции publish/poll/ack/fail (§5, §7). |
| `metadata` | any | Произвольные дополнительные данные модуля (необязательно). |

В таблице выше — только **объявляемые** поля (второй аргумент `Heap.Table`).

**Системные поля.** Heap добавляет ко **всем** таблицам три зарезервированных поля автоматически; объявлять их в схеме нельзя (см. `inner/docs/008-heap.md`). Здесь они используются так:

| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | string | Системный первичный ключ строки. Бизнес-идентичность модуля ведётся по `moduleKey`, не по `id` ([ADR-0001](../ADR/0001-modulekey-natural-key.md)). |
| `createdAt` | `Date` | Момент первой регистрации модуля (проставляет Heap при создании строки). |
| `updatedAt` | `Date` | Момент последнего upsert (обновляет Heap автоматически). |

### 3.2. `BrokerEvents` — журнал опубликованных событий

Неизменяемый журнал фактов: каждая строка — одно опубликованное событие. Это центральное хранилище брокера и источник истины о том, что произошло (§1). Строки **только добавляются** (append-only): фактическая часть события (тип, версия, `payload`, продюсер) после публикации не редактируется — меняется лишь служебная отметка `dispatchedAt` (разовый переход `null → timestamp` при завершении fan-out, §3.2). Событие не теряется никогда. Журнал хранится в Heap как **двухуровневый якорь истины** ([ADR-0011](../ADR/0011-event-history-hybrid-heap-clickhouse.md)): «горячая» `BrokerEvents` (~последний месяц) → холодный инлайновый архив `BrokerEventsArchive` (строки ~8 КБ, §3.4). retention-джоб (§3.5) перекладывает старые события в инлайновый архив — иначе неизменяемый журнал упёрся бы в лимит 1 млн строк на таблицу (§3.4). Параллельно fan-out (§5.8) **best-effort зеркалит** событие в **ClickHouse** — вторичная копия для аналитики и удобного реплея, **не источник истины** (§3.4). При удалении модуля-производителя его уже опубликованные события остаются (в горячей таблице или в архиве; каскад §5.5). Материализация доставок по подписчикам — отдельная таблица `BrokerDeliveries` (§3.3), куда при fan-out копируется снимок события. **Рантайм-обработка идёт по самодостаточным доставкам, журнал на этом пути не читается** — `BrokerEvents` (и её холодные уровни) служат неизменяемой историей: аудит, ре-материализация (новый подписчик / ре-подписка), ручной разбор.

| Поле | Тип | Назначение |
|------|-----|-----------|
| `eventType` | string | Доменный тип события (`tasks.created`, `finance.payment.created`). При публикации сверяется с `allowedPublishTypes` продюсера, при материализации доставок — с `allowedSubscribeTypes` подписчиков (glob, §5). Неизменяем. |
| `schemaVersion` | number | Версия схемы `payload` **внутри данного `eventType`** (целое, `1` по умолчанию). Продюсер бампает при несовместимом изменении формата. Брокер несёт пару `(eventType, schemaVersion)` непрозрачным тегом — не хранит и не проверяет сам контракт (§1, §8): структуру по этой паре знает код подписчика. Отдельным полем — чтобы фильтровать/маршрутизировать по версии через `where`, не парся `payload` ([ADR-0003](../ADR/0003-schema-version-field.md)). |
| `producerModuleKey` | string | `moduleKey` модуля-производителя — обычной строкой, **не** `RefLink` (связь с `BrokerModules` ведётся строковым ключом, §5.5). Один тип могут публиковать разные модули (multi-producer, §5.2). |
| `payload` | any | Полезная нагрузка — данные произошедшего факта (JSON-объект). Структуру согласуют между собой продюсер и подписчики в коде по паре `(eventType, schemaVersion)` — это соглашение вне брокера, не таблица брокера (§1, §8). Неизменяема после публикации. |
| `idempotencyKey` | string | Ключ дедупликации от продюсера (опционально; пусто → без дедупа). Повтор публикации того же факта (сетевые ретраи, at-least-once у продюсера) не создаёт дубль: уникальность — пара `(producerModuleKey, idempotencyKey)`, проверка при публикации через `runWithExclusiveLock` (§5, семантика ниже). Неизменяем. |
| `dispatchedAt` | number \| null | Отметка «fan-out завершён»: timestamp ставится **после создания всех строк `BrokerDeliveries`** по событию; `null` — рассылка не завершена. Единственное мутируемое поле события (разовый `null → timestamp`); фактическая часть неизменна. Основа краш-безопасности и восстановления (семантика ниже). |

В таблице выше — только **объявляемые** поля.

**Событие — маленький структурированный факт, обычно сотни байт** (`id`, тип, версия, ключевые поля произошедшего вроде `{ orderId, userId, amount, status }`) — не документ и не блоб. Оптимум строки Heap ~8 КБ всех полей в JSON, крупнее — заметно медленнее на Postgres (`inner/docs/008-heap.md`); но эти ~8 КБ здесь **жёсткий потолок, а не ориентир**: событие, раздувшееся до килобайтов, — сигнал, что в `payload` попал контент, которому там не место. Крупное (тексты, файлы, base64, длинные коллекции) кладётся в файловое хранилище (`ctx.storage`) или отдельную сущность, а в событие идёт только ссылка/метаданные. Тот же принцип держит компактными строки холодного архива (§3.4) — только там ~8 КБ набирается **пачкой** таких фактов в одной строке, а не одним событием.

**Системные поля** (Heap добавляет автоматически ко всем таблицам, см. §3.1):

| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | string | Системный первичный ключ — он же идентификатор события, на который ссылаются строки `BrokerDeliveries` (тоже строкой, не `RefLink`). |
| `createdAt` | `Date` | Момент публикации события. Служит отметкой времени для best-effort порядка доставки внутри подписки (§1). |
| `updatedAt` | `Date` | Технически обновляется Heap, семантически не используется: журнал неизменяем (= `createdAt`). |

**Служебные поля публикации — семантика (сам flow — операция `publishEvent` и fan-out-дренер, §5.8).** Поля объявлены в схеме выше; заполняет их операция публикации:

- **Дедупликация события (`idempotencyKey`).** Защищает от повторной публикации *одного и того же факта* продюсером (его at-least-once ретраи). Ключ — пара `(producerModuleKey, idempotencyKey)` (у разных модулей ключи не пересекаются). Проверка — `runWithExclusiveLock` по этому ключу, **а не уникальный индекс**: под одним замком идут «проверить наличие → вставить», поэтому две гонки-копии не проходят обе. Пустой ключ → дедупликации нет.
- **Завершение рассылки (`dispatchedAt`).** Публикация двухфазная: `publishEvent` записывает событие (`dispatchedAt = null`, §5.8), а fan-out (создание строк `BrokerDeliveries`) выполняет **асинхронный дренер** — по завершении всех доставок события он ставит `dispatchedAt`. `null` = fan-out не завершён; дренер это и подхватывает (он же recovery: краш продюсера/дренера доставку не теряет). Детали — §5.8.
- **Идемпотентность повторного fan-out — по ключу ДОСТАВКИ, не события.** Тонкость: `idempotencyKey` события здесь *не* работает — он один на все доставки события и не различает, какие строки уже созданы. Дедуп повторной рассылки идёт по естественному ключу доставки **`(eventId, subscriberModuleKey)`** (одна доставка на подписчика на событие): fan-out под `runWithExclusiveLock` по событию для каждого подписчика делает `findOneBy({ eventId, subscriberModuleKey })` и создаёт строку, только если её нет — уже созданные при повторе не дублируются, недостающие дозаписываются. (Heap `createOrUpdateBy` ключует по одному полю, поэтому пару проверяем явно под замком либо материализуем отдельным уникальным полем — деталь §5.) Итого два слоя идемпотентности с разными ключами: событие — `(producerModuleKey, idempotencyKey)`, доставка — `(eventId, subscriberModuleKey)`.
- **Порядок.** Строгого тотального порядка нет — счётчик `seq` не заводим (не окупает глобальный лок, сериализующий все публикации). Порядок доставки — **best-effort по `createdAt`** (§1); детерминированный обход журнала (аудит / ре-материализация) — по `(createdAt, id)` (`id` уникален), без отдельного счётчика.

**Версия ≠ новый тип.** Бамп `schemaVersion` не меняет `eventType`, поэтому вайтлисты `allowedPublishTypes`/`allowedSubscribeTypes` (glob по типу) и модерация (§5.6) не затрагиваются — версия это вторичный дискриминатор *внутри* типа. Подписчик, получив незнакомую (более новую) версию, действует по своей политике (отложить/отклонить) — это его дело, брокер версии не интерпретирует.

**История операций — системные события в журнале (решено, [ADR-0009](../ADR/0009-operational-audit-events.md)).** Изменения состояния модулей (переходы `status`, §3.1) и решения модерации (§5.6) фиксируются **не отдельной таблицей**, а **системными событиями типов `broker.*`** (`broker.module.registered`, `broker.module.status-changed`, `broker.module.moderated`, `broker.module.deleted`, `broker.*-types.changed`) в этом же журнале `BrokerEvents`: брокер публикует их сам как системный продюсер (обычным модулям `broker.*` эмитить нельзя). Они проходят обычный путь публикации (`dispatchedAt`, retention/архив §3.4/§3.5) и **фанаутятся единообразно** — обычно подписчиков по `broker.*` нет (ноль доставок), но модуль-дашборд может подписаться на `broker.module.*` и получать аудит-фид тем же механизмом. Отдельная таблица (`BrokerOpsAudit`) намеренно **не заводится** — иначе получился бы второй event-store в обход инварианта «факт → событие → журнал».

**Системный продюсер `broker.*` и защита namespace.** Эти события публикует **внутренний код брокера**, не модульный publish-API: под зарезервированным `producerModuleKey = 'broker'` (строка-сентинел, **не** запись в `BrokerModules` — брокер не участник обмена, а его хозяин), без токен-аутентификации и **в обход `assertCanPublish`** (брокер — источник полномочий). Ключ `broker` зарезервирован и как `moduleKey` — регистрация под ним запрещена (§5.2). Namespace `broker.` **зарезервирован под систему**: модуль его публиковать **не может** — публикация любого `broker.`-префиксного типа модулем отклоняется **независимо** от его `allowedPublishTypes` (даже `*` не даёт права), а явный `broker.*` в `allowedPublishTypes` отвергается валидацией (§5.2/§5.3). **Подписка** на `broker.*` модулю разрешена (аудит-фид, см. выше) — резервирование касается только публикации.

### 3.3. `BrokerDeliveries` — материализованные доставки

**Самодостаточный рабочий элемент, по которому подписчик ведёт свой pull.** При публикации события (§3.2) на шаге материализации (fan-out) создаётся **по одной строке на каждого подписчика, чья подписка матчит `eventType`**, и в неё **кладётся снимок события** (`eventType` + `schemaVersion` + `payload`). Модуль обрабатывает доставку целиком из своей таблицы доставок — **к общей `BrokerEvents` на рантайме не обращается** («толстая» доставка, обоснование — [ADR-0006](../ADR/0006-fat-delivery-payload-snapshot.md)). Жизненный цикл ведёт **сам подписчик через pull** (§1), брокер лишь хранит статус.

| Поле | Тип | Назначение |
|------|-----|-----------|
| `eventId` | string | `id` события из `BrokerEvents` (§3.2) — обычной строкой, **не** `RefLink`. Обратная ссылка на журнал для трассировки; **не** для дозагрузки данных (они уже в строке). Вместе с `subscriberModuleKey` — **естественный уникальный ключ доставки** `(eventId, subscriberModuleKey)`: по нему fan-out идемпотентен (§3.2). |
| `eventType` | string | Тип события (снимок): фильтр pull-выборки по типу и вход бизнес-логики подписчика. |
| `schemaVersion` | number | Версия схемы `payload` (снимок, §3.2): подписчик по паре `(eventType, schemaVersion)` знает, как интерпретировать данные. Копируется при fan-out вместе с payload. |
| `payload` | any | **Снимок данных события** (§3.2), скопированный при fan-out. Делает доставку самодостаточной — подписчик обрабатывает, не читая журнал. Маленький факт (обычно сотни байт; ~8 КБ — потолок записи Heap, граница а не цель, §3.2): крупный контент — ссылкой на файловое хранилище, не инлайном. |
| `subscriberModuleKey` | string | `moduleKey` подписчика — строкой, не `RefLink` (§5.5). |
| `status` | enum | `pending` (fan-out создал, ждёт pull) → `claimed` (подписчик взял в обработку) → `acked` (успех, терминальное) / `dead` (подписчик исчерпал свои попытки, терминальное). Отдельного `failed` нет: неподтверждённая `claimed` — это и есть «повтор» (перезабор после таймаута). |
| `claimedAt` | number | Момент последнего `claim`. Основа перезабора: `fetchDeliveries` отдаёт подписчику его `pending` и **просроченные** `claimed` — те, где `claimedAt` старше эффективного таймаута модуля (`claimTimeoutMs ?? DEFAULT_CLAIM_TIMEOUT_MS`, §3.1). До первого claim — не задан. |
| `claimCount` | number | **Сколько раз доставку выдавали.** Инкрементирует **брокер** при каждом `claim` — в той же записи, что `claimedAt`/`claimToken`, поэтому обходится бесплатно (лишнего обращения к Heap нет). Считает именно **выдачи**, а не попытки обработки: приступил ли модуль к работе после захвата, брокер не знает (отсюда имя — не `attempts`, §5.9 О7). Назначение — **строго диагностическое**: высокое значение выдаёт ядовитое сообщение либо циклически падающего подписчика, причём видно это и по строкам в статусе `claimed`, которые иначе выглядят штатными. **На решения брокера не влияет**: автоматического перевода в `dead` по достижении порога нет и не будет — когда сдаться, решает только модуль (§1). До первого claim — `0`. |
| `lastError` | string \| null | Причина финальной сдачи — необязательный параметр `deadDelivery` (§5.9, О7). Делает список dead-доставок в админке самообъясняющим, иначе причину пришлось бы искать в логах чужого приложения по `eventId`. Ограничение длины — `LAST_ERROR_MAX` (~1000 символов); **при превышении текст обрезается, вызов не отклоняется** (модуль в этот момент в аварийной ветке — отказать ему в закрытии значит подвесить доставку из-за длины строки). Диагностика для человека: брокер текст не парсит и не классифицирует. **Не история ошибок** — только последняя причина: промежуточные сбои брокеру не сообщаются вовсе (при неудаче модуль просто не подтверждает, захват протухает). Живёт до очистки `dead` в retention (§3.5). |
| `claimToken` | string \| null | **Метка текущего захвата.** Генерируется заново (`nanoid`, `@app/nanoid`) при **каждом** `claim`, включая перезабор просроченной, и отдаётся подписчику вместе с доставкой. `ackDelivery`/`deadDelivery` обязаны предъявить её обратно; не совпала с текущей — закрытие отклоняется как перехваченное (§5.9, О6). До первого claim — не задан. После терминального статуса **сохраняется** (не обнуляется) — на ней держится идемпотентность повторного закрытия. |

Жизненный цикл (ведёт подписчик, не брокер):

1. **fan-out** при публикации создаёт строки `pending` по подписчикам, копируя в каждую снимок события (`eventType` + `schemaVersion` + `payload`). Идемпотентно по естественному ключу `(eventId, subscriberModuleKey)` (под `runWithExclusiveLock` по событию: `findOneBy` пары → создать недостающие) — повторный fan-out после сбоя (`dispatchedAt = null`, §3.2) не плодит дубли. По завершении всех строк брокер ставит `dispatchedAt` события.
2. На **pull** подписчик выбирает свои `pending` + просроченные `claimed` (порядок — по `createdAt` доставки ≈ время публикации, best-effort §1), помечает их `claimed` с текущим `claimedAt`, **новой `claimToken`** и инкрементом `claimCount` (одна запись), кладёт в свою внутреннюю очередь и обрабатывает **прямо по строке** (payload уже в ней).
3. Успех → `acked` (с предъявлением своей `claimToken`). Упал/не подтвердил → строка осталась `claimed` и вернётся в выдачу на следующем pull после истечения эффективного таймаута (§3.1); перезабор выдаёт **новую** метку, и отчёт по старой уже не примут (§5.9, О6).
4. Число попыток, backoff и очередь обработки — **политика модуля** (вне брокера, описывается в pull-части модуля). Исчерпав попытки, модуль ставит терминальный `dead`. Брокер сам не ретраит и повторный push не шлёт.

Таблица выше — **полный объявляемый состав** (диагностические `claimCount`/`lastError` зафиксированы в §5.9, О7). Поля `attempts` нет намеренно: счётчик попыток обработки брокеру недоступен — он видит только выдачи, это и есть `claimCount`. Поля `subscriptionId` нет: отдельной сущности-подписки не существует, доставка матчится по типу через whitelist `allowedSubscribeTypes` ([ADR-0008](../ADR/0008-subscription-routing-type-only.md)); при нужде трассировать — `eventType` уже в строке. `eventCreatedAt` не нужен: порядок берётся по `createdAt` самой доставки, дозагрузка события не выполняется.

**Системные поля** (Heap, §3.1): `id` — ключ строки доставки; `createdAt` — момент материализации (основа retention); `updatedAt` — момент последнего перехода статуса.

`BrokerDeliveries` — самая быстрорастущая таблица брокера (один fan-out → N строк по числу подписчиков), поэтому подпадает под retention-джоб (§3.5): `acked` старше 24 часов **удаляются**, `dead` **удаляются** после окна хранения (не архивируются — оперативная диагностика, а не история; сам факт события хранит `BrokerEvents`), `pending`/`claimed` не трогаются.

### 3.4. `BrokerEventsArchive` — холодный архив журнала

Неизменяемый журнал (§3.2) не влезает в лимит **1 млн строк на таблицу** (`inner/docs/008-heap.md`, тарифная граница), а удалять события нельзя → двухуровневое хранение. **Модель ([ADR-0011](../ADR/0011-event-history-hybrid-heap-clickhouse.md)):** два уровня Heap как **якорь истины** — горячая `BrokerEvents` (~последний месяц + небольшой «хвост») → **инлайновый архив** здесь (события старше месяца, упакованы по размеру ~8 КБ, данные в самой таблице). Перекладку делает retention-джоб (§3.5, проход 2). Обоснование уровней и размерной упаковки — [ADR-0005](../ADR/0005-two-tier-event-storage.md).

**ClickHouse-зеркало (вторичная копия, не источник истины).** Параллельно якорю fan-out-дренер (§5.8) **best-effort зеркалит** каждое событие в ClickHouse (`writeWorkspaceEvent`/`writeMetricEvent`, `inner/docs/049-clickhouse.md`). Назначение — **аналитика/дашборды** (throughput по `eventType`, объём fan-out, доля dead-letter, лаг «публикация → доставка») и **удобный (негарантирующий) реплей/аудит** через `queryAi`. Зеркало **не авторитетно**: запись best-effort/эвентуальна и без подтверждения коммита, повтор дренера может задублировать — поэтому его **потеря/дубль на корректность брокера не влияют** (правда — в Heap-якоре), точные числа берут из Heap, а по CH делают дедуп по `eventId` (`argMax`/`GROUP BY`) на чтении. `payload` в CH пишется одним непрозрачным JSON-полем (лосслесс-реплей; агрегация по его полям — вне задач зеркала). **Удаление события из Heap на CH-запись не завязывается** — якорная копия всегда остаётся в Heap (§3.5).

| Поле | Тип | Назначение |
|------|-----|-----------|
| `batchFrom` | number | Минимальный `createdAt` в батче (начало временного диапазона), **epoch ms** — заполняется из системного `Date` через `.getTime()` (§3, соглашение о типах). |
| `batchTo` | number | Максимальный `createdAt` в батче (конец диапазона), **epoch ms** — так же. |
| `count` | number | Число упакованных событий — переменное, сколько уместилось в ~8 КБ. |
| `events` | any | Упакованный массив архивируемых строк `BrokerEvents` (JSON, с сохранением исходных `id`). Данные лежат в самой таблице и запрашиваются через `where`. |

Системные поля: `id` — ключ строки архива; `createdAt` — момент архивации; `updatedAt` не используется (архив неизменяем).

**Упаковка инлайновой строки — по размеру, целевой ~8 КБ.** События добавляются, пока набранный размер (сумма `JSON.stringify(event).length`, символы, O(n)) вмещает очередное и не достигнут потолок `MAX_COUNT`; иначе строка сбрасывается. Сбрасываем **только полную** строку — признак полноты «очередное событие не влезло либо `MAX_COUNT`» (набранный размер всегда чуть меньше таргета, поэтому сравнивать сумму с таргетом на «≥» нельзя); неполный «хвост» ждёт в горячей до накопления (отсюда «хвост» сверх последнего месяца). Байт-точность не нужна — таргет с запасом под 8 КБ (`Buffer`/`TextEncoder` в UGC может не быть, `047-base64.md`). Число событий в строке переменное (обычно единицы-десятки). Ёмкость архива — порядка единиц-десятков млн событий: при упаковке по ~8 КБ лимит в 1 млн строк соответствует такому объёму, и для масштаба FLOW он недостижим.

**Что делать при приближении к лимиту таблицы.** Если архив всё же начнёт подходить к 1 млн строк, потребуется вынос старых партий за пределы Heap — в файловое хранилище либо в другое хранилище целиком. Механизм не спроектирован: на текущем масштабе он не нужен, а проектировать его заранее значило бы тащить в спеку конструкцию, которая устареет раньше, чем понадобится. Пункт зафиксирован в §0.1.

**Архив — только история; на рантайме доставки не читаются** (доставка самодостаточна, §3.3 / [ADR-0006](../ADR/0006-fat-delivery-payload-snapshot.md)). Журнал и архив — **неизменяемая история**: аудит, повторная материализация (новый подписчик / ре-подписка), ручной разбор. Читаются только диагностикой и админкой.

Строка архива читается по диапазону `batchFrom`/`batchTo` со сканом ≤1000-батча в коде. Запрос `where` внутрь `any`-поля Heap умеет (вложенный путь, частичный матч, `$includes` по вложенным **скалярным** массивам — `008-heap.md`, «Поля AnyKind»), но `events` здесь — **массив объектов**, а матч элемента массива-объектов по полю поддержкой не подтверждён; к тому же запрос внутрь JSON неиндексирован (скан). Поэтому остаёмся на «диапазон + скан батча в коде»; при надобности точечной выборки — типизировать поле и проверить на рантайме.

`eventId` на доставке (§3.3) — обратная ссылка на журнал для трассировки/аудита, не для дозагрузки данных.

### 3.5. Retention-джоб брокера (ежедневно в полночь)

Один джоб `@app/jobs` (`app.job`) обслуживает растущие таблицы и запускается **каждую полночь**, в конце сам планируя следующий запуск на ближайшие `00:00` через `scheduleJobAt` (самоперепланирование; таймзона аккаунта). Первый запуск — bootstrap при деплое / разовым админ-действием. **Два прохода:** очистка доставок и архивация событий.

**Проход 1 — очистка доставок (`BrokerDeliveries`).** Два удаления, каждое одним вызовом `deleteAll` с `limit: null` и `hard: true`:

- `acked` старше **24 часов** — успешно доставлены, ценности не несут: `where({ status: 'acked', createdAt: { $lt: new Date(now - DAY_MS) } })`;
- `dead` старше **окна хранения** (30 дней) — оперативная диагностика с истёкшим сроком; **не архивируются** (сам факт события хранит `BrokerEvents`): `where({ status: 'dead', updatedAt: { $lt: new Date(now - MONTH_MS) } })`.

Пороги здесь — объекты `Date`, а не числа: сравнение идёт с системными полями (§3, соглашение о типах).

`pending`/`claimed` (в работе / ждут перезабора) не трогаются.

**Почему `limit: null`, а не батчи.** Числовой `limit` у `deleteAll` — это **предохранитель**, а не размер партии: если под `where` попадает больше строк, чем указано, вызов падает с `Accidental mass-delete protection` и не удаляет **ничего** (проверено прогоном 20-07-2026; `008-heap.md` в одном месте описывает это верно, в другом — как «удалит максимум N», см. правку доков). Поэтому «удалять партиями по 1000 в цикле» не работает в принципе — джоб упал бы на первом же прогоне, где накопилось больше тысячи строк, то есть ровно тогда, когда он нужен.

`limit: null` снимает вопрос: удаляется всё подходящее одним вызовом (проверено на 1001 строке — скрытого лимита в 1000 нет). Побочно это **устойчивее** любого числового варианта: предохранителю нечего срабатывать, поэтому рост множества между вызовами безразличен. Рекомендуемый документацией `countBy` → `limit: count` этим свойством не обладает — между двумя вызовами доставка, созданная 25 часов назад, может быть подтверждена и попасть под то же условие, после чего предохранитель сработает.

**`hard: true` обязателен.** По умолчанию `hard = false`. Retention существует, чтобы таблица не упиралась в лимит строк, а не чтобы прятать записи из выдачи, — поэтому мягкое удаление здесь бессмысленно: строки останутся, и лимит будет достигнут при формально работающей чистке.

**Нарезка по временным окнам сознательно не вводится.** Она понадобилась бы, только если удаление перестанет укладываться в лимит джоба (60 с). Оценивать это заранее нечем, а строить машинерию под неизмеренный риск — та же ошибка, что и преждевременный дизайн выноса архива (§3.4). Вместо этого джоб **логирует число удалённых строк и длительность прохода**: если однажды длительность начнёт приближаться к лимиту, вопрос вернётся с реальными цифрами на руках.

**Проход 2 — архивация журнала событий старше месяца, строки по размеру ~8 КБ.** События из `BrokerEvents` (§3.2) старше **1 месяца** переносятся в холодный архив `BrokerEventsArchive` (§3.4) строками, набранными по размеру (~8 КБ), затем удаляются из горячей таблицы.

Алгоритм: тянуть самые старые строки (`updatedAt < new Date(now - MONTH_MS)`, порядок `updatedAt asc`; для неизменяемого журнала `updatedAt = createdAt`, §3.2), набирать батч, пока очередное событие влезает в **целевые ~8 КБ** (и не достигнут потолок `MAX_COUNT`). Строка считается **полной**, когда остался невлезший кандидат либо набран `MAX_COUNT` (§3.4). Если все старые события поместились в одну недобранную строку — **ничего не архивируем** (остаток ждёт в горячей таблице). Иначе на каждый набранный полный батч: создать строку архива → **только после успешной записи архива** удалить исходные строки по их `id`. Порядок «сначала архив, потом удаление» исключает потерю при сбое (худший случай — повтор архивации того же диапазона, а не утрата). Цикл — до исчерпания полных строк либо до лимита строк-архива за прогон (ограничивает время джоба; остаток догоняется следующими полуночами).


```ts
const DAY_MS = 24 * 60 * 60 * 1000
const MONTH_MS = 30 * DAY_MS
const SIZE_TARGET = 7 * 1024 // ~7 КБ символов на строку архива — с запасом под оптимум записи Heap ~8 КБ (008-heap.md)
const MAX_COUNT = 1000 // защитный потолок числа событий в строке
const FETCH = 1000 // максимум выборки findAll за раз (008-heap.md); хватает на одну строку-архива за итерацию
const MAX_ARCHIVE_ROWS_PER_RUN = 50 // предел времени прогона; остаток догонит следующая полночь

export const brokerRetentionJob = app.job('/broker/retention', async (ctx) => {
  const now = Date.now()
  const startedAt = now

  // Проход 1: очистка доставок — acked старше 24ч и dead старше 30 дней.
  // limit: null — удаляем всё подходящее одним вызовом; числовой limit это
  // предохранитель (падает, если строк больше), а не размер партии.
  // hard: true — иначе строки останутся в таблице и лимит будет достигнут.
  const ackCutoff = new Date(now - DAY_MS)
  const deadCutoff = new Date(now - MONTH_MS)
  const purge = (where) => BrokerDeliveries.deleteAll(ctx, { where, limit: null, hard: true })

  const deletedDeliveries =
    (await purge({ status: 'acked', createdAt: { $lt: ackCutoff } })) +
    (await purge({ status: 'dead', updatedAt: { $lt: deadCutoff } }))

  // Проход 2: архивация журнала событий строками по размеру ~8 КБ
  const archiveCutoff = new Date(now - MONTH_MS)
  let archivedEvents = 0
  for (let row = 0; row < MAX_ARCHIVE_ROWS_PER_RUN; row++) {
    // самые старые кандидаты (для журнала updatedAt = createdAt)
    const candidates = await BrokerEvents.findAll(ctx, {
      where: { updatedAt: { $lt: archiveCutoff } },
      order: [{ updatedAt: 'asc' }],
      limit: FETCH,
    })
    if (candidates.length === 0) break

    // набираем одну строку архива по размеру ~8 КБ (или до потолка MAX_COUNT)
    const pack = []
    let size = 0
    for (const ev of candidates) {
      const evSize = JSON.stringify(ev).length
      if (pack.length > 0 && (size + evSize > SIZE_TARGET || pack.length >= MAX_COUNT)) break
      pack.push(ev)
      size += evSize
    }

    // строка полная, если очередной кандидат в неё уже не влез (по размеру) либо достигнут потолок MAX_COUNT;
    // набранный size при этом чуть МЕНЬШЕ таргета — сравнивать size с SIZE_TARGET нельзя (почти никогда не равен).
    // неполный «хвост» (все кандидаты поместились, а новых нет) оставляем в горячей до накопления
    const full = pack.length < candidates.length || pack.length >= MAX_COUNT
    if (!full) break

    // сначала пишем архив...
    await BrokerEventsArchive.create(ctx, {
      // createdAt — системное поле Date, batchFrom/batchTo — наши number (epoch ms): конвертируем явно (§3)
      batchFrom: pack[0].createdAt.getTime(),
      batchTo: pack[pack.length - 1].createdAt.getTime(),
      count: pack.length,
      events: pack, // JSON в самой таблице (запрашивается через where), не выносим в ctx.storage
    })
    // ...и только потом удаляем исходные строки по id
    // limit здесь безопасен: where по списку id попадает ровно в pack.length строк,
    // предохранитель срабатывает только когда подходящих БОЛЬШЕ лимита.
    // hard: true — иначе строки останутся и архивация не разгрузит таблицу
    await BrokerEvents.deleteAll(ctx, {
      where: { id: pack.map((e) => e.id) },
      limit: pack.length,
      hard: true,
    })
    archivedEvents += pack.length
  }

  // durationMs — не для отчётности, а чтобы знать, когда понадобится
  // нарезка удаления: приближение к лимиту джоба (60 с) и есть сигнал
  ctx.account.log('broker retention', {
    level: 'info',
    json: { deletedDeliveries, archivedEvents, durationMs: Date.now() - startedAt },
  })

  // следующий запуск — в ближайшую полночь (таймзона аккаунта)
  const next = new Date()
  next.setHours(24, 0, 0, 0)
  await brokerRetentionJob.scheduleJobAt(ctx, next, {})
})

```

Все Heap-таблицы брокера согласованы: `BrokerModules` (§3.1), `BrokerEvents` (§3.2), `BrokerDeliveries` (§3.3), `BrokerEventsArchive` (§3.4). Отдельных таблиц под контракты, подписки и операционный аудит **не заводится**: контракт `payload` — вне брокера (§1, §8); подписка — whitelist по типу на строке модуля ([ADR-0008](../ADR/0008-subscription-routing-type-only.md)); операционная история — системные `broker.*` события в журнале ([ADR-0009](../ADR/0009-operational-audit-events.md)).

### Операции гейтвея и эндпоинты

> Имена ниже — предложенные (на ревью вместе с §5.3/§5.4). Гейтвей `core` — единственная точка записи в `BrokerModules`; модуль обращается только к этим операциям.

Ниже — операции **реестра модулей**. Остальные операции модульного API описаны в своих разделах: публикация — §5.8, pull-API подписчика (`fetchDeliveries`, `ackDelivery`, `deadDelivery`) — §5.9.1. У всех у них та же пара транспортов.

Каждая операция имеет **два транспортных эндпоинта** на одну и ту же логику; от того, какой из них вызван, гейтвей выводит `source` (§5.2):

| Операция | Назначение | internal — `app.function` | external — HTTP-роут |
|----------|------------|---------------------------|----------------------|
| `registerModule` | Регистрация, только create (§5.2) | `registerModule` | `POST /broker/register` |
| `updatePublishTypes` | Обновление `allowedPublishTypes`, только mutate (§5.3) | `updatePublishTypes` | `POST /broker/publish-types` |
| `updateSubscribeTypes` | Обновление `allowedSubscribeTypes`, только mutate (§5.4) | `updateSubscribeTypes` | `POST /broker/subscribe-types` |
| `deleteModule` | Удаление регистрации, только existing (§5.5) | `deleteModule` | `POST /broker/delete` |

- Вызов через internal-эндпоинт (`app.function`) → `source = internal`; через external HTTP-роут → `source = external`. На обновлениях (§5.3/§5.4) `source` берётся из строки и каналом **не** переопределяется — канал там влияет только на политику модерации.
- `registerModule` токен **не** принимает (только создаёт строку); `updatePublishTypes` / `updateSubscribeTypes` токен **требуют** и работают лишь по существующей строке. Объединять их в один эндпоинт с опциональным токеном решено **не** (conditional auth на границе доверия).
- **Админ-операции — третий, отдельный канал, вне развилки `source`.** Модерация (§5.6) и выключение/включение модуля (§5.7) вызываются не модулем, а админ-страницей самого брокера: серверный роут с `// @shared-route`, который клиент дёргает через `.run()`. У них нет ни `app.function`-, ни HTTP-эндпоинта для модулей, они не принимают токен модуля и защищены `requireAccountRole(ctx, 'Admin')`. `source` к ним неприменим в принципе — он описывает происхождение **модуля**, а не вызова.

### 5.1. Хэширование auth-токена (SHA-256)

**Решение (зафиксировано):** `authTokenHash` вычисляется настоящим SHA-256 через `@npm/node-forge`. Обоснование выбора — что вообще доступно в Chatium и почему именно SHA-256, а не шифрование / FNV-1a / медленные KDF / HMAC — в [ADR-0002](../ADR/0002-token-hashing-sha256.md).

Правила реализации:

- **Зависимость.** В `package.json` workspace должен быть объявлен `"@npm/node-forge": "*"` (пакет в платформенном белом списке; по умолчанию в проекте не подключён — рядом объявлены только `@npm/date-fns*`). Доступность подтверждается типчеком/деплоем перед первым использованием.
- **Алгоритм.** `forge.md.sha256` → шестнадцатеричная строка из 64 символов.
- **Вход.** Хэшируется доменно-разделённая строка `broker-module-auth:<moduleKey>:<authToken>`: префикс отделяет пространство имён брокера, `moduleKey` привязывает хэш к конкретному модулю (защита от переноса хэша между модулями).
- **Хранение.** Результат пишется в выделенное поле `authTokenHash` строки `BrokerModules` (§3.1). Сам токен в открытом виде нигде не хранится и не логируется.

```ts
import * as forge from '@npm/node-forge'

function moduleAuthHash(moduleKey: string, authToken: string): string {
  const md = forge.md.sha256.create()
  md.update(`broker-module-auth:${moduleKey}:${authToken}`)
  return md.digest().toHex() // 64 hex-символа
}
```

### 5.2. Регистрация модуля

Модуль не пишет себя в `BrokerModules` напрямую — он регистрируется через гейтвей, который определяет источник, проверяет заявку и выдаёт токен доступа.

Шаги:

1. Модуль вызывает операцию `registerModule` (internal `app.function` или external `POST /broker/register`), передавая `moduleKey`, `allowedPublishTypes`, `allowedSubscribeTypes` и **опционально** `claimTimeoutMs` (переопределение таймаута перезабора; не передан → дефолт брокера 1 минута, §3.1/§3.3). Токен не передаётся — операция только создаёт строку.
2. **Источник определяет гейтвей по каналу запроса, не из тела:**
   - запрос пришёл внутренним `app.function`-вызовом → `source = internal`;
   - запрос пришёл на внешний HTTP-роут (`app.post /broker/register`) → `source = external`.

   **Почему это надёжно.** Дело не в конкретном способе роутинга, а в том, что **канал нельзя подделать**: `app.function` доступен только коду внутри платформы, внешний HTTP-роут — всем. Поэтому назваться `internal`, придя снаружи, невозможно, и `source` не нужно принимать на веру из тела запроса. Это же правило определяет, куда относить любой новый транспорт, если он появится: доступен ли он извне периметра.
3. Гейтвей валидирует заявку: корректность glob-синтаксиса заявленных `allowedPublishTypes` / `allowedSubscribeTypes` — синтаксис **посегментный** (§3.1): каждый dot-сегмент это литерал или `*` целиком; частично-сегментные паттерны (`tasks.cr*`), `**` и регэкспы **отвергаются** (это гарантирует перечислимость предков для fan-out, [ADR-0008](../ADR/0008-subscription-routing-type-only.md)). Зарезервировано системой (§3.2): `moduleKey = broker` и префикс `broker.` в `allowedPublishTypes` — отвергаются (в `allowedSubscribeTypes` `broker.` разрешён — аудит-подписка). **Эксклюзивности типа нет** — один и тот же тип события могут публиковать несколько модулей (multi-producer: например, гейты к разным банкам, эмитящие `payment.created`). Продюсер для подписчика **анонимен** ([ADR-0008](../ADR/0008-subscription-routing-type-only.md)): брокер источник в доставку не кладёт и по нему не роутит. Если подписчику нужно различать источник — продюсер **добровольно** помещает свою идентификацию в `payload`; это контракт между продюсером и потребителем, брокер его **не регулирует**. Отдельного правила «занятости» не вводится: спуфинг типов закрывается модерацией `external` (шаг 4) и запретом повторной регистрации (см. ниже).
4. При успешной валидации гейтвей:
   - генерирует случайный **токен доступа**, сохраняет в `authTokenHash` только его `SHA-256` (§5.1), а сам токен возвращает модулю один раз;
   - фиксирует строку модуля с проставленным `source` и начальным `status` по источнику:
     - `internal` → `status = active` (активен сразу);
     - `external` → `status = onModeration` («На модерации»); заявка попадает на **однократную модерацию** в админке гейтвея, до одобрения модуль в обмене не участвует.
5. Токен модуль хранит у себя и предъявляет при каждой операции; гейтвей хэширует входящий токен и сверяет с `authTokenHash`. Хэш в БД при утечке рабочего креденциала не даёт.

**Токен выдаётся всем модулям одинаково — и `internal`, и `external`.** Шаг 4 не ветвится по источнику. `source` на аутентификацию не влияет вообще: его единственная роль — политика модерации (начальный статус здесь и необходимость одобрения при расширении наборов, §5.3/§5.4).

Изменение набора разрешённых типов после регистрации — отдельные операции: публикации — §5.3, подписки — §5.4. **Повторная регистрация уже существующего `moduleKey` запрещена**: `registerModule` на занятый ключ → отказ; существующая строка только обновляется (§5.3/§5.4) или удаляется (§5.5). Это и снимает спуфинг — нельзя пере-зарегистрировать чужой модуль на новый токен/набор типов в обход модерации. Ротация токена (отдельно от update) — §5, в проработке.

### 5.3. Обновление `allowedPublishTypes`

Смена набора типов, которые модулю разрешено публиковать, — самостоятельная операция, а не повторная регистрация: `moduleKey`, `source`, `authTokenHash` и токен не меняются.

Шаги:

1. Модуль вызывает операцию `updatePublishTypes` (internal `app.function` или external `POST /broker/publish-types`) с новым полным набором `allowedPublishTypes` (полная замена, не дельта) и предъявляет свой токен. Операция применима только к **существующей** строке `moduleKey`; создание идёт исключительно через `registerModule` (§5.2).
2. **Аутентификация.** Гейтвей хэширует входящий токен и сверяет с `authTokenHash` строки `moduleKey` (§5.1). Чужой/неверный токен → отказ. `source` берётся из существующей строки, **не** переопределяется каналом запроса (в отличие от регистрации `source` неизменяем).
3. **Валидация.** Проверяется корректность glob-синтаксиса — **посегментного** (§3.1/§5.2: частично-сегментные паттерны и регэкспы отвергаются); префикс `broker.` в `allowedPublishTypes` **отвергается** — зарезервированный системный namespace (§3.2). Несколько модулей могут публиковать один тип (multi-producer — см. §5.2, шаг 3).
4. **Модерация по `source`:**
   - `internal` → изменение применяется сразу, `status` не меняется;
   - `external` · **сужение** (только удаление паттернов) → применяется сразу, без модерации;
   - `external` · **расширение** (есть паттерн, которого не было) → заявка уходит на модерацию: гейтвей пишет запрошенный набор в **`pendingPublishTypes`**, а боевой `allowedPublishTypes` и `status` **не трогает**. Модуль продолжает работать на одобренных правах; новые вступают в силу только после approve (§5.6).
5. **Применение.** Одобренное изменение (`internal`, либо `external`-сужение) пишется в `allowedPublishTypes`; запрошенное `external`-расширение — в `pendingPublishTypes`. В обоих случаях обновляется `updatedAt`, изменение фиксируется `broker.*` событием (§3.2/[ADR-0009](../ADR/0009-operational-audit-events.md)) — история живёт в событии, не на строке. `status` при обновлении наборов не меняется никогда.

Пустой массив = публикация запрещена (как в §3.1): корректный способ полностью отозвать право на публикацию.

### 5.4. Обновление `allowedSubscribeTypes`

Смена набора типов, на которые модуль подписан, — самостоятельная операция; `moduleKey`, `source`, `authTokenHash` и токен не меняются.

Шаги:

1. Модуль вызывает операцию `updateSubscribeTypes` (internal `app.function` или external `POST /broker/subscribe-types`) с новым полным набором `allowedSubscribeTypes` (полная замена) и предъявляет токен. Операция применима только к **существующей** строке `moduleKey`; создание — исключительно через `registerModule` (§5.2).
2. **Аутентификация** — идентично §5.3 (сверка хэша токена; `source` из строки, не из канала).
3. **Валидация.** Проверяется корректность glob-паттернов — **посегментный** синтаксис (§3.1/§5.2: частично-сегментные паттерны и регэкспы отвергаются; это условие перечислимости предков на fan-out).
4. **Модерация по `source`:**
   - `internal` → применяется сразу, `status` не меняется;
   - `external` · **сужение** → применяется сразу, без модерации;
   - `external` · **расширение** → на модерацию (подписка на новый тип = доступ к новым данным, граница доверия): запрошенный набор пишется в **`pendingSubscribeTypes`**, боевой `allowedSubscribeTypes` не меняется. **Ключевое следствие:** fan-out по-прежнему резолвит модуль по одобренному набору, поэтому его текущие подписки продолжают работать, а доставки — забираться и подтверждаться (§5.9, О5). Не одобрено — просто не добавилось.
5. **Применение** — идентично §5.3 (одобренное в боевое поле, запрошенное в `pending*`, `updatedAt`, `broker.*` событие-история, `status` не меняется).

Пустой массив = подписка запрещена (§3.1).

### 5.5. Удаление регистрации модуля

Удаление снимает модуль с обмена и **освобождает `moduleKey`**: после удаления ключ снова можно занять через `registerModule` (§5.2). Это и есть способ «переоформить» модуль, раз повторная регистрация занятого ключа запрещена (§5.2).

Шаги:

1. Модуль вызывает операцию `deleteModule` (internal `app.function` или external `POST /broker/delete`), передавая `moduleKey` и предъявляя токен. Операция применима только к **существующей** строке.
2. **Аутентификация.** Гейтвей хэширует входящий токен и сверяет с `authTokenHash` строки `moduleKey` (§5.1). Чужой/неверный токен → отказ. Несуществующий `moduleKey` → отказ (нечего удалять).
3. **Без модерации.** Удаление — сокращение собственного присутствия модуля (как «сужение» в §5.3/§5.4), не расширение прав; и для `internal`, и для `external` применяется сразу. Админ-инициированное снятие — это статус `disabled` («Выключен», §3.1), отдельный механизм, не `deleteModule`.
4. **Применение.** Строка `BrokerModules` удаляется (состояние на строке — только текущее, §3.1). Факт удаления фиксируется отдельным событием брокера — история «когда / кто» живёт в событии, не на строке; сам `moduleKey` после этого свободен.

**Каскад при удалении (решено, конкретизировано по §3.2/§3.3):**

- **События** (`BrokerEvents` §3.2 и архив §3.4) — **остаются**: исторический факт; `producerModuleKey` хранится обычной строкой, ссылочной целостности нет, удаление строки модуля журнал не задевает.
- **Доставки** (`BrokerDeliveries` §3.3) — удаляются **все строки с `subscriberModuleKey` удаляемого модуля, независимо от статуса**: `deleteAll({ where: { subscriberModuleKey }, limit: null, hard: true })`. Причины: рабочая очередь и dead-letter имеют смысл только при живом подписчике; главное — `moduleKey` после удаления **освобождается**, и оставленные строки при повторном занятии ключа (§5.2) мисатрибутировались бы новому модулю. Доставки, где модуль был лишь продюсером события, не трогаются — они принадлежат подписчикам.

  `limit: null` и `hard: true` — по тем же причинам, что в §3.5: числовой `limit` это предохранитель, а не размер партии, а мягкое удаление оставило бы строки, из-за чего освободившийся `moduleKey` при повторной регистрации получил бы чужой хвост.

  **Удаление доставок здесь обязано быть синхронным, и это не про удобство.** Оно завершается **до** того, как `moduleKey` станет свободен — иначе между освобождением ключа и концом уборки открывается окно, в котором новый владелец того же ключа получает **чужую очередь**: он аутентифицируется своим токеном, проходит гейт статуса, а выборка `fetchDeliveries` идёт по `subscriberModuleKey`, который у него теперь тот же самый. Ему выдадут доставки предшественника, включая события типов, на которые он не подписан.

  **Отличие от retention — лимит времени:** здесь 20 секунд (вызов из запроса), а не 60. Оценка объёма: retention уже снёс `acked` старше суток и `dead` за окном, поэтому у модуля стоят только `pending`, `claimed`, сутки подтверждённых и `dead` в пределах окна — для масштаба FLOW сотни строк. Большая очередь означает не нагрузку, а сломанного подписчика, который давно не разгребает.

  ⚠️ **Если каскад когда-нибудь перестанет укладываться в 20 с — наивный вынос в джоб («снять строку модуля сразу, доставки почистить фоном») открывает описанное окно.** Асинхронная версия обязана закрыть его целиком: (1) джоб удаляет с отсечкой `createdAt < deletedAt`, иначе снесёт доставки нового владельца ключа; и (2) ключ **не освобождается** до конца уборки — либо строка модуля живёт до завершения джоба, либо регистрация занятого-в-уборке ключа отклоняется. Пункт (1) без пункта (2) проблему не решает: он защищает уборку от переудаления, но не новый модуль от чтения старых строк.
- **Подписки** — отдельной сущности нет: список подписки живёт полем `allowedSubscribeTypes` на строке `BrokerModules` и удаляется **вместе с ней** ([ADR-0008](../ADR/0008-subscription-routing-type-only.md)). Отдельного каскада не требуется.

### 5.6. Модерация: очередь, аппрув и дезлайн

**Очередь модерации собирается по двум разным признакам** (оба — только `external`; у `internal` всё применяется сразу, в модерацию он не попадает). Идентификатор сущности — `moduleKey`.

| Случай | Признак | Состояние модуля | Что рассматривает админ |
|--------|---------|------------------|--------------------------|
| **Первичная заявка** | `status = 'onModeration'` | не запущен, доставок не получал | набор, заявленный при регистрации (§5.2) |
| **Заявка на расширение** | непустой `pendingPublishTypes` / `pendingSubscribeTypes` | **работает как обычно**, `status = 'active'` | запрошенное расширение (§5.3/§5.4) |

Разница принципиальная и раньше терялась. Первичная заявка — это модуль, которого **ещё нет** в обмене: одобрение его запускает. Заявка на расширение — модуль, который **уже работает** по одобренным правам: одобрение лишь добавляет ему новые. Поэтому первый случай выражается статусом (операционное состояние), а второй — отдельным полем, и работу модуля не затрагивает.

Прежде оба случая кодировались статусом (`onModeration` / `reModeration`), из-за чего запрос расширения останавливал работающий модуль до решения админа — побочный эффект, а не замысел. Поправка к [ADR-0010](../ADR/0010-module-status-enum.md) от 19-07-2026 разводит эти оси; значение `reModeration` упразднено.

> **§5.6 ещё требует расширения:** фиксированные запросы каждой операции (Request/Response контракты) и примеры с реальными полями из `BrokerModules`; пока здесь только логика и псевдокод.

Операции (admin-only, `requireAccountRole(ctx, 'Admin')`). Это не модульный API — у `internal`/`external` модулей к ним нет доступа в принципе. Вызываются из админ-страницы самого брокера: серверный роут с `// @shared-route`, клиент дёргает его напрямую через `.run()`, без `app.function` и без отдельного HTTP-эндпоинта для модулей.

| Операция | Назначение | Роут (`// @shared-route`) |
|----------|------------|----------------------------|
| `listPendingModeration` | Модули, ожидающие решения | `/broker/admin/moderation` |
| `approveModeration` | Одобрить модуль по `moduleKey` | `/broker/admin/moderation/approve` |
| `declineModeration` | Отклонить модуль по `moduleKey` | `/broker/admin/moderation/decline` |

- **`listPendingModeration`** — объединяет оба признака: `status = 'onModeration'` **или** непустой `pending*`. Условие с `$or` — форму `where` см. §5.9, О4 (смешивать поле и `$or` на одном уровне типы не позволяют, нужен явный `$and`). Возвращает `moduleKey`, случай (первичная / расширение), рассматриваемый набор (заявленный при регистрации либо `pending*`) и — для расширений — текущий одобренный набор, чтобы админ видел разницу. Плюс `updatedAt`.
- **`approveModeration(moduleKey)`** — действие зависит **от случая**:
  - первичная заявка (`status = 'onModeration'`) → `status = 'active'`, модуль допущен в обмен;
  - заявка на расширение → `pending*` переносится в соответствующее боевое поле (`allowedPublishTypes` / `allowedSubscribeTypes`), `pending*` очищается. `status` не трогается — модуль и так `active`. С этого момента fan-out начинает резолвить его и по новым типам.

  Решение фиксируется `broker.*` событием (кто/когда — в событии, не на строке, §3.2).
- **`declineModeration(moduleKey, reason?)`** — действие зависит **от случая**:
  - первичная заявка → строка `BrokerModules` удаляется, `moduleKey` освобождается (модуль может подать заявку заново; ср. §5.5);
  - заявка на расширение → `pending*` очищается, и на этом всё. Модуль как работал на одобренных правах, так и работает — **откат теперь автоматический и полный**, потому что прежний набор никуда не девался. (Раньше отката не было вовсе: запрошенный набор затирал одобренный, и модуль оставался приостановленным до тех пор, пока не подаст корректную заявку заново.)

  `reason` — необязательная причина для диагностики; решение фиксируется событием брокера.

**Повторная заявка до решения.** Модуль вправе подать расширение ещё раз, пока предыдущее висит: `pending*` перезаписывается целиком (наборы всегда полные, не дельты — §5.3/§5.4). Админ видит последнюю версию заявки.

### 5.7. Выключение и включение модуля админом

Помимо модерации (§5.6), админ может **приостановить** и **вернуть** уже работающий модуль — независимо от его набора типов и без ре-модерации. Обе операции — admin-only (`requireAccountRole(ctx, 'Admin')`), вызываются из админ-страницы брокера так же, как §5.6 (роут `// @shared-route`, `.run()`), не модульный API.

| Операция | Переход | Роут (`// @shared-route`) |
|----------|---------|----------------------------|
| `disableModule` | `active` → `disabled` | `/broker/admin/disable` |
| `enableModule` | `disabled` → `active` | `/broker/admin/enable` |

- **`disableModule(moduleKey, reason?)`** — переводит `active` → `disabled` («Выключен», §3.1). Применяется **только из `active`**: модуль в `onModeration` и так не участвует (для отклонения первичной заявки — `declineModeration` §5.6; для полного снятия — `deleteModule` модулем, §5.5). Висящая заявка на расширение выключению не мешает и им не отменяется: `pending*` остаётся, и админ может решить по ней отдельно — оси независимы (§5.6). Выключенный модуль в обмене не участвует: его публикации отклоняются, при fan-out доставки ему **не создаются** (резолв подписчиков берёт только `status = active`, §3.1). Уже созданные доставки остаются и обработаются после включения — доставка самодостаточна. `reason` — опц. диагностика.
- **`enableModule(moduleKey)`** — возвращает `disabled` → `active` на его текущем (уже одобренном) наборе whitelist; ре-модерация не требуется.
- Оба перехода фиксируются `broker.*` событием (§3.2/[ADR-0009](../ADR/0009-operational-audit-events.md)) — кто/когда/почему в событии, не на строке.

### 5.8. Публикация события: `publishEvent` и fan-out-дренер

Центральная операция брокера (§1): производитель сообщает, что произошёл факт. Публикация **двухфазная, fan-out — асинхронный**: `publishEvent` быстро и надёжно **записывает событие**, а материализацию доставок (fan-out) выполняет **дренер** контролируемым темпом. Так брокер поглощает всплески публикаций (тысячи-десятки тысяч запросов) без backpressure на Heap и без потери событий.

**Каналы и авторизация.** Как у прочих пост-регистрационных операций (§5.3): вызывается зарегистрированным продюсером через internal `app.function` или external `POST /broker/publish`; канал — **транспорт**, `source`/идентичность берутся из строки, не из канала (в отличие от `registerModule`). Аутентификация — **по токену, единообразно для всех модулей** (`authTokenHash`, §5.1); `source` на неё не влияет. Требуется `status = active`; применяется `assertCanPublish` (тип матчит `allowedPublishTypes` по glob); тип из зарезервированного namespace `broker.*` от модуля отклоняется (§3.2). Оверсайз `payload` (сверх ~8 КБ потолка строки Heap, §3.2) отклоняется — крупное кладётся ссылкой в файловое хранилище (конкретный порог — в проработке).

**Параметры:** `eventType`, `payload`, `schemaVersion` (опц., default `1`), `idempotencyKey` (опц.).

**Фаза 1 — `publishEvent` (синхронно, минимум работы):**

1. Auth + `assertCanPublish` + проверки выше.
2. **Дедуп** (если задан `idempotencyKey`): под `runWithExclusiveLock((producerModuleKey, idempotencyKey))` — `findOneBy`; при совпадении возвращаем существующее событие, нового не создаём. Окно дедупа — горячая `BrokerEvents` (~месяц ретеншена §3.4); для сетевых/чанковых ретраев продюсера достаточно.
3. Запись строки `BrokerEvents` (`dispatchedAt = null`).
4. Триггер fan-out-дренера asap (`scheduleJobAsap`, §5-jobs).
5. **Возврат** `eventId` (+ признак «новое / дедуп-хит»).

Синхронной работы — минимум (одна запись + дедуп-проверка), поэтому скачок запросов поглощается, а не транслируется в скачок fan-out-записей на Heap.

**Фаза 2 — fan-out-дренер (`app.job`, асинхронно):**

- Под одиночным замком (`tryRunWithExclusiveLock` — если проход уже идёт, выйти; корректность и так держит идемпотентный ключ доставки) сканирует `BrokerEvents where dispatchedAt = null`, старые первыми (`createdAt asc`), батчами **≤1000** (лимит Heap).
- По каждому событию: резолв **активных** подписчиков (`where { status: 'active', allowedSubscribeTypes: { $includes: { $any: [развёртка предков `eventType`] } } }`, §3.1) → создание строк `BrokerDeliveries` со снимком (`eventType`+`schemaVersion`+`payload`), **идемпотентно** по `(eventId, subscriberModuleKey)` (`lock + findOneBy` по событию, §3.3) → простановка `dispatchedAt` → **best-effort зеркалирование события в ClickHouse** (вторичная аналитическая копия, §3.4 / [ADR-0011](../ADR/0011-event-history-hybrid-heap-clickhouse.md); на корректность не влияет, при повторе прохода допускает дубль — дедуп по `eventId` на чтении, `payload` пишется JSON-полем) → best-effort push-триггер подписчикам.
- **Самоперепланирование**, пока в журнале есть `dispatchedAt = null`; backlog пуст — выходит.
- **Он же — recovery.** Отдельного восстановительного джоба нет: любое незавершённое (краш продюсера/дренера) остаётся `dispatchedAt = null` и подхватится следующим проходом. `dispatchedAt = null`-события журнала — это и есть durable-очередь работы fan-out (отдельная `@app/jobs`-очередь тут не нужна, была бы вторым backlog'ом).

**Надёжность.** at-least-once (§1): событие durably в журнале уже после Фазы 1 (источник истины); fan-out гоняется, пока не проставлен `dispatchedAt`. Два слоя идемпотентности (§3.2): `(producerModuleKey, idempotencyKey)` — от дублей при ретраях продюсера; `(eventId, subscriberModuleKey)` — от дублей при повторном fan-out.

**Латентность.** Между Фазой 1 и материализацией доставок — небольшой лаг (событие записано → дренер прогнал → доставки есть → push). Секунды при asap-триггере; приемлемо для pull-driven best-effort (§1).

**Системные `broker.*`** идут той же механикой по внутреннему пути (сентинел `producerModuleKey = 'broker'`, без токена и `assertCanPublish`, §3.2).

**Bulk-публикация.** Одиночный `publishEvent` ограничен по построению (одно событие); цикл ×N у продюсера убивает **его** запрос (лимит вызова роута / `app.function` — **20 секунд**; 60 с относятся только к `app.job`, `005-jobs.md`), не брокер. Batch-`publishEvent` (пачка за вызов) пока **не вводится**: продюсеру, генерящему всплеск (парсинг тысяч сообщений и т.п.), — чанковать свою генерацию **собственной** custom job queue (`inner/docs/005-jobs.md`) + `idempotencyKey` на событие для безопасных ретраев чанка. Если batch понадобится — вводить **с кэпом** размера либо через брокерскую очередь приёма (в проработке).

### 5.9. Получение и обработка доставок (pull-API подписчика)

Потребительская половина round-trip'а: `publishEvent` (§5.8) — входная дверь продюсера, fan-out (§3.2/§3.3) материализует доставки, а **pull-API — выходная дверь подписчика**: как модуль забирает свои доставки, помечает взятыми и закрывает (успех/`dead`). Жизненный цикл уже описан прозой в §3.3 (`pending → claimed → acked/dead`, ведёт подписчик, не брокер) и в глоссарии («poll → claim → ack»); здесь он **оформляется как операции гейтвея** — по образцу §5.2–§5.8 (таблица транспортов internal `app.function` / external HTTP-роут, пошаговый флоу, Request/Response-контракты).

**Что задано ранее и не пересматривается (§3.3):** доставка самодостаточна (снимок `eventType`+`schemaVersion`+`payload`, [ADR-0006](../ADR/0006-fat-delivery-payload-snapshot.md)) — подписчик обрабатывает по строке, журнал не читает. Жизненный цикл ведёт **сам подписчик** через pull; брокер хранит статус и не ретраит. Число попыток, backoff, внутренняя очередь и дедупликация двойной обработки — политика модуля вне брокера (§1).

#### 5.9.1. Операции, каналы и авторизация

**Три операции** (О1): забор — батчем, закрытие — поодиночке.

| Операция | Назначение | Гранулярность | External-роут |
|----------|------------|---------------|---------------|
| `fetchDeliveries` | забрать свои доставки и застолбить их | пачка | `POST /broker/deliveries/fetch` |
| `ackDelivery` | подтвердить успешную обработку | одна | `POST /broker/deliveries/ack` |
| `deadDelivery` | признать необрабатываемой | одна | `POST /broker/deliveries/dead` |

**Каналы.** Как у прочих пост-регистрационных операций (§5.3/§5.8): internal — `app.function`, external — HTTP-роут из таблицы. Канал здесь **транспорт**; `source` и идентичность берутся из строки модуля, не из канала (в отличие от `registerModule`, §5.2).

**Аутентификация** — по токену, единообразно для всех модулей (`authTokenHash`, §5.1); `source` на неё не влияет (О5).

**Гейт статуса** — все три операции требуют `status = 'active'` (О5). `onModeration` не запущен ни разу; `disabled` — админ-стоп, его доставки ждут включения (§5.7). Висящая заявка на расширение (`pending*`, §5.6) работе не мешает: модуль остаётся `active`.

#### 5.9.2. `fetchDeliveries` — забрать и застолбить

**Request:**

| Поле | Тип | Назначение |
|------|-----|-----------|
| `limit` | number (опц.) | Сколько доставок забрать за вызов. По умолчанию `FETCH_LIMIT_DEFAULT = 50`, потолок `FETCH_LIMIT_MAX = 200`; превышение — отсечка до потолка, не отказ. |

**Флоу:**

1. Аутентификация по токену; гейт `status = 'active'`.
2. Вычислить порог просрочки: `cutoff = Date.now() - (claimTimeoutMs ?? DEFAULT_CLAIM_TIMEOUT_MS)` из строки модуля (О2). **Один и тот же `cutoff` используется и в выборке, и в условиях CAS** — иначе окно проверки разъедется.
3. Выборка кандидатов одним запросом (О4): свои `pending` плюс просроченные `claimed`, порядок `createdAt asc`, лимит из Request.
4. По каждому кандидату — условный захват через `updateAll`, где `where` повторяет прочитанное состояние (О3, [ADR-0014](../ADR/0014-claim-race-per-row-cas.md)):

   ```typescript
   const won = await BrokerDeliveries.updateAll(ctx, {
     patch: {
       status: 'claimed',
       claimedAt: now,
       claimToken,                      // новая метка на каждый захват (§3.3)
       claimCount: row.claimCount + 1,  // считаем в коде: patch принимает только литералы
     },
     where: { id: row.id, status: 'pending' },        // для просроченной:
     limit: 1,                                        // { id, status: 'claimed', claimedAt: { $lt: cutoff } }
   })
   ```

5. `won === 1` — строка захвачена, добавляем в результат. `won === 0` — перехватили, **пропускаем** и переходим к следующему кандидату.
6. Вернуть захваченные. Блокировки на этом пути нет: параллельные воркеры одного модуля забирают доставки одновременно и расходятся по непересекающимся строкам.

**Отдано может быть меньше, чем запрошено** — проигранные гонки просто выпадают. Это штатно и саморегулируется: воркер получил меньше работы ровно потому, что её взял другой. Повторять забор внутри того же вызова не нужно.

**Response** — массив доставок, по каждой:

| Поле | Назначение |
|------|-----------|
| `id` | идентификатор доставки; им же оперируют `ackDelivery`/`deadDelivery` |
| `eventId` | обратная ссылка на журнал для трассировки (**не** для дозагрузки — данные уже здесь) |
| `eventType`, `schemaVersion`, `payload` | снимок события: вход бизнес-логики подписчика |
| `claimToken` | метка захвата — **обязана быть возвращена** при закрытии (О6) |
| `claimExpiresAt` | когда захват протухнет (`claimedAt + эффективный таймаут`). Отдаёт брокер, чтобы модуль не угадывал — свой `claimTimeoutMs` он мог не задавать, и тогда действует дефолт |
| `claimCount` | сколько раз доставку выдавали; высокое значение — сигнал о ядовитом сообщении или циклически падающем воркере (§3.3) |
| `createdAt` | момент материализации; основа best-effort порядка (§1) |

**Как выбирать `limit`.** У числа **две независимые границы**, и при правках надо держать в голове обе.

**Сверху — обработка.** Вся пачка столбится сразу, поэтому модуль должен успеть разгрести её за `claimTimeoutMs`; иначе хвост протухнет и уйдёт на перезабор раньше, чем он до него дойдёт.

**Снизу — стоимость самого забора.** Это N+1 последовательных обращений к Heap (выборка кандидатов плюс условное обновление на каждого), и все они укладываются в бюджет **одного вызова — 20 секунд** (§5.8). Очередями это не дробится: в отличие от публикации, где N событий дают N отдельных запросов со своим бюджетом каждый, здесь один вызов подписчика порождает N+1 запросов внутри себя.

Отсюда дефолт 50 и потолок 200. Если замер покажет, что забор подходит к бюджету вызова, есть **два рычага**: дешёвый — снизить потолок (подписчик просто просит меньше за раз, и вот тогда каждый вызов получает свой бюджет); основательный — перейти на батчевый захват по списку `id`, который сводит забор к ~5 запросам. Схема батчевого захвата описана целиком в [ADR-0014](../ADR/0014-claim-race-per-row-cas.md) вместе с единственным её препятствием (`claimCount` требует группировки кандидатов по текущему значению счётчика).

**Логирование.** `fetchDeliveries` пишет через `ctx.account.log()` длительность забора, число запрошенных, число реально захваченных и число проигранных гонок. Длительность здесь не отчётность, а сигнал: приближение к 20 секундам и есть условие включения батчевого захвата.

#### 5.9.3. `ackDelivery` — подтвердить успех

**Request:** `deliveryId`, `claimToken`.

**Флоу:**

1. Аутентификация; гейт `status = 'active'`.
2. Прочитать строку доставки.
3. **Проверка принадлежности модулю:** строка существует **и** её `subscriberModuleKey` == вызывающий. Любой из двух промахов — **один и тот же** `AccessDeniedError`; какой именно, вызывающему не сообщается (см. «Недоступная доставка» в §5.9.5).
4. **Проверка принадлежности захвату:** предъявленная `claimToken` == текущая в строке. Не совпала — отказ `claimSuperseded`: захват перехвачен, работу закрывает уже не этот воркер.
5. **Проверка статуса** по матрице О6: `claimed` → перевести в `acked`; уже `acked` → идемпотентный успех без записи; `dead` или `pending` → отказ.

**Response:** `{ result: 'acked' | 'alreadyAcked' }`. Повтор отдаётся успехом (иначе модуль решит, что подтверждение не прошло), но **различимым** — чтобы всплеск `alreadyAcked` был виден в диагностике как потеря ответов, а не тонул в общем «ок».

#### 5.9.4. `deadDelivery` — признать необрабатываемой

**Request:** `deliveryId`, `claimToken`, `lastError` (опц.).

**Флоу** — шаги 1–4 идентичны `ackDelivery`. Далее:

5. Статус по матрице О6: `claimed` → `dead`; уже `dead` → идемпотентный успех; `acked` или `pending` → отказ.
6. `lastError`, если передан, обрезается до `LAST_ERROR_MAX` (~1000 символов) и пишется в строку. **Превышение длины вызов не отклоняет** (О7).

**Response:** `{ result: 'dead' | 'alreadyDead' }`.

Брокер на этом ничего не предпринимает: `dead` — решение модуля, повторных попыток и уведомлений за ним не следует (§1). Строка живёт до очистки в retention (§3.5) и служит материалом для разбора.

#### 5.9.5. Ошибки

| Ситуация | Ошибка | Комментарий |
|----------|--------|-------------|
| Токен отсутствует или не совпал | `AccessDeniedError` | §5.1 |
| `status` модуля не `active` | `AccessDeniedError` | `onModeration` / `disabled` (§5.7) |
| **Недоступная доставка** — строки с таким `id` нет **либо** она принадлежит другому модулю | `AccessDeniedError` | единый ответ на оба случая, см. ниже |
| `claimToken` не совпал | `CustomError` `claimSuperseded` | захват перехвачен по таймауту; воркеру следует прекратить работу и **не** повторять вызов |
| Статус не допускает операцию | `CustomError` `invalidDeliveryStatus` | по матрице О6; сигнал рассинхрона внутри модуля |

**Почему `NotFoundError` здесь не используется вовсе.** Дело не в сокрытии, а в том, что это неверное утверждение. Область, которой оперирует модуль, — **его собственные** доставки; `id` за её пределами для него не адресуем, и существует ли где-то такая строка, к его запросу отношения не имеет. «Не найдено» звучит как «поищи другой идентификатор» и отправляет разработчика разбираться с поиском строки, тогда как ответ по существу один: **эта доставка тебе недоступна**. `AccessDeniedError` формулирует это точно и одинаково верно в обоих случаях — и когда строки нет, и когда она чужая.

Различие между двумя случаями при этом не теряется — оно **уходит в лог брокера**, а не в ответ. Код ошибки и её причина не обязаны совпадать по детальности: вызывающему достаточно знать, что действие невозможно, разбирающему нужна точная причина.

**Соображения «не дать перебрать чужие доставки» здесь нет.** Системные `id` в Heap неперебираемы — длинная случайная строка (§3), поэтому подобрать чужой `id` нельзя, и различающиеся коды ошибок ничего бы не выдали. Решение опирается **только** на корректность формулировки, и ссылаться на него как на меру защиты не следует.

Обратная сторона, которую принимаем осознанно: опечатка в `deliveryId` теперь читается как проблема с правами. Ситуация маловероятна — модуль оперирует только теми `id`, которые ему сию минуту выдала `fetchDeliveries`, — а при разборе точная причина берётся из лога.

Отдельный видимый случай той же ветки: очень поздний повтор закрытия, когда строку уже удалил retention (§3.5 чистит `acked` через 24 ч, `dead` — после окна). Идемпотентного «уже закрыто» тут не выйдет — записи нет, и подтвердить нечего; модуль получит отказ. Сделать с ним он всё равно ничего не может, доставки давно нет.

`claimSuperseded` и `invalidDeliveryStatus`, наоборот, разделены намеренно: первое — штатное следствие превышения таймаута, воркер просто опоздал; второе — противоречие («сдался, а потом получилось»), которое стоит расследовать. Оба случая относятся к **своей** доставке аутентифицированного модуля, так что скрывать нечего, а различие в диагностике ценно.

**Логирование.** Каждая ветка отказа пишется через `ctx.account.log()` — и для «недоступной доставки» лог фиксирует, какой именно это был случай: строки нет или она чужая. Уровни: `claimSuperseded` — предупреждение (штатно, но при массовости требует внимания); `invalidDeliveryStatus` и обращение к чужой строке — ошибка; обращение к несуществующей — предупреждение.

#### 5.9.6. Принятые решения (О1–О7)

Ниже — запись решений с обоснованием и отклонёнными альтернативами. Все закрыты 19–20-07-2026; §5.9.1–5.9.5 выше есть их развёртка.

**О1. Набор операций и гранулярность.** Сколько операций, слиты ли `poll` и `claim`, одиночные ли ack/dead.

**Забор — слитый (`poll` + `claim` в одной операции). Решено ([ADR-0012](../ADR/0012-fetch-deliveries-atomic-poll-claim.md)).** Единая атомарная `fetchDeliveries` выбирает строки и тут же переводит их в `claimed`; подписчику они возвращаются уже застолблёнными. Отдельной операции «застолбить» в pull-API нет.

Ключевое основание — разнесение ломает сам смысл `claim`: между ответом `poll` и вызовом `claim` строки уже у подписчика (возможно, в обработке), а в брокере всё ещё `pending`, и падение в этом промежутке даёт состояние **«отдано, но не помечено»** — следующий pull выдаст те же доставки повторно. При слиянии такого состояния не существует: упало до фиксации — строки остались `pending` и никем не получены; упало после — `claimed` и перезаберутся по `claimTimeoutMs` (§3.3). Оба исхода покрыты принятой моделью at-least-once. Побочно: гонка параллельного забора локализуется в одной серверной точке (О3), а не размазывается между двумя вызовами.

Прямые следствия для контрактов: `fetchDeliveries` **обязана** нести лимит числа строк за вызов (выдача сразу столбит — иначе один воркер заберёт всю очередь); read-only просмотр очереди «не трогая» на потребительском пути невозможен по построению и при нужде выносится в админ-операцию §5.6.

**Закрытие — поодиночке. Решено.** `ackDelivery(id)` и `deadDelivery(id)` работают с **одной** доставкой за вызов; батчевых `ackDeliveries`/`deadDeliveries` нет.

Основание — **единица обработки подписчика есть одна доставка**. Множествами оперирует брокер: он материализует fan-out, он держит очередь, он отдаёт пачку на `fetchDeliveries`. Подписчик пачку разбирает и обрабатывает элементы по одному (§3.3: кладёт в свою внутреннюю очередь и обрабатывает построчно) — значит и закрывает по одному. Работа со списками событий в код подписчика не протекает.

Асимметрия с забором намеренная и следует из того же принципа: забор обязан быть батчевым, потому что это операция брокера над его очередью (и атомарность выдачи+claim, [ADR-0012](../ADR/0012-fetch-deliveries-atomic-poll-claim.md)); закрытие — отчёт подписчика о результате обработки **конкретного** факта, и агрегировать его не во что.

Прямое следствие: **вопрос семантики частичного отказа не возникает** — нечему быть частичным. Каждый вызов имеет ровно один исход, ownership и идемпотентность (О6) проверяются на одной строке, `lastError` (О7) относится к одной доставке. Цена — N вызовов на пачку из N доставок; принимается осознанно как плата за то, что подписчик не ведёт списочную логику.

**Итог О1 — 3 операции:** `fetchDeliveries` (батч, poll+claim слиты), `ackDelivery` (одна), `deadDelivery` (одна).  
_Статус: закрыт._

**О2. Кто применяет `claimTimeoutMs` (расхождение §3.1 ↔ §3.3). Решено.**

Таймаут применяет **серверная выборка `fetchDeliveries`**: она вычисляет просрочку по эффективному значению `claimTimeoutMs ?? DEFAULT_CLAIM_TIMEOUT_MS` из строки `BrokerModules` вызывающего модуля (§3.1) и возвращает просроченные `claimed` вместе с `pending`. Перезабор = повторный claim, обновляющий `claimedAt`. Прежняя формулировка §3.1 «visibility-timeout применяет сам подписчик, не брокер» снята как неверная.

При этом инициатива остаётся за модулем — и в этом смысле «ведёт подписчик» (§3.3) верно: брокер **не** сбрасывает `claimed → pending` по таймеру и **не** шлёт повторный push. Просроченная доставка не возвращается в оборот сама — она возвращается ровно тогда, когда подписчик придёт за очередной пачкой. Брокер лишь считает просрочку в момент запроса.

**Значение таймаута — дефолт брокера 1 минута с переопределением на модуле.** `DEFAULT_CLAIM_TIMEOUT_MS = 60_000` действует, когда модуль не задал своё; заданное при регистрации значение (§5.2) его перекрывает. Логика от этого не усложняется — резолв `?? DEFAULT` в одном месте, в выборке.

Обоснование пары «малый дефолт + переопределение»: раз у модуля есть собственная ручка, закладывать большой запас «на всякий случай» в дефолт не нужно. Минуты хватает подавляющему большинству обработчиков, а быстрый дефолт даёт быстрое восстановление после падения воркера — доставки упавшего не простаивают. Модуль с заведомо долгой обработкой (скачивание и разбор крупного файла по ссылке из `payload`, §3.2) объявляет свой таймаут явно — это его знание о себе, а не догадка брокера.

Граница, принимаемая осознанно: обработка, вышедшая за объявленный таймаут, приводит к перезабору и повторной обработке. Корректность это не нарушает (at-least-once + обязательная идемпотентность потребителя, §1), но означает лишнюю работу — модуль обязан либо укладываться в свой таймаут, либо поднять его.  
_Статус: закрыт._

**О3. Гонка двойного claim. Решено — построчный CAS** ([ADR-0014](../ADR/0014-claim-race-per-row-cas.md)). Захват идёт условным обновлением на строку: `where` повторяет прочитанное состояние, перехваченная строка ему не удовлетворяет и возвращает `0` — её пропускают. Блокировки на пути забора нет.

Природа гонки: `fetchDeliveries` атомарна снаружи, но изнутри это чтение `pending`/просроченных `claimed` плюс отдельная запись статуса — между ними зазор в два обращения к Heap. Два параллельных забора одного подписчика (два воркера либо плановый pull, совпавший с pull по push-триггеру) читают базу до первой записи, видят одну строку и оба её столбят; вторая запись перезаписывает первую теми же данными, ошибки нет. Итог — **одновременная** обработка одной доставки двумя воркерами. Это опаснее обычного дубля: идемпотентность потребителя пишется под последовательный повтор и при одновременном не срабатывает (оба проверяют «уже обработано?» в один момент и оба получают «нет»). Поэтому гонку закрывает брокер, а не подписчик.

Почему CAS работает: условие включает прочитанное состояние, и если строку перехватили — у `pending` сменился статус, у просроченной `claimed` обновился `claimedAt` — условие не сойдётся. Предохранитель `limit` не мешает: `where` содержит `id`, попадает максимум в одну строку. Атомарность `updateAll` подтверждена прогоном 20-07-2026 (10 параллельных попыток → ровно один победитель, проверено и через `Promise.all`, и через независимые HTTP-запросы).

Отвергнуты две альтернативы. **Эксклюзивная блокировка** по `subscriberModuleKey` на время fetch-and-claim — работает, но сериализует заборы одного модуля: его воркеры выстраиваются в очередь при том же числе записей в Heap. **Батчевый захват** (захват всей пачки одним `updateAll` по списку `id`, затем выборка по `claimToken` — ~5 запросов вместо N+1) **возможен и не отвергнут, но отложен**: прежнее обоснование «у `updateAll` нет `order`» оказалось неверным — порядок обеспечивает отдельная выборка кандидатов, а предохранитель `limit` не срабатывает, потому что список `id` ограничивает число совпадений. Единственное препятствие — `claimCount`: литеральный `patch` требует группировки кандидатов по текущему значению счётчика. Схема и условие включения — [ADR-0014](../ADR/0014-claim-race-per-row-cas.md).

> **Отдельно — как это решалось.** Сначала был выбран замок, а CAS отклонён с обоснованием «условного апдейта в Heap нет». Обоснование было ложным: `updateAll` существует, он лишь не описан в `008-heap.md`. Ошибка методическая — вывод «в документации нет ⇒ в API нет» неверен, проверять надо типы в `node_modules/@app/types` и, для семантики, рантайм. Стоит помнить при следующем платформенном вопросе.

Цена решения: заборы одного модуля сериализуются (два его воркера не заберут пачки одновременно). Приемлемо — под замком нет обработки; разные модули друг друга не блокируют (ключ разный). Дробить ключ имело бы смысл только при высоком потоке у отдельного модуля, вне масштаба FLOW.  
_Статус: закрыт._

**О4. Форма Heap-выборки `pending ∪ просроченные claimed`. Решено — один запрос.**

```typescript
where: {
  $and: [
    { subscriberModuleKey: moduleKey },
    {
      $or: [
        { status: 'pending' },
        { status: 'claimed', claimedAt: { $lt: cutoff } },
      ],
    },
  ]
},
order: [{ createdAt: 'asc' }],
limit: batchSize
```

где `cutoff = Date.now() - (claimTimeoutMs ?? DEFAULT_CLAIM_TIMEOUT_MS)` (§3.1, О2).

**Форма `where` обязана быть именно такой — это типовое требование, не стиль.** `HsFilter` — union из `HsFilterInputObject` (mapped type **без** index signature: только поля таблицы), `$and`/`$or`/`$not`-операторов. Ключ `$or` полем таблицы не является, поэтому запись `{ subscriberModuleKey, $or: [...] }` на одном уровне даёт excess property error; смешивать поле и логический оператор нельзя — поле заворачивается в свой элемент `$and`. Внутри же одного объекта два условия соединяются неявным AND, поэтому `{ status: 'claimed', claimedAt: { $lt: cutoff } }` корректно. Форма `order: [{ createdAt: 'asc' }]` подтверждена.

Опасение из прежней формулировки О4 («range требует `$and`») **не применимо**: правило запрещает два оператора сравнения на одном поле (`{ $gte, $lte }` молча теряет одну границу), а здесь граница одна — `claimedAt < cutoff`.

**Почему один запрос, а не два с merge.** Альтернатива — два `findAll` через `Promise.all` (`pending` и просроченные `claimed` отдельно) с последующей сортировкой и отсечением по `batchSize` — рекомендуется как более дружественная планировщику: PostgreSQL хуже оптимизирует `OR` и без подходящего индекса может уйти в seq scan, а индексами Heap управлять не даёт (§3). Отклонена, потому что аргумент опирается на большую таблицу, которой здесь не будет: `BrokerDeliveries` — **рабочая очередь, а не журнал** (retention §3.5 удаляет `acked` через 24 ч и `dead` после окна; копятся только `pending`/`claimed`), забор идёт мелкими регулярными батчами, а крупные выборки адресуются ClickHouse-зеркалу (§3.4), не Heap. Разбухание этой таблицы до объёмов, где план запроса начинает решать, означает вставшего подписчика — состояние, которое надо замечать и чинить, а не обслуживать оптимизацией.

На выбранных критериях (простота и очевидная корректность) выигрывает один запрос: нет merge, нет сортировки со срезом и рассуждения о том, почему «первые N из каждой выборки» дают верные N старейших. Надёжности два запроса не добавляют — оба варианта целиком лежат под `runWithExclusiveLock` (О3), гарантии одинаковы.

Запасной ход дешёвый: переход на два запроса — локальная правка внутри `fetchDeliveries`, контракт операции не меняется. Медленная выборка трактуется как **эксплуатационный сигнал** о разбухшей очереди, а не как повод усложнить запрос.  
_Статус: закрыт._

**О5. Auth и гейт статуса. Решено.**

**Аутентификация — токеном, единообразно для всех модулей.** Никаких исключений по источнику: `fetchDeliveries`, `ackDelivery` и `deadDelivery` требуют токена так же, как publish (§5.8) и операции §5.3–§5.5. Гейтвей хэширует предъявленное и сверяет с `authTokenHash` строки `moduleKey` (§5.1); несовпадение или отсутствие токена → отказ.

Это ровно то, что уже описано в §5.2 (токен выдаётся всем при регистрации, шаг 4 не ветвится по источнику) и в §3.1 (`authTokenHash` сверяется при каждой операции). `source` в аутентификации не участвует — он влияет только на модерацию.

**Гейт статуса — `status = 'active'`, все три операции.** `fetchDeliveries`, `ackDelivery`, `deadDelivery` требуют активного модуля:

- `onModeration` — не запущен ни разу, доставок ему не создавалось (fan-out берёт только `active`, §3.1). Отказ.
- `disabled` — админ-стоп. Отказ; уже созданные доставки лежат и обрабатываются после включения, ровно как обещает §5.7.

Третьего случая больше нет: модуль с висящей заявкой на расширение остаётся `active` и работает без ограничений (§5.6, поправка к [ADR-0010](../ADR/0010-module-status-enum.md)). Прежняя редакция замораживала его через статус `reModeration` — заявка на новые права отбирала уже одобренные, и очередь модуля стояла, пока админ не дойдёт до неё. Разведение осей «работает» и «есть заявка» убрало это, и гейт свёлся к одному значению без исключений и перечислений.  
_Статус: закрыт._

**О6. Ownership и идемпотентность ack/dead. Решено.**

Закрытие проходит **три проверки**, все по одной строке (закрытие поодиночке, О1 — частичных исходов не бывает):

1. **Принадлежность модулю.** `subscriberModuleKey` строки == аутентифицированный вызывающий (О5). Чужую доставку закрыть нельзя — инвариант авторизации: иначе один модуль уничтожал бы работу другого, и она не была бы выполнена никогда.
2. **Принадлежность захвату.** Предъявленная `claimToken` == текущая в строке (§3.3). Отсекает опоздавшего: воркер, чей захват перехвачен по таймауту, закрыть строку уже не может.
3. **Статус.** Требуется `claimed`. Исключения — в таблице ниже.

**Метка захвата (`claimToken`) — существо решения.** Проверка по `subscriberModuleKey` различает модули, но **не различает воркеров одного модуля**, а именно между ними и возникает конфликт. Сценарий: воркер A взял доставку, не уложился в таймаут; воркер B законно перезабрал её; A заканчивает и закрывает строку — из-под B. Раньше это проходило беспрепятственно, и хуже всего был случай, когда **B падал и слал `dead`**: строка уже `acked`, отказ трактовался как повтор и молча терялся. Метка это снимает: каждый `claim` (включая перезабор) пишет новое значение, и принимается отчёт только от текущего держателя. Побочно A получает **информацию** — отказ «захват перехвачен» вместо ложной уверенности в успехе; он может прекратить работу и не писать результат.

Рассмотренная альтернатива — использовать как метку существующий `claimedAt`: ничего не добавляем в схему. Отклонена из-за разрешения в миллисекунду: два захвата одной строки в одну миллисекунду дали бы неразличимые метки. Практически почти невозможно (захваты модуля сериализованы замком О3), но «почти» в механизме корректности неприемлемо; отдельное случайное поле снимает вопрос за одну колонку.

**Матрица исходов** (при пройденных проверках 1 и 2):

| Статус строки | `ackDelivery` | `deadDelivery` |
|---|---|---|
| `claimed` | → `acked` | → `dead` |
| `acked` | **успех, no-op** (повтор) | отказ |
| `dead` | отказ | **успех, no-op** (повтор) |
| `pending` | отказ | отказ |

Правило одной строкой: **повтор того же закрытия — успех, всё прочее не из `claimed` — отказ.**

Почему повтор обязан быть успехом: модуль послал ack, брокер выполнил, но ответ потерялся по сети. Модуль повторяет. Ответь брокер ошибкой — модуль решит, что подтверждение не прошло, и либо обработает событие ещё раз, либо пометит `dead` уже успешно закрытое. Ошибка на ровном месте. Метка при этом не мешает: она сохраняется после терминального статуса, поэтому повтор от того же держателя её проходит.

Почему `pending` — отказ: вернуться в `pending` по жизненному циклу невозможно (перезабор ставит `claimed` заново, брокер сам статус не сбрасывает — О2). Значит закрытие `pending` это баг в модуле, отчёт о работе, которую он не брал. Отказ делает ошибку заметной.

Почему переход между терминальными — отказ: «сдался, а потом получилось» и «получилось, а потом сдался» — противоречия, признак рассинхрона внутри модуля. Проглатывать молча не следует.  
_Статус: закрыт._

**О7. Диагностические поля доставки. Решено — оба вводятся** (§3.3). Предварительный дефолт в части счётчика **пересмотрен**.

**`claimCount` — ведёт брокер, а не модуль.** Дефолт предлагал оставить счётчик модулю (durable, если нужен across-restart). Отклонено: чтобы счётчик пережил рестарт воркера, модулю пришлось бы завести собственную таблицу, ключуемую по `id` доставки, — теневую копию строки, которая уже есть у брокера, и так в каждом подписчике. Брокеру же инкремент не стоит ничего: он и так перезаписывает строку на каждом захвате (`claimedAt` + `claimToken`, О6), `+1` идёт той же записью.

Имя изменено с `attempts` намеренно. Брокер знает число **выдач**, а не попыток обработки: приступил ли модуль к работе после захвата, ему неизвестно. Разница не косметическая — в сценарии двойного захвата (О6) счётчик вырастет на два при одной фактической попытке.

Ценность — в наблюдаемости, доступной **только брокеру**: высокий `claimCount` означает ядовитое сообщение или циклически падающего подписчика, и заметить это со стороны модуля некому — он в такие моменты как раз не работает. Видно даже по строкам в `claimed`, которые без счётчика выглядят штатно работающими.

**Ограничение зафиксировано жёстко: счётчик диагностический и на поведение брокера не влияет.** Автоперевода в `dead` по достижении порога нет. Наличие числа такой соблазн создаёт, но это прямо противоречит §1 — когда сдаться, решает модуль, брокер лишь хранит статус.

**`lastError` — принимается, с двумя оговорками.** Необязательный параметр `deadDelivery`; делает список dead-доставок самообъясняющим, иначе причина ищется в логах чужого приложения сопоставлением по `eventId` — и только если модуль вообще потрудился залогировать.

Первая оговорка — **лимит длины** (`LAST_ERROR_MAX`, ~1000 символов): строка доставки уже несёт снимок payload и живёт под потолком ~8 КБ (§3.2), стек-трейс её испортит. Вторая, важнее — **при превышении обрезаем, а не отклоняем вызов**: модуль в этот момент в аварийной ветке, он уже сдался и закрывает доставку. Отказ по длине текста подвесил бы строку в `claimed` навсегда по формальной причине.

Поле диагностическое: брокер текст не парсит, не классифицирует и решений по нему не принимает. И это **не история ошибок** — только причина финальной сдачи. Промежуточные сбои брокер не видит в принципе: при временной неудаче модуль ничего не вызывает, он просто не подтверждает, а захват протухает. Собирать историю можно было бы только через операцию «не получилось, верни в очередь» — четвёртую в API, которой О1 не предусмотрел.  
_Статус: закрыт._

## 8. Правила эволюции

Контракт `payload` (структура данных внутри пары `(eventType, schemaVersion)`, §3.2) — соглашение между модулем-продюсером и модулями-подписчиками на уровне их собственного кода. Брокер контракт не хранит, не валидирует и не версионирует за них: `schemaVersion` — непрозрачный тег, который брокер лишь копирует из события в доставку (§3.2/§3.3), не заглядывая внутрь `payload`. Эволюция схемы (совместимые/несовместимые изменения, когда бампать `schemaVersion`, как долго подписчики обязаны понимать старые версии) — ответственность договаривающихся сторон вне спецификации брокера.
