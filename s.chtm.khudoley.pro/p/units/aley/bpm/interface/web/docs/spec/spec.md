## 1. Назначение и стадийность

`p/units/aley/bpm/interface/web` — веб-интерфейс личной BPM-системы (бренд во UI — `FLOW · Личная BPM-система`). Это рабочее место одного пользователя для GTD/PARA-цикла: захват → уточнение → организация → действие. Интерфейс объединяет задачи, журнал захвата, диалоги (в т.ч. с ИИ), финансы, библиотеку, сервисы-интеграции и инструменты в одной «терминальной» оболочке.

### 1.1 Стадия 1 — дизайн на моках (текущая)

Проект строится **mock-first**: сначала переносится дизайн и интеракции, данных бэкенда нет. Все экраны работают на статичных mock-данных, живущих в `shared/mocks/*`, и на клиентском реактивном состоянии. Heap, repos, реальные API в стадии 1 **не создаются**.

Цель стадии 1 — пиксельно и поведенчески воспроизвести `BPM Терминал v2` как Vue-приложение: оболочка, навигация между экранами, все экраны и их под-виды, дизайн-система, интерактивные элементы (переключатели видов, сортировка/группировка таблицы, чек-боксы, тогглы сервисов, открытие детальных карточек).

### 1.2 Стадия 2 — функциональность (будущая, вне этой редакции)

После приёмки дизайна mock-слой замещается реальными данными: появляются `tables/`, `repos/`, `api/`, серверное логирование и интеграции. Контракт компонентов при этом не меняется — меняется только источник данных (provider). Раздел 12 фиксирует точки замены.

Проект не является самостоятельным Node/Vite-приложением. Он работает внутри Chatium, где `app`, `ctx`, file-based routing, SSR-рендер Vue, auth и публикация предоставляются платформой.

## 2. Платформенные инварианты

- `app` и `ctx` — глобальные объекты Chatium, не импортируются из локальных файлов.
- Новые npm-зависимости не добавляются. Только платформенные модули `@app/*` и CDN-шрифты (Inter, Space Grotesk, JetBrains Mono).
- File-based routing: один файл route entrypoint = один route, предпочтительный путь внутри файла — `'/'`.
- Ссылки между маршрутами — через `config/routes.tsx` и route helpers (`withProjectRoot`). Домен не хардкодится.
- Vue-компоненты не импортируют `tables/`, `repos/`, `lib/`. Допустимы импорты из `shared/*`, помеченных `// @shared`.
- В стадии 1 источник данных Vue — mock-модули из `shared/mocks/*` (client-safe) и SSR-пропсы. В стадии 2 — SSR-пропсы и `fetch`/`api/*`.
- `// @ts-ignore` допустим только для системных модулей Chatium без локальных типов; на собственный код не ставится.

- Приоритет (`pr`): `high` / `med` / `low` → точечные цвета (карта `prMap`), отображаются точкой-индикатором.

### 4.4 Декоративная система (референс «Терминал»)

Навешивается через атрибуты/классы и не несёт данных:

- `[data-deco]` со значениями `grid` (акцентная угловая сетка), `gridL` (нейтральная сетка слева), `dots` (точки снизу), `corner`/`corner2` (HUD-уголки), `scan` (сканлиния).
- `.deco-label` — моно-лейбл секции формата `[NN] — рейл — ПОДПИСЬ`.
- `.deco-rail` — вертикальный акцент-бар слева от заголовка.
- `.gcard` + `.gut` — карточка с левым «гаттер»-полем и моно-номером (`01`, `02`).
- `.statcard` — стат-плитка с HUD-уголками и hover-подъёмом; `.liftcard` — кликабельная карточка с подъёмом.
- `.deco-val` — крупное числовое значение; `.deco-foot` — подпись-сноска под значением.
- `[data-role=appbg]` — фоновый слой: сетка 34px + SVG-шум + радиальные свечения акцентом.
- Анимации: `fadeUp` (появление экранов), `softpulse` (пульсация индикаторов).

### 4.5 Адаптивность

Брейкпоинты из `.dc.html`: `≤880px` — sidebar превращается в выезжающий drawer с backdrop, появляется кнопка-меню, метрики 2 колонки, home-grid в 1 колонку, диалоги/таблицы перестраиваются; `≤560px` — метрики в 1 колонку, уплотнение паддингов, hero в колонку. Адаптивность реализуется через `shared/responsive` + media-queries в `styles.tsx`.

## 5. Оболочка, навигация и состояние

### 5.1 Оболочка

`AppShell.vue` = фон (`AppBackground`) + `Sidebar` (224px, на мобиле drawer) + основная колонка (`Topbar` 58px + скролл-область с активным экраном).

- **Sidebar**: логотип/бренд, пункт «Главная»; раздел «Пространство» (Журнал, Задачи, Диалоги — со счётчиками/индикатором); раздел «Модули» (Инструменты, PARA, Финансы, Сервисы, Библиотека); футер (Дизайн-система, карточка пользователя со статус-точкой).
- **Topbar**: код экрана (`[CODE]`, моно), заголовок экрана, глобальный поиск, кнопка «Захват», кнопка-меню на мобиле.

### 5.2 Клиентская маршрутизация экранов

Активный экран — поле состояния `screen` (значения: `home`, `journal`, `tasks`, `dialogs`, `finances`, `para`, `tools`, `services`, `library`, `components`, `detail`, `stub`). Переключение — действия `open.<screen>` / `go(scr)`; нереализованные модули ведут на `stub` с именем. Заголовок и код экрана берутся из карт `titles`/`codes` в `config/project.tsx`.

### 5.3 Модель состояния (frontend store)

Состояние оболочки и экранов хранится в реактивном store (`shared/store.ts`), портированном из реактивной модели `.dc.html` (`setState` → Vue `reactive`/`ref`). Ключевые поля:

| Поле | Назначение |
| ---- | ---------- |
| `screen`, `stubName` | Активный экран и имя заглушки. |
| `navOpen` | Открыт ли мобильный drawer. |
| `density`, `accent` | Плотность (`comfy`/`compact`) и акцент темы. |
| `taskView` | Вид задач: `board` / `table` / `timeline` / `gtd`. |
| `tableFilter`, `tableSortKey`, `tableSortDir`, `tableGroup` | Состояние таблицы задач. |
| `activeThread`, `messages`, `composer` | Состояние диалогов. |
| `detailType`, `detailId`, `editMode`, `detailFrom` | Открытая детальная карточка (`task`/`library`/`project`/`ref`), режим правки, экран возврата. |
| `tasks`, `lib`, `projs`, `refs` | Ленивая копия mock-сторов для редактирования (см. §7.2). |
| `todayTasks`, `habits`, `services`, `libFilter`, `pomoSec` | Состояния отдельных экранов. |

Производные значения (биндинги для шаблонов) вычисляются из state + mock-данных в computed-слое (аналог `renderVals()` в `.dc.html`).

## 6. Библиотека компонентов: хранение и использование

Это ядро стадии 1. Компоненты организованы по уровням ответственности; правила ниже нормативны.

### 6.1 Уровни компонентов

1. **`components/ui/` — атомы и молекулы дизайн-системы.** Чистые презентационные Vue-компоненты без знания о доменах и состоянии приложения. Получают всё через props, наружу отдают события (`emit`). Не импортируют store, mock-данные, `lib/`, `repos/`. Это переиспользуемый «kit», на котором собираются экраны. Предполагаемый состав (точный список фиксируется при реализации в §3.1):
   - `DecoLabel`, `DecoRail` — моно-лейбл секции и акцент-рейл.
   - `GutterCard` — карточка с гаттер-полем и моно-номером.
   - `StatCard` / `MetricTile` — стат-плитка с HUD-уголками и трендом.
   - `Pill` / `StatusPill` — чип статуса/тега.
   - `SegmentedControl` — переключатель видов (Доска/Таблица/…).
   - `NavItem` — пункт сайдбара (иконка, лейбл, счётчик/индикатор, active).
   - `Avatar` — аватар контакта/ИИ.
   - `Toggle` — тоггл-переключатель (сервисы).
   - `Checkbox` — чек-бокс задачи/привычки.
   - `ProgressBar` — полоса прогресса (бюджеты, чтение).
   - `BarChart` — столбчатый мини-график (активность недели, траты).
   - `SourceChip` — чип источника записи журнала.
   - `IconButton`, `PrimaryButton` — кнопки.
2. **`components/screens/` — экраны.** Компонуют атомы/молекулы под конкретный экран, читают данные из store/computed-слоя, разводят события в действия store. Один файл = один экран из §5.2.
3. **`components/tasks/` — под-виды экрана задач.** `TaskBoard`, `TaskTable`, `TaskTimeline`, `TaskGtd`; монтируются внутри `TasksScreen` по `taskView`.
4. **`components/shell/` — каркас.** `Sidebar`, `Topbar`, `AppBackground`, `MobileDrawer`; знают о навигации, но не о внутренностях экранов.

### 6.2 Правила компонентов

- **Стили — токенами.** Цвета, отступы, типографика — только через CSS-переменные темы (§4) и общие классы декоративной системы. Хардкод hex/px цветов запрещён, кроме определения самих токенов в `shared/theme.ts`.
- **Презентационные компоненты `ui/` не знают о данных.** Никаких импортов store/mock внутри `ui/`. Доменная привязка — на уровне `screens/`.
- **Контракт props/emit стабилен.** Имена и форма props фиксируются здесь при добавлении компонента; экраны зависят от контракта, а не от внутренностей.
- **Витрина в экране «Дизайн-система».** Экран `components` (`ComponentsScreen`) — живой каталог: палитра токенов, статус-пилюли, примеры атомов/молекул. Любой новый `ui/`-компонент добавляется в витрину.
- **Импорт-граница.** Vue-компоненты импортируют только из `shared/*` (`// @shared`) и других компонентов. Серверные слои недоступны.

## 7. Mock-данные: хранение и использование

### 7.1 Размещение

Все mock-данные стадии 1 живут в `shared/mocks/*.mock.ts` (client-safe, `// @shared`) и типизируются через `shared/types.ts`. Каждый домен — отдельный модуль, экспортирующий seed-набор. Экраны читают моки через computed-слой store, а не импортируют `.mock.ts` напрямую в произвольных местах — это упрощает замену провайдера в стадии 2.

### 7.2 Семантика «seed → store»

Как в `.dc.html`: моки — это **seed**. При первом действии, меняющем данные (открытие/создание/правка), store лениво копирует seed в редактируемые сторы (`tasks`, `lib`, `projs`, `refs`) и далее работает с копией (`ensureStores`/`seedTasks`/`seedLib`/`seedProjs`/`seedRefs`). Это даёт интерактивность (чек-боксы, добавление, правка) без бэкенда и изолирует исходные моки от мутаций.

### 7.3 Доменные сущности (frontend-типы)

| Сущность | Ключевые поля | Mock-модуль |
| -------- | ------------- | ----------- |
| `Task` | `id, title, status(inbox/todo/doing/wait/done), project, client, context(@…), pr(high/med/low), due, start, end, desc, checklist[]` | `tasks.mock.ts` |
| `Project` | `id, name, para, area, client, deadline, desc, taskFilter` | `para.mock.ts` |
| `Ref` (PARA area/resource/archive) | `id, name, kind, desc, tags[], items[]` | `para.mock.ts` |
| `Thread` | `id, name, ai, last, time, unread, messages[]{role(me/ai/them), text, time}` | `dialogs.mock.ts` |
| `JournalEntry` | `src, txt, time` | `journal.mock.ts` |

## 9. Права доступа

Стадия 1 — единственный пользователь, аутентификация платформы Chatium. Разграничения ролей внутри интерфейса нет; админских экранов из `template_project` (admin/tests) в стадии 1 нет — они вне продуктовой роли BPM-терминала. Если позже понадобится защищённый вход — фиксируется отдельной редакцией §9 и §3.1.

## 10. Логирование

В стадии 1 (frontend на моках) серверного логирования нет. Допустим минимальный браузерный диагностический лог при необходимости отладки UI. Полное серверное/браузерное логирование по образцу `template_project` (`shared/logger`, `api/logger/*`) вводится в стадии 2 вместе с бэкендом и описывается отдельной редакцией.

## 11. Тесты

Стадия 1: предмет проверки — соответствие дизайну и работоспособность интеракций. Минимально — типы (`vue-tsc`) и стиль (Prettier) через `/check`. Юнит/интеграционные раннеры из `template_project` подключаются в стадии 2 при появлении бизнес-логики.

## 12. Точки замены mock → backend (стадия 2)

Зафиксировано заранее, чтобы стадия 1 не закладывала тупиков:

- `shared/mocks/*` → `repos/*` + `api/*`; computed-слой store меняет источник, контракт компонентов сохраняется.
- `shared/store.ts` локальные мутации → server actions/`fetch`; `ensureStores`-семантика заменяется загрузкой с сервера.
- `tables/*` (Heap-схемы) появляются под доменные сущности §7.3.
- Раздел §9/§10 дополняется auth-гейтами и логированием.

## 13. Известные ограничения и допущения

- Реализация — Vue (по образцу `template_project`); `.dc.html` — референс, не копируется дословно. Решение фиксируется здесь и может быть пересмотрено отдельной редакцией до начала кодирования.
- Single-page-shell с клиентской маршрутизацией экранов — намеренное отклонение от «страница на маршрут» `template_project`, продиктованное природой дизайна (одна оболочка, экран в состоянии).
- Все данные стадии 1 — моки; ни одно действие не персистится между перезагрузками.
