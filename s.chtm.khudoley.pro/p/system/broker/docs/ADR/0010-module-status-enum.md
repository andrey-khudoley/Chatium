# ADR-0010: Состояние модуля — единый `status`, заявка на расширение — отдельные поля

Дата: 2026-07-07

## Контекст

Строка `BrokerModules` должна отвечать на два разных вопроса: **работает ли модуль в обмене** и **висит ли у него заявка на расширение прав**. Вопрос — как их закодировать.

Напрашивающийся вариант — пара булевых (`enabled` для участия в обмене + `adminDisabled` для админ-стопа) — не годится:

1. **`enabled` тянет две роли** — «допущен» и «приостановлен до одобрения»: §5.3/§5.4 при заявке на расширение whitelist гасили бы его. Признак становится **немонотонным** (активный модуль снова `false`).
2. Как следствие, §5.6 не может по `enabled` различить **первичную регистрацию** и **расширение прав** — обе дают `false`. Пришлось бы выводить «был ли модуль активен» из истории событий.
3. Два ортогональных булевых дают **двусмысленные комбинации** (`enabled = false` + `adminDisabled = true` — какую причину показывать?).

Обобщённо: и то и другое — признак, который вместе с состоянием кодирует что-то ещё.

## Решение

**Две независимые оси, две разные формы хранения.**

**Ось 1 — операционное состояние.** Поле `status` (enum) с взаимоисключающими значениями; описывает **только** то, работает модуль или нет:

| Значение | Подпись | Смысл |
|---|---|---|
| `onModeration` | «На модерации» | `external`, первая заявка, ещё ни разу не запущен, ждёт первичного одобрения (§5.2) |
| `active` | «Активен» | участвует в обмене |
| `disabled` | «Выключен» | админ-стоп |

**Ось 2 — наличие заявки на расширение.** Поля `pendingPublishTypes` / `pendingSubscribeTypes` (§3.1), `null` = заявки нет. Читает их **исключительно админка модерации**; ни publish-проверка, ни fan-out их не видят.

Правила:

- **`disabled` применяется только из `active`** (админ гасит работающий модуль). `onModeration` и `disabled` не совмещаются.
- `internal` при регистрации → сразу `active`; `external` → `onModeration`.
- «Активен сейчас» = `status == 'active'`, без исключений и перечислений.
- **Заявка статусом не выражается:** работающий модуль, попросивший расширение, остаётся `active`.
- Переходы фиксируются `broker.*` событиями ([ADR-0009](0009-operational-audit-events.md)); на строке — только текущее состояние.

## Обоснование

1. **Единый признак-состояние вместо двух ортогональных булевых** — исчезают двусмысленные комбинации, консистентнее под гонки (одно поле, а не два).
2. **Заявка не отменяет работу.** Если бы наличие заявки кодировалось значением статуса, модуль, попросивший **дополнительные** права, терял бы **уже одобренные**: его очередь замирала бы до решения админа, хотя доставки в ней нарезаны законно. Наказание за просьбу, а не за нарушение. Разведение осей это устраняет — новые права вступают в силу после approve, старые работают всё это время.
3. **Отклонение заявки — полный откат.** `pending*` очищается, боевой набор не пострадал. Если бы запрошенный набор писался прямо в боевое поле, отката бы не существовало: `declineModeration` оставлял бы модуль приостановленным до подачи новой корректной заявки.
4. **Различие «первичка / расширение» — без обращения к истории:** первичка по статусу (`onModeration`), расширение по непустому `pending*`. Ни флага `everActive`, ни скана журнала.
5. **`active` означает ровно «работает».** Гейт pull-операций (§5.9, О5) и резолв подписчиков на fan-out (§3.1) — однозначный `status = 'active'`.

## Последствия

Плюсы:

- §5.6 читает решение (`approve`/`decline`, delete-vs-keep) прямо из строки;
- одно поле честно описывает «в каком состоянии модуль», второе — «что он попросил»;
- `onModeration` на оси статуса конфликта не создаёт: новичок-`external` действительно не работает — он ни разу не одобрен, доставок ему не создавалось. Это операционное состояние, а не заявка, притворяющаяся состоянием.

Минусы / границы:

- **Очередь модерации не берётся одним условием.** Это `status = 'onModeration'` **или** непустой `pending*` — запрос с `$or` (форма — §5.9, О4);
- **две колонки** `pending*` вместо одного значения enum;
- теряется ортогональность «админ-стоп ⟂ состояние модерации» — но она не нужна: модуль на первичной модерации и так не участвует, «выключать» его поверх бессмысленно (отсюда «`disabled` только из `active`»);
- полная хронология статусов — в `broker.*` журнале (ADR-0009), не на строке.

Отвергнуто:

- **Пара булевых `enabled`/`adminDisabled`** — немонотонный `enabled`, двусмысленные комбо, нужда в выводе «ever active» из истории (см. Контекст).
- **Значение статуса под заявку на расширение** (`reModeration` и подобные) — статус начинает означать одновременно операционное состояние и наличие заявки, со всеми последствиями из пп. 2–3 обоснования.
- **Отдельный флаг `everActive`** — денормализация исторического факта на строку, против принципа «строка — текущее, история — в событиях».

Связано: [ADR-0009](0009-operational-audit-events.md), §3.1, §5.2–§5.6, §5.9 О5.
