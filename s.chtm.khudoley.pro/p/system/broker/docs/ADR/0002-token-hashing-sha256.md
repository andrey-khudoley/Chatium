# ADR-0002: Хеширование auth-токена модуля через SHA-256 (`@npm/node-forge`)

Дата: 2026-06-30

## Контекст

При регистрации модуля гейтвей выдаёт случайный **auth-токен**, а в `BrokerModules.authTokenHash` хранит только его необратимый отпечаток; при каждой операции входящий токен хешируется и сверяется с сохранённым (§5.1, §5.2 спецификации). `authTokenHash` — граница доверия (задел под внешнюю недоверенную регистрацию модулей), поэтому отпечаток должен быть криптографически стойким.

Что реально доступно в Chatium (сверено по `inner/docs`):

| Вариант | Доступность | Пригодность для задачи |
|---|---|---|
| Нативный `crypto` (Node) | **Отсутствует** в платформе | — |
| `@app/security` | Есть, но это **только CSRF**-токены (`generate/verifyDynamicCsrfToken`) | Не хеш-функция общего назначения |
| `@app/nanoid` | Есть | Генерация случайности (нужна для **самого токена**), но не хеширование |
| Прочие `@app/*` (30 модулей) | — | Криптопримитивов не предоставляют |
| `@npm/node-forge` | В белом списке npm | Полноценная крипта: SHA-256/512, HMAC, PBKDF2, AES, RSA |
| Самописный FNV-1a | Реализуемо без зависимостей | Некриптографический, 32 бита — отвергнут |

Внутри `node-forge` рассмотрены конкретные примитивы под задачу «необратимая проверка высокоэнтропийного токена».

## Решение

Считать `authTokenHash` настоящим **SHA-256 из `@npm/node-forge`** по доменно-разделённой строке `broker-module-auth:<moduleKey>:<token>` (реализация — §5.1).

### Почему именно SHA-256, а не другие доступные варианты

- **Не шифрование (AES/RSA).** Нужна односторонняя проверка, а не обратимость. Хранить шифртекст токена хуже: утечка ключа = утечка всех токенов. Хеш необратим by design.
- **Не `@app/security`.** Это CSRF-механизм с собственной семантикой, не примитив хеширования произвольной строки.
- **Не FNV-1a.** 32-битный некриптографический отпечаток не устойчив к подбору и коллизиям — недопустимо для границы доверия.
- **Не медленные KDF (bcrypt/scrypt/PBKDF2/argon2).** Они нужны для **низкоэнтропийных паролей**, где намеренное замедление защищает от перебора. Здесь токен — случайный, высокоэнтропийный (≥128 бит), перебор невозможен в принципе; платить CPU за медленный хеш на каждой операции брокера нерационально.
- **Не SHA-512.** Та же семья и стойкость для нашей задачи, длиннее хеш без практической выгоды; SHA-256 — отраслевой дефолт, 64 hex-символа удобны для хранения.
- **Не HMAC-SHA256.** Дал бы стойкость к length-extension, но потребовал бы управления серверным секретным ключом (хранение, ротация). Для высокоэнтропийного токена доменный префикс + привязка к `moduleKey` уже дают разделение пространств имён и защиту от переноса хеша между модулями; вводить ключевое хозяйство преждевременно. (Оставляем как возможный апгрейд — см. ниже.)

## Последствия

Плюсы:

- криптографически стойкий отпечаток на границе доверия;
- утечка строки `BrokerModules` не даёт рабочего креденциала (сам токен не хранится и не логируется);
- доменный префикс + `moduleKey` привязывают хеш к конкретному модулю;
- нулевая стоимость управления ключами (в отличие от HMAC/шифрования).

Минусы / риски:

- появляется зависимость от `@npm/node-forge`. Пакет входит в фиксированный список из трёх, зашитый в платформу (`date-fns`, `date-fns-tz`, `node-forge`), — объявлять его в `package.json` не требуется и на резолв это не влияет. Типов у пакета нет, поэтому типчек его доступность не подтверждает (`inner/docs/003-auth.md`);
- голый SHA-256 уязвим к length-extension в общем случае — для нашего сценария (фиксированный формат входа, сравнение целиком) неэксплуатируемо, но если в будущем формат входа усложнится, апгрейд до **HMAC-SHA256** (тоже есть в `node-forge`) — предусмотренный путь развития.
