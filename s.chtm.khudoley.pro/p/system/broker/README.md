# Broker — глобальный брокер событий уровня аккаунта

Dev-копия (окружение stage) глобального брокера событий уровня аккаунта: модули публикуют события по типам, брокер разносит их подписчикам через pull-модель с доставками (`BrokerDeliveries`) и claim/ack/dead-жизненным циклом. Prod-копия появится в `p/system/broker` при переносе через `/to-prod`/`/to-sync`.

**Источник истины** — [`docs/spec/spec.md`](docs/spec/spec.md) (карта разделов — [`docs/spec/spec-map.md`](docs/spec/spec-map.md)). Данный README и остальные `docs/*.md` — обзорная навигация, при расхождении приоритет у спеки.

## Тесты

Гейт приёмки — `GET ./web/tests/ai` (нужна админ-сессия): возвращает `{ project: 'broker', summary: { total, passed, failed, duration, success }, results }`; зелёный прогон = `summary.success === true`. Параметр `?category=` запускает срез по одной категории (`database`, `functional`, `api`, `integration`, `concurrency`, `limits`) вместо полного набора. Интерактивная страница — `./web/tests`.

Прогоны нельзя параллелить — набор использует общий sweep тестовых строк, общий probe-key лог-теста и общий `log_level`; **один прогон за раз**.

## Окружения

Каждая из 5 сущностей брокера (`modules`, `events`, `deliveries`, `events-archive`, `settings`) хранится в паре Heap-таблиц stage/prod (`__stage_*`/`__prod_*`). Выбор конкретной таблицы — статический селектор по `IS_PROD`, вычисленному из `PROJECT_ROOT` в `config/env.ts`; наружу из `tables/*.table.ts` экспортируется только репозиторий выбранного окружения. Текущий каталог (`s.chtm.khudoley.pro/d/system/broker`) — окружение stage; при переносе в `p/system/broker` тот же код автоматически адресуется к `_prod`-таблицам.
