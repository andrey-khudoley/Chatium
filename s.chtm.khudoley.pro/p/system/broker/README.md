# Broker — глобальный брокер событий уровня аккаунта

Dev-копия (окружение stage) глобального брокера событий уровня аккаунта: модули публикуют события по типам, брокер разносит их подписчикам через pull-модель с доставками (`BrokerDeliveries`) и claim/ack/dead-жизненным циклом. Prod-копия появится в `p/system/broker` при переносе через `/to-prod`/`/to-sync`.

**Источник истины** — [`docs/spec/spec.md`](docs/spec/spec.md) (карта разделов — [`docs/spec/spec-map.md`](docs/spec/spec-map.md)). Данный README и остальные `docs/*.md` — обзорная навигация, при расхождении приоритет у спеки.

## Текущее состояние

Волна 2 (MVP) реализована полностью: регистрация и жизненный цикл модулей, publish/fanout/pull/ack/dead, лог-инфраструктура, полный набор тестов. Волна 2.5 «Наблюдаемость» (§5.11 спеки) реализована на dev: админ-страница `web/admin` (Admin-only) с тремя панелями — живой монитор логов (WS-подписка + история), статус брокера (backlog fan-out, доставки по статусам, реестр модулей, disable/enable, `log_level`) и метрики (Heap-агрегаты по событиям); 5 новых API-роутов `api/broker/admin/*`. Приёмка §9.5.2 спеки — зелёная: `web/tests/ai` → 46/46, `summary.success: true`, повторяемо (2 круга подряд), Runtime Verification браузером пройдена. Prod-копия (`p/system/broker`) пока соответствует волне 2 — перенос волны 2.5 предстоит отдельной командой `/to-prod`.

Вне волны 2.5 (запланировано на волну 3 — Beta, §0.1/§9.5.3 спеки):

## Тесты

Гейт приёмки — `GET ./web/tests/ai` (нужна админ-сессия): возвращает `{ project: 'broker', summary: { total, passed, failed, duration, success }, results }`; зелёный прогон = `summary.success === true`. Параметр `?category=` запускает срез по одной категории (`database`, `functional`, `api`, `integration`, `concurrency`, `limits`) вместо полного набора. Интерактивная страница — `./web/tests`. Реестр — 46 тестов (было 39: +1 functional `readlogs_history`, +6 api `admin_*`, физически в `suites/api-admin.ts`); гейт зелёного прогона не менялся.

Прогоны нельзя параллелить — набор использует общий sweep тестовых строк, общий probe-key лог-теста и общий `log_level`; **один прогон за раз**.

## Окружения

Каждая из 5 сущностей брокера (`modules`, `events`, `deliveries`, `events-archive`, `settings`) хранится в паре Heap-таблиц stage/prod (`__stage_*`/`__prod_*`). Выбор конкретной таблицы — статический селектор по `IS_PROD`, вычисленному из `PROJECT_ROOT` в `config/env.ts`; наружу из `tables/*.table.ts` экспортируется только репозиторий выбранного окружения. Текущий каталог (`s.chtm.khudoley.pro/d/system/broker`) — окружение stage; при переносе в `p/system/broker` тот же код автоматически адресуется к `_prod`-таблицам.
