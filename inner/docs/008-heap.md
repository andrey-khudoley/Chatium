## Основные концепции

**Heap** — встроенная база данных Chatium для хранения структурированных данных.

### Ключевые понятия

- **Heap.Table** — определение структуры таблицы
- **HeapTableRepo** — объект для работы с данными таблицы
- **HeapObject** — запись (строка) таблицы
- **Типы полей** — String, Number, Boolean, Money, RefLink и др.

### Важные особенности

- ✅ Таблицы доступны **только на backend**
- ✅ Для клиента создавайте API роуты
- ✅ Автоматическая типизация TypeScript
- ✅ Поддержка транзакций
- ✅ Полнотекстовый поиск с эмбеддингами
- ⚠️ **Эмбединги** (`embeddings: false`) — по умолчанию **ВЫКЛЮЧЕНЫ**, включайте только при необходимости семантического поиска

### Лимиты (тариф «Бизнес»)

Ограничения тарифа платформы Chatium, не самого Heap-движка — источник: страница тарифов Chatium («Включённые ресурсы»), актуально для тарифа **Бизнес** (используется в этом workspace):

- **Максимальный объём базы данных:** 15 Гб. Превышение → тариф «Крупный бизнес».
- **Максимальное количество строк в одной Heap-таблице:** **1 000 000** (не 10 млн). Превышение → тариф «Private».

### Размер одной записи (важно для производительности)

Ориентиры от поддержки Chatium (не тарифный лимит, а особенность хранения на Postgres):

- **Оптимально — не больше ~8 КБ** на запись: все поля строки, упакованные в JSON. Это «здоровый» размер.
- **Больше 8 КБ — заметно медленнее** из-за внутренних особенностей Postgres (TOAST и т.п.). По возможности не превышать.
- Если действительно нужно больше — **пара мегабайт считается терпимой**, но всё равно нежелательна.
- Всё, что крупнее, — **обращаться в поддержку** со своим конкретным кейсом.
- **Heap — для структурированных данных, не для больших блобов.** Блобы (крупные тексты, файлы, base64) храните в **файловом хранилище** (`ctx.storage`, см. `009-files.md`), а в Heap — только ссылку/метаданные.

Практические следствия:

- Поля `Object`/`Array`/`any` с произвольным JSON держите компактными; крупный контент выносите в файловое хранилище.
- При упаковке нескольких логических записей в одну строку (архивные батчи, агрегаты) **ориентируйтесь на ~8 КБ на строку**, а не на максимум, — набирайте батч по размеру, а не по фиксированному числу.

При проектировании append-only/журнальных таблиц (историю событий, логи, доставки и т.п.) закладывайте retention/очистку заранее — таблица без чистки на 1 млн строк упрётся в лимит.

### Латентность операций

Замеры 21-07-2026, 10 независимых прогонов + 1000 отдельных вызовов.

| Операция | Длительность |
|---|---|
| **Первая** операция Heap в вызове — любая, включая `countBy` | **35–150 мс** |
| Каждая последующая: `findAll` (10 строк) | 2–3 мс |
| Каждая последующая: `updateAll` (CAS по `id`) | 2–33 мс, медиана 4–5 мс |
| `create` пачкой (`Promise.all` по 200) | ~1900 строк/с |
| `deleteAll` (`limit: null, hard: true`) | ~0.8 мс на строку |

**Первое обращение к Heap в вызове стоит на порядок дороже остальных** — воспроизводится стабильно, во всех 10 прогонах, и не зависит от типа операции: медленным оказывается именно первый вызов, будь то `countBy`, `findAll` или `updateAll`. Дальше латентность падает до единиц миллисекунд.

Практическое следствие ровно одно: **не оценивайте стоимость операции по одному замеру** — разовый вызов покажет ~100 мс там, где в цикле он стоит 4 мс. На бюджет обработчика (20 с) эта надбавка не влияет: она разовая и не накапливается.

⚠️ **Зависимость от простоя не проверена.** Растёт ли эта первая задержка после минут бездействия — неизвестно: `setTimeout` на сервере нет, а из workspace-инструментария выдержать реальную паузу между вызовами не получается. Наблюдения сделаны на вызовах, идущих подряд.

## Создание таблицы

### Heap.Table - определение таблицы

**Базовый синтаксис**: первый аргумент — **идентификатор таблицы Heap** (строка вида `t__project__table__xyz`), второй — объект полей, третий (опционально) — опции таблицы (`customMeta: { title, description }`). Поля при необходимости оборачиваются в `Heap.Optional()`.

```typescript
import { Heap } from '@app/heap'

export const TableName = Heap.Table(
  't__project__table__xyz',
  {
    fieldName: Heap.Optional(Heap.FieldType(options))
    // ... другие поля
  },
  { customMeta: { title: 'Название таблицы', description: 'Описание' } }
)
```

**Полный пример**:

```typescript
import { Heap } from '@app/heap'

export const Products = Heap.Table(
  't__project__products__Ab1Cd2',
  {
    name: Heap.Optional(
      Heap.String({
        customMeta: { title: 'Название' },
        searchable: { langs: ['ru', 'en'], embeddings: false } // ✅ По умолчанию ВЫКЛЮЧЕНО
      })
    ),
    price: Heap.Optional(
      Heap.Money({
        customMeta: { title: 'Цена' }
      })
    ),
    inStock: Heap.Optional(
      Heap.Boolean({
        customMeta: { title: 'В наличии' },
        defaultValue: true
      })
    ),
    description: Heap.Optional(
      Heap.String({
        customMeta: { title: 'Описание' },
        searchable: { langs: ['ru'], embeddings: false } // ✅ По умолчанию ВЫКЛЮЧЕНО
      })
    )
  },
  { customMeta: { title: 'Товары', description: 'Каталог товаров' } }
)
```

### Типы полей

| Тип               | Описание         | TypeScript тип |
| ----------------- | ---------------- | -------------- |
| `Heap.String()`   | Строка           | `string`       |
| `Heap.Number()`   | Число            | `number`       |
| `Heap.Boolean()`  | Булево           | `boolean`      |
| `Heap.Money()`    | Деньги           | `Money`        |
| `Heap.DateTime()` | Дата и время     | `Date`         |
| `Heap.RefLink()`  | Ссылка на запись | `RefLink<T>`   |
| `Heap.Array()`    | Массив           | `Array<T>`     |
| `Heap.Object()`   | Объект           | `Object`       |
| `Heap.Enum()`     | Перечисление     | `union type`   |
| `Heap.Any()`      | Любой тип        | `any`          |

**Зарезервированные поля**: имена `id`, `createdAt`, `updatedAt` являются системными и добавляются ко всем таблицам автоматически. **Не объявляйте их** в объекте полей (второй аргумент `Heap.Table`) — см. [Ошибка #4](#-ошибка-4-ручное-добавление-полей-id-createdat-updatedat).

### Пример таблицы

**⚠️ КРИТИЧЕСКИ ВАЖНО**: Heap таблицы создаются как **файлы с расширением `.table.ts`**, но при импорте расширение `.ts` **НЕ указывается**!

```typescript
// tables/users.table.ts (С расширением .ts!)
import { Heap } from '@app/heap'

export const Users = Heap.Table(
  't__project__users__Xy1Zw2',
  {
    firstName: Heap.Optional(
      Heap.String({
        customMeta: { title: 'Имя' },
        searchable: { langs: ['ru', 'en'], embeddings: false }
      })
    ),
    lastName: Heap.Optional(
      Heap.String({
        customMeta: { title: 'Фамилия' }
      })
    ),
    email: Heap.Optional(
      Heap.String({
        customMeta: { title: 'Email' }
      })
    ),
    age: Heap.Optional(
      Heap.Number({
        customMeta: { title: 'Возраст' },
        defaultValue: 0
      })
    ),
    isActive: Heap.Optional(
      Heap.Boolean({
        customMeta: { title: 'Активен' },
        defaultValue: true
      })
    )
  },
  { customMeta: { title: 'Пользователи', description: 'Пользователи системы' } }
)

export default Users
```

**Ключевые правила**:

- ✅ Файл: `tables/users.table.ts` (С расширением .ts)
- ✅ Импорт: `import { Users } from '../tables/users.table'` (БЕЗ .ts)
- ❌ Импорт: `import { Users } from '../tables/users.table.ts'` (НЕ указывайте .ts при импорте!)
- ✅ Используйте TypeScript и `Heap.Table()` в файле таблицы
- ✅ Экспортируйте таблицу: `export const Users = ...`

### Окружения stage/prod: сегмент окружения в id таблицы

**Динамические имена таблиц невозможны.** Первый аргумент `Heap.Table()` — статичный строковый литерал; вычислить имя таблицы в рантайме (на запрос, на тестовый прогон) нельзя.

**Идентичность таблицы — её строковый id, глобальный на аккаунт.** При этом **id может быть объявлен ровно в одном файле аккаунта**: повторный вызов `Heap.Table` с тем же id из другого файла — даже с идентичной схемой, даже из другого проекта — даёт BuildError сборки UGC (подтверждено прогоном 22-07-2026):

```
Detected duplicate heap table name 't__…' in file '<файл-2>',
conflicting heap table is defined in '<файл-1>' (variable name '…')
```

Коварство: у «проигравшего» файла ошибка может не всплыть явно — импортированные из него репозитории приходят `undefined`, и падает первый же вызов метода (`Cannot read properties of undefined`). Внутри **одного** файла два вызова с разными id — легальны.

**Следствие: паттерн «пара таблиц в одном файле + селектор» для двух копий проекта не работает** — обе копии объявляли бы одни и те же id, и вторая копия получает `undefined`-репозитории. (Пара в одном файле билдится, пока копия одна, — но блокирует появление второй копии; не применять.)

**Действующий паттерн — сегмент окружения в id; файл объявляет только таблицу своего окружения:**

```typescript
// tables/orders.table.ts — dev-копия (d/<project>)
import { Heap } from '@app/heap'

const orderFields = {
  userId: Heap.Optional(Heap.String({ customMeta: { title: 'Пользователь' } }))
  // ... поля — ровно один раз, общие для окружений по построению переноса
}

// Весь остальной код импортирует ЭТО и про окружения не знает
export const Orders = Heap.Table('t__project__orders__stage_Ab1Cd2', orderFields, {
  customMeta: { title: 'Заказы (stage)' }
})
```

Перенос кода в прод-копию (`p/<project>`) **механически трансформирует** сегмент id в `tables/*.table.ts`: `__stage_` → `__prod_` (и пометки `(stage)` → `(prod)` в customMeta) — наряду с `PROJECT_ROOT` в `config/routes.tsx` это два трансформируемых класса файлов переноса (`006-arch.md` → «Окружения stage/prod: два каталога проекта»). Схемы окружений не разъезжаются, потому что файлы копий идентичны с точностью до сегмента.

**Селектор `IS_PROD`** — статический, из `PROJECT_ROOT` (`config/env.ts`) — остаётся рычагом **не-табличных** аккаунт-глобальных идентичностей (ключи замков, сокет-каналы):

```typescript
// config/env.ts
import { PROJECT_ROOT } from './routes'

export const IS_PROD = PROJECT_ROOT.startsWith('p/')
```

**Общая таблица между окружениями напрямую невозможна** (id объявляется один раз): если данные должны быть общими, таблицу объявляет один выделенный проект, а остальные ходят к нему через его API/`app.function` — не через повторное объявление id.

**Каверзы**:

- **Эволюция схемы**: аддитивные изменения (добавить поле) безопасны; переименования и удаления полей — только синхронно с переносом кода между окружениями (схема привязана к id глобально).
- **Лимиты аккаунта общие**: stage-набор ест те же 15 Гб аккаунта; лимит 1 млн строк — на каждую таблицу отдельно. Закладывайте чистку stage-данных.
- **Для изоляции тестов паттерн не подходит**: тест должен проверять боевой путь, а не его копию — используйте неймспейс строк на боевых таблицах (изоляция строками) с чисткой по префиксу.
- **Проверьте остальные аккаунт-глобальные идентичности.** Сегмент в id таблиц не разделяет ключи замков (`runWithExclusiveLock`, `028-sync.md`), имена сокет-каналов (`sendDataToSocket`) и очередей — они тоже глобальны на аккаунт: без сегмента окружения stage- и prod-копии молча вытесняют друг друга. Правила — `006-arch.md` → «Окружения stage/prod: два каталога проекта».

## CRUD операции

### create - создание записи

```typescript
const record = await TableName.create(ctx, {
  field1: value1,
  field2: value2
})
```

**Пример**:

```typescript
// ✅ ПРАВИЛЬНО: Импорт БЕЗ расширения .ts
import { Products } from '../tables/products.table' // Именованный импорт
import { Money } from '@app/heap'

// ❌ НЕПРАВИЛЬНО: НЕ указывайте .ts при импорте
// import { Products } from '../tables/products.table.ts'

const product = await Products.create(ctx, {
  name: 'iPhone 15',
  price: new Money(999, 'USD'),
  inStock: true,
  description: 'Latest iPhone model'
})

ctx.account.log('Product created', {
  level: 'info',
  json: { productId: product.id }
})
```

### findAll - получение списка

```typescript
const records = await TableName.findAll(ctx, {
  where?: Where<T>,
  limit?: number,        // По умолчанию 1000, максимум 1000
  offset?: number,
  order?: Array<Order<T>>
})
```

**Примеры**:

```typescript
// Все записи
const products = await Products.findAll(ctx, {
  limit: 100
})

// С фильтром
const inStockProducts = await Products.findAll(ctx, {
  where: { inStock: true },
  limit: 50
})

// С сортировкой
const sortedProducts = await Products.findAll(ctx, {
  where: { inStock: true },
  order: [{ createdAt: 'desc' }],
  limit: 100
})
```

### findById - получение по ID

```typescript
const record = await TableName.findById(ctx, id)
// Возвращает null если не найдена
```

**Или getById** (выбрасывает ошибку если не найдена):

```typescript
const record = await TableName.getById(ctx, id)
// Выбросит ошибку если не найдена
```

**Пример**:

```typescript
const product = await Products.findById(ctx, 'product_id')

if (!product) {
  return { success: false, error: 'Product not found' }
}

return { success: true, product }
```

### findOneBy - поиск одной записи

```typescript
const record = await TableName.findOneBy(ctx, where)
// Возвращает null если не найдена
```

**Пример**:

```typescript
const product = await Products.findOneBy(ctx, {
  name: 'iPhone 15'
})

if (product) {
  ctx.account.log('Product found', {
    level: 'info',
    json: { id: product.id }
  })
}
```

### update - обновление записи

**Сигнатура**: метод принимает **два аргумента** — `ctx` и один объект с полем `id` и обновляемыми полями. Не три аргумента `(ctx, id, data)`.

```typescript
const updated = await TableName.update(ctx, {
  id: 'record_id',
  field1: newValue1,
  field2: newValue2
})
```

**Типичная ошибка TypeScript**: «Expected 2 arguments, but got 3» — если вызывать `TableName.update(ctx, id, data)`. В Heap нужно передавать один объект: `TableName.update(ctx, { id, ...data })`. Подробнее см. [Ошибка #9: update с тремя аргументами](#-ошибка-9-update-с-тремя-аргументами-ctx-id-data).

**Пример**:

```typescript
import { Money } from '@app/heap'

const updated = await Products.update(ctx, {
  id: 'product_id',
  price: new Money(899, 'USD'),
  inStock: false
})

ctx.account.log('Product updated', {
  level: 'info',
  json: { productId: updated.id }
})
```

### updateAll - условное обновление

Обновляет **все** записи, подходящие под `where`, и возвращает **число реально изменённых строк**. В отличие от `update`, работает не по `id`, а по условию.

```typescript
const changed = await TableName.updateAll(ctx, {
  patch: { field1: newValue },   // Что записать. ТОЛЬКО литеральные значения
  where?: Where<T>,              // Какие строки обновить
  limit: number | null           // Порог защиты, как у deleteAll
})
```

**Ключевые свойства**:

1. **Возвращает число изменённых строк.** Если под `where` не попало ничего — вернёт `0`, ошибки не будет.
2. **Операция атомарна.** Проверено параллельным прогоном: 10 одновременных попыток обновить одну строку с условием `{ id, status: 'pending' }` дают ровно **одного** победителя (вернул `1`) и девять проигравших (вернули `0`); записанное значение соответствует победителю. Это позволяет строить **оптимистичную блокировку (CAS)** без `runWithExclusiveLock`.
3. **`patch` принимает только литералы.** Выражения вида «прибавить единицу к текущему значению» не поддерживаются — новое значение нужно вычислить в коде и передать готовым.
4. **Порядок задать нельзя** — у `updateAll` нет параметра `order`. Поэтому «обновить N самых старых» одним вызовом не выйдет: под `where` попадёт произвольное подмножество.
5. **`limit` — предохранитель, а не размер партии** (проверено прогоном 21-07-2026). Если под `where` подходит **больше** строк, чем указано в `limit`, вызов падает и **не меняет ничего**:

   ```
   Accidental mass-update protection: going to update 10 records while limit is 5
   ```

   | `limit` | Подходит строк | Результат |
   |---|---|---|
   | `5` | 10 | ❌ ошибка, изменено **0** |
   | `10` | 10 | ✅ изменено 10 |
   | `20` | 10 | ✅ изменено 10 |
   | `null` | 10 | ✅ изменено 10 |
   | не указан | 10 | ❌ `Limit must be greater than 0 or explicit null` |

   Отсюда: «обновлять партиями по N в цикле» **не работает в принципе** — на первом же проходе, где подходящих строк больше N, вызов упадёт. Чтобы обновить всё подходящее, передавайте `limit: null`.

6. **`limit` обязателен.** Опустить параметр нельзя — будет `Limit must be greater than 0 or explicit null`. Это строже, чем у `findAll`, где `limit` необязателен (по умолчанию 1000).

7. **Стоимость вызова** (замер 21-07-2026): медиана **4–5 мс**, разброс 2–33 мс. Проверено на 1000 последовательных вызовов (5 прогонов × 200) — значений выше 33 мс не встретилось ни разу. 200 последовательных CAS-захватов занимают **0.9–1.0 с**. Отсюда практический потолок в обработчике роута (бюджет 20 с) — порядка **2000 итераций**; это расчёт от медианы, а не наблюдённый обрыв.

**CAS — захват строки без блокировки**:

```typescript
// Условие включает id (попадёт максимум одна строка, предохранитель не сработает)
// и текущее состояние — если его перехватили, условие не сойдётся
const won = await Tasks.updateAll(ctx, {
  patch: { status: 'processing', ownerId, attempts: task.attempts + 1 },
  where: { id: task.id, status: 'pending' },
  limit: 1
})

if (won === 1) {
  // Строка наша — обрабатываем
} else {
  // Перехватили, берём следующую
}
```

Такой CAS дешевле `runWithExclusiveLock` там, где нужно захватывать **отдельные строки**: параллельные обработчики не выстраиваются в очередь, а расходятся по разным строкам. Блокировка остаётся нужной для «проверить и создать» (см. [Предотвращение race condition](#предотвращение-race-condition)) — там защищать нечего условием, строки ещё нет.

**Смежное**: в типах есть также `updateMaybe(ctx, patch & { id })`. Поведение не проверялось, документации нет.

### deleteAll - массовое удаление

**⚠️ КРИТИЧЕСКИ ВАЖНО**: Метод `deleteAll` имеет защиту от случайного массового удаления!

```typescript
const deletedCount = await TableName.deleteAll(ctx, {
  where?: Where<T>,  // Условие фильтрации
  limit: number | null,  // ⚠️ КРИТИЧЕСКИ ВАЖНО: по умолчанию limit = 1
  hard?: boolean  // Жёсткое удаление (по умолчанию false)
})
```

**Ключевые правила**:

1. **По умолчанию `limit = 1`** — защита от случайного удаления всех записей
2. **Если записей больше чем `limit`** — удаление **БЛОКИРУЕТСЯ** и выбрасывается ошибка
3. **Для удаления всех записей** используйте `limit: null`
4. **Всегда проверяйте количество** через `countBy` перед удалением

**Примеры**:

```typescript
// ❌ НЕПРАВИЛЬНО - удалит только 1 запись (по умолчанию limit = 1)
await Products.deleteAll(ctx, {
  where: { inStock: false }
})

// ✅ ПРАВИЛЬНО - удалит все записи с inStock: false
await Products.deleteAll(ctx, {
  where: { inStock: false },
  limit: null // Удаляем все записи
})

// ❌ НЕПРАВИЛЬНО - это НЕ «удалить максимум 100»
const deleted = await Products.deleteAll(ctx, {
  where: { inStock: false },
  limit: 100
})
// Если под where попадает 101 запись — НИЧЕГО не удалится, будет выброшена ошибка
// «Accidental mass-delete protection». limit — это ПОРОГ СРАБАТЫВАНИЯ ЗАЩИТЫ,
// а не размер партии. Батчевое удаление через limit невозможно.

// ⚠️ РИСКОВАННО - гонка между countBy и deleteAll
const count = await Products.countBy(ctx, { inStock: false })
if (count > 0) {
  await Products.deleteAll(ctx, {
    where: { inStock: false },
    limit: count
  })
}
// Если между двумя вызовами под where попадёт новая запись, фактическое
// количество превысит limit — и удаление упадёт. Годится только там,
// где параллельная запись в таблицу исключена.

// ✅ ПРАВИЛЬНО для больших объёмов - партиционирование сужением where
// Режем набор на срезы, каждый удаляем целиком через limit: null.
// Предохранитель не срабатывает в принципе, объём работы за вызов ограничен
// шириной среза, параллельная запись безопасна.
for (const [from, to] of timeWindows) {
  await Products.deleteAll(ctx, {
    where: { $and: [{ createdAt: { $gte: from } }, { createdAt: { $lt: to } }] },
    limit: null
  })
}
```

**Проверено на рантайме (2026-07-20):** `deleteAll({ where, limit: 3 })` при 10 подходящих записях выбрасывает `Accidental mass-delete protection: going to delete 10 records while limit is 3, delete is cancelled!` и **не удаляет ничего**. `limit: null` на 1001 записи удаляет все за один вызов — скрытого ограничения в 1000 строк у `deleteAll` нет (в отличие от `findAll`).

**Стоимость удаления** (замер 21-07-2026, `limit: null`, `hard: true`):

| Строк | Длительность | На строку |
|---|---|---|
| 1 000 | 1.0 с | ~1.0 мс |
| 5 000 | 3.3 с | ~0.66 мс |
| 20 000 | 13.5 с | ~0.67 мс |
| 50 000 | 39.4 с | ~0.79 мс |

Рост близок к линейному, **≈0.8 мс на строку** (прямой замер до 50 000). Отсюда ориентиры: в обработчике роута — порядка **25 000 строк** за 20 с, в `app.job` — порядка **75 000** за 60 с. Прогон на 150 000 строк наблюдался косвенно (через опрос `countBy`): удаление завершилось на **90–95-й секунде**, что даёт ~0.6 мс/строку — того же порядка. Для планирования держитесь консервативных 0.8 мс/строку; точная скорость на больших объёмах и её зависимость от нагрузки кластера не измерены.

**Долгая операция дорабатывает до конца, но код после неё не выполняется.** Это тонкое, важное и проверенное разделение (прогоны 21–22-07-2026, объёмы 50 000–150 000 из роутов и `app.job`):

- сам `deleteAll` при превышении бюджета **доходит до конца** — таблица в итоге пуста во всех прогонах, отката и застрявших частичных состояний не наблюдалось;
- но **обработчик прерывается сразу после возврата операции** — код, идущий за `await deleteAll(...)`, не исполняется. В прогоне это проявилось так: отметка «начал», записанная до удаления, есть; отметка «закончил», которую обработчик пишет после, — отсутствует, хотя данные удалены.

Практические следствия, каждое с ценой:

- **Ничего важного после потенциально долгой операции.** Логирование итога, планирование следующего джоба (`scheduleJobAt`), запись «готово», любой второй шаг — всё это не выполнится, если предыдущая операция переросла бюджет. Самоперепланирующийся джоб, ставящий следующий запуск **в конце**, при переросшем проходе **остановится навсегда**. Ставьте перепланирование/критические side-effect'ы **до** тяжёлой работы или разбивайте её так, чтобы каждый вызов заведомо укладывался в бюджет.
- **Таймаут вызова ≠ провал операции.** Клиент, получивший ошибку по времени, не знает, выполнилась работа или нет — чаще всего выполняется. Перед повтором **проверяйте состояние**, а не повторяйте вслепую: повторная запись может задвоить данные.
- **Платформа не отличает «доработал, но код после не успел» от «упал с ошибкой».** Системного «job success» у переросшего джоба нет. Наблюдаемость такого прохода — только через ваши собственные логи, записанные **до** тяжёлого шага.

⚠️ **На какой секунде наступает прерывание — не измерено.** Джоб с бюджетом 60 с был прерван на задаче ~90 с, но точная граница неизвестна: возможно, бюджет — не жёсткий таймер, а ориентир. Клиент в прогонах обрывался раньше (по своему лимиту 5 с), так что момент прерывания именно платформой не наблюдался прямо.

Промежуточное состояние во время удаления: `countBy` девятью замерами по ходу операции возвращал полное исходное число, затем сразу ноль (150 000 → 0 без промежуточных значений). Это сильнее, чем один замер, но всё же **не гарантия атомарности видимости** — считайте состояние во время долгого `deleteAll` неопределённым.

Для сравнения — **вставка** тем же прогоном: ~1900 строк/с (параллельные батчи `Promise.all` по 200), то есть создание примерно вдвое дешевле удаления того же объёма.

**Ошибка "Accidental mass-delete protection"**:

Если вы видите эту ошибку, значит количество записей, соответствующих `where`, превышает `limit`. Решения:

1. Используйте `limit: null` для удаления всех записей
2. Увеличьте `limit` до нужного значения
3. Используйте точное количество через `countBy`

**Пример безопасного удаления всех зависимых записей**:

```typescript
// Удаляем все записи кэша для датасета
async function deleteDatasetCache(ctx: app.Ctx, datasetId: string): Promise<number> {
  // 1. Проверяем количество
  const totalCount = await CacheTable.countBy(ctx, {
    dataset_id: datasetId
  })

  if (totalCount === 0) {
    return 0
  }

  // 2. Удаляем все записи сразу
  const deleted = await CacheTable.deleteAll(ctx, {
    where: { dataset_id: datasetId },
    limit: null // Удаляем все записи
  })

  // 3. Финальная проверка
  const remaining = await CacheTable.countBy(ctx, {
    dataset_id: datasetId
  })

  if (remaining > 0) {
    throw new Error(`Не удалось удалить все записи. Осталось: ${remaining}`)
  }

  return deleted
}
```

**⚠️ ВАЖНО: Порядок удаления зависимых записей**

При удалении записей со связями `RefLink` **ВСЕГДА** удаляйте зависимые записи **ПЕРЕД** удалением родительской записи:

```typescript
// ❌ НЕПРАВИЛЬНО - вызовет "Data consistency error"
async function deleteDataset(ctx: app.Ctx, datasetId: string) {
  // Сначала удаляем датасет
  await Datasets.delete(ctx, datasetId) // ← ОШИБКА!

  // Потом пытаемся удалить кэш (но датасет уже удалён!)
  await deleteDatasetCache(ctx, datasetId)
}

// ✅ ПРАВИЛЬНО - сначала зависимые записи
async function deleteDataset(ctx: app.Ctx, datasetId: string) {
  // 1. Сначала удаляем все зависимые записи (кэш)
  await deleteDatasetCache(ctx, datasetId)

  // 2. Проверяем, что кэш действительно пустой
  const remainingCache = await CacheTable.countBy(ctx, {
    dataset_id: datasetId
  })

  if (remainingCache > 0) {
    throw new Error(`Невозможно удалить датасет: осталось ${remainingCache} записей в кэше`)
  }

  // 3. Только после полной очистки удаляем датасет
  await Datasets.delete(ctx, datasetId)
}
```

**Ошибка "Data consistency error! Reason: Heap object with idx = X has dependent links"**:

Эта ошибка возникает, когда вы пытаетесь удалить запись, на которую ссылаются другие записи через `RefLink`. Решение:

1. **Всегда удаляйте зависимые записи первыми**
2. **Проверяйте `countBy` после удаления** зависимых записей
3. **Удаляйте родительскую запись только после полной очистки зависимых**

## Фильтрация данных

**⚠️ ВАЖНО**: Для фильтрации используется параметр `where`, НЕ `filter`!

### Индексация: только плоские поля

**Индексируются поля плоских типов на корневом уровне записи** (подтверждено поддержкой Chatium). Всё остальное — составные типы и любая вложенность внутри них — запрашивается сканированием.

| Что | Индекс |
|---|---|
| `Heap.String`, `Heap.Number`, `Heap.Boolean`, `Heap.Enum`, `Heap.Date`, `Heap.RefLink`, `Heap.Money` — на корневом уровне | ✅ |
| `Heap.Array` — вхождение через `$includes`/`$any`/`$all` | ❌ скан |
| `Heap.Object` — вложенный путь (`{ props: { color: 'red' } }`) | ❌ скан |
| `Heap.Any` — любой запрос внутрь JSONB | ❌ скан |
| Системные `id`, `createdAt`, `updatedAt` | ✅ |

**Управлять индексами из кода нельзя.** В API нет ни создания индексов (`createIndex`, опция `indexes` у таблицы), ни хинтов у `findAll`, ни `explain()`/`plan()` у селект-билдера. Планы строит PostgreSQL, и наблюдать его выбор из workspace невозможно.

**Единственный доступный рычаг — форма схемы.** Объявление поля решает, попадёт ли оно под индекс вообще:

```typescript
// ❌ фильтр по версии уйдёт в скан JSONB
const Events = Heap.Table('events', { payload: Heap.Any() })
await Events.findAll(ctx, { where: { payload: { schemaVersion: 2 } } })

// ✅ версия вынесена в корневое плоское поле — индекс
const Events = Heap.Table('events', { schemaVersion: Heap.Number(), payload: Heap.Any() })
await Events.findAll(ctx, { where: { schemaVersion: 2 } })
```

Отсюда практическое правило: **поле, по которому фильтруют на горячем пути, должно быть корневым и плоского типа.** Объявить `Heap.Object`/`Heap.Array` вместо `Heap.Any` ради индекса бессмысленно — типобезопасность это даст, индекс нет.

Скан сам по себе не приговор: на таблице в сотни строк (реестр, справочник, настройки) он неотличим от индексного поиска. Проблема возникает только там, где таблица растёт.

### Простые фильтры

**Точное совпадение**:

```typescript
const products = await Products.findAll(ctx, {
  where: { inStock: true } // ✅ where, НЕ filter!
})
```

**IN запрос** (массив значений):

```typescript
const products = await Products.findAll(ctx, {
  where: {
    // ✅ where, НЕ filter!
    category: ['electronics', 'computers', 'phones']
  }
})
```

**IN по системному `id`** — работает так же, отдельного оператора не требуется (проверено прогоном 21-07-2026; `$in` не нужен):

```typescript
// Массовая операция по заранее собранному списку строк
await Deliveries.updateAll(ctx, {
  patch: { status: 'claimed' },
  where: { id: [...ids], status: 'pending' },   // → WHERE id IN (...)
  limit: null,
})
```

Это делает возможной **пакетную обработку в два запроса** там, где иначе понадобился бы цикл `1 + N`: сначала `findAll` собирает `id`, затем один `updateAll` меняет весь набор. Замер на 100 строках: **60 мс** против **704 мс** у последовательного варианта — почти в 12 раз быстрее. Цена — в `patch` только литералы, поэтому «прибавить единицу к текущему значению каждой строки» так не сделать (см. [updateAll](#updateall---условное-обновление)).

**❌ НЕПРАВИЛЬНО** (не используйте `filter`):

```typescript
// ❌ Параметр 'filter' НЕ существует в Chatium!
const products = await Products.findAll(ctx, {
  filter: { inStock: true } // ← НЕ РАБОТАЕТ!
})
```

### Операторы сравнения

```typescript
// Меньше
{ price: { $lt: 1000 } }

// Меньше или равно
{ price: { $lte: 1000 } }

// Больше
{ price: { $gt: 500 } }

// Больше или равно
{ price: { $gte: 500 } }
```

**⚠️ Диапазон на одно поле — только через `$and`.** Значение поля принимает **ровно один** оператор сравнения (в типах это `HsCompareOperators` — union одиночных операторов). Форма `{ price: { $gte: 500, $lte: 1000 } }` возвращает результат только по **одной** из границ (какая именно — не гарантируется), вторая молча игнорируется — **подтверждено поддержкой Chatium**. Для диапазона всегда пишите через `$and`:

```typescript
const midrange = await Products.findAll(ctx, {
  where: {
    $and: [{ price: { $gte: 500 } }, { price: { $lte: 1000 } }]
  }
})
```

**Пример**:

```typescript
const affordableProducts = await Products.findAll(ctx, {
  where: {
    price: { $lte: new Money(500, 'USD') },
    inStock: true
  }
})
```

### Логические операторы

**$and** - все условия должны выполняться:

```typescript
const products = await Products.findAll(ctx, {
  where: {
    $and: [{ inStock: true }, { price: { $lt: new Money(1000, 'USD') } }]
  }
})
```

**$or** - хотя бы одно условие:

```typescript
const products = await Products.findAll(ctx, {
  where: {
    $or: [{ category: 'electronics' }, { featured: true }]
  }
})
```

**$not** - отрицание:

```typescript
const products = await Products.findAll(ctx, {
  where: {
    $not: { inStock: false }
  }
})
```

**Комбинирование**:

```typescript
const products = await Products.findAll(ctx, {
  where: {
    $and: [
      { inStock: true },
      {
        $or: [{ category: 'electronics' }, { price: { $lt: new Money(500, 'USD') } }]
      }
    ]
  }
})
```

### Оператор $not на уровне поля

Внутри значения поля `$not` работает как `!=` (или `NOT IN` для массива). Это не то же самое, что логический `$not` на уровне всего `where` (он инвертирует условие целиком, см. выше).

```typescript
// Все заказы, кроме отменённых
await Orders.findAll(ctx, { where: { status: { $not: 'cancelled' } } })

// Кроме нескольких статусов (NOT IN)
await Orders.findAll(ctx, { where: { status: { $not: ['cancelled', 'refunded'] } } })
```

### Фильтрация BooleanKind (три состояния)

Поле `Heap.Boolean()` имеет три состояния: `true`, `false` и **не задано** (`null`/`undefined`). Частая ловушка: `{ isActive: false }` находит только строки, где поле **явно** `false`, и пропускает те, где значения нет.

```typescript
// ❌ Пропустит записи, где isActive не задано
await Subscriptions.findAll(ctx, { where: { isActive: false } })

// ✅ «Не активно» = false ИЛИ не задано
await Subscriptions.findAll(ctx, { where: { isActive: { $not: true } } })
```

**Правило:** для семантики «не включено / не активно / не выполнено» используйте `{ field: { $not: true } }`, а не `{ field: false }`.

### Вложенные поля ObjectKind

Для поля типа `Heap.Object({...})` фильтр по вложенному свойству задаётся **вложенным объектом**, а не строкой с точкой:

```typescript
// Таблица: settings: Heap.Object({ theme: Heap.String(), notifications: Heap.Boolean() })
await Users.findAll(ctx, {
  where: {
    settings: { theme: 'dark' } // ✅ вложенный объект
  }
})
```

> В `where` путь к вложенному полю — это **вложенный объект** (`{ settings: { theme: 'dark' } }`). В `select` тот же путь задаётся **массивом** (`['settings', 'theme']`, см. раздел про `select`). Не путать эти две формы.

### Строки и массивы: $ilike, $includes

```typescript
// Строка по шаблону (LIKE); % — любое число символов
await Users.findAll(ctx, { where: { email: { $ilike: '%@gmail.com' } } })

// Массив: вхождение одного значения
await Posts.findAll(ctx, { where: { tags: { $includes: 'javascript' } } })

// Массив: хотя бы одно из / все из
await Posts.findAll(ctx, { where: { tags: { $includes: { $any: ['js', 'ts'] } } } })
await Posts.findAll(ctx, { where: { tags: { $includes: { $all: ['js', 'ts'] } } } })
```

### Поля AnyKind (Heap.Any)

Фильтрация по `Heap.Any()` работает **той же нотацией, что и по обычным полям** (вложенный объект, операторы, `$includes`), но **менее эффективно**: значение лежит в JSONB без выделенного индекса, запрос идёт сканированием.

Для данных, по которым часто фильтруют, поле выносят **на корневой уровень плоским типом** — только так оно получает индекс (см. «Индексация: только плоские поля»). Замена `Heap.Any` на `Heap.Object`/`Heap.Array` даёт типобезопасность, но **не индекс**: вложенное в составные типы сканируется так же.

Что доступно по `Heap.Any()` (**подтверждено поддержкой Chatium**):

```typescript
// Таблица: payload: Heap.Any()

// Частичный матч по объекту — НЕ требует полного совпадения значения.
// Найдёт запись со значением { theme: 'dark', fontSize: 14 }:
await Events.findAll(ctx, { where: { payload: { theme: 'dark' } } })

// Фильтр по вложенному пути — вложенным объектом (не строкой с точкой):
await Events.findAll(ctx, { where: { payload: { user: { role: 'admin' } } } })

// Операторы сравнения $gt/$lt/$gte/$lte работают; тип сравнения
// определяется по ПРАВОЙ части: справа число → значение приводится к числу,
// справа Date → к дате:
await Events.findAll(ctx, { where: { payload: { amount: { $gt: 1000 } } } })

// Массивы внутри — через $includes / $any / $all, как у ArrayKind:
await Events.findAll(ctx, { where: { payload: { tags: { $includes: 'vip' } } } })
await Events.findAll(ctx, { where: { payload: { tags: { $includes: { $any: ['vip', 'paid'] } } } } })
await Events.findAll(ctx, { where: { payload: { tags: { $includes: { $all: ['vip', 'paid'] } } } } })
```

> Диапазон на одном вложенном поле внутри `Any` подчиняется тому же правилу, что и обычные поля: два оператора сразу ненадёжны, для диапазона — `$and` (см. выше).

### Шпаргалка: фильтры по типам полей

| Тип поля               | Пример фильтра                                                | Примечание                       |
| ---------------------- | ------------------------------------------------------------ | -------------------------------- |
| String                 | `{ name: 'Alice' }`                                          | точное совпадение                |
| String                 | `{ name: ['Alice', 'Bob'] }`                                | IN                               |
| String                 | `{ name: { $not: 'Alice' } }`                               | !=                               |
| String                 | `{ email: { $ilike: '%@gmail.com' } }`                      | поиск по шаблону (LIKE)          |
| Number                 | `{ age: { $gte: 18 } }`                                     | один оператор на поле            |
| Number (диапазон)      | `{ $and: [{ age: { $gte: 18 } }, { age: { $lte: 65 } }] }`  | диапазон через `$and`            |
| Money                  | `{ price: { $gt: new Money(1000, 'USD') } }`               | сравнение работает               |
| DateTime               | `{ createdAt: { $gte: new Date(...) } }`                   | объект `Date`                    |
| Boolean (включён)      | `{ isActive: true }`                                        | только `true`                    |
| Boolean (не включён)   | `{ isActive: { $not: true } }`                             | `false` + не задано              |
| Enum                   | `{ status: 'paid' }` / `{ status: ['paid', 'shipped'] }`   | значение или IN                  |
| RefLink                | `{ category: 'id' }` / `{ category: ['id1', 'id2'] }`      | ID строкой, не объект            |
| UserRefLink            | `{ userId: ctx.user.id }`                                  | ID пользователя                  |
| Array                  | `{ tags: { $includes: 'js' } }`                            | вхождение; `$any`/`$all` для множеств; **скан** (не индексируется) |
| Object (вложенное)     | `{ props: { color: 'red' } }`                              | вложенный объект (частичный матч); **скан** |
| Any (`Heap.Any`)       | `{ payload: { theme: 'dark' } }` / `{ payload: { tags: { $includes: 'vip' } } }` | как обычные поля: частичный матч, вложенный путь, сравнения, `$includes`; **скан** (JSONB без индекса) |

Индексируются только плоские типы на корневом уровне — см. «Индексация: только плоские поля».

## Сортировка

**⚠️ КРИТИЧЕСКИ ВАЖНО**: Chatium использует специфический синтаксис сортировки!

### Правильный синтаксис

```typescript
order: [{ fieldName: 'asc' | 'desc' }]
```

**Ключевые правила**:

- ✅ Имя поля = ключ объекта
- ✅ Направление сортировки = значение ('asc' или 'desc')
- ✅ Только одно поле в каждом объекте массива
- ❌ НЕ используйте `{ field: ..., direction: ... }` (это синтаксис других ORM!)

### Примеры правильной сортировки

**По одному полю**:

```typescript
const products = await Products.findAll(ctx, {
  order: [{ createdAt: 'desc' }]
})

const users = await Users.findAll(ctx, {
  order: [{ lastName: 'asc' }]
})
```

**По нескольким полям**:

```typescript
const products = await Products.findAll(ctx, {
  order: [
    { priority: 'desc' }, // Сначала по приоритету (высокий первый)
    { name: 'asc' } // Потом по имени (алфавит)
  ]
})

const orders = await Orders.findAll(ctx, {
  order: [
    { status: 'asc' }, // По статусу
    { createdAt: 'desc' } // По дате создания (новые первые)
  ]
})
```

### ❌ НЕПРАВИЛЬНЫЕ примеры (НЕ ДЕЛАЙТЕ ТАК!)

```typescript
// ❌ НЕПРАВИЛЬНО - синтаксис TypeORM/Sequelize
const products = await Products.findAll(ctx, {
  order: [{ field: 'title', direction: 'asc' }] // ← НЕ РАБОТАЕТ!
})

// ❌ НЕПРАВИЛЬНО - несколько полей в одном объекте
const products = await Products.findAll(ctx, {
  order: [{ priority: 'desc', name: 'asc' }] // ← РАБОТАЕТ, НО НЕ РЕКОМЕНДУЕТСЯ
})

// ❌ НЕПРАВИЛЬНО - строка вместо массива
const products = await Products.findAll(ctx, {
  order: 'createdAt DESC' // ← НЕ РАБОТАЕТ!
})
```

### Почему это важно

Chatium **НЕ** использует стандартный синтаксис TypeORM, Sequelize или других популярных ORM. Не путайте синтаксис разных фреймворков! Использование неправильного синтаксиса приведет к ошибкам или непредсказуемому поведению.

## Подсчёт записей

### countBy - подсчёт с фильтром

```typescript
const count = await TableName.countBy(ctx, where?)
```

**Примеры**:

```typescript
// Всего записей
const total = await Products.countBy(ctx)

// С фильтром
const inStockCount = await Products.countBy(ctx, {
  inStock: true
})

// Сложный фильтр
const count = await Products.countBy(ctx, {
  $and: [{ inStock: true }, { price: { $lte: new Money(500, 'USD') } }]
})
```

**Важно**: Не используйте `findAll` + `.length` для подсчёта! `findAll` ограничен 1000 записями (лимит по умолчанию и максимум), поэтому `.length` вернёт неверное число на больших таблицах. Для счётчика — `countBy`. Когда нужен не просто счётчик, а сумма/среднее/группировка или несколько агрегатов за один проход — используйте query builder `select` (см. [Агрегации и вычисления](#агрегации-и-вычисления-select)).

## Агрегации и вычисления (select)

`countBy` считает строки. Когда нужны сумма, среднее, минимум/максимум, конкатенация, группировка или вычисляемые колонки — используется query builder `select`. Запрос выполняется **на стороне БД** (не тянет строки в память), поэтому не подчиняется лимиту 1000 записей `findAll`.

Метод `select` (как и `selectAll`, `countBy`) есть у каждой таблицы Heap.

### Синтаксис select

```typescript
await Table
  .select({ alias: expression /* , ... */ }) // псевдонимы → выражения
  .where(/* фильтр ДО агрегации */)
  .group(/* псевдонимы */)
  .having(/* фильтр ПОСЛЕ агрегации */)
  .order(/* сортировка */)
  .limit(10) // и .offset(n) — пагинация
  .run(ctx) // выполнить → Promise<Array<{ alias: ... }>>
```

- Ключи объекта в `select({...})` — **псевдонимы (aliases)** колонок результата.
- Значение псевдонима — выражение (`HqExpr`): имя поля, путь, функция, агрегат или литерал.
- `.run(ctx)` **обязателен** — без него вернётся билдер, а не данные.
- Типы результата **выводятся** из select-клаузы (это не `any[]`).

```typescript
// Объектная форма — псевдонимы + выражения
const rows = await Orders.select({
  st: 'status', // колонка как есть
  total: { $sum: ['amount'] } // агрегат
})
  .group('st')
  .run(ctx)
// rows: Array<{ st: string; total: number }>  ← типы выведены

// Краткая форма — только имена колонок
const cols = await Orders.select('id', 'status').run(ctx)

// Все поля
const all = await Orders.selectAll().where({ status: 'paid' }).run(ctx)
```

### Виды выражений

Каждое выражение внутри `select` — одно из:

| Вид                     | Синтаксис                | Пример                          |
| ----------------------- | ------------------------ | ------------------------------- |
| Колонка                 | строка — имя поля        | `'amount'`, `'status'`          |
| Путь к вложенному полю  | массив имён              | `['settings', 'theme']`         |
| Литерал                 | `{ $dyn: value }`        | `{ $dyn: ' ' }`, `{ $dyn: 0.9 }`|
| Функция                 | `{ $fn: [args] }`        | `{ $concat: [...] }`            |
| Агрегат                 | `{ $agg: [args] }`       | `{ $count: ['*'] }`             |

Аргументы функций и агрегатов — массив выражений, поэтому их можно вкладывать друг в друга.

> **Путь vs where.** В `select` вложенное поле — это **массив** (`['settings', 'theme']`). В `where` тот же путь задаётся **вложенным объектом** (`{ settings: { theme: 'dark' } }`).

### Агрегатные функции

Все агрегаты принимают массив-аргумент и поддерживают опциональный сиблинг-ключ `$distinct: true` (не аргумент, а соседний ключ объекта).

| Агрегат        | Возвращает            | Аргумент                          |
| -------------- | --------------------- | --------------------------------- |
| `$count`       | number                | `['*']`, `[]` или `['field']`     |
| `$sum`         | number                | `['numField']`                    |
| `$avg`         | number                | `['numField']`                    |
| `$max`         | number/string/date    | `['field']`                       |
| `$min`         | number/string/date    | `['field']`                       |
| `$string_agg`  | string                | `['strField', 'разделитель']`     |
| `$bool_and`    | boolean               | `['boolField']`                   |
| `$bool_or`     | boolean               | `['boolField']`                   |

```typescript
// Подсчёт всех строк ('*' или пустой массив [] — эквивалентны)
const totalRes = await Orders.select({ total: { $count: ['*'] } }).run(ctx)
const total = totalRes[0]?.total ?? 0

// Подсчёт непустых значений поля
await Customers.select({ withEmail: { $count: ['email'] } }).run(ctx)

// Подсчёт уникальных ($distinct — сиблинг-ключ)
await Orders.select({ buyers: { $count: ['userId'], $distinct: true } }).run(ctx)

// Несколько агрегатов за один проход
const stats = await Orders.select({
  revenue: { $sum: ['amount'] },
  avgCheck: { $avg: ['amount'] },
  maxCheck: { $max: ['amount'] },
  minCheck: { $min: ['amount'] }
})
  .where({ status: 'paid' })
  .run(ctx)

// Склейка значений в строку через разделитель (разделитель — литерал $dyn)
await Orders.select({ ids: { $string_agg: ['id', { $dyn: ', ' }] } }).run(ctx)
```

> **Важно про аргументы.** Строка внутри массива-аргумента — это **имя поля** (`'amount'` → колонка). Константу передавайте литералом `{ $dyn: ... }` (`{ $dyn: ', ' }`), иначе она будет истолкована как имя колонки.
>
> **Money.** Агрегируются числовые поля (`Heap.Number`/`Heap.Integer`). Для `Heap.Money` берите числовую часть по пути: `{ $sum: [['price', 'amount']] }`. В примерах ниже `amount` — числовое поле.

### Скалярные функции и арифметика

Работают над значениями строки (не агрегируют):

- **Числовые:** `$abs`, `$ceil`, `$floor`, `$round`, `$length` (длина строки).
- **Арифметика** (2+ числовых аргумента): `$add`, `$sub`, `$mul`, `$div`, `$mod`, `$exp` (степень); `$neg` — унарный минус.
- **Строковые:** `$concat` (склейка), `$lower`, `$upper`, `$coalesce` (первое непустое).
- **Дата:** `$now` (`[]`), `$date_trunc` (`[{ $dyn: 'month' }, 'dateField', { $dyn: 'UTC' }]`), `$coalesce`.
- **Литерал:** `$dyn` — константа (строка/число/boolean/Date); нужна, чтобы смешивать литералы с именами полей.

```typescript
// Конкатенация с литералом $dyn
await Users.select({
  fullName: { $concat: ['firstName', { $dyn: ' ' }, 'lastName'] }
}).run(ctx)

// Первое непустое значение
await Users.select({ name: { $coalesce: ['displayName', 'firstName'] } }).run(ctx)

// Вычисляемая колонка над числовым полем: значение × 0.9
await Products.select({ net: { $mul: ['weight', { $dyn: 0.9 }] } }).run(ctx)
```

### Группировка group и autoGroup

- `.group('alias')` или `.group(['a', 'b'])` — группирует по **псевдонимам** из `select` (не по именам полей таблицы!).
- `.autoGroup()` — автоматически группирует по всем неагрегатным псевдонимам.

```typescript
// Количество и сумма по каждому статусу
const byStatus = await Orders.select({
  st: 'status', // псевдоним 'st'
  count: { $count: ['id'] },
  total: { $sum: ['amount'] }
})
  .group('st') // ← псевдоним, НЕ имя поля 'status'
  .run(ctx)
// byStatus: Array<{ st: string; count: number; total: number }>
```

**⚠️ Частая ошибка:** в `group` передаётся псевдоним (ключ из `select`), а не имя поля таблицы. При псевдониме `st` вызов `.group('status')` не пройдёт типизацию.

### where, having, order, limit

- `.where(filter)` — фильтр **до** агрегации; тот же синтаксис `where`, что у `findAll`.
- `.having(filter)` — фильтр **после** агрегации, по псевдонимам (в т.ч. агрегатам).
- `.order(order)` — сортировка по псевдониму или полю; формат как у `findAll` (`[{ alias: 'desc' }]`).
- `.limit(n)` / `.offset(n)` — пагинация результата.

```typescript
// Категории, где продано больше 5 штук; топ-3 по выручке
const top = await Orders.select({
  cat: 'category',
  sold: { $count: ['id'] },
  revenue: { $sum: ['amount'] }
})
  .where({ status: 'paid' }) // до агрегации
  .group('cat')
  .having({ sold: { $gt: 5 } }) // после агрегации
  .order([{ revenue: 'desc' }])
  .limit(3)
  .run(ctx)
```

> `.having()` и `.order()`/`.limit()` существуют — сортировку и постфильтр агрегатов **не нужно** делать вручную в JavaScript.

### Запуск запроса: run и варианты

`.run(ctx)` возвращает массив с выведенными типами. Для агрегата без группировки это массив из одного элемента.

| Метод                      | Возвращает                              |
| -------------------------- | --------------------------------------- |
| `run(ctx)`                 | `Array<{...}>`                          |
| `runTakeFirst(ctx)`        | первый элемент или `null`               |
| `runTakeFirstOrThrow(ctx)` | первый элемент или ошибка               |
| `runFlatten(ctx)`          | массив значений первой колонки          |
| `runJson(ctx)` / `run*Json`| те же данные как «плоский» JSON         |

```typescript
// Одиночный агрегат удобно брать через runTakeFirst
const res = await Orders.select({ n: { $count: ['*'] } }).runTakeFirst(ctx)
const count = res?.n ?? 0

// runFlatten — быстро получить список значений одной колонки
const ids = await Orders.select({ id: 'id' })
  .where({ status: 'paid' })
  .runFlatten(ctx) // string[]
```

### Практические примеры

**Подсчёт: `countBy` vs `select`.**

```typescript
// Простой подсчёт — countBy короче
const paid = await Orders.countBy(ctx, { status: 'paid' })

// Несколько агрегатов сразу — только через select
const m = await Orders.select({
  cnt: { $count: ['*'] },
  sum: { $sum: ['amount'] }
})
  .where({ status: 'paid' })
  .runTakeFirst(ctx)
```

**Витрина: топ-5 клиентов по сумме покупок (без N+1).**

```typescript
const top5 = await Orders.select({
  userId: 'userId',
  spent: { $sum: ['amount'] },
  purchases: { $count: ['id'] }
})
  .where({ status: 'paid' })
  .group('userId')
  .order([{ spent: 'desc' }])
  .limit(5)
  .run(ctx)

// Обогащение данными пользователей одним batch-запросом
const users = await Users.findAll(ctx, { where: { id: top5.map((r) => r.userId) } })

```

**Антипаттерны select:**

```typescript
// ❌ Обращение к [0] без проверки — упадёт на пустом результате
// const n = (await Orders.select({ n: { $count: ['*'] } }).run(ctx))[0].n
```

## Работа со связями RefLink

### Определение RefLink

**Синтаксис**: `Heap.RefLink(tableIdentifier, options)` — первый аргумент **идентификатор таблицы Heap** целевой таблицы (строка вида `t__project__table__xyz`), второй — объект с **customMeta** и **onDelete**. Для опционального поля оборачивайте в `Heap.Optional(Heap.RefLink(...))`.

```typescript
import { Heap } from '@app/heap'

export const Orders = Heap.Table(
  't__project__orders__Mn3Op4',
  {
    customer: Heap.Optional(
      Heap.RefLink('t__project__customers__Qr5St6', {
        customMeta: { title: 'Клиент' },
        onDelete: 'none'
      })
    ),
    product: Heap.Optional(
      Heap.RefLink('t__project__products__Ab1Cd2', {
        customMeta: { title: 'Продукт' },
        onDelete: 'none'
      })
    ),
    quantity: Heap.Optional(
      Heap.Number({
        customMeta: { title: 'Количество' }
      })
    )
  },
  { customMeta: { title: 'Заказы' } }
)
```

**Опция onDelete** (для RefLink обязательна в рабочем коде): в текущей реализации используется `'none'` — не проверять ссылки при удалении целевой записи.

**Опциональный RefLink**: оборачивайте в `Heap.Optional(Heap.RefLink(...))`. При проверке используйте `if (record.refField?.id)`.

### Определение UserRefLink

**UserRefLink** — ссылка на пользователя системы. В опциях **обязательно** поле **onDelete**: `'restrict' | 'none'` (тип HsLinkOptions требует его). В рабочем коде используется `onDelete: 'none'`. Поле оборачивается в `Heap.Optional(Heap.UserRefLink(...))`.

```typescript
import { Heap } from '@app/heap'

export const Campaigns = Heap.Table(
  't__saas-ref__campaign__8Hn4Lx',
  {
    title: Heap.Optional(Heap.String({ customMeta: { title: 'Название' } })),
    ownerUserId: Heap.Optional(
      Heap.UserRefLink({
        customMeta: { title: 'Владелец' },
        onDelete: 'none'
      })
    )
  },
  { customMeta: { title: 'Кампании', description: 'Кампании' } }
)

export const CampaignMembers = Heap.Table(
  't__saas-ref__campaign_member__2Km5Ny',
  {
    campaignId: Heap.Optional(
      Heap.RefLink('t__saas-ref__campaign__8Hn4Lx', {
        customMeta: { title: 'Кампания' },
        onDelete: 'none'
      })
    ),
    userId: Heap.Optional(
      Heap.UserRefLink({
        customMeta: { title: 'Пользователь' },
        onDelete: 'none'
      })
    ),
    role: Heap.Optional(Heap.String({ customMeta: { title: 'Роль' } }))
  },
  { customMeta: { title: 'Участники кампании', description: 'Участники кампании' } }
)
```

### Создание с RefLink

При создании передавайте **ID строкой**:

```typescript
const order = await Orders.create(ctx, {
  customer: customerId, // Передаем ID строкой
  product: productId, // Передаем ID строкой
  quantity: 2
})
```

### Получение связанных данных

При чтении получаете **объект RefLink** с методами:

```typescript
const order = await Orders.findById(ctx, orderId)

// Прямой доступ к ID

// Получить полную запись
```

### Фильтрация по RefLink

Используйте **ID строкой** в фильтрах:

```typescript
// Один ID
const orders = await Orders.findAll(ctx, {
  where: {
    customer: customerId // Используй ID, не объект RefLink!
  }
})

// Множественная фильтрация (IN)
const orders = await Orders.findAll(ctx, {
  where: {
    customer: ['customer1_id', 'customer2_id', 'customer3_id']
  }
})

const orders = await Orders.findAll(ctx, {
```

Перед использованием в фильтрах проверяйте наличие RefLink: `if (record.refField?.id)` — иначе при пустом RefLink будет `undefined`.

### Удаление связи (установка в null)

Если поле RefLink опциональное, связь можно сбросить, передав `null` при обновлении:

```typescript

  id: product.id,
  author: null
})
```

### Batch loading — оптимизация при работе с RefLink

Избегайте N+1 запросов: не вызывайте `.get(ctx)` в цикле. Соберите все ID, загрузите связанные записи одним запросом и сопоставьте по map:

```typescript
const products = await Products.findAll(ctx, { limit: 100 })
const categoryIds = [...new Set(products.map((p) => p.category?.id).filter(Boolean))]
const categories = await Categories.findAll(ctx, { where: { id: categoryIds } })
const categoriesMap = new Map(categories.map((c) => [c.id, c]))

const productsWithCategories = products.map((product) => ({
  ...product,
  categoryData: categoriesMap.get(product.category?.id)
}))
```

### Циклические зависимости таблиц

Не импортируйте одну таблицу из файла другой таблицы. В RefLink указывайте **только идентификатор таблицы Heap** (строка вида `t__project__table__xyz`) первым аргументом, без импорта целевой таблицы — так циклические зависимости не возникают.

### RefLink runtime interface

```typescript
interface RefLink<T> {
  id: string // ID связанной записи
  get(ctx): Promise<T | null> // Получить полную запись
  getTitle(ctx): Promise<string | null> // Получить заголовок
  getTargetTableRepo(ctx): Promise<HeapTable<T>> // Получить таблицу
  toJSON(): string // Возвращает ID
}
```

## Предотвращение race condition

### Проблема race condition

**Race condition** (состояние гонки) возникает, когда несколько параллельных запросов одновременно работают с одними и теми же данными, что может привести к некорректным результатам.

**Типичные сценарии**:

- Два запроса одновременно проверяют существование записи через `findOneBy`, затем оба создают её через `createOrUpdateBy`
- Между `findOneBy` и `createOrUpdateBy` другой процесс может изменить данные
- Параллельные обновления одной записи могут перезаписать изменения друг друга

**Пример проблемного кода**:

```typescript
// ❌ НЕПРАВИЛЬНО - возможна race condition
const existingChat = await TelegramChats.findOneBy(ctx, { chatId: chatIdString })

if (existingChat) {
  // Между findOneBy и createOrUpdateBy другой запрос может создать запись
  await TelegramChats.createOrUpdateBy(ctx, 'chatId', {
    chatId: chatIdString,
    firstSeenAt: existingChat.firstSeenAt, // Может быть перезаписано!
    lastSeenAt: new Date()
  })
} else {
  // Два параллельных запроса могут оба создать запись
  await TelegramChats.createOrUpdateBy(ctx, 'chatId', {
    chatId: chatIdString,
    firstSeenAt: new Date(), // Может быть установлено неправильно!
    lastSeenAt: new Date()
  })
}
```

### Решение: runWithExclusiveLock

Используйте `runWithExclusiveLock` из `@app/sync` для гарантии, что только один процесс может выполнить критическую секцию кода для конкретного ключа. Полный справочник по модулю: [028-sync.md](028-sync.md).

**Синтаксис**:

```typescript
import { runWithExclusiveLock } from '@app/sync'

await runWithExclusiveLock(ctx, lockKey, options, async () => {
  // ВСЕ операции с БД должны быть ВНУТРИ блокировки
  // Это гарантирует атомарность выполнения
})
```

**Параметры**:

- `ctx` — контекст приложения
- `lockKey` — уникальный ключ блокировки (строка). **ВАЖНО**: Ключ должен быть одинаковым для всех параллельных запросов, работающих с одними данными
- `options` — опции блокировки (обычно пустой объект `{}`)
- `callback` — функция, которая выполнится с блокировкой

**Пример правильного использования**:

```typescript
// ✅ ПРАВИЛЬНО - с эксклюзивной блокировкой
import { runWithExclusiveLock } from '@app/sync'

const chatIdString = String(chatInfo.id)
const lockKey = `telegram-chat-${chatIdString}` // Ключ должен быть одинаковым для одного chatId

await runWithExclusiveLock(ctx, lockKey, {}, async () => {
  // ВСЕ операции с БД ВНУТРИ блокировки
  const existingChat = await TelegramChats.findOneBy(ctx, { chatId: chatIdString })

  if (existingChat) {
    // Обновляем существующий чат
    await TelegramChats.createOrUpdateBy(ctx, 'chatId', {
      chatId: chatIdString,
      firstSeenAt: existingChat.firstSeenAt, // Гарантированно сохранится оригинальное значение
      lastSeenAt: new Date()
    })
  } else {
    // Создаём новый чат (гарантированно только один раз)
    await TelegramChats.createOrUpdateBy(ctx, 'chatId', {
      chatId: chatIdString,
      firstSeenAt: new Date(),
      lastSeenAt: new Date()
    })
  }
})
```

### Ключевые правила

1. **Ключ блокировки должен быть одинаковым**:

   ```typescript
   // ✅ ПРАВИЛЬНО - одинаковый ключ для одного chatId
   const lockKey = `telegram-chat-${chatIdString}`

   // ❌ НЕПРАВИЛЬНО - разные ключи для одного chatId
   const lockKey = `telegram-chat-${chatIdString}-${Date.now()}` // Каждый запрос получит свой ключ!
   ```

2. **Все операции с БД должны быть внутри блокировки**:

   ```typescript
   // ❌ НЕПРАВИЛЬНО - findOneBy снаружи блокировки
   const existingChat = await TelegramChats.findOneBy(ctx, { chatId: chatIdString })

   await runWithExclusiveLock(ctx, lockKey, {}, async () => {
     // Между findOneBy и createOrUpdateBy может произойти другой запрос!
     await TelegramChats.createOrUpdateBy(ctx, 'chatId', { ... })
   })

   // ✅ ПРАВИЛЬНО - все операции внутри
   await runWithExclusiveLock(ctx, lockKey, {}, async () => {
     const existingChat = await TelegramChats.findOneBy(ctx, { chatId: chatIdString })
     await TelegramChats.createOrUpdateBy(ctx, 'chatId', { ... })
   })
   ```

3. **Блокировка работает только для одинаковых ключей**:

   ```typescript
   // Запрос 1: lockKey = "telegram-chat-123"
   await runWithExclusiveLock(ctx, "telegram-chat-123", {}, async () => { ... })

   // Запрос 2: lockKey = "telegram-chat-123" → будет ждать завершения запроса 1
   await runWithExclusiveLock(ctx, "telegram-chat-123", {}, async () => { ... })

   // Запрос 3: lockKey = "telegram-chat-456" → выполнится параллельно с запросом 1
   await runWithExclusiveLock(ctx, "telegram-chat-456", {}, async () => { ... })
   ```

### Когда использовать

✅ **Используйте `runWithExclusiveLock` когда**:

- Несколько параллельных запросов могут работать с одной записью
- Между проверкой (`findOneBy`) и изменением (`createOrUpdateBy`) может произойти другой запрос
- Нужно гарантировать атомарность последовательности операций
- Важно сохранить оригинальные значения полей (например, `firstSeenAt`)

❌ **НЕ используйте `runWithExclusiveLock` когда**:

- Операции уже атомарны (например, один `createOrUpdateBy` без предварительной проверки)
- Работаете с разными записями (разные ключи блокировки)
- Нет риска параллельного доступа

### Производительность

- Блокировка выполняется на уровне приложения, не блокирует всю БД
- Запросы с разными ключами блокировки выполняются параллельно
- Запросы с одинаковым ключом выполняются последовательно (второй ждёт завершения первого)
- Внутри блокировки можно выполнять любые операции: работу с таблицами, внешние запросы и т.д.

## Частые ошибки при работе с Heap Tables

### ❌ Ошибка #1: Включение embeddings без необходимости

**НЕПРАВИЛЬНО**:

```typescript
// ❌ Embeddings включены без явной необходимости
export const Products = Heap.Table('products', {
  name: Heap.String({
    searchable: {
      langs: ['ru', 'en'],
      embeddings: true // ← Увеличивает нагрузку и затраты!
    }
  }),
  description: Heap.String({
    searchable: {
      langs: ['ru'],
      embeddings: true // ← Без необходимости!
    }
  })
})
```

**ПРАВИЛЬНО**:

```typescript
// ✅ По умолчанию embeddings ВЫКЛЮЧЕНЫ
export const Products = Heap.Table('products', {
  name: Heap.String({
    searchable: {
      langs: ['ru', 'en'],
      embeddings: false // ✅ По умолчанию
    }
  }),
  description: Heap.String({
    searchable: {
      langs: ['ru'],
      embeddings: false // ✅ По умолчанию
    }
  })
})

// ✅ Включайте только при необходимости семантического поиска
export const Articles = Heap.Table('articles', {
  content: Heap.String({
    searchable: {
      langs: ['ru'],
      embeddings: true // Осознанное решение для смыслового поиска
    }
  })
})
```

**Почему это важно**:

- Эмбединги значительно увеличивают нагрузку на систему
- Увеличивают стоимость операций записи и поиска
- Требуются только для семантического (смыслового) поиска
- Для обычного полнотекстового поиска достаточно `embeddings: false`

**Когда включать embeddings: true**:

- ✅ Поиск по смыслу, а не по точным словам
- ✅ Поиск синонимов и близких по смыслу запросов
- ✅ Многоязычный поиск с автоматическим переводом смысла

**Когда НЕ нужны embeddings**:

- ❌ Обычный поиск по названиям, артикулам, именам
- ❌ Поиск по точным совпадениям
- ❌ Простой полнотекстовый поиск

### ❌ Ошибка #2: Неправильный импорт таблиц (с расширением .ts)

**НЕПРАВИЛЬНО**:

```typescript
// ❌ НЕ указывайте .ts при импорте!
import { Services } from '../tables/services.table.ts'
import { Settings } from '../tables/settings.table.ts'
```

**ПРАВИЛЬНО**:

```typescript
// ✅ Импорт БЕЗ расширения .ts
import { Services } from '../tables/services.table'
import { Settings } from '../tables/settings.table'
```

**Структура файлов**:

```
tables/
├── services.table.ts     ← Файл С расширением .ts
└── settings.table.ts     ← Файл С расширением .ts
```

**Содержимое файла таблицы (ПРАВИЛЬНО)**:

```typescript
// tables/services.table.ts
import { Heap } from '@app/heap'

export const Services = Heap.Table('services', {
  title: Heap.String({
    customMeta: { title: 'Название сервиса' }
  })
})

export default Services
```

**Почему это важно**:

- Файл создаётся С расширением `.table.ts`
- При импорте расширение `.ts` НЕ указывается (стандарт TypeScript/JavaScript)
- Неправильный импорт приведет к ошибкам разрешения модулей

### ❌ Ошибка #3: Неправильный синтаксис параметра order

**НЕПРАВИЛЬНО** (как в других ORM):

```typescript
// ❌ Это синтаксис других фреймворков, НЕ Chatium!
const services = await ServicesTable.findAll(ctx, {
  order: [{ field: 'title', direction: 'asc' }], // ← НЕПРАВИЛЬНО!
  limit: 1000
})

const settings = await SettingsTable.findAll(ctx, {
  order: [{ field: 'key', direction: 'desc' }], // ← НЕПРАВИЛЬНО!
  limit: 1000
})
```

**ПРАВИЛЬНО** (синтаксис Chatium):

```typescript
// ✅ В Chatium имя поля = ключ, направление = значение
const services = await ServicesTable.findAll(ctx, {
  order: [{ title: 'asc' }], // ← ПРАВИЛЬНО!
  limit: 1000
})

const settings = await SettingsTable.findAll(ctx, {
  order: [{ key: 'desc' }], // ← ПРАВИЛЬНО!
  limit: 1000
})
```

**Сравнение синтаксиса**:

```typescript
// ❌ НЕПРАВИЛЬНО (синтаксис TypeORM/Sequelize)
order: [{ field: 'title', direction: 'asc' }]
order: [{ field: 'key', direction: 'desc' }]
order: [{ field: 'updatedAt', direction: 'asc' }]

// ✅ ПРАВИЛЬНО (синтаксис Chatium)
order: [{ title: 'asc' }]
order: [{ key: 'desc' }]
order: [{ updatedAt: 'asc' }]
```

**Сортировка по нескольким полям**:

```typescript
// ❌ НЕПРАВИЛЬНО
order: [
  { field: 'category', direction: 'asc' },
  { field: 'title', direction: 'desc' }
]

// ✅ ПРАВИЛЬНО
order: [{ category: 'asc' }, { title: 'desc' }]
```

**Почему это важно**:

- Chatium использует собственный синтаксис, отличный от TypeORM, Sequelize и других ORM
- Формат `{ field: ..., direction: ... }` просто не будет работать
- Не путайте синтаксис разных фреймворков!

### ❌ Ошибка #4: Ручное добавление полей id, createdAt, updatedAt

Имена `id`, `createdAt`, `updatedAt` **зарезервированы**: их нельзя объявлять в схеме таблицы и нельзя передавать в `create()` или `update()`.

**НЕПРАВИЛЬНО — объявление в схеме таблицы**:

```typescript
// ❌ Зарезервированные поля не объявляют в Heap.Table!
export const BadTable = Heap.Table('t__project__bad__Xy1', {
  title: Heap.Optional(Heap.String({ customMeta: { title: 'Название' } })),
  createdAt: Heap.Optional(Heap.DateTime({ customMeta: { title: 'Создано' } })), // ← ОШИБКА!
  updatedAt: Heap.Optional(Heap.DateTime({ customMeta: { title: 'Обновлено' } })) // ← ОШИБКА!
})
```

**НЕПРАВИЛЬНО — передача в create/update**:

```typescript
// ❌ Поля id, createdAt, updatedAt добавляются АВТОМАТИЧЕСКИ!
await Table.create(ctx, {
  key: 'some_key',
  value: 'some_value',
  updatedAt: new Date() // ← ОШИБКА! Это поле добавляется автоматически
})

await Table.update(ctx, {
  id: recordId,
  value: 'new_value',
  updatedAt: new Date() // ← ОШИБКА! Обновляется автоматически
})
```

**Ошибка** (при передаче в create/update):

```
HeapTableRepo.create: given data doesn't match the table schema:
"must be string", "data": {}
```

**ПРАВИЛЬНО**:

```typescript
// ✅ В схеме — только пользовательские поля; id, createdAt, updatedAt не объявляем
export const GoodTable = Heap.Table('t__project__good__Ab2', {
  title: Heap.Optional(Heap.String({ customMeta: { title: 'Название' } }))
  // id, createdAt, updatedAt добавятся автоматически
})

// ✅ В create/update эти поля не передаём
await Table.create(ctx, { key: 'some_key', value: 'some_value' })
await Table.update(ctx, { id: recordId, value: 'new_value' })
```

**Почему это важно**:

- Chatium автоматически добавляет поля `id`, `createdAt`, `updatedAt` ко всем таблицам
- Эти поля **зарезервированы**: не объявляйте их в объекте полей `Heap.Table` и не передавайте в `create()` или `update()`
- При объявлении в схеме или передаче вручную может возникнуть ошибка валидации

**Автоматические поля**:

- `id: string` — уникальный идентификатор записи (генерируется автоматически)
- `createdAt: Date` — дата создания записи (устанавливается при create)
- `updatedAt: Date` — дата последнего обновления (обновляется при create/update)

**Формат `id`** (проверено на рантайме 2026-07-20): строка длиной **25 символов** из алфавита base64url (`A–Z`, `a–z`, `0–9`). Не UUID и не последовательный счётчик. Последние символы — суффикс аккаунта (одинаковый у всех записей аккаунта), остальное случайно. Практические следствия:

- **`id` неперебираем** — подобрать чужой идентификатор нельзя, поэтому его можно отдавать наружу как непрозрачную ссылку;
- по `id` **нельзя** судить о времени создания или о соседних записях — для сортировки по времени используйте `createdAt`.

**`createdAt`/`updatedAt` — это объекты `Date`, а не числа** (проверено: `instanceof Date === true`). В `where` их сравнивают с `Date`: `{ createdAt: { $lt: new Date(...) } }`, а не с `Date.now()`. Если храните собственные метки времени числом (epoch ms), конвертируйте на стыке явно — `row.createdAt.getTime()`.

### ❌ Ошибка #5: Использование filter вместо where

**НЕПРАВИЛЬНО** (синтаксис других фреймворков):

```typescript
// ❌ Параметр 'filter' НЕ существует в Chatium!
const config = await SettingsTable.findAll(ctx, {
  filter: {
    // ← НЕПРАВИЛЬНЫЙ ПАРАМЕТР!
    key: {
      $in: ['amocrm_subdomain', 'amocrm_client_id']
    }
  }
})

const products = await ProductsTable.findAll(ctx, {
  filter: { inStock: true } // ← НЕПРАВИЛЬНО!
})
```

**ПРАВИЛЬНО** (синтаксис Chatium):

```typescript
// ✅ В Chatium используется параметр 'where'
const config = await SettingsTable.findAll(ctx, {
  where: {
    // ← ПРАВИЛЬНЫЙ ПАРАМЕТР!
    key: {
      $in: ['amocrm_subdomain', 'amocrm_client_id']
    }
  }
})

const products = await ProductsTable.findAll(ctx, {
  where: { inStock: true } // ← ПРАВИЛЬНО!
})
```

**Все методы используют where, не filter**:

```typescript
// ✅ findAll с where
const items = await Table.findAll(ctx, { where: { status: 'active' } })

// ✅ findOneBy принимает where напрямую
const item = await Table.findOneBy(ctx, { email: 'user@example.com' })

// ✅ countBy принимает where напрямую
const count = await Table.countBy(ctx, { status: 'pending' })

// ✅ searchBy поддерживает where
const results = await Table.searchBy(ctx, {
  query: 'search term',
  where: { category: 'electronics' }
})
```

**Почему это важно**:

- В Chatium параметр называется `where`, не `filter`
- Использование `filter` приведет к тому, что фильтрация просто не будет работать
- Все операции фильтрации используют единый параметр `where`

### ❌ Ошибка #6: Использование default импорта вместо именованного

**НЕПРАВИЛЬНО** (default импорт):

```typescript
// ❌ Default импорт может привести к ошибке "findOneBy is not a function"
import ProjectSettings from '../tables/settings.table'

// При использовании возникает ошибка:
// settings_table_1.default.findOneBy is not a function
const existing = await ProjectSettings.findOneBy(ctx, { key: 'log_level' })
```

**Ошибка в логах**:

```
[WARN]: [logging] Не удалось обновить уровень логов (admin-page):
settings_table_1.default.findOneBy is not a function
```

**ПРАВИЛЬНО** (именованный импорт):

```typescript
// ✅ Используйте именованный импорт для таблиц Heap
import { ProjectSettings } from '../tables/settings.table'

// Теперь все методы доступны корректно
const existing = await ProjectSettings.findOneBy(ctx, { key: 'log_level' })
```

**Почему это важно**:

- При компиляции TypeScript default импорт может не сохранить все методы таблицы Heap
- Именованный импорт гарантирует доступ ко всем методам: `findOneBy`, `findAll`, `create`, `update`, `delete`, `createOrUpdateBy` и т.д.
- Эта ошибка типична при работе с таблицами Heap и может проявляться не сразу

**Как определить проблему**:

- Если видите ошибку `...default.findOneBy is not a function` или `...default.findAll is not a function`
- Проверьте импорт таблицы в файле, где возникает ошибка
- Замените `import TableName from ...` на `import { TableName } from ...`

```typescript
// tables/settings.table.ts
import { Heap } from '@app/heap'

export const ProjectSettings = Heap.Table('t__project__settings__a1b2c3d4', {
```

```typescript
// lib/logging.ts
// ✅ ПРАВИЛЬНО - именованный импорт
import { ProjectSettings } from '../tables/settings.table'

// ❌ НЕПРАВИЛЬНО - default импорт может не работать
```

### ❌ Ошибка #7: Циклическая зависимость в файлах таблиц

**Симптомы**:

```
TypeError: Cannot read properties of undefined (reading 'findOneBy')
TypeError: Cannot read properties of undefined (reading 'findAll')
TypeError: Cannot read properties of undefined (reading 'create')
```

При этом таблица определена корректно, именованный импорт используется правильно, но методы таблицы недоступны.

**Причина**: Файл таблицы импортирует модуль, который (прямо или косвенно) импортирует другую таблицу. Это создаёт **циклическую зависимость** при инициализации модулей JavaScript/TypeScript.

**НЕПРАВИЛЬНО** — файл таблицы с дополнительными импортами:

```typescript
// tables/settings.table.ts
import { Heap } from '@app/heap'
import { Debug } from '../shared/debug' // ❌ Debug импортирует logs.table.ts!

export const ProjectSettings = Heap.Table('settings', {
  key: Heap.String({ customMeta: { title: 'Ключ' } }),
  value: Heap.Any({ customMeta: { title: 'Значение' } })
})

// ❌ Функция использует Debug, который зависит от другой таблицы
export async function ensureDefaultSettings(ctx: RichUgcCtx): Promise<void> {
  Debug.info(ctx, 'Инициализация...')
  const existing = await ProjectSettings.findOneBy(ctx, { key: 'log_prefix' })
  // ...
}

export default ProjectSettings
```

**Цепочка зависимостей, вызывающая ошибку**:

```
logging.ts
  → imports { ProjectSettings } from settings.table.ts
    → settings.table.ts imports { Debug } from debug.ts
      → debug.ts imports ProjectLogs from logs.table.ts
        → logs.table.ts определяет таблицу

При загрузке модулей: ProjectSettings = undefined (ещё не создан к моменту использования)
```

**ПРАВИЛЬНО** — файл таблицы содержит ТОЛЬКО определение таблицы:

```typescript
// tables/settings.table.ts
import { Heap } from '@app/heap'

// ✅ Только определение таблицы, никаких других импортов!
export const ProjectSettings = Heap.Table('t__project__settings__a1b2c3d4', {
  key: Heap.String({
    customMeta: { title: 'Ключ настройки' }
  }),
  value: Heap.Any({
    customMeta: { title: 'Значение настройки' }
  })
})

export default ProjectSettings
```

**ПРАВИЛЬНО** — вспомогательные функции в отдельном файле:

```typescript
// lib/settings-init.ts
import { ProjectSettings } from '../tables/settings.table'
import { Debug } from '../shared/debug'

// ✅ Функции, использующие Debug, вынесены в отдельный файл
export async function ensureDefaultSettings(ctx: RichUgcCtx): Promise<void> {
  Debug.info(ctx, '[settings] Инициализация настроек по умолчанию')

  const existing = await ProjectSettings.findOneBy(ctx, { key: 'log_prefix' })

  if (!existing) {
    await ProjectSettings.createOrUpdateBy(ctx, 'key', {
      key: 'log_prefix',
      value: '[PROJECT]'
    })
  }
}
```

**Правила для файлов таблиц**:

- ✅ Файл таблицы должен содержать **ТОЛЬКО** определение `Heap.Table()` и его экспорт
- ✅ Импортировать можно **ТОЛЬКО** `{ Heap } from '@app/heap'`
- ❌ НЕ импортируйте Debug, логгеры или другие модули в файл таблицы
- ❌ НЕ добавляйте вспомогательные функции в файл таблицы
- ❌ НЕ импортируйте одну таблицу из файла другой таблицы
- ❌ НЕ импортируйте утилиты, валидаторы или бизнес-логику в файл таблицы

**Почему это важно**:

- При загрузке модуля JavaScript/TypeScript сначала выполняет все импорты
- Если импорт A зависит от B, а B (через цепочку) зависит от A — возникает цикл
- В момент цикла экспортируемая переменная ещё не инициализирована (`undefined`)
- Результат: `Cannot read properties of undefined (reading 'findOneBy')`
- Эта проблема сложно диагностируется, так как код выглядит правильным

**Как избежать**:

1. **Держите файлы таблиц минимальными** — только `import { Heap }` и `Heap.Table()`
2. **Выносите всю логику** в отдельные файлы: `lib/`, `helpers/`, `utils/`
3. **Не используйте Debug** в файлах таблиц
4. **Проверяйте цепочки импортов** при появлении ошибок про `undefined`

**Пример правильной организации**:

```
tables/
  settings.table.ts        # Только Heap.Table()
  logs.table.ts            # Только Heap.Table()
  users.table.ts           # Только Heap.Table()

lib/
  settings-init.ts         # Функции работы с settings (с Debug)
  logging.ts               # Функции логирования (с Debug)
  user-helpers.ts          # Функции работы с users (с Debug)

shared/
  debug.ts                 # Debug класс (может импортировать logs.table.ts)
```

**Диагностика**:
Если видите `Cannot read properties of undefined (reading 'methodName')` для таблицы Heap:

1. Проверьте файл таблицы — есть ли там импорты кроме `@app/heap`
2. Проверьте, не используется ли Debug или другие модули в файле таблицы

### ❌ Ошибка #8: UserRefLink без onDelete; RefLink с targetTablePath вместо идентификатора таблицы

**Симптомы (TypeScript / runtime)**:

- `Property 'onDelete' is missing in type '{ customMeta: ... }' but required in type 'HsLinkOptions'.`
- При создании записей: «Undefined values are not allowed» или ошибки, связанные с `startsWith`.

**Правило 1 — UserRefLink требует onDelete.**  
Тип **HsLinkOptions** в Heap требует обязательное поле **onDelete**: `'restrict' | 'none'`. В рабочем коде используется `onDelete: 'none'`.

**НЕПРАВИЛЬНО**:

```typescript
userId: Heap.UserRefLink({
  customMeta: { title: 'Пользователь' }
  // ❌ Нет onDelete — ошибка типов
})
```

**ПРАВИЛЬНО** (в соответствии с рабочим кодом):

```typescript
userId: Heap.Optional(
  Heap.UserRefLink({
    customMeta: { title: 'Пользователь' },
    onDelete: 'none'
  })
)
```

**Правило 2 — RefLink: два аргумента — идентификатор таблицы Heap и опции.**  
**RefLink** вызывается как **Heap.RefLink(tableIdentifier, options)**. Первый аргумент — **идентификатор таблицы Heap** целевой таблицы (строка вида `t__project__table__xyz`), а не путь к файлу (`targetTablePath`). Использование пути к файлу или объекта с `targetTablePath` приводит к ошибкам при создании записей.

**НЕПРАВИЛЬНО**:

```typescript
campaignId: Heap.RefLink({
  targetTablePath: 'tables/campaigns.table',
  customMeta: { title: 'Кампания' }
})
```

**ПРАВИЛЬНО** (в соответствии с рабочим кодом):

```typescript
campaignId: Heap.Optional(
  Heap.RefLink('t__saas-ref__campaign__8Hn4Lx', {
    customMeta: { title: 'Кампания' },
    onDelete: 'none'
  })
)
```

**Значения onDelete**: `'restrict'` — не разрешать удаление целевой записи при наличии ссылок; `'none'` — не проверять ссылки (в рабочем коде используется `'none'`).

**Почему это важно**: Без корректного onDelete для UserRefLink и без идентификатора таблицы в RefLink типы и runtime Heap не совместимы с объявлением, возможны ошибки при создании записей («Undefined values are not allowed», «Cannot read properties of undefined (reading 'startsWith')»).

### ❌ Ошибка #9: update с тремя аргументами (ctx, id, data)

**Симптом (TypeScript)**:

```
Expected 2 arguments, but got 3.
```

**Причина**: В других ORM (TypeORM, Sequelize и т.п.) часто используется сигнатура `update(id, data)` или `update(ctx, id, data)`. В Heap метод `update` принимает **только два аргумента**: `ctx` и **один объект**, в котором обязательно есть поле `id` и поля для обновления.

**НЕПРАВИЛЬНО** (три аргумента):

```typescript
// ❌ Типичная ошибка — 3 аргумента
await Users.update(ctx, id, data)
await Products.update(ctx, productId, { price: new Money(100, 'USD') })
```

**ПРАВИЛЬНО** (два аргумента: ctx и объект с id и полями):

```typescript
// ✅ В Heap — два аргумента: ctx и один объект
await Users.update(ctx, { id, ...data })
await Products.update(ctx, {
  id: productId,
  price: new Money(100, 'USD')
})
```

**В репозитории** удобно сохранять сигнатуру `update(ctx, id, data)` для вызывающего кода, а внутри вызывать Heap с одним объектом:

```typescript
export async function update(ctx: app.Ctx, id: string, data: UsersUpdateData): Promise<UsersRow> {
  return Users.update(ctx, { id, ...data })
}
```

**Почему это важно**: TypeScript выдаёт «Expected 2 arguments, but got 3» при вызове `TableName.update(ctx, id, data)`. Нужно всегда передавать один объект с `id` и обновляемыми полями.

### 📝 Чек-лист перед созданием таблицы

Перед тем как создать Heap таблицу, проверьте:

- ✅ Файл имеет расширение `.table.ts` (С `.ts`)
- ✅ Импорт БЕЗ `.ts`: `import { TableName } from '../tables/tableName.table'`
- ✅ Используете `Heap.Table()` для определения таблицы
- ✅ Используете `where` для фильтрации (НЕ `filter`)
- ✅ Используете синтаксис `{ fieldName: 'asc' }` для сортировки (НЕ `{ field: ..., direction: ... }`)
- ✅ Добавили `customMeta: { title: ... }` для читаемых названий полей
- ✅ **Embeddings выключены** (`embeddings: false`) по умолчанию, включены только при необходимости
- ✅ Используете **именованный импорт** (`import { TableName } from ...`) вместо default импорта
- ✅ Файл таблицы содержит **ТОЛЬКО** `import { Heap }` и определение таблицы (без Debug, логгеров, функций)
- ✅ **RefLink**: вызываете как `Heap.RefLink(tableIdentifier, { customMeta?, onDelete: 'none' })` — первый аргумент идентификатор таблицы Heap целевой таблицы
- ✅ **UserRefLink**: в опциях обязательно указываете **onDelete**: `'restrict'` или `'none'` (в рабочем коде — `'none'`)
- ✅ **update**: вызываете с **двумя** аргументами — `TableName.update(ctx, { id, ...поля })`, не тремя `(ctx, id, data)` (см. [Ошибка #9](#-ошибка-9-update-с-тремя-аргументами-ctx-id-data))

### 🔍 Пример правильной таблицы

```typescript
// tables/products.table.ts
import { Heap } from '@app/heap'

export const Products = Heap.Table(
  't__project__products__Ab1Cd2',
  {
    title: Heap.Optional(
      Heap.String({
        customMeta: { title: 'Название' }
      })
    ),
    price: Heap.Optional(
      Heap.Money({
        customMeta: { title: 'Цена' }
      })
    ),
    inStock: Heap.Optional(
      Heap.Boolean({
        customMeta: { title: 'В наличии' },
        defaultValue: true
      })
    )
  },
  { customMeta: { title: 'Товары', description: 'Каталог товаров' } }
)

export default Products
```

### 🔍 Пример правильного использования

```typescript
// api/products.ts
// ✅ ПРАВИЛЬНО: Импорт БЕЗ .ts
import { Products } from '../tables/products.table'

// ❌ НЕПРАВИЛЬНО: НЕ указывайте .ts
// import { Products } from '../tables/products.table.ts'

export const apiGetProductsRoute = app.get('/products/list', async (ctx) => {
  // ✅ ПРАВИЛЬНО: where + правильный синтаксис order
  const products = await Products.findAll(ctx, {
    where: { inStock: true }, // НЕ filter!
    order: [{ title: 'asc' }], // НЕ { field: 'title', direction: 'asc' }!
    limit: 100
  })

  return { success: true, products: products || [] }
})
```
