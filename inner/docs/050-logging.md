## Коротко

| Вопрос | Ответ |
|---|---|
| Чем логировать | `ctx.account.log(msg, params)` |
| Куда попадает | ClickHouse, таблица `chatium_ai.account_logs` |
| Запись синхронна? | **Нет.** Возвращает `void`, в том же запросе прочитать нельзя |
| Чем читать | `queryAi` из `@traffic/sdk`, **только динамический import** |
| Ретеншен | не ваша забота — таблица platform-managed |
| `console.log` | запрещён |
| `ctx.console.log` | **не логирование** — консоль браузера разработчика |

## Два разных «log»

На `ctx` есть два похожих метода, и путаница между ними — самая частая ошибка в этой теме.

```typescript
ctx.account.log('сообщение', { level: 'info', json: {...} })  // ✅ логирование
ctx.console.log('отладка', value)                              // ⚠️ НЕ логирование
```

`ctx.console` (в части контекстов доступен как `ctx.log`) — отладочный канал. Из типов платформы (`CtxConsole`):

> Debug API for UGC developers to write output to the developer's browser/client console from which the request is made

| | `ctx.account.log` | `ctx.console.log` |
|---|---|---|
| Куда пишет | ClickHouse `account_logs` | консоль браузера разработчика |
| Переживает запрос | да | нет |
| Виден в проде | да | нет |
| Годится для разбора инцидента | да | нет |
| Когда применять | всегда | отладка во время разработки |

Методы `ctx.console`: `debug`, `log`, `info`, `warn`, `error`. Аргументы сериализуются через `JSON.stringify` перед отправкой клиенту.

**Строить логирование на `ctx.console` нельзя** — ничего не сохраняется. Если запрос пришёл не из среды разработки, вывод не увидит никто.

## ctx.account.log — единственное настоящее логирование

```typescript
ctx.account.log('Пользователь выполнил действие', {
  level: 'info',
  json: { amount: 1500, orderId: '123' },   // произвольная структура, НЕ JSON.stringify
  kv: { source: 'payment', userId: ctx.user?.id ?? '' },
})

ctx.account.log('Текст сообщения')            // просто строка
ctx.account.log(new Error('Ошибка'))           // Error
ctx.account.log('Ошибка', new Error('msg'))    // строка + Error
```

### LogParams

```typescript
type LogParams = {
  msg?: string | number
  level?: LogLevel   // 'fatal' | 'error' | 'warn' | 'info' | 'debug' | 'trace' | 'unknown'
  json?: unknown     // произвольная структура → колонка json_str
  kv?: { [key: string]: string | number | undefined }   // → строка "key=value key2=value2"
  err?: Error
}

// Перегрузки:
log(params: LogParams): void
log(msg: string | number, params?: LogParamsNoMsg): void
log(msg: string | number, err?: Error): void
log(err: Error): void
```

> ⚠️ Полей `source`, `userId`, `extra_json` в `LogParams` **нет**, хотя одноимённые колонки в таблице есть — они read-side. Плоские метки кладите в `kv`, структуры в `json`.

### ⚠️ Запись асинхронна

`ctx.account.log` возвращает `void` и **не ждёт подтверждения**. Запись доходит до ClickHouse за сотни миллисекунд — секунды.

- **В том же запросе, где вы залогировали, `queryAi` эту запись не найдёт.** Сценарий «записал → сразу прочитал» не работает; нужен отдельный запрос или отложенный джоб.
- Отметка времени при этом **точна**: `ts64` соответствует моменту вызова, а не моменту, когда строка стала видна.
- Для пользовательских сценариев безразлично — следующий запрос увидит данные.

> Не пытайтесь дождаться записи активным ожиданием (`while (Date.now() - t < ms) {}`). Оно блокирует event loop, и запись не уйдёт вообще. `setTimeout` на сервере отсутствует, задержки — только через `scheduleJobAfter` ([005-jobs.md](005-jobs.md)).

## Куда попадает запись

Таблица **`chatium_ai.account_logs`**, ~39 колонок. Раскладка при записи:

| Параметр | Колонка | Формат хранения |
|---|---|---|
| `msg` (первый аргумент) | `msg` | как есть |
| `level` | `level` | строка уровня |
| `json` | `json_str` | сериализованный JSON **строкой** |
| `kv` | `kv` | строка `"key=value key2=value2"` |
| — | `ts64` | `DateTime64(3, 'UTC')`, проставляет платформа |
| — | `dt` | дата, проставляет платформа |
| — | `workspace_path` | путь проекта, проставляет платформа |

**Таблица общая на аккаунт** — туда пишут все приложения аккаунта, плюс сама платформа.

Ретеншеном и лимитами занимается платформа: своей таблицы логов заводить не нужно и не следует — это была бы вторая копия тех же данных, подпадающая под лимит Heap в 1 млн строк.

## Чтение логов

### Импорт — только динамический

```typescript
// @ts-ignore — @traffic/sdk внедряется платформой, локальных типов нет
const { queryAi } = await import('@traffic/sdk')
```

Модуля **нет** в `node_modules`, и декларации в `@app/types/modules.d.ts` тоже нет — платформа внедряет его в рантайме. Статический импорт не резолвится, типов нет, `@ts-ignore` обязателен.

Проверено: модуль отдаёт 19 экспортов — `queryAi`, `query`, `raw`, `filteredQueryAi`, `filterByDate`, `groupByDate`, `writeAccessLog` и другие. Ответ `queryAi` — `{ rows: Array<Record<string, any>> }`. Только `SELECT`, никакого DDL.

### Базовый запрос

```typescript
// @ts-ignore
const { queryAi } = await import('@traffic/sdk')

const logs = await queryAi(ctx, `
  SELECT ts64, level, msg, kv, json_str
  FROM chatium_ai.account_logs
  WHERE workspace_path = 'p/units/myproject'
    AND level IN ('error', 'warn')
  ORDER BY ts64 DESC
  LIMIT 50
`)

return logs.rows
```

### Фильтр по workspace_path обязателен

Значение — **относительный путь проекта от корня аккаунта, без ведущего слеша**: `p/units/myproject`, `temp/qna2007`.

⚠️ **`workspace_path` — это корень workspace, а не каталог конкретного роут-файла.** Проверено прогоном 22-07-2026: код в `p/temp/qna2107/trace/` и в `p/temp/qna2107/bulk-abort/` пишет логи с **одним и тем же** `workspace_path = 'temp/qna2107'` — подкаталоги `trace`/`bulk-abort` в значение не попадают. Отсюда практическое следствие: **не собирайте фильтр из пути к файлу обработчика** — подкаталог схлопнется, и запрос с `.../trace` в условии не найдёт ничего, хотя записи есть (ровно так «логи не находились» в прошлом прогоне). Точное значение для своего проекта узнавайте **эмпирически**: напишите лог и прочитайте его обратно без фильтра (`WHERE msg LIKE '%<метка>%'`), посмотрите, какой `workspace_path` проставила платформа.

**Логи из `app.job` работают так же, как из роутов** (тот же прогон): тот же `workspace_path`, видны так же быстро (в пределах секунд), запись `job-first` и `job-last` появляются обе. Отдельного поведения у джобов нет.

Без этого фильтра вы получите логи чужих приложений. Заметная доля строк имеет пустой `workspace_path` — это системные записи платформы, фильтр отсекает и их.

### Колонка ts64

**Тип — `DateTime64(3, 'UTC')`** (`SELECT toTypeName(ts64)` подтверждает). В ответе `queryAi` приходит отрендеренной строкой `"2026-07-21 10:49:43.834"` — это форма выдачи, а не тип.

Все временные операции делаются средствами SQL, а не разбором строк в коде:

```sql
ORDER BY ts64 DESC                                                   -- ✅
WHERE ts64 >= now() - INTERVAL 1 HOUR                                -- ✅
WHERE ts64 BETWEEN '2026-07-21 00:00:00' AND '2026-07-21 23:59:59'   -- ✅
WHERE ts64 >= '2026-07-21 00:00:00'                                  -- ✅
```

### Запросы для страницы логов

Проверено прогоном — всё работает:

| Задача | Запрос |
|---|---|
| Пагинация | `ORDER BY ts64 DESC LIMIT 50 OFFSET 100` |
| Число страниц | `SELECT count() FROM … WHERE <те же условия>` |
| Фильтр по уровню | `WHERE level IN ('error', 'warn')` |
| Поиск по тексту | `WHERE msg LIKE '%текст%'` |
| Поиск без учёта регистра | `WHERE msg ILIKE '%ТЕКСТ%'` |
| Поиск по содержимому payload | `WHERE json_str LIKE '%значение%'` |
| Извлечение поля payload | `JSONExtractString(json_str, 'orderId')` |

Фильтрация по меткам `kv` — текстовая: они хранятся одной строкой `"key=value key2=value2"`.

**Лимит на объём выдачи — ~20 МБ сырого JSON**, ограничения на число строк нет (замеры и подробности — [049-clickhouse.md](049-clickhouse.md), «Лимиты queryAi»). Для страницы логов это означает: `LIMIT` берите любой разумный, а следите за **шириной** `SELECT` — `json_str` может быть большим, и тянуть его в список записей не стоит. Забирайте его отдельным запросом при раскрытии конкретной строки.

### ❌ queryAccountLogs и listAccountLogs не работают

Обе объявлены в `@app/ugc/index.d.ts`, но в рантайме падают:

```
Method not found (from sysCall 'ugc.queryAccountLogs')
Method not found (from sysCall 'ugc.listAccountLogs')
```

Подтверждено на двух независимых прогонах. Использовать нельзя, несмотря на наличие типов.

### Альтернатива для разработки

Платформенный dev-логгер `/s/dev/logs` — JSON с последними записями, удобен при отладке и не требует `@traffic/sdk`.

## Модуль-обёртка над логированием

Прямые вызовы `ctx.account.log` по всему проекту неудобны: нельзя выключить болтливость на проде, нельзя добавить дополнительные выходы. Типовое решение — тонкая обёртка `writeServerLog`, через которую логирует весь проект.

Что в ней обычно есть:

**Отсечка по настраиваемому уровню.** Настройка приложения (`log_level`) задаёт нижнюю границу интереса; записи ниже неё **не доходят до платформы вообще**, а не фильтруются при чтении. Порядок строгости: `fatal` > `error` > `warn` > `info` > `debug` > `trace`.

**Политика payload.** Разумное правило — отдавать `json`/`kv` только на уровне `Debug`: на рабочем уровне остаётся карта вызовов (что происходило и в каком порядке), но в журнал не оседает содержимое запросов и пользовательских данных. При диагностике уровень поднимают временно.

**Дополнительные выходы** помимо `ctx.account.log`:
- `sendDataToSocket` в канал админки — живой монитор логов (не хранится, только текущее подключение);
- `POST` на внешний URL для алертинга — обязательно fire-and-forget: `throwHttpErrors: false`, таймаут, результат не ожидается, ошибка глотается. Недоступный приёмник не должен ронять запись лога;
- `ctx.console.log` — если хочется видеть вывод в браузере при разработке.

**⚠️ Запрет рекурсии.** Если обёртка читает настройки (уровень, URL вебхука), функции их чтения **нельзя** логировать через саму обёртку — получится бесконечная рекурсия. Цена ошибки: `RangeError: Maximum call stack size exceeded` примерно на 1300-м вызове, то есть мгновенное падение запроса. Заметно сразу, но лучше не допускать.

**Имя канала для сокета.** WebSocket-каналы не изолированы по проекту: если два приложения используют одинаковый идентификатор, они окажутся в одном канале. Добавляйте в имя уникальный суффикс.

## Частые ошибки

### ❌ console.log

Не работает в серверном рантайме Chatium. Только `ctx.account.log`.

### ❌ ctx.console вместо ctx.account.log

Компилируется, не падает, ничего не сохраняет. См. [Два разных «log»](#два-разных-log).

### ❌ Ожидание, что запись сразу читается

`ctx.account.log` асинхронен. Тест «записал → прочитал в том же обработчике» всегда провалится. Нужен отдельный запрос.

### ❌ Статический импорт queryAi

```typescript
import { queryAi } from '@traffic/sdk'   // ❌ не резолвится
```

Только `await import('@traffic/sdk')` с `@ts-ignore`.

### ❌ Запрос к account_logs без workspace_path

Вернёт логи всех приложений аккаунта вперемешку с системными.

### ❌ Разбор ts64 в коде

Колонка `DateTime64`, все сравнения и сортировки делаются в SQL.

### ❌ Своя Heap-таблица логов

Дублирует `account_logs` и упирается в лимит 1 млн строк, требуя собственного ретеншена. Логи уже хранятся платформой.

## Замеры

Прогон 2026-07-21, прототип с включёнными сокетом и вебхуком:

| Операция | Длительность |
|---|---|
| Одна запись через обёртку (с сокетом и вебхуком) | ~5 мс |
| Пакет из 5 записей | ~20 мс (4 мс на запись) |
| Простой `SELECT` через `queryAi` | ~120 мс |
| Страница логов (выборка + `count()`) | ~230 мс |
| Вебхук на живой URL | ~480 мс (сеть) |
| Вебхук на мёртвый URL | ~120 мс (отказ DNS) |

Производительности достаточно для админ-страницы логов с поиском и пагинацией.
