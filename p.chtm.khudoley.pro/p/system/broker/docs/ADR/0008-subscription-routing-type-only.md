# ADR-0008: Роутинг подписок — только по типу, продюсер анонимен (подписка = whitelist на модуле)

Дата: 2026-07-07

## Контекст

Брокер на fan-out должен ответить на вопрос «какие модули подписаны на событие типа `T`». Встали два связанных вопроса:

1. **Гранулярность подписки.** Подписываться только по **типу** события (`tasks.created`) или по паре **тип + источник** (тот же тип, но только от конкретного продюсера)? Второе имеет смысл лишь при multi-producer (один тип публикуют несколько модулей, §5.2) и когда подписчику нужна не вся выборка по типу.
2. **Форма хранения.** Достаточно ли whitelist-поля `allowedSubscribeTypes` на строке `BrokerModules` (§3.1), или нужна отдельная сущность-подписка (по плоской строке на каждую подписку в своей таблице)?

Дополнительный контекст платформы: `allowedSubscribeTypes` — типизированный `string[]` (ArrayKind), фильтруемый оператором `$includes`/`$any`/`$all` (`inner/docs/008-heap.md`). Матч элемента **массива-объектов** по полю платформой не подтверждён, поэтому запрашиваемую пару `(тип, источник)` нельзя выразить одним полем-массивом объектов — она потребовала бы либо плоских строк-подписок в отдельной таблице, либо композитного ключа-строки в том же `string[]`.

## Решение

**Подписка — только по типу события; продюсер для подписчика анонимен; отдельной таблицы-подписок нет.**

- Подписка = whitelist `allowedSubscribeTypes: string[]` на строке `BrokerModules` (§3.1). Отдельной сущности-подписки не заводим.
- Маршрутизация на fan-out **не зависит от продюсера**: подписчик получает факт по типу и не знает (и не указывает), какой модуль его опубликовал.
- **Резолв подписчиков — одним запросом.** Publish-код разворачивает конкретный `eventType` в конечный набор предков-глобов (напр. `tasks.created` → `{ *, tasks.*, tasks.created }`) и делает один `where` по `BrokerModules`:
  ```ts
  where: {
    allowedSubscribeTypes: {
      $includes: {
        $any: ['*', 'tasks.*', 'tasks.created']
      }
    }
  }
  ```
- **Грамматика глоба — посегментная** (каждый dot-сегмент = литерал или `*`), чтобы набор предков был конечным и перечислимым из строки типа. Частичные глобы внутри сегмента (`tasks.cr*`) и регэкспы не поддерживаются.

Отсев «нужное/ненужное» сверх типа — на стороне подписчика (по `payload` в его обработчике). Брокер отдаёт факт события, не решает за подписчика.

## Обоснование

1. **Развязка pub/sub.** Подписчик связан с **контрактом события (типом)**, а не с продюсером. Это и есть роль брокера как развязки; иначе он вырождается в point-to-point «кто-кому».
2. **Расширяемость.** Нового продюсера того же типа можно добавить, **не трогая подписчиков** — они его не знают.
3. **Стоимость резолва ограничена размером реестра, а не формой запроса.** Фильтр `$includes`/`$any` по `string[]` идёт **сканом**: индексируются только плоские поля на корневом уровне, массивы — нет (§3, `inner/docs/008-heap.md`, подтверждено поддержкой Chatium). Это горячий путь — на каждую публикацию, — но `BrokerModules` не растёт: одна строка на модуль, единицы-десятки, сотни в пределе (§3.1). На таком объёме скан неотличим от индексного поиска. Индексируемость здесь **не несущая, опираться на неё нельзя** — несёт инвариант размера реестра.
4. **Простота.** Нет отдельной сущности с собственным жизненным циклом (пауза/курсор/модерация подписки) там, где это не требуется.

## Последствия

Плюсы:

- одна выборка подписчиков на fan-out, без отдельной таблицы;
- подписчик не зависит от состава продюсеров; добавление продюсера не задевает подписки;
- меньше сущностей и каскадов (подписка живёт полем на модуле, удаляется вместе с ним, §5.5).

Минусы / границы:

- **Нет фильтра по источнику.** При multi-producer подписчик получает событие по типу от **любого** продюсера и отсеивает лишнее сам. Продюсер **анонимен**: брокер источник в доставку не кладёт. Если различение источника нужно — продюсер **добровольно** помещает свою идентификацию в `payload`; это контракт продюсер↔потребитель, брокер его не регулирует (§5.2). Если однажды понадобится роутинг «тип от конкретного источника» **как запрос брокера** — это отдельное решение (композитный ключ-строка в `string[]` либо плоские строки-подписки в отдельной таблице), пересматривающее данный ADR.
- **Грамматика глоба ограничена** посегментными wildcard'ами (цена перечислимости предков).
- whitelist-тип поля фиксирован как `string[]` — при обоих возможных будущих исходах он не меняется.

Отвергнуто:

- **Пара `(тип, источник)` как подписка** — YAGNI: в системе тип по факту принадлежит своему продюсеру, а отсев «не от того» дешевле оставить подписчику; добавляет брокеру сложность без спроса.
- **Отдельная таблица-подписок** — оправдана только под источник-фильтр или per-subscription состояние, чего текущий scope не требует. Её неочевидное преимущество: поле `type` там было бы **корневым плоским**, то есть резолв шёл бы по индексу. Поэтому именно она — путь отхода, если реестр однажды перестанет быть маленьким (подписки пообъектные, а не помодульные); композитный ключ-строка в том же `string[]` таким свойством не обладает, он остался бы сканом.
- **`allowedSubscribeTypes` как `Heap.Any()`** — фильтр внутрь `any` работает, но теряются типобезопасность и явность схемы. На скорость это не влияет: индекса не получает ни `Heap.Any`, ни `string[]`.
