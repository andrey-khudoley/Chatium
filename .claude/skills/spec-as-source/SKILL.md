# spec-as-source — написание и пересборка спецификаций

Методология: **спецификация — источник истины; код не меняется, пока изменение не описано в спеке.** Скилл ведёт два сценария: (А) спека нового проекта с нуля и (Б) пересборка спеки существующего проекта (обычно перед рефакторингом/переносом). После сборки черновика — протокол закрытия открытых вопросов с пользователем по одному.

**Живые эталоны в воркспейсе (читать перед работой, стиль копировать оттуда):**

- `s.chtm.khudoley.pro/d/system/broker/docs/spec/` — greenfield-эталон (спека писалась до кода);
- `s.chtm.khudoley.pro/d/gateways/getcourse/docs/spec/` — эталон пересборки (спека переописывает действующую реализацию);
- их `docs/ADR/` — формат ADR.

Эталоны большие: читай заголовки (`grep -n '^#'`), словарь, §0–§0.1 и один-два содержательных раздела — этого достаточно, чтобы держать стиль.

---

## 1. Артефакты

| Файл | Роль |
| --- | --- |
| `docs/spec/spec.md` | **Источник истины.** Все решения с обоснованиями, волны с чек-листом |
| `docs/spec/spec-map.md` | Производный обзор: слои + mermaid-диаграммы. Новых правил не вводит; при расхождении приоритет за spec.md |
| `docs/ADR/NNNN-slug.md` | Крупные архитектурные развилки: контекст → решение → последствия. Нумерация сквозная, продолжается от существующих |

Даты везде — только через shell (`date "+%d-%m-%Y"`), не из головы.

## 2. Каноническая структура spec.md

```markdown
# Spec-as-source: `<имя-проекта>`

Статус: в разработке. / Последнее обновление: <дата>.
Область действия: <каталоги dev/prod-копий>.

## Словарь терминов          ← только оправданные англицизмы, остальное переводить
## 0. Правило предварительной спецификации
## 0.1. Волны разработки и чек-лист готовности
### Принцип распределения    ← таблица классов цены позднего решения
### Волна 1 … Волна 4        ← Код / Гейт / открытые пункты `- [ ]` / Приёмка
## 1. Назначение             ← что делает и — обязательно — чего НЕ делает
## 2. Доменная модель        ← сущности, состояния, жизненные циклы
## 3. Хранилище (Heap)       ← ограничения платформы, таблицы по одной, идентификаторы
## 4. API                    ← публичный и админ-слои, транспорт, конверт ответа
## 5. Бизнес-логика          ← по операциям/конвейерам, §5.x на каждую
## 6. Интеграции
## 7. Права доступа          ← таблица «слой → гейт → кто»
## 8. Правила эволюции       ← что ломающее, что аддитивное, как расширять
## 9. Тестирование и приёмка ← модель исполнения, категории, приёмка §9.5.N по волнам
```

Разделы, для которых пока нет материала, не выдумывать — оставить заготовкой и внести пунктом в чек-лист волны (так сделан §2 у брокера).

## 3. Волны и цена позднего решения

Ядро методологии. Пункт попадает в волну **по цене позднего решения**, не по важности:

| Класс | Что стоит поздний ответ | Крайний срок |
| --- | --- | --- |
| **Необратимое** | форма/смысл записанной строки, факт удаления данных — правится миграцией, при неудаче потерей | до первой боевой записи в таблицу |
| **Ломающее** | внешний контракт: операции, коды ошибок, заголовки — правится согласованным обновлением потребителей | до первого внешнего потребителя |
| **Аддитивное** | ничего: новая операция, поле с дефолтом, экран | в любой момент, сознательно позже |

Правила, которые держат систему:

- **Запрет сдвига:** пункт первых двух классов нельзя перенести позже «ради скорости» — именно там рождаются переписывания и потери. Аддитивное, наоборот, отодвигается сознательно: ранний ответ устареет раньше, чем понадобится.
- **Закрытые пункты удаляются из чек-листа**, а не помечаются галочкой: решение живёт в профильном разделе спеки (с датой и обоснованием), в §0.1 — только оставшаяся работа.
- Типовой порядок волн: прототип/каркас ничего не замораживает → MVP замораживает форму строк → beta замораживает внешний контракт → релиз только добавляет.
- **Если внешний контракт уже опубликован** (пересборка живого сервиса) — он входит в спеку как «ломающее, уже замороженное»: наследуется как данность, волна его воспроизводит, а не проектирует; ломать — только новой версией + ADR.
- Приёмка волны — машинный факт, где возможно (штатная система тестов, `./web/tests/ai` → `summary.success = true`), а не «посмотрели глазами».

## 4. Стиль текста

- **Решение + почему.** Каждое зафиксированное решение несёт обоснование и отвергнутые альтернативы одной-двумя фразами («вариант X отвергнут: …»). Спека без «почему» не переживает первый спор.
- Плотный деловой русский; кросс-ссылки на разделы (`§5.9`, `ADR-0004`) и на платформенные доки (`inner/docs/NNN-*.md`) — везде, где решение на них опирается.
- Решения датировать: «**Решено (DD-MM-YYYY):** …».
- Константы кода называть по имени с значением (`GW_OUTBOUND_TIMEOUT_MS = 10_000`) и фиксировать, что меняются только правкой спеки.
- Не дублировать в спеку целиком унаследованные детальные нормативы (manual, реестры, JSON-схемы) — они остаются приложениями, спека ссылается и объявляет иерархию: **spec.md > унаследованные нормативы > код**; при расхождении правится нижний уровень.
- В spec-map — только визуализация уже согласованного: слои снизу вверх (методология → назначение/границы → хранилище → бизнес-логика → тесты) + сводка открытых вопросов; каждая диаграмма подписана ссылками на §.

## 5. Сценарий А — спека с нуля

1. Каркас файла по структуре §2 этого скилла; словарь и §0 сразу.
2. §1 «Назначение» с явными границами («чего проект НЕ делает» — не менее ценно).
3. Все нерешённые вопросы — в §0.1 по волнам; не решать за пользователя необратимое и ломающее — формулировать как пункты с вариантами.
4. Дальше — протокол закрытия вопросов (§7 этого скилла).

## 6. Сценарий Б — пересборка спеки существующего проекта

Порядок выверен на getcourse-gateway:

1. **Скопировать унаследованные спецификации** в docs/ нового проекта (обычно `docs/gateway/`-подобные нормативы + `docs/ADR/`), **без кода**. LLM-логи и описания кода (architecture/api/data) не копировать — они документируют то, что остаётся в старом дереве. Согласовать состав с пользователем.
2. **Изучить действующую реализацию как референс, не как истину:** README, architecture/api/data, ключевые исходники (таблицы, каталоги, конвейеры). Унаследованные нормативы читать полностью — они основной источник решений.
3. **Сверять доки с фактическим кодом** — расхождения и есть главная добыча анализа. Типовые находки: «README обещает больше, чем включено» (проверить фактические enum-значения grep'ом), незакрытые пункты старого плана, недоснесённые shim'ы, отсутствующий retention растущих таблиц, устаревшие после поздних ADR разделы manual. Каждую находку — либо в спеку решением, либо в чек-лист вопросом.
4. **Собрать spec.md**: решения из manual/ADR/кода фиксируются как принятые (со ссылками), всё спорное и всё, что воркспейс с тех пор делает иначе, — в §0.1 по волнам.
5. **Написать spec-map.md**, обновить/дополнить ADR.
6. Отчитаться пользователю: что скопировано, что написано, **список вскрытых аномалий** и главные развилки — это материал для его ревизии.

### Канонические развилки волны 1 (сеять в чек-лист пересборки)

Паттерны, к которым воркспейс уже пришёл; старый проект их обычно не имеет — каждый даёт вопрос:

- **Способ переноса кода:** копия / пересборка / гибрид «ядро копией (обкатанные протокольные тонкости), обвязка пересборкой». Фиксировать ADR-ом.
- **Пары Heap-таблиц stage/prod** + статический селектор `IS_PROD` по `PROJECT_ROOT` (`006-arch.md`, реализация — брокер) vs одиночные таблицы. Идентификаторы неизменяемы — фиксируются в §3 до первой записи (скоуп + сегмент + случайный хвост через shell).
- **Судьба данных старых таблиц:** чистый лист (настройки руками, журналы — архив) vs миграция.
- **Логи:** ClickHouse через `ctx.account.log` без своей таблицы (модель брокера §5.10, `050-logging.md`; `workspace_path` подтверждать прогоном) vs Heap-таблица логов.
- **Тесты:** штатная система Chatium (`020-testing.md`, машинный гейт) vs самописная страница. Сьюты, бьющие по внешним системам, — вне машинного автопрогона.
- **Retention** любой таблицы, растущей с каждым запросом (лимит 1 млн строк): ежедневный `app.job`, самоперепланирование **первым действием**, `deleteAll({ limit: null, hard: true })`, порог — константа.
- **Плоские корневые поля** под горячие фильтры (только они индексируются) — включая поля, нужные будущим метрикам (тенант/группировка), пока таблица пуста и поле бесплатно.

## 7. Протокол закрытия вопросов (с пользователем)

Когда пользователь говорит «поехали по вопросам» (или сразу после черновика — предложить):

1. **По одному, сверху вниз.** Формат подачи: «**Вопрос N из M** (волна K): **название**» → суть понятным языком → факты/находки → **моё предложение** с обоснованием → `AskUserQuestion` с 2–4 вариантами, рекомендация первой с «(Recommended)». В вариантах честно называть цену каждого.
2. **Фиксация сразу после ответа, до следующего вопроса:**
   - удалить пункт из чек-листа §0.1 (или переформулировать в подтверждённый объём работы, если осталась реализация);
   - записать решение в профильный раздел спеки: «**Решено (дата):** …» + обоснование;
   - обновить spec-map (текст и mermaid: `✓` у закрытого);
   - крупная развилка → ADR со следующим номером;
   - решение может тянуть правки в нескольких разделах (таблица §3, API §4, наблюдаемость §5.x) — провести все, спека не должна противоречить сама себе.
3. **Решение пользователя против рекомендации — норма.** Не переспаривать; зафиксировать его выбор и укрепить дизайном (контракты ключей, обязательная деградация, порог пересмотра).
4. Вопросы-подтверждения объёма работ (не развилки) проходить быстро: короткая суть + «включить в волну N?».
5. Закрыв волну — короткий рубеж: «волна K закрыта по решениям, осталась работа: …». Закрыв список — сводка всех решений + следующие шаги (коммит, старт реализации через `/ppN`).
6. Считать вопросы по исходному списку (нумерация «N из M» не плывёт от появления производных подпунктов; новые крупные вопросы — в конец с ростом M).

## 8. Чего не делать

- Не писать код и не менять код проекта — спека первична; реализация — отдельная задача (`/ppN`).
- Не закрывать необратимые/ломающие вопросы «по умолчанию» без пользователя — это его решения, скилл готовит варианты и рекомендацию.
- Не оставлять решение только в чек-листе или только в ответе чата — оно обязано жить в разделе спеки.
- Не переносить в новую спеку исторические ошибки молча (устаревшие разделы manual помечать «историческими» с указанием заместившего решения).
- Не раздувать spec-map: карта — производная, новых правил не вводит.
